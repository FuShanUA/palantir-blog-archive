环境创建

这是一篇深入探讨环境初始化过程的高级指南。面向希望了解可能影响初始化性能的技术考量的用户。

有关常见环境问题的一般性指导，请参阅《环境故障排查指南》。

Conda 是一款开源的、与语言无关的包和环境管理器。Mamba 是 Conda 包管理器的开源重新实现。Code Workbook 已改用 Mamba 来解析包依赖关系，并将包集合安装到独立的环境中。与 Conda 相比，Mamba 在包解析方面具有多项优势，最显著的特点是速度提升且错误信息更易读。

本页介绍了最重要的概念并概述了环境创建过程；更多信息请查阅 Conda 官方文档 ↗ 和 Mamba 文档 ↗。

重要术语

  • 包
  • 频道
  • 环境

包

包是一个文件集合，通常包含元数据、库和/或二进制文件。Code Workbook 提供了对多种包（例如 numpy）的访问，以补充核心语言功能。

包是带有版本控制的，几乎总是有一组依赖项——即为确保其正常运行而必须同时安装的其他包。依赖项可以是包的特定版本、可接受的版本范围或任意版本。

频道

频道，有时也称为仓库，是存储包的任何位置。一个频道可以是本地文件系统中的一个目录，另一个频道可以是托管在网络服务器上的目录。

无论何种类型，每个频道都是一棵按平台架构划分包的目录树。每个平台子目录都包含一个名为 repodata.json 的文件，这是该子目录下所有包的索引。

Conda 在需要获取包时，会搜索一组预配置的频道。有关 Foundry 中频道管理的更多信息，请参阅《代码仓库文档》。

环境

Conda 环境是一个包含特定包集合的目录。系统会将环境配置面板中指定的包传递给 Conda，为每个 Workbook 创建相应的环境。Conda 会构建一组满足配置及所有依赖关系的包，并将这些包安装到为 Workbook 提供底层支持的 Spark 模块上。

性能

以下关于性能的说明借鉴了 Anaconda 的这篇博文 ↗，该文详细讨论了 Conda 的性能，同样适用于 Mamba 的实现。接下来的两节总结了这些内容，并概述了与 Code Workbook 最相关的性能因素：

  • 创建环境
  • 局限性

创建环境

环境创建包含两个主要步骤：解析步骤和安装步骤。

解析步骤

在解析步骤中，指定的包管理器（Conda 或 Mamba）会尝试找到满足所有传递依赖关系的包及版本。传递依赖关系由环境配置面板中指定的包的依赖项、这些依赖项自身的依赖项等构成。此步骤包含四个阶段：

  1. 下载并处理包索引。Conda 会从每个配置的频道下载相关的 repodata.json 文件，并将索引条目转换为内存中的对象。
  2. 缩减索引。Conda 会构建一组可能用于该环境的所有包。为完成此操作，算法从提供的包规范开始，递归遍历所有依赖项。所有不需要的包（主要是那些不在依赖图中的包）都会被剪除。
  3. 将依赖约束表达为布尔可满足性（SAT）问题。Conda 倾向于某些类型的解决方案（例如使用尽可能新的包版本），这些偏好已被固化到子句构建过程中。
  4. 运行 SAT 求解器。

安装步骤

如果求解成功，接下来就是安装步骤。在此步骤中，将从相应的通道中检索每个构件，并使用它们来构建环境。此步骤包含三个阶段：

  1. 下载并解压已求解环境中的所有包。
  2. 验证包内容。根据配置，Conda 将使用校验和或验证文件大小是否正确。
  3. 将包链接到环境中。

局限性

在特定情况下，上述所有步骤都可能出现速度缓慢的问题。导致缓慢的原因通常分为以下三类：

  • 上游变更
  • 环境规格
  • 包大小

上游变更

很大一部分缓慢问题是由 Foundry 之外的因素引起的。

  • 下载和处理通道索引的耗时随索引文件总大小而增加；需要考虑的通道越多且这些通道越大，这些步骤花费的时间就越长。
  • 索引缩减的耗时也随传递依赖的数量而增加；传递依赖的数量取决于包选择声明的依赖项。

由于这些因素是外部且不透明的，因此很难对性能回退进行根因分析。某个环境可能突然需要更长的加载时间，是因为某个通道最近增加了容量，或者是因为某个包在其最新版本中声明了几个新的依赖项。

环境规格

更常见的情况是，初始化缓慢与环境规格本身直接相关。求解步骤的耗时随环境大小呈超线性增长，因此根据一般经验，包含更多包的环境将花费不成比例的更长初始化时间。

有两种方法可以缓解这些情况。

  • 首先，从环境定义中移除不需要的包。拥有小型的、专用环境比拥有大型的、通用环境的性能要好得多。
  • 其次，尝试在环境配置面板中为某些包添加版本约束。对于像 python 这样存在大量构建版本的包，以及像 scipy 这样具有复杂依赖关系图的包，锁定版本最为有效。这样做可以让 Conda 更积极地缩减索引，意味着 SAT 求解器将不需要考虑那么多的包版本。

包大小

包大小通常比本节中的其他因素造成的问题要少，但在某些情况下仍然具有相关性。

下载和解压哪怕是一个包也可能并非易事。例如，pytorch 包的大小约为 460 MB，解压可能需要 35 秒以上。

下载、解压和验证的耗时随环境中包的大小和数量呈线性增长。由于传递依赖的存在，已求解环境通常包含比环境定义中明确指定的多得多的包，包数量的增加可能会导致缓慢。

这种情况下的缓解措施与环境规格的建议类似：尽可能保持环境最小化。

求解步骤优化：规格文件

如果环境的请求包集合未发生更改，则无需再次解析环境，因为这会得到完全相同的已解析环境。因此，Code Workbook 会将成功解析的结果存储在 spec file 中，从而避免执行环境初始化中的 解析步骤。下次需要初始化相同的环境时，Code Workbook 会直接引用该 spec file，跳过解析步骤。这使得初始化时间得到显著优化，因为 Code Workbook 只需执行安装步骤即可。作为应对极少数情况下先前成功环境失效的预防措施，Code Workbook 默认会在 24 小时后使 spec file 失效。此后，在下次成功解析环境时，将生成新的 spec file。

如上图所示，要初始化环境，Code Workbook 必须先获取 Spark 模块，然后在模块上安装包。与 Code Workbook 将成功解析的结果存储在 spec file 中类似，有效的环境也可以存储为 Docker 镜像。使用此镜像，Code Workbook 可以请求后续的 Spark 模块，且模块中已包含所有必需的包。这绕了解析步骤和模块上的 Conda 安装步骤，从而大幅缩短环境初始化时间。Code Workbook 默认会在 24 小时后使 Docker 镜像失效。在下次成功解析环境时，将生成新的镜像。

禁用 Conda Docker

在极少数情况下，如果环境中的包包含 链接前或链接后脚本 ↗，使用 Conda Docker 的环境在运行时可能会失败。如果包含此类脚本的环境似乎出现故障，你可以在 Code Workbook 环境配置选项卡的左下角手动禁用此优化。禁用 Conda Docker 后，环境仍可受益于其他可用的优化。对于所使用的配置文件与切换该设置所在的工作簿分支，此配置将持续生效。

预热

虽然上述优化可以缩短环境的初始化时间，但你可以通过主动初始化若干模块来完全免去了等待环境的时间。这些模块被称为“暖”模块。通过配置暖模块队列，Code Workbook 可确保一组已预初始化的 Spark 模块始终处于就绪状态，随时可供分配。有关如何配置暖模块的更多信息，请参阅 Code Workbook 配置文档中的 预热 部分。

如果你经常遇到等待环境初始化耗时过长的情况，建议为相关的配置文件实施暖模块队列。

上述优化仅对 Code Workbook 已存储的环境有效。请求自定义环境时，Code Workbook 必须执行标准初始化过程，无法受益于任何优化。因此，自定义环境的初始化时间通常较长。若要排查自定义配置文件初始化过程缓慢的问题，请参阅 疑难解答 文档。
