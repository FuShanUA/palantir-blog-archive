会话历史与固定会话

会话

在 Spark 模块中初始化 Code Workbook 环境时，系统会为该模块关联一系列元数据。这些元数据可分为两类：计算信息与环境信息。会话是这些设置在 Spark 模块生命周期中的一次实例化，通俗来讲，也就是“特定 Spark 模块在其生命周期内的真实运行状态”。

计算信息包含了分配给该 Spark 模块的各项 Spark 设置详情，以及其他相关信息，例如 jar 依赖项、资源标识符和所启动模块的类型。另一方面，环境信息可进一步细分为两类：请求环境与解析环境。请求环境是一组 conda 规范，例如要求 pandas=2.* 或 python<3.10；而解析环境则是满足请求环境所设约束的一组已解析包。需要注意，请求环境具有非确定性，而解析环境按定义是对特定请求环境的持久化解决方案。因此，两个相同的请求环境可能会解析出不同的结果。例如，在 2017 年请求 python>=3.6 很可能会解析为 Python 3.6，而如今提出相同请求则可能解析为 Python 3.10。

会话历史

Code Workbook 提供了工作簿近期所有会话的相关信息。要查看某个工作簿的会话历史，请选择 Code Workbook 的 Environment 下拉菜单，然后选择 View session history。

Session history 窗口通过三个不同的选项卡提供信息：Compute information、Requested environment 和 Resolved environment。左窗格显示会话 ID 以及该会话初始化时的时间戳。会话 ID 左侧的图标会指示其是否成功初始化（绿色）、失败（红色）或其他状态（蓝色）。蓝色通常表示会话尚未完成初始化，因此既未达到失败状态也未达到成功状态。

计算信息

Compute information 选项卡提供了为特定会话所请求的 Spark 模块类型信息：

  • moduleAssignmentInfo: 关于环境是由特定用户为交互式工作簿使用而启动，还是由后台作业（例如计划构建）启动的常规信息。有关交互式模块与构建模块之间区别的更多信息，请参阅《批处理与交互式构建》。
  • moduleLaunchInfo: 关于分配给 Spark 模块本身属性的元数据。包括以下字段：
    • sparkModuleRid: Spark 模块的唯一标识符。鉴于模块与会话之间存在 1:1 的映射关系，您会注意到 sparkModuleRid 和 sessionId 将共享同一个唯一标识符。
    • profileRid: 初始化 Spark 模块时所使用配置文件的唯一标识符。
    • isCustomEnv: 指示该会话是否使用自定义环境进行初始化。请回忆一下，Code Workbook 中的自定义环境是指在 Code Workbook 界面的 Environment 配置菜单中直接修改过的任何标准配置文件。
    • jarDependencies: 在《控制面板》中手动添加到配置文件的所有 jar 依赖项列表。
    • moduleLaunchType: 取值为 WARM_MODULE 或 ON_DEMAND_MODULE。指示会话是使用了已预热模块队列中已完成初始化的模块，还是请求了一个全新的模块来启动其初始化过程。
    • moduleResources: 与该模块绑定的所有计算信息，包括驱动程序和执行程序的内存、执行程序的数量等。

请求环境

Requested Environment 选项卡提供了在开始初始化之前所需的环境设置信息：

  • Initialization Mode: 为该会话执行的 Code Workbook 初始化类型。取值可以是 solve、file 或 docker。有关 Code Workbook 中初始化类型的更多信息，请参阅有关《环境优化》的文档。
  • 环境仓库：存储相关环境所用的键。对于非自定义环境，这是会话使用的 profileRid。对于自定义环境，这是配置文件所在工作簿的 workbookRid。
  • 请求的包：初始化过程中提交的请求包列表。该列表既包含配置文件明确请求的包，也包含 Code Workbook 为实现正常运行而自动请求的附加包。
  • 后备仓库：已安装包的来源通道列表。

已解析环境

已解析环境选项卡提供了初始化过程中使用的环境包信息。这包括环境初始化耗时，以及最终安装到模块上的包及其版本。这些信息至关重要，因为 Code Workbook 的会话回滚功能复用的是先前会话的已解析环境，而非请求的环境。

比较会话

比较两个给定会话有助于理解特定环境发生了哪些变化。会话历史窗口支持将当前会话与任意历史会话进行对比。在会话历史窗口右上角选择比较会话，即可进入会话历史比较选项卡。

会话比较菜单会并排显示两个窗口。右侧展示当前会话的信息，左侧则可从会话列表中选择任意会话显示以便对比。如需切换左侧用于比较的会话，直接从会话列表中选择即可。在窗口右上角选择退出比较可随时退出比较视图。退出后，左侧选定的会话将保持选中状态。

所有三个可用选项卡均支持会话比较。比较两个会话的计算信息有助于了解模块可用内存的变化。比较请求的环境有助于了解两次会话环境间进行了哪些手动更改，而比较已解析环境则有助于理解两次会话中安装的版本差异。

在以下示例中，通过会话比较的各个选项卡，可以揭示两次会话的如下信息：

  • 使用了不同的配置文件。
  • 配置文件已自定义。
  • 模块采用了更小的内存设置。

  • 请求了更新版本的 Python 和 pandas 等包。

  • 环境中最终安装了不同版本的 Python。

会话固定

某些情况下，您可能希望临时回滚到历史会话的完全相同设置。Code Workbook 支持通过固定会话来实现这一操作。固定会话意味着使用与历史会话相同的元数据初始化全新会话和 Spark 模块，从而复现一个看似完全相同的环境。固定会话会借用历史会话的已解析信息来初始化新模块。这一点至关重要，因为使用相同的已解析环境能确保安装的包完全一致，而仅使用相同的请求环境则无法保证。因此，固定会话能够模拟回滚到先前可用环境的效果。少数几项元数据（如初始化模式）不会从历史会话中借用。

如何固定会话

要固定会话，请导航至 Session history 窗口并选择您想要固定的会话。然后，在面板右下角选择 Pin session。工作簿的当前分支将具有固定的会话覆盖，该覆盖将持续最多 24 小时。横幅将显示覆盖的剩余时间，以及移除固定的选项。在此期间，随后的每次交互式环境初始化都将借用该固定会话的信息。当该期间到期或手动取消固定会话时，工作簿将恢复使用其当前配置的环境。

会话固定专为调试目的而设计，不应将其用于长期、生产就绪的流水线。因此，固定会话仅影响交互式环境。在 Code Workbook 界面之外执行的构建（例如计划构建）不受固定会话覆盖的影响。我们建议您将会话固定限制在偶尔的故障排除和实验中。相反，请使用会话历史功能来了解不同会话之间的差异，并直接编辑环境定义。

会话固定的限制

固定会话时存在某些限制。固定会话不是无限期持续的，并非所有会话都能被固定，也并非所有初始化都会受到固定会话的影响。以下是考虑固定会话时需要注意的限制列表：

  • 会话固定最多可持续 24 小时。要进一步延长此时间，您需要重新固定会话。重新固定会话会导致启动一个新的模块，并且该模块上的所有本地状态都将丢失。
  • 固定会话不会从其附加的历史会话中借用所有元数据。初始化模式等设置可能会有所不同。
  • 只有之前成功的会话才能被固定。不允许固定未完成或不成功的会话。此外，由于版本不兼容、包版本被列入黑名单、API 破坏等，Code Workbook 可能会防御性地阻止您选择要固定的版本。红色横幅会通知您无法固定的会话。
  • 固定会话仅影响交互式作业 - 不影响构建作业。
  • 只有使用Artifacts Profiles 的会话才能被固定

由于上述原因，固定会话是一项调试功能，不应依赖其用于长期、生产就绪的流水线。

使用会话历史对环境进行故障排除

上述提到的会话历史、会话比较和会话固定功能可用于对故障环境进行故障排除。特别是，它们有助于解决以前正常工作的环境出现的故障。请按照以下步骤来纠正此类情况：

  1. 环境以前是否正常工作过？

    • 如果没有，请参阅《环境故障排除指南》。
    • 如果是，但现在失败了，请继续下一步。

  2. 使用Compare sessions 功能，将当前失败的会话与之前成功的会话进行选择，并观察环境中的差异。

    • 如果计算信息详细信息发生变化，模块设置可能没有足够的内存供模块正确初始化。
    • 如果请求的环境详细信息发生变化，则说明对环境引入了手动破坏性更改，从而导致失败。
    • 如果请求的环境详细信息没有变化，但已解析环境中的变化导致了失败，则所使用的其中一个包的新版本可能包含破坏性更改。这些包通常来自开源。您可以直接调查有问题的包，或修改您的环境以避免请求相关的包或版本。
    • 如果环境中没有明显变化，或者上述步骤没有帮助，请参阅《环境故障排除指南》或联系 Palantir 支持以获取进一步的帮助。

  3. （可选）在上述步骤中进行故障排除时，使用Session pinning 功能首先确保固定的环境版本有效，并在寻找更持久的解决方案时暂时恢复工作簿的功能。

  4. 找到环境失败的根本原因后，相应地调整配置文件的设置，以永久性地解决该情况。
