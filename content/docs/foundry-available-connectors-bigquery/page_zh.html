将 Foundry 连接至 Google BigQuery，以便在 BigQuery 表和 Foundry 数据集之间读取和同步数据。

设置

  1. 打开“数据连接”应用程序，在屏幕右上角选择+ 新建数据源。
  2. 从可用的连接器类型中选择BigQuery。
  3. 按照后续的配置提示，结合下方各章节的信息，继续完成连接器的设置。

详细了解如何在 Foundry 中设置连接器。

您必须拥有一个Google Cloud IAM 服务账号 ↗才能继续进行 BigQuery 身份验证和设置。

身份验证

使用 BigQuery 连接器需要具备以下身份与访问管理 (IAM) 角色：

读取 BigQuery 数据：

  • BigQuery Read Session User：在 BigQuery 项目中授予
  • BigQuery Data Viewer：在 BigQuery 数据上授予，用于读取数据和元数据
  • BigQuery Job User（可选）：授予以用于引入视图并运行自定义查询

将数据从 Foundry 导出到 BigQuery：

  • BigQuery Data Editor：在 BigQuery 数据集或项目中授予
  • BigQuery Job User：在 BigQuery 项目中授予
  • Storage Object Admin：如果数据通过 Google Cloud Storage 导出，则在存储桶上授予

使用临时表：

  • BigQuery Data Editor：如果数据集由连接器自动创建，则在 BigQuery 项目中授予
  • BigQuery Data Editor：在指定的数据集上授予，用于存放临时表

请在Google Cloud 关于访问控制的文档 ↗中详细了解所需角色。

从以下可用的身份验证方法中选择一种：

  • GCP 实例账号：有关如何设置基于实例的身份验证的信息，请参阅Google Cloud 文档 ↗。

    • 请注意，GCP 实例身份验证仅适用于通过运行在 GCP 中已正确配置的实例上的代理程序进行操作的连接器。
    • 请注意，虚拟表不支持 GCP 实例身份验证凭据。

  • 服务账号密钥文件：有关如何设置服务账号密钥文件身份验证的信息，请参阅Google Cloud 文档 ↗。密钥文件可以以 JSON 或 PKCS8 凭据的形式提供。

  • 工作负载身份联合 (OIDC)：按照显示的源系统配置说明设置 OIDC。有关工作负载身份联合的详细信息，请参阅Google Cloud 文档 ↗；有关 OIDC 如何与 Foundry 协同工作的详细信息，请参阅我们的文档。

网络连接

BigQuery 连接器需要通过 443 端口访问以下域名：

  • bigquery.googleapis.com
  • bigquerystorage.googleapis.com
  • storage.googleapis.com
  • www.googleapis.com

可能还需要对以下域名进行额外访问：

  • oauth2.googleapis.com
  • accounts.google.com

如果要在 Google Cloud Platform (GCP) 上的 Foundry 与 GCP 上的 BigQuery 之间建立直接连接，还必须通过相关的 VPC 服务控制启用该连接。如果您的设置需要此连接，请联系 Palantir 支持获取更多指导。

从 BigQuery 同步数据

要设置 BigQuery 同步，请在源概览屏幕的右上角选择探索并创建同步。接下来，选择要同步到 Foundry 中的表。准备好同步后，选择为 x 个数据集创建同步。

探索可用的同步并将其添加到连接器后，导航至编辑同步。在左侧的同步面板中，找到要配置的同步，并选择右侧的>。

数据模型

来自 BigQuery 的表导入到 Foundry 时，数据以 Avro 格式保存。导入时不支持 BIGNUMERIC 和 TIME 类型的列。

将数据从 Foundry 导出到 BigQuery 时，支持除 MAPS、STRUCTS 和 ARRAYS 之外的所有列类型。

导入设置

同步全表

选项是否必填？描述
BigQuery project Id否表所属项目的 ID。

自定义 SQL

可以运行任意查询，结果将保存在 Foundry 中。查询输出必须小于 20GB（BigQuery 表的最大大小），或者必须启用临时表。查询必须以关键字 select 或 with 开头。例如：SELECT * from table_name limit 100;。

增量同步

通常，同步会从目标表导入所有匹配的行，而不论同步期间数据是否发生了变化。相比之下，增量同步会保存有关最近一次同步的状态，并仅从目标摄取新的匹配行。

从 BigQuery 摄取大型表时可以使用增量同步。要使用增量同步，表必须包含一个严格单调递增的列。此外，正在读取的表或查询必须包含具有以下数据类型之一的列：

  • INT64
  • FLOAT64
  • NUMERIC
  • BIGNUMERIC
  • STRING
  • TIMESTAMP
  • DATE
  • TIME
  • DATETIME

示例：一个 5 TB 的表包含数十亿行数据，我们要将其同步到 BigQuery。该表包含一个名为 id 的单调递增列。可以将同步配置为每次摄入 5000 万行数据，使用 id 列作为增量列，初始值为 -1，配置的限制为 5000 万行。

首次运行同步时，前 5000 万行（基于 id 升序排列）且 id 值大于 -1 的数据将被摄入 Foundry。例如，如果该同步运行了多次，且上一次同步期间摄入的最大 id 为 19384004822，则下一次同步将摄入接下来的 5000 万行数据，从第一个大于 19384004822 的 id 开始，依此类推。

增量同步需要以下配置：

选项是否必填？描述
列是选择用于增量摄入的列。如果表不包含任何受支持的列类型，下拉菜单将为空。
初始值是开始同步数据的值。
限制否单次同步中要下载的记录数。

自定义查询的增量同步

要为自定义查询同步启用增量查询，必须将查询更新为以下格式：

SELECT * from table_name where incremental_column_name > @value order by incremental_column_name asc limit @limit

临时表

从非标准 BigQuery 表同步超过 20GB 的数据时，必须启用临时表。临时表允许 BigQuery 从大型查询输出中导入结果，从视图和其他非标准表中导入数据，并允许进行大型增量导入和摄入。

单次同步导入的数据量上限为磁盘可用空间大小。对于在 Foundry worker 上运行的同步，通常限制为 600 GB。请使用增量同步来导入更大的表。

要使用临时表，请在连接器配置中启用临时表设置，并根据需要配置以下设置：

  • 自动创建数据集：选择此选项将创建一个 Palantir_temporary_tables 数据集来存储临时表。此选项要求 BigQuery 账号在该项目中拥有 BigQuery Data Editor 角色。
  • 提供要使用的数据集：选择此选项可手动添加用于存储临时表的数据集的项目 ID 和数据集名称。此选项要求 BigQuery 账号在所提供的数据集上拥有 BigQuery Data Editor 角色。

请注意，在将 BigQuery 视图 ↗ 与虚拟表结合使用时，必须启用临时表。

将数据导出至 BigQuery

对于更复杂的场景，请使用纯代码替代方案。

导出至 BigQuery 是通过导出任务完成的。

该连接器可以通过两种方式导出至 BigQuery：

  • 通过 Google Cloud Storage 作为中间存储（推荐）。有关更多信息，请参阅下文中通过 Cloud Storage 导出章节。
  • 通过 BigQuery API，有关所用 API 的详细信息，请参阅从本地文件读取数据 ↗

通过 API 写入适用于数百万行规模的数据。此模式的预期性能是大约两分钟导出一百万行。如果您的数据规模达到数十亿行，请改用 Google Cloud Storage。

任务配置

要开始导出数据，必须配置导出任务。导航到包含要导出到的连接器的项目文件夹，右键单击连接器名称，然后选择 Create Data Connection Task。

在数据连接视图的左侧面板中，验证 源 名称与要使用的连接器是否匹配。然后，将输入数据集添加到 输入 字段。输入数据集必须命名为 inputDataset，它就是正在导出的 Foundry 数据集。导出任务还需要配置一个 输出。输出数据集必须命名为 outputDataset 并指向 Foundry 数据集。输出数据集用于运行、调度和监控任务。

在数据连接视图的左侧面板中：

  1. 验证 源 名称与要使用的连接器是否匹配。
  2. 添加一个名为 inputDataset 的 Input。输入数据集即为正在导出的 Foundry 数据集。
  3. 添加一个名为 outputDataset 的 Output。输出数据集用于运行、调度和监控任务。
  4. 最后，在文本字段中添加一个 YAML 块来定义任务配置。

左侧面板中显示的连接器和输入数据集标签与 YAML 中定义的名称不一致。

创建导出任务 YAML 时，请使用以下选项：

选项是否必填？默认值描述
project否连接器的 Project ID目标表所属的 Project ID。
dataset是表所属的数据集名称。
table是数据将导出到的表的名称。
incrementalType是SNAPSHOT该值可以为 SNAPSHOT 或 REQUIRE_INCREMENTAL。
• 快照模式导出：目标表的内容将被替换。
• 增量模式导出：内容将被追加到现有表中。

要设置增量导出，请将首次导出设置为 SNAPSHOT，然后将 incrementalType 更改为 REQUIRE_INCREMENTAL。

配置导出任务后，请在右上角选择保存。

向输出表追加行

如果需要向目标表追加行，可以使用 REQUIRE_INCREMENTAL，而不是替换数据集。

增量同步要求行只能追加到输入数据集中，并且 BigQuery 中的目标表必须与 Foundry 中的输入数据集模式匹配。

通过云存储导出（推荐）

要通过 Cloud Storage 导出，必须在连接器的配置设置中配置 Cloud Storage 存储桶。此外，该 Cloud Storage 存储桶必须仅供连接器存放临时表，以确保写入存储桶的任何临时数据仅对极少数用户可见。

建议通过 Cloud Storage 而非 BigQuery API 导出到 BigQuery；Cloud Storage 在大规模运行时表现更佳，且不会在 BigQuery 中创建临时表。

通过 BigQuery API 导出

导出作业会在目标表旁边创建一个临时表；该临时表不会应用额外的访问限制。此外，SNAPSHOT 导出会删除并重建表，这意味着额外的访问限制也会被丢弃。

通过 BigQuery API 导出时，数据会导出到临时表 datasetName.tableName_temp_$timestamp 中。导出完成后，这些行会自动从临时表转移到目标表。

通过 BigQuery API 导出不支持 Hive 表分区。对于使用 Hive 表分区的数据集，请改用通过 Cloud Storage 导出。

如果导出以 REQUIRE_INCREMENTAL 模式运行，则可以共享目标表。以 SNAPSHOT 模式运行时，每次运行都会重建表，因此需要重新应用共享设置。

要通过 BigQuery API 成功导出，请勿对导出的表应用 BigQuery 行级或列级权限。

在代码中使用 BigQuery 源

对于更复杂的场景，可以使用专业代码替代方案连接到 BigQuery 源。

以下示例演示了如何在外部转换中使用 Google BigQuery 的 Python 客户端 ↗ google-cloud-bigquery 连接到 BigQuery 源。

这些示例基于存储在 BigQuery 中的 Shipments 表，该表表示包裹货运列表及其重量、承运商和跟踪单号。

通过外部转换从 BigQuery 读取

本示例从给定承运商列表中读取超过特定重量的货运信息。

1from pandas import DataFrame
2from transforms.api import lightweight, Output, Input, transform_pandas
3from transforms.external.systems import ResolvedSource, external_systems, Source
4from google.cloud import bigquery
5import json
6
7@lightweight
8@external_systems(
9    bigquery_source=Source("<source_rid>")
10)
11@transform_pandas(
12    Output("<dataset_rid>"),
13    carriers_df=Input("<dataset_rid>") # dataframe with schema [carrier_name: String, carrier_details: String]
14)
15def read(bigquery_source: ResolvedSource, carriers_df: DataFrame) -> DataFrame:
16    # Authenticate BigQuery client
17    account_info = json.loads(bigquery_source.get_secret("jsonCredentials"))
18    client = bigquery.Client.from_service_account_info(account_info)
19
20    # Table reference
21    bigquery_project = "<bigquery_project_id>"
22    bigquery_dataset = "<bigquery_dataset_id>"
23    bigquery_table = "<bigquery_table_id>"
24    table_ref = f"{bigquery_project}.{bigquery_dataset}.{bigquery_table}"
25
26    # Filter parameters
27    carrier_list = carriers_df['carrier_name'].drop_duplicates().tolist()
28    min_weight = 10
29
30    job_config = bigquery.QueryJobConfig(
31        query_parameters=[
32            bigquery.ArrayQueryParameter("carrier_list", "STRING", carrier_list),
33            bigquery.ScalarQueryParameter("min_weight", "FLOAT64", min_weight)
34        ]
35    )
36
37    query = f"""
38        SELECT shipment_id, carrier, weight, tracking_number
39        FROM `{table_ref}`
40        WHERE carrier IN UNNEST(@carrier_list) AND weight >= @min_weight
41    """
42
43    query_job = client.query(query, job_config=job_config)
44    return query_job.to_dataframe()

Write to BigQuery with an external transform

This function updates tracking numbers for shipments based on the provided shipment information, and then returns the updated shipment details.



Copied!
1from pandas import DataFrame
2from transforms.api import lightweight, Output, Input, transform_pandas
3from transforms.external.systems import ResolvedSource, external_systems, Source
4from google.cloud import bigquery
5import json
6
7@lightweight
8@external_systems(
9    bigquery_source=Source("<source_rid>")
10)
11@transform_pandas(
12    Output("<dataset_rid>"),
13    shipments_df=Input("<dataset_rid>"), # DataFrame with schema [shipment_id: Long, tracking_number: String]
14)
15def update_tracking_numbers(bigquery_source: ResolvedSource, shipments_df: DataFrame) -> DataFrame:
16    # Authenticate BigQuery client
17    account_info = json.loads(bigquery_source.get_secret("jsonCredentials"))
18    client = bigquery.Client.from_service_account_info(account_info)
19
20    # Table reference
21    bigquery_project = "<bigquery_project_id>"
22    bigquery_dataset = "<bigquery_dataset_id>"
23    bigquery_table = "<bigquery_table_id>"
24    table_ref = f"{bigquery_project}.{bigquery_dataset}.{bigquery_table}"
25
26    # 1. Clear temporary table if it exists
27    temp_table_id = f"{bigquery_project}.{bigquery_dataset}.temp_update_tracking"
28    client.delete_table(temp_table_id, not_found_ok=True)
29
30
31    # 2. Upload shipments_df to a temporary table
32    load_job_config = bigquery.LoadJobConfig(
33        schema=[
34            bigquery.SchemaField("shipment_id", "INT64"),
35            bigquery.SchemaField("tracking_number", "STRING"),
36        ]
37    )
38    job = client.load_table_from_dataframe(shipments_df, temp_table_id, job_config=load_job_config)
39    job.result()  # Wait for load to finish
40
41    # 3. MERGE to update tracking_number in main table
42    merge_query = f"""
43    MERGE `{table_ref}` T
44    USING `{temp_table_id}` S
45    ON T.shipment_id = S.shipment_id
46    WHEN MATCHED THEN
47      UPDATE SET tracking_number = S.tracking_number
48    """
49
50    client.query(merge_query).result()
51
52    # 3. (Optional) Clean up temporary table
53    client.delete_table(temp_table_id, not_found_ok=True)
54
55    # Optionally, return the updated rows as a DataFrame for confirmation
56    updated_ids = shipments_df["shipment_id"].tolist()
57    query = f"""
58        SELECT shipment_id, tracking_number
59        FROM `{table_ref}`
60        WHERE shipment_id IN UNNEST(@updated_ids)
61    """
62    job_config = bigquery.QueryJobConfig(
63        query_parameters=[
64            bigquery.ArrayQueryParameter("updated_ids", "INT64", [int(x) for x in updated_ids])
65        ]
66    )
67    updated_rows = client.query(query, job_config=job_config).to_dataframe()
68    return updated_rows

Virtual tables

This section provides additional details around using virtual tables with a BigQuery source. This section is not applicable when syncing to Foundry datasets.

The table below highlights the virtual table capabilities that are supported for BigQuery.

CapabilityStatus
Bulk registration🟢 Generally available
Automatic registration🟢 Generally available
Table inputs🟢 Generally available: tables, views, materialized views in Code Repositories, Pipeline Builder
Table outputs🟢 Generally available: tables in Code Repositories, Pipeline Builder
Incremental pipelines🟢 Generally available: APPEND only [1]
Compute pushdown🟢 Generally available

Consult the virtual tables documentation for details on the supported Foundry workflows where BigQuery tables can be used as inputs or outputs.

[1] To enable incremental support for pipelines backed by BigQuery virtual tables, ensure that Time Travel ↗ is enabled with the appropriate retention period. This functionality relies on Change History ↗ and is currently append-only. The current and added read modes in Python Transforms are supported. The _CHANGE_TYPE and _CHANGE_TIMESTAMP columns will be made available in Python Transforms.

It is critical to ensure that incremental pipelines backed by virtual tables are built on APPEND-only source tables, as BigQuery does not provide UPDATE or DELETE change information. See official BigQuery documentation ↗ for more details.

Source configuration requirements

When using virtual tables, remember the following source configuration requirements:

  • You must use a Foundry worker source. Virtual tables do not support use of agent worker connections.
  • Ensure that bi-directional connectivity and allowlisting is established as described in the Networking section of this documentation.
  • If using virtual tables in Code Repositories, refer to the Virtual Tables documentation for details of additional source configuration required.
  • Use service account key file credentials or Workload Identity Federation (OIDC). Instance-based authentication is not supported with virtual tables.
  • Temporary tables must be enabled in order to register BigQuery views ↗, and the relevant roles must be added to the authentication credentials.
  • Configure Spark settings to read unparameterized BIGNUMERIC columns in Spark.

See the Connection details and Temporary tables section above for more details.

Compute pushdown

Foundry offers the ability to push down compute to BigQuery when using virtual tables. Virtual table inputs leverage the BigQuery Spark connector ↗ which has built-in support for predicate pushdown.

When using BigQuery virtual tables registered to the same source as inputs and outputs to a pipeline, it is possible to fully federate compute to BigQuery. This feature is currently available in Python transforms. See the Python documentation for details on how to push down compute to BigQuery.

Troubleshooting

Not found: Dataset <dataset> was not found in location <location>

BigQuery determines the location where a given query will be run based on either the inputs used in the query or where the results of the query are being stored. When temporary tables are used, the output is set to a temporary table in the temporary tables dataset; the location of this dataset will determine where the query is run. Ensure that all inputs of the sync and the temporary tables dataset are in the same region. If the Automatically create dataset. setting is enabled, use the Google Cloud console or Google's SDKs/APIs to determine the location of the dataset called Palantir_temporary_tables.

gRPC message exceeds maximum size

If syncing data with large content like JSON columns, the transfer may fail with the above error. Adjust BigQuery's Maximum inbound message size in gRPC Settings on the source to increase data transfer in a single API call. Remember, a single call fetches multiple rows, so setting it to the largest row size may not be enough.

You can find this configuration option by navigating to Connection Settings > Connection Details, then scrolling to More options and selecting gRPC Settings.

Preview fails to load for a dataset or virtual table based on a BigQuery view

If your preview fails to load and the target BigQuery table is a view, this may be the result of a schema mismatch. Views' schemas are determined by the result of the SQL query defined at the time of creation. Subsequent changes to the underlying table's structure are not automatically applied to the view's schema and may lead to unexpected behavior when trying to consume the view (such as via a sync/virtual table in the Palantir platform.)

You can verify this by comparing the schemas between the view and the underlying table in the BigQuery console, checking for any discrepancies. This can include:

  • Columns present in the view but not in the underlying table (and vice versa)
  • Having two different data types for the same column

In the case that any discrepancies are identified, you can address them by recreating or refreshing the view using CREATE OR REPLACE VIEW or ALTER VIEW.
