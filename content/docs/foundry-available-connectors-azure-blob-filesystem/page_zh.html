OneLake and Azure Blob Filesystem (ABFS)

Connect Foundry to OneLake ↗, Azure Data Lake Storage Gen2 (ADLS Gen2), and other eligible Azure products using Azure Blob Filesystem (ABFS) ↗. The ABFS connector allows files to be read into Foundry and written from Foundry to Azure.

Connections to Azure Data Lake Storage Gen1 are not supported.

Data model

The connector can transfer files of any type into Foundry datasets. File formats are preserved and no schemas are applied during or after the transfer. Apply any necessary schema to the output dataset, or write a downstream transformation to access the data.

Performance and limitations

There is no limit to the size of transferable files. However, network issues can result in failures of large-scale transfers. In particular, syncs running on a Foundry worker that take more than two days to run will be interrupted. To avoid network issues, we recommend using smaller file sizes and limiting the number of files that are ingested in every execution of the sync. Syncs can be scheduled to run frequently.

Setup

  1. Open the Data Connection application and select + New Source in the upper right corner of the screen.
  2. Select ABFS—Azure Data Lake Storage Gen2 from the available connector types.
  3. Follow the additional configuration prompts to continue the set up of your connector using the information in the sections below.

We recommend using a Foundry worker with direct connection policies with the ABFS connector to simplify setup and configuration.

Learn more about setting up a connector in Foundry.

Authentication

To import data, the connector must list the filesystem contents and read the file contents in Azure. This behavior can be achieved through authorization patterns of the principal associated with the provided connection credentials.

  • In the case of role-based access control (RBAC), a coarse-grain form of access control, the principal needs the Storage Blob Data Reader ↗ role on the container where the data is located.
  • For access control lists (ACL), a fine-grained form of access control, the principal needs the Read ACL on the ingest files, and the Execute ACL on all path directories, from the container root to the file location.

Learn more about access control in ADLS Gen2 ↗.

The ABFS connector supports the following configurations options for authentication, listed from most to least recommended:

  • Client credentials (recommended)
  • Shared access signature
  • Username and password
  • Refresh token
  • Shared key
  • Workload Identity Federation
  • Azure managed identity (legacy — for existing agent worker sources)

选项是否必填？描述

网络配置

在 Foundry 中运行 Azure 连接时，需要设置网络访问权限，以便 Foundry 与源进行通信。这是一个两步过程：

  1. 设置从 Foundry 到源的网络出口访问权限。可以在数据连接应用中对源应用相应的出口策略来完成此操作。Azure 连接器需要通过 端口 443 访问存储帐户的域。例如，如果要连接到 abfss://file_system@account_name.dfs.core.windows.net，则存储帐户的域为 account_name.dfs.core.windows.net。数据连接应用会根据源中提供的连接详情建议合适的出口策略。
  2. 将 Foundry 添加到 Azure 存储容器的允许列表中。可以按照 Azure 文档 ↗ 中的说明，将 Foundry IP 添加到 Azure 存储帐户的允许列表中。可以在控制面板应用的 Network Egress 下找到 Foundry IP 详情。

如果要访问与 Foundry 注册位于同一 Azure 区域的 Azure 存储桶，则需要通过 PrivateLink 设置与上述不同的网络配置。如果属于这种情况，请联系 Palantir 管理员寻求帮助。

要开始导出数据，必须配置导出任务。导航至包含要导出连接器的项目文件夹，右键单击连接器名称，然后选择 Create Data Connection Task。

  2. 添加名为 inputDataset 的 Input。输入数据集即正在导出的 Foundry 数据集。

rewritePaths

如果第一个键与文件名匹配，键中的捕获组将被替换为对应的值。该值本身可以包含额外部分，用于向文件名添加元数据。

如果值包含：

  • ${dt:javaDateExpression}：值的此部分将被替换为文件导出时的时间戳。javaDateExpression 遵循 DateTimeFormatter ↗ 模式。
  • ${transaction}：这部分值将被替换为包含该文件的事务的 Foundry 事务 ID。
  • ${dataset}：这部分值将被替换为包含该文件的数据集的 Foundry 数据集 ID。

示例：

假设 Foundry 数据集中有一个名为 "spark/file_name" 的文件，其事务 ID 为 transaction_id，数据集 ID 为 dataset_id。如果使用表达式 fi.*ame 作为键，并使用 file_${dt:DD-MM-YYYY}-${transaction}-${dataset}_end 作为值，当该文件写入 Azure 时，其存储名称将为 spark/file_79-03-2023-transaction_id-dataset_id_end。

标志文件

对于给定的构建任务，一旦所有数据复制完毕，连接器即可向 Azure 存储写入一个空的标志文件。该空文件表明内容已就绪可供使用，且不会再进行修改。标志文件将写入 directoryPath。然而，如果启用了 createTransactionFolders，则每个写入内容的文件夹都将生成一个标志文件。如果启用了标志文件，且该标志文件名为 confirmation.txt，所有标志文件将在本次构建中导出的文件写入完成后一并写入。

标志文件在构建结束时写入，而不是在子文件夹导出完成时写入。

如果 Azure 中的文件比标志文件还要新，通常意味着上一次导出未成功，或者该文件夹正在进行导出。

配置完导出任务后，在右上角选择 保存。

源配置要求

使用虚拟表时，请注意以下源配置要求：

  • 必须使用 Foundry worker 源。虚拟表不支持使用 agent worker 连接。
  • 确保已按照本文档的网络部分所述建立双向连接性并完成白名单设置。
  • 如果在代码仓库中使用虚拟表，请参阅《虚拟表文档》了解所需额外源配置的详细信息。
  • 设置源凭证时，必须使用 client credentials、username/password 或 workload identity federation 其中之一。使用虚拟表时不支持其他凭证选项。

Delta

要为由虚拟表支撑的管道启用增量支持，请确保在源 Delta 表上启用了更改数据馈送 ↗。支持 Python 转换中的 current 和 added 读取模式。_change_type、_commit_version 和 _commit_timestamp 列将在 Python 转换中可用。

Iceberg

加载由 Apache Iceberg 表支撑的虚拟表需要 Iceberg 目录。要了解有关 Iceberg 目录的更多信息，请参阅《Apache Iceberg 文档》 ↗。在同一个源上注册的所有 Iceberg 表必须使用相同的 Iceberg 目录。

默认情况下，将使用 S3 中的 Iceberg 元数据文件创建表。注册表时必须提供用于指示这些元数据文件位置的 warehousePath。

《Unity 数据目录》 ↗ 在 Databricks 平台 中使用 Delta Universal Format (UniForm) 时，可用作 Iceberg 数据目录。有关此集成的更多详情，请参阅 《Databricks 官方文档》 ↗。可在数据源的 连接详情 选项卡中配置该数据目录。连接至 Unity 数据目录 需提供端点地址和 个人访问令牌。表注册应采用 catalog_name.schema_name.table_name 命名模式。

增量支持依赖 Iceberg 的 《增量读取》 ↗，且目前仅支持仅追加模式。支持 《Python 转换》 中的 current 和 added 读取模式。

Parquet

使用 Parquet 的虚拟表依赖模式推断。系统最多使用 100 个文件来确定模式。
