环境创建概览

这是一份深入探讨环境初始化过程的高级指南。面向希望了解影响初始化性能技术细节的用户。如需有关常见环境问题的通用指导，请参阅《环境故障排除指南》。

Conda 是一款开源的、与语言无关的包和环境管理器。Mamba 是 Conda 包管理器的开源重新实现。Hawk 是 Palantir 开发的 Conda 包管理器的重新实现。代码仓库在提交后运行检查时，使用 Hawk 解析包依赖关系。与 Mamba 和 Conda 相比，Hawk 在包解析方面具有多项优势，最显著的是速度更快，且处于活跃维护和开发状态。

本页介绍了最重要的概念并概述了环境创建过程；更多信息，请参阅《Conda 官方文档 ↗》和《Mamba 文档 ↗》。

重要术语

包

包是文件的集合，通常包含元数据、库和/或二进制文件。代码仓库提供了丰富的包（例如 numpy）供访问，以补充核心语言功能。

包是带有版本控制的，且几乎总有一组依赖项——为了让该包正常运行，必须同时安装其他包。依赖项可以是包的特定版本、可接受的版本范围，或者是任意版本。

频道

频道，有时也称为仓库，是存储包的任何位置。一个频道可以是本地文件系统中的一个目录，另一个频道可以是托管在网络服务器上的目录。

无论何种类型，每个频道都是一棵目录树，按平台架构对包进行划分。每个平台子目录都包含一个名为 repodata.json 的文件，它是该子目录中所有包的索引。

Conda 在需要获取包时，会搜索一组预配置的频道。有关 Foundry 中频道管理的更多信息，请参阅《包文档》。

环境

Conda 环境是一个包含特定包集合的目录。系统会将环境配置面板中指定的包传递给 Conda，为每个仓库创建一个环境。Conda 会构建一组满足配置及所有依赖关系的包，并将这些包安装到支持该仓库的 Spark 模块上。

性能

以下关于性能的说明借鉴了这篇《Anaconda 博文 ↗》，该文详细讨论了 Conda 的性能，但也同样适用于 Mamba 或 Hawk 的实现。接下来两节对这部分内容进行了总结，并概述了与代码仓库最相关的性能因素：

  • 创建环境
  • 局限性

创建环境

环境创建包含两个主要步骤：解析步骤和安装步骤。

解析步骤

在解析步骤中，指定的包管理器（Conda、Mamba 或 Hawk）会尝试找到满足所有传递依赖的包及版本。传递依赖由环境配置面板中指定的包的依赖、这些依赖的依赖等共同构成。此步骤包含四个阶段：

  1. 下载并处理包索引。包管理器会从每个配置的频道下载相关的 repodata.json 文件，并将索引条目转换为内存中的对象。
  2. 缩减索引。包管理器会构建一组可能用于该环境的所有包。为此，算法从提供的包规范开始，递归遍历所有依赖项。所有不需要的包——主要是那些不在依赖图中的包——都会被剪除。
  3. 将依赖约束表示为布尔可满足性 (SAT) 问题。包管理器偏好特定类型的解决方案（例如包的最新可用版本），这些偏好会内化到子句构造中。
  4. 运行 SAT 求解器。

安装步骤

若求解成功，接下来便是安装步骤。在此阶段，每个构件将从相应渠道获取，包管理器利用这些构件构建环境。此步骤包含三个阶段：

  1. 下载并解压已求解环境中的所有包。
  2. 校验包内容。根据配置，Conda 会使用校验和或验证文件大小是否正确。
  3. 将包链接至环境中。

局限性

在特定情况下，上述步骤均可能出现延迟。导致缓慢的原因通常归为以下三类：

  • 上游变更
  • 环境规格
  • 包大小

上游变更

大部分延迟源于 Foundry 外部的因素。

  • 下载与处理频道索引的开销随索引文件总大小增加而增加；需考量的频道越多、频道规模越大，此步骤耗时越长。
  • 索引缩减的开销也随传递依赖的数量增加而增加；传递依赖的数量取决于包声明了哪些依赖项。

由于这些因素具有外部性和不可见性，针对性能退化的根因分析往往面临困难。某个环境可能突然加载变慢，原因可能在于某频道近期体积变大，或某包在其最新版本中声明了数个新依赖。

环境规格

更常见的情况是，初始化缓慢与环境规格直接相关。求解步骤的开销随环境规模呈超线性增长，因此作为一般经验法则，包含更多包的环境初始化时间将不成比例地延长。

有两种方法可缓解此类情况。

  • 首先，从环境定义中移除不需要的包。采用小型的、专用的环境远比大型的、通用的环境性能更佳。
  • 其次，尝试在环境配置面板中为部分包添加版本约束。对于像 python 这样存在大量现存构建的包，以及像 scipy 这样具有复杂依赖图的包，锁定版本最为有效。此举可使 Conda 更积极地缩减索引，从而减少 SAT 求解所需考量的包版本数量。

包大小

在此部分讨论的因素中，包大小通常引发的问题较小，但在某些情况下仍需关注。

即便仅下载并解压单个包，也可能并非易事。例如，pytorch 包体积约 460 MB，解压耗时可能超过 35 秒。

下载、解压与校验的开销均随环境中包的规模和数量呈线性增长。受传递依赖影响，求解后的环境通常包含比环境定义中显式指定的数量更多的包，包数量的增加可能导致延迟。

针对此情况的缓解措施与环境规格的建议类似：尽量保持环境精简。
