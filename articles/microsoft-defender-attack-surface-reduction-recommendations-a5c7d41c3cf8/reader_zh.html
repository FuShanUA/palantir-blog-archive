Microsoft Defender Attack Surface Reduction Recommendations

2021-01-11 · Original →

本篇博文基于 Palantir 信息安全团队过去两年从 Windows Defender 攻击面减少 (ASR) 系列安全控制机制中收集的审计数据，提供了一套建议。希望能为正在考虑部署该功能的其他安全团队提供参考。我们将结合生产环境的部署经验，着重说明各项设置的注意事项。

对于初次接触该主题的读者，Windows Defender 攻击面减少 (ASR) 是微软赋予一组控制机制的名称，用于限制 Windows 终端上常见的恶意软件和漏洞利用技术。

与 Windows Defender 漏洞利用防护不同，ASR 控制机制就像是简单的开关，管理员利用组策略或 Intune 即可在极短时间内完成部署，尤其是在计划使用“仅审计”模式的情况下。

微软网站上的这段话提供了一个很好的概述：

ASR 针对的是经常被攻击者滥用的软件行为，例如：

  • 启动试图下载或运行文件的可执行文件和脚本
  • 运行混淆的或其他可疑的脚本
  • 执行应用程序在日常工作中通常不会发起的行为

合法应用程序有时也会出现这类行为；但由于它们常被恶意软件滥用，因此被视为存在风险。攻击面减少规则可以约束此类高风险行为，帮助保护您的组织安全。

建议摘要

我们在本帖中力求提出一些明确的主张，以便为读者提供实用价值，但同时也承认所有设置都受具体环境特性的影响。请参阅本摘要下方的描述，了解我们提出这些建议的原因。您在所保护的网络中面临的一组约束条件很可能与我们的有所不同。

对于那些希望直接切入自身环境日志的读者，相关信息将记录在两个不同的事件中：

  • 仅审计模式：Windows 事件 1122
  • 阻止模式：Windows 事件 1121

以阻止模式配置任何 ASR 设置，将导致 Windows 拒绝该行为并记录事件。面临的明显挑战在于，Windows 可能会在某些环境中限制合法用例，这正是我们希望通过此文帮助大家避免的情况。仅审计模式本身就能为防御者提供极佳的数据源，且在大多数企业环境中，利用组策略不到一小时即可完成全网部署。

关于 ASR 部署和事件日志转发的说明：在 Palantir，我们大量使用 Windows 事件转发 (WEF) 来收集 1121 和 1122 事件以进行分析。我们的 WEF 指南可在此处找到，配置示例可在此处获取。我们的 ASR 部署是通过配置此处描述的组策略设置来完成的。您可以为所有 ASR 规则使用单一策略；只需针对配置的每个设置，在阻止模式和审计模式之间做出选择即可。另一种方法是创建多个策略，为需要授予阻止模式规则例外的情况配置例外项，将其作为更高优先级的组策略，并通过组策略对象上的“应用组策略”访问控制条目进行限制。

规则解析

阻止从 USB 运行不受信任且未签名的进程

以阻止模式配置“阻止从 USB 运行不受信任且未签名的进程”未造成任何问题。我们在第一天就以阻止模式配置了此规则，在过去 18 个月中未记录到任何与此 ASR 规则相关的事件。我们认为，在几乎任何企业环境中，以阻止模式部署此规则都是安全的。

建议：阻止模式。

阻止 Adobe Reader 创建子进程

在 ASR 实施初期，我们发现“阻止 Adobe Reader 创建子进程”确实会导致一个轻微问题，这与 Adobe 目前为某些产品进行更新的方式有关。

以下是一个示例事件：

Windows Defender Exploit Guard 阻止了您的 IT 管理员不允许的操作。如需更多信息，请联系您的 IT 管理员。
ID: 7674BA52-32EB-4A4F-A9A1-F0F9A1619A2C
检测时间: 2020-10-16T19:35:03.723Z
用户: DOMAIN\User
路径: C:\xxx\Acrobat_DC_Set-Up.exe
进程名: C:\xxx\AcroRd32.exe
安全智能版本: 1.325.883.0
引擎版本: 1.1.17500.4
产品版本: 4.18.2009.7

Adobe 应用程序的更新机制尝试启动子进程来执行操作。这对 ASR 规则而言构成了问题。

我们并未回退该规则，而是选择通过中心的软件补丁和维护服务来执行 Adobe 更新，这样 Adobe 就没有合法理由启动子进程。为了获得与 PDF 恶意软件相关的额外安全保障，我们决定接受这一微小的运维成本。

建议：阻止模式。
提示：注意 Adobe 更新过程。确保您备有替代的更新方案。

阻止来自电子邮件客户端和网页邮箱的可执行内容

在我们收集的数据中，未记录到与此控制机制相关的事件。我们的许多用户确实使用了该规则所针对的电子邮件客户端和网页邮箱。我们的理论是，电子邮件传递路径中的其他控制机制在保护用户免受电子邮件中可执行内容侵害方面做得相当不错。显然，我们仍建议保留此控制机制，作为保护终端的最后一道防线。

建议：阻止模式。

阻止 JavaScript 或 VBScript 启动下载的可执行内容

在我们收集的 18 个多月的数据中，未记录到与此控制机制相关的事件。

微软提供了以下建议：

“虽然不常见，但业务线应用程序有时会使用脚本下载并启动安装程序。”

然而，作为本文基础的 Windows 环境包含大约 1000 台终端，我们并未记录到任何指示此类行为的迹象。因此，我们认为它对大多数企业网络来说极有可能是安全的，但管理员应留意上述警告。

建议：阻止模式。

阻止通过 WMI 事件订阅实现持久化

在我们收集的 18 个多月的数据中，未记录到与此控制机制相关的事件。实际上，我们最初是在审计模式下运行的，因为我们忽略了关于它是仅限阻止模式控制机制的说明（审计模式默认为阻止）。

微软对此的描述为：“此规则可防止恶意软件滥用 WMI 在设备上实现持久化。”

这种持久化行为已在许多 C2 框架中实现，使其成为攻击者唾手可得的战术。正因如此，我们虽然已经开发了与此行为相关的额外告警，但也十分欣赏这一控制机制的简洁性。我们配置了阻止模式，且未记录到任何问题。

建议：阻止模式。

阻止从 Windows 本地安全机构子系统 (lsass.exe) 窃取凭据

lsass 保护规则是我们遇到的最常见的 ASR 审计模式事件之一。在我们的环境中，它每六个月大约产生 1200 万个事件。

许多安全进程也会触发 lsass.exe 规则的 ASR 告警，从防御者的角度来看，很难区分合法用例与攻击者的攻击手法。

在规划阻止模式实施时，由于审计模式下记录的事件数量庞大，我们最初决定不对该规则强制执行阻止。

经过一番深思熟虑，并对部分用例和告警进行深入挖掘后，我们认为捕获到的主要是微软在此说明中描述的事件：

在某些应用中，代码会枚举所有正在运行的进程，并尝试以穷举权限打开它们。此规则将拒绝应用的进程打开操作，并将详细信息记录到安全事件日志中。此规则可能会产生大量噪音。如果某个应用只是枚举 LSASS，但对功能没有实际影响，则无需将其添加到排除列表中。仅凭此事件日志条目本身并不一定表明存在恶意威胁。

我们还应提及，环境中已经部署了 Credential Guard。这让我们更加确信，跨进程访问 lsass 不太可能存在合法且关键的用例。

基于这些原因，我们将 lsass 规则移至阻止模式，并取得了成效。截至撰写本文时，该规则已部署三个月，未记录到任何用户影响。如果出现新的考量因素，我们定会更新此文。

微软还在此博文中提供了一种统计方法，用于区分信号与噪音。

建议：阻止模式。
提示：您会看到此设置产生大量审计模式事件。不要因此放弃向阻止模式过渡。

阻止 Office 应用程序创建可执行内容

我们在六个月内为一小部分用户记录了大约 100 个审计事件。经过仔细调查，我们了解到导致 ASR 规则“阻止 Office 应用程序创建可执行内容”出现问题的文件是：

  • “microsoft.office.smartlookup.ssr.js”

此文件似乎是 Office 智能查找功能的一部分，微软在此处对该功能进行了描述。

利用该 .js 文件进行智能查找的 Office 进程分布如下：

  • MS Excel 占 75%
  • MS OneNote 占 20%
  • MS Word 占 5%

作为防御者，我们自问的一个问题是：攻击者是否会利用这个 .js 文件？虽然可能性不大，但至少这个文件似乎提供了持久化的机会。因此，我们目前尚未将该文件从 ASR 审计中排除。我们针对有业务需求的用户设置了例外，将规则移至阻止模式，并更新了内部维基，以便员工根据业务需求更轻松地申请例外。

建议：阻止模式。
提示：如果 Microsoft Office 智能查找功能对您的用户很重要，请务必谨慎。

阻止 Office 应用程序向其他进程注入代码

令人惊讶且遗憾的是，我们存在合法用例，导致无法立即以阻止模式推进此规则。这本应是个简单的决定。

微软对该规则的解释如下：

此规则阻止 Office 应用程序向其他进程进行代码注入的尝试。攻击者可能会尝试利用 Office 应用程序通过代码注入将恶意代码迁移到其他进程中，从而使代码伪装成干净进程。使用代码注入没有任何已知的合法业务目的。

在六个月期间，我们收集了大约 10,000 个与此行为相关的事件。

大多数事件与环境中所识别出的一款供极少数用户使用的专有/商业非微软应用有关。为了说明进程注入的使用有多么不规则：

  • 该专有/商业应用占了环境中此行为的 97%。
  • Adobe Acrobat 占 1%（大约每 6 个月 100 个事件）
  • Microsoft Word 告警额外占 1%
  • 其余事件合计占 1%，分布在 Office 套件中。

目前，我们无法对大多数设备以阻止模式推进此规则，不得不创建单独的组策略来满足这一小部分用户的需求。

建议：阻止模式，需设置小规模例外组。
提示：这是您需要仔细评估审计数据的控制机制之一。如果您不幸遇到具有此行为且业务至关重要的应用程序，而只有少数人需要该加载项，我们建议针对这一小部分用户使用单独的例外组策略。

阻止 Office 宏调用 Win32 API

如果您读到了这里，非常感谢，但您肯定会想：“这本该是个轻易就能决定阻止的选项。”我们当时也是这么想的，却对数据感到惊讶。

目前，我们每半年在此类别中记录约 15,000 个事件。

所有事件都与旧式 Office 宏的使用有关，事件指向以下变体：

  • C:\Program Files\Microsoft Office\root\Office16\STARTUP\ProprietaryThing.dotm → word/vbaProject.bin

我们找到的对此行为最简明的解释在这里。

该页面解释道：

“vbaProject.bin 文件是一个二进制 OLE COM 容器。这是 Excel 2007 之前的旧版 xls 版本所使用的格式。与 xlsx/xlsm 文件的其他所有组件不同，其数据并非以 XML 格式存储。相反，函数和宏是以预解析的二进制格式存储的。”

在我们的环境中，我们可以将此行为归因于一小部分用户所需的旧版插件。有趣的是，单个严重依赖该插件的用户产生了约 30% 的事件量。

顺便一提：我们在此网络中阻止了大多数用户的宏，并建议与我们合作的其他环境也采取同样做法。@Oddvarmoe 最近发布了这篇优秀的博文，描述了企业环境中宏的安全选项。

建议：阻止模式，需设置小规模例外组。
提示：这是您需要仔细评估数据的第二个控制机制。如果您不幸遇到具有此行为且业务至关重要的应用程序，我们建议您在敦促供应商更新的同时，尝试为这些用户使用单独的例外组策略。

阻止所有 Office 应用程序创建子进程

此控制机制每六个月在少量终端设备上产生约 150 个事件。这些终端设备由共享一组通用工作流的团队成员使用。有趣的是，一半的事件与团队使用的一款商业/专有应用有关。

我们查阅了该应用程序的供应商文档，发现这种子进程方式实际上是在两个不利选择中为了保证应用程序正常运行而采取的损害较小的一种，无法避免，否则无法满足业务需求。供应商提供的替代选项利用 rundll32.exe 来实现类似目标。

此 ASR 类别中剩余 50%（每 6 个月约 75 个）的事件分布在知名的 Office 二进制文件（Word、Excel 等）以及套件中一些不常见应用程序的安装程序（officeconnectsetup、officeconnectupdate 等）之间。

建议：阻止模式，需设置小规模例外组。
提示：这是另一个需要您仔细评估数据的控制机制。我们建议针对有业务需求的用户使用单独的例外组策略。

阻止执行潜在混淆的脚本

这是一类数量不多但分布广泛的事件。我们每六个月观察到大约 70 个事件。这些事件分布在多台不同的终端设备上。

每个六个月期间内的大多数事件（80% 以上）属于：

  • C:\Windows\System32\WindowsPowerShell\v1.0\Modules\SmbShare\Smb.types.ps1xml

Smb.types.ps1xml 文件很有意思。这个 XML 结构化文件中的叶属性是 PowerShell 命令，但除了作为这种不常见文件类型的一部分之外，它们实际上似乎并未被混淆。

例如：


[Microsoft.PowerShell.Cmdletization.GeneratedTypes.SmbShare.ShareState]
($this.PSBase.CimInstanceProperties['ShareState'].Value)

还有一个未签名的 PowerShell 模块，与其余数据相比经常被标记。您可以在以下位置找到它：

  • C:\Windows\System32\WindowsPowerShell\v1.0\Modules\ServerManagerTasks\ServerManagerTasks.psd1

与 SMBTypes 文件一样，该文件也是由 Microsoft 创建的，似乎不构成威胁。

Author of this module

Author = 'Microsoft Corporation'

它似乎没有以任何方式被混淆，但绝对触发了检测。

作为防御者，我们并不热衷于将未签名的脚本排除在 ASR 规则集之外，即使它位于相当安全的路径中。我们不喜欢攻击者轻易修改它们并可能躲过我们检测的可能性。

除了这两个有趣的微软文件外，我们还收到了与内部开发工作流相关的“潜在混淆”告警。

建议：在企业环境中采用阻止模式，但对于开发人员机器需谨慎。
提示：我们并不担心上述微软文件被阻止，而是更担心确保我们庞大的开发人员群体不会受到负面影响。如果您所处的环境能够从组策略角度轻松将开发人员机器与企业机器分开，那可能是处理此控制机制的明智之举。您或许可以基于审计模式事件构建检测机制。

阻止可执行文件运行，除非它们满足流行度、时间或可信列表条件

我们立刻意识到，在 Palantir，此 ASR 控制机制在阻止模式下不太可能提供价值，原因很简单：我们的开发人员会频繁生成新文件。我们仍然觉得分享我们收集的关于此控制机制的数据可能对他人有所帮助：

  • 我们每六个月产生约 180,000 个此类别的 1122 ASR 事件。
  • 大约 110,000 个（60%）事件直接与内部开发工作流相关。
  • 其余事件归因于非常常见且频繁更新的外部包。

在实际操作中，我们接受在此项上无法超越审计模式。不过，此控制机制在审计模式下仍具有很大价值。通过非常简单的查询，我们就能看到 Microsoft Defender 因其年龄和其他信任启发式机制而产生疑虑的所有二进制文件。

建议：审计模式。
提示：只有最严格控制的环境才能以阻止模式支持此 ASR 设置。除了开发人员工作流外，我们收集到审计事件的应用程序都非常常见，只是更新频繁。

使用针对勒索软件的高级防护

以下是微软对该控制机制的描述：

此规则提供了针对勒索软件的额外保护层。它扫描进入系统的可执行文件，以确定其是否可信。如果文件与勒索软件高度相似，此规则将阻止其运行，除非它们位于可信列表或排除列表中。

在我们审计的环境中，此控制机制类似于“阻止可执行文件运行，除非它们满足流行度……”控制机制，确实拦截了一些开发工作流。我们每六个月看到约 500 个事件，所有事件都与团队进行 Windows 应用程序开发的内部产品有关。

因此，让全公司强制执行此 ASR 规则并不合适，但我们能够针对需要例外的群体利用单独的策略，并鼓励其他人也这样做。减少摩擦的关键是在启用规则后监控事件，并具备处理例外的流程。

建议：阻止模式。
提示：如果您支持开发工作流，请务必谨慎。

阻止源自 PSExec 和 WMI 命令的进程创建

我们最喜欢的横向移动技术之一（有时由与我们合作的 SpecterOps 攻击者模拟团队使用）是利用“网络登录”类型和 WMI/psexec 风格的横向移动。之所以说是“最喜欢”，是因为它非常难以防范。

我们在此处写过一篇关于该主题的长文，但在考虑此 ASR 控制机制时，请注意：

  • 网络登录类型将在客户端绕过通过组策略进行的“允许登录”权限管理等机制，操作可以简单到只需执行“runas /user:contoso\bob-t1 'psexec.exe \tier1server.contoso.com cmd.exe'”
  • 网络登录类型和 psexec 通常会绕过寻找层级违规的网络检测设备。
  • 修改 DisableRestrictedAdmin 注册表键在许多情况下可用于禁用 MFA，这可以通过 PSExec/WMI 使用单一身份验证因子来实现。

但最重要的是（也很遗憾）：

  • 这个看似完美的 ASR 规则，如果在环境中使用了 SCCM，则无法移入阻止模式。

微软提供的段落如下：

此规则与通过 Microsoft Endpoint Configuration Manager 进行的管理不兼容，因为此规则会阻止 Configuration Manager 客户端正常运行所需的 WMI 命令。

不出所料，此事件每六个月产生约 1,000,000 个事件，分布在约 1000 台设备上。使用 SCCM 时，这是一个大容量事件。通过仔细过滤，我们仍然能够从这些事件中提取价值：任何不属于 SCCM 用例的事件都会凸显出来。

此处的建议是分析不属于 SCCM 的事件，以快速识别此规则旨在阻止的异常情况。

建议：审计模式，特别是在使用 SCCM 或 Intune 时。
提示：如果您使用 SCCM 且担心事件量，可能希望不配置此 ASR 规则。如果您不使用 SCCM 或 Intune，请从审计模式开始，审查数据，并争取过渡到阻止模式。

阻止 Office 通信应用程序创建子进程

在包含 Microsoft Outlook 和 Microsoft Teams 的环境中，我们每六个月追踪到约 5000 个事件。

  • 50% 的事件来自 Outlook，
  • 40% 来自 Teams。

其余事件如预期般与通信应用相关，源自具有集成功能的进程。

事件分布与我们在“阻止 Office 宏调用 Win32 API”中经历的非常相似；主要归因于一小部分用户群使用的第三方商业集成。所有事件中有一半属于单个高级用户，记录此类事件的用户不到 50 人。

建议：对于使用 Office 集成的用户采用审计模式。
提示：这在大多数环境中会有所不同。请先调查审计事件，如果没有记录到任何事件，再过渡到阻止模式。

文件和路径例外

攻击面减少策略可以配置文件和文件夹排除项。该过程在此处描述。

您应注意以下三个重要事项：

  • 排除项适用于所有 ASR 规则，不存在每个规则单独的 ASR 文件/文件夹排除项。
  • 排除项同样适用于审计事件。
  • 阻止通过 WMI 事件订阅实现持久化不支持排除。

我们不喜欢仅审计事件也被排除的想法。我们已经限制了与 ASR 相关的文件排除项的使用。

ASR 审计模式

值得一提的是，审计模式事件能带来很大价值，虽然我们的目标是在可能的情况下对所有内容实施阻止模式，但我们并不认为停留在审计模式是一个完全负面的结果。我们的应急响应团队花时间了解了我们未阻止的每个类别的“常态”，并能够从这些事件中提取高保真信号。

以审计模式部署 ASR 可以快速且相对安全地完成。关键提示是要留意可能引发性能问题或导致日志管道量激增的过度记录。

关于审查审计日志的说明：在搜索事件日志时，跟踪和评估单个 ASR 规则的方法是通过每条规则的唯一 GUID。每条规则的 GUID 可在微软文档中找到。例如，要审查上面的“Adobe Reader”规则，我们将搜索包含“7674ba52–37eb-4a4f-a9a1-f0f9a1619a2c”的 1121 事件。
