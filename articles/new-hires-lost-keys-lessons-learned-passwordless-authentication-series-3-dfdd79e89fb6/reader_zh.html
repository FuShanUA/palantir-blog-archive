New Hires, Lost Keys & Lessons Learned (Passwordless Authentication Series, #3)

2023-01-17 · Original →

编者注：这是《无密码身份验证系列》的第三篇也是最后一篇文章，分享了我们在 Palantir 全公司范围内通过硬件身份验证器推行 FIDO2 身份验证的历程与洞察。第一篇文章重点关注《硬件选择与物流》；第二篇文章涵盖了《技术控制、推行与边缘情况》。尽管 Palantir 强制执行强多因素身份验证已有十余年之久，但采用 FIDO2 的硬件支持身份验证代表了目前可用的最强大的现代身份验证方式。

新员工入职

组织在转向无密码身份验证后，面临的首要问题之一就是如何打造顺畅的新员工入职体验。无论贵组织选择哪种身份提供商，都需要确保新用户能够快速注册，并开始使用他们的 FIDO2 设备进行身份验证，而无需依赖传统密码。你很快就会发现，这演变成了一个经典的“鸡生蛋还是蛋生鸡”的问题：新用户需要拥有已注册的 FIDO2 身份验证器，才能向其账户中添加新的 FIDO2 身份验证器。

不同的身份提供商在处理此问题时的方式差异很大。例如，Okta 的解决方式是提供一个控制台，管理员可以登录并手动将 FIDO2 身份验证器添加到特定用户的账户中。虽然乍看之下这似乎是个无缝解决方案，但你会发现它很快就会面临可扩展性方面的隐患，因为管理员必须为每一批新入职的员工逐一手动注册 FIDO2 身份验证器。对于某些组织而言，这种入职群体的规模可能高达数百人！此外，远程办公在全行业日益普及也带来了另一个问题：谁来负责将这些 FIDO2 设备邮寄给每位远程用户？如何处理入职前的物流延迟或包裹在邮寄过程中丢失的潜在风险？

在 Palantir，我们使用 Azure Active Directory (Azure AD) 作为所有用户身份验证的主要身份提供商。尽管有多种方法可以通过自动化来解决此注册问题，但我们在全公司推行 FIDO2 身份验证的指导原则是降低门槛。因此，本文主要探讨如何在 Azure 内置身份验证功能的框架内解决新员工注册问题。Azure 目前提供的这种集成解决方案称为临时访问通行码 (TAP)。

TAP 码

TAP 码作为一种限时令牌，允许 Azure 中的用户绕过诸如 FIDO2 和多因素身份验证 (MFA) 等身份验证要求。此外，可以在 Azure 门户中对其进行配置，以调整其长度、有效时长以及是单次使用还是多次使用。

TAP 码可为用户提供一种注册 FIDO2 设备的方法，无需其账户中已注册现有设备，从而解决了我们前文所述的“鸡生蛋还是蛋生鸡”的问题。通过赋予支持团队管理员在入职期间为用户生成 TAP 码的权限，新员工可以利用这些绕过令牌首次登录其账户的安全页面，随后即可在那里注册他们的 FIDO2 安全密钥。然而，TAP 码单次使用变体的一项局限在于，身份验证后生成的会话令牌仅在 10 分钟内有效。因此，一旦 TAP 码被使用，用户只有 10 分钟的时间来完成 FIDO2 密钥与账户的绑定，否则就需要获取新的 TAP 码。完善的新用户入职文档和充足的现场支持可以有效缓解这一隐患，使整个流程运行得更加顺畅快捷。

尽管 TAP 码对于在组织内部强制推行无密码身份验证策略大有裨益，但该功能也并非毫无缺点。如果未能限制 TAP 码配置的作用范围、缺乏供支持团队遵循以核实用户身份的标准操作程序 (SOP)，或者未提供向最终用户提供这些绕过令牌的安全渠道，都极易引发安全事件。必须再三强调，TAP 码提供了一种身份验证方式，可以绕过您在 Azure AD 中配置的其他安全控制（包括 MFA），因此必须极其谨慎地处理。

配置

首先，导航至 Azure 租户中的 TAP 功能。

Azure Active Directory > Security > Authentication Methods > Policies > Temporary Access Pass

接下来，你需要启用 TAP 码并将其作用范围限定在应有权限使用该功能的用户组中。在此示例中，目标设为应用于 Azure AD 租户中的所有用户。

Azure Active Directory > Security > Authentication Methods > Policies > Temporary Access Pass > Enable and Target

最后，可能也是最重要的一点，你需要决定 TAP 码的配置方式。这将极大地取决于贵组织的安全态势以及您希望借助此功能支持的工作流程。鉴于 TAP 码具备绕过安全身份验证的特性，我们建议将其限制为单次使用，并将最长生命周期（TAP 码生成后的有效期限）设置得尽可能严格。

Azure Active Directory > Security > Authentication Methods > Policies > Temporary Access Pass > Configure

丢失或损坏的身份验证器

组织在依赖硬件令牌进行身份验证时遇到的另一个常见问题是用户丢失身份验证器。有多种情况会导致用户被锁定在账户之外，包括将身份验证器遗忘在家中、处于无法随身携带身份验证器的场所，或是身份验证器完全损坏而无法继续使用。拥有一套强大的标准操作程序 (SOP) 来应对这些情况至关重要，以便尽快安全地帮助用户恢复访问。

在用户丢失 FIDO2 设备的情况下，能够选择性地将用户排除在 FIDO2 要求策略之外，将是解除用户封锁的关键所在。一种实现方式是通过回滚组：在核实用户身份后，支持团队管理员可以将用户放入一个特定组，使其退回到传统的用户名和密码加强多因素身份验证模式。我们建议引入一定程度的自动化机制，在特定时间后（例如每 7 天）自动将用户从该例外组中移除，以免用户被永久排除在 FIDO2 身份验证策略之外。你甚至可以更进一步，设置一个聊天机器人，在用户被加入该组时向其发送消息，并在其 FIDO2 例外权限即将到期时予以通知。

应对丢失或损坏的 FIDO2 身份验证器的另一种方法是使用多次使用的 TAP 码。多次使用的 TAP 码会为用户提供一个随机生成的代码，在令牌的整个生命周期内可用于绕过 FIDO2 和 MFA 控制以登录其账户。作为管理员，你可以配置代码的长度以及该绕过身份验证要求的令牌应保持有效的时间。如前所述，这些 TAP 码会绕过所有身份验证策略，因此其本身的安全性天然低于传统的“密码 + MFA”身份验证方法。

如果想要启用多次使用的 TAP 码，你需要稍微调整一下 TAP 设置，取消勾选“要求单次使用”限制。

无论你决定采用哪种策略来实现 FIDO2 绕过，都应仔细权衡其安全利弊。无论是使用 TAP 码、排除组还是其他方法，你都需要一种手段来核实申请 FIDO2 身份验证例外的用户身份。如果你的支持团队规模足够大，或许可以靠人工逐一核实每位用户的身份。然而，也有一些像 Berbix | Instant and Accurate ID Checks 这样的产品，能够将身份验证工作从管理员手中完全自动化，在组织中每次有用户需要 FIDO2 例外时，大幅减少支持团队的手动工作量。

虽然这看似会削弱 FIDO2 身份验证策略的有效性，但事实恰恰相反。用户难免会丢失身份验证密钥，如果没有办法为这些丢失安全密钥的用户解除封锁，你的 FIDO2 推行计划很可能会以失败告终。制定一套完善的行动计划来应对用户丢失身份验证密钥和被锁定在账户之外的情况，是至关重要的。

在推行无密码验证时，你应该考虑推行的另一个方面是向用户提供多个 FIDO2 安全密钥。Palantir 鼓励所有用户购置两把安全密钥，以便在其中一把损坏或丢失/被盗时作为备用。你还可以建议用户购买两种不同形态的 FIDO2 设备（例如一把 USB-C 接口和一把 USB-A 接口），以缓解设备端口与 FIDO2 安全密钥之间的任何兼容性问题。

在各办公地点储备一批 FIDO2 安全密钥也是个好主意，这样用户可以更方便地获取新密钥，而不必自行订购并等待物流派送。

确保向用户提供非常明确的指引，说明如何获取新的 FIDO2 安全密钥以及密钥丢失或被盗后的应对措施。当用户不在办公室且无法在等待新设备邮寄送达期间快速获得批准的 FIDO2 安全密钥时，FIDO2 绕过请求应作为最后的兜底手段。

用户教育

用户教育是无密码身份验证策略中最容易被忽视的方面之一，但在决定 FIDO2 推行的整体成败时，它可能也是最重要的一环。缺乏全面且准确的指引，用户面对这项新技术会手足无措，在强制推行无密码验证期间你很可能会遭遇重大阻碍，这种摩擦甚至可能让你的推行计划半路夭折！

在编写用户指南时，务必涵盖在组织推行中所选 FIDO2 密钥的实际操作基础。就 YubiKey 而言，这应包括图示以及（最好）简短的视频片段，展示如何将密钥插入设备以及在不同形态的密钥上触摸金属触点的位置。虽然听起来很简单，但从一开始就考量这项技术可能带来的摩擦是至关重要的。与密码不同，这是一种全新的身份验证形式，用户很可能难以理解。因此，你的用户可能会对“带有 PIN 码的 FIDO2 设备为何比密码加 MFA 更安全”感到困惑，所以最好提供一些高层次文档，介绍这项技术在安全性上的提升。向用户普及有关 FIDO2 及其工作原理的基础知识还有一个额外好处，即能够提升用户的认同感，有助于在全组织内部的推广应用。

最后，我们建议针对不同技术水平的用户，提供在范围和复杂度上有所差异的文档。例如，一套文档可以围绕基础最终用户展开，提供一套基础操作说明，涵盖如何订购密钥、如何使用密钥进行身份验证以及密钥丢失后的替换流程。对于管理员和技术水平较高的用户，可以辅以高级文档，涵盖 FIDO2 技术的工作原理以及可使用的其他功能，例如面向开发者的 OpenPGP 代码签名。为了更好地支持那些在使用 FIDO2 时可能遇到较大困难的用户，我们建议设立答疑时间或现场交流会，让用户可以提问并获得清晰的解答。

经验教训

Palantir 走向无密码身份验证的道路并非一帆风顺。它要求我们的用户转变对身份验证机制的认知；而且由于该技术相对较新，我们不得不应对在全面推行无密码验证过程中涌现的似乎永无休止的边缘情况。尽管我们从中汲取了许多经验教训，但以下列出的是我们在这一过程中遇到的一些最大的漏洞、陷阱和痛点。

OATH 令牌。 如果你的组织有员工在敏感信息隔离设施 (SCIF) 内工作，那么你必须提供一种允许用户绕过 FIDO2 身份验证的工作流程，因为大多数 SCIF 不允许用户将 FIDO2 硬件设备带入，更不可能允许用户将安全密钥插入现场的计算机中。因此，你可能会尝试通过 FIDO2 例外工作流程结合传统密码 + OATH 令牌进行 MFA 的方式来弥补这一身份验证缺口。取决于你所使用的身份提供商，支持 OATH 令牌工作流程的体验会有所不同。例如，Azure 在预配 OATH 令牌方面的支持功能并不完善，而且截至撰稿时，每个 OATH 令牌都必须手动加载到 Azure 中并与最终用户关联。Microsoft 正在开发一些新的 API，以扩展 OATH 令牌注册到用户账户的功能，但在这些功能上线之前，你必须手动将便携式对称密钥容器 (PSKC) 文件转换为 .csv 格式并上传到 Azure，这个过程极其繁琐。此外，除了主要零部件供应商之外，你很难找到声誉卓著的 OATH 令牌品牌，而且你会发现它们极难采购。我们最好的建议是，除非绝对必要，否则不要支持 OATH 令牌工作流程。

物流配送。 在全球疫情期间，要将 YubiKey 交到遍布世界各地的每一名员工手中，充满了巨大的挑战。密钥在寄往其他国家时被边检人员没收，或者包裹在邮寄途中彻底丢失的情况屡见不鲜。遗憾的是，在向用户寄送密钥的过程中，有很多因素是你无法控制的。我们最好的建议是在办公定点提供密钥，并在强制推行之前给予用户充裕的时间来获取他们的 FIDO2 安全密钥。

边缘情况。 在 YubiKey 推行过程中，我们遇到了一些极具挑战性的边缘情况，这在本系列的第二篇博文中已作讨论。遗憾的是，在将推行范围扩大到足够多的用户以使这些边缘情况浮现之前，没有简便的方法能检测出组织中所有不支持 FIDO2 身份验证的应用程序。我们建议将 FIDO2 推行组保持在合理的规模，并在每批组强制推行之间预留缓冲时间，以便及时识别问题。

文档。 除了用于身份验证的 FIDO2 模块外，Palantir 还使用 YubiKey 的其他模块，例如用于代码签名的 OpenPGP 模块。这给用户带来了不少摩擦，因为他们不清楚各模块之间的区别或细微差异。你可以通过为每个模块提供极其详尽的文档和调试说明，来减少最终用户遇到的此类摩擦。

可用性。 在决定向用户提供哪种形态的 FIDO2 设备时，建议考虑避免采用非 nano 形态的设备。在 Palantir 的推行过程中，我们发现非 nano 形态的设备更容易发生损坏（例如，用户在插入设备使用时不慎将其折断），相比之下，较小形态的设备受损概率更低。此外，向仅有一个 USB-C 端口的笔记本电脑用户提供 USB-C 设备，也会导致密钥丢失，因为用户需要拔下 FIDO2 设备以插入其他外设。最后，如果用户不要求立即使用 FIDO2 密钥，他们更有可能忘记自己设置的唯一 PIN 码。这在 YubiKey 强制推行时引发了大量痛点，用户需要重置 PIN 码，而这会抹除所有 FIDO2 密钥凭证信息，并要求对设备进行重新注册。

结语

FIDO2 仍是一项相对较新的技术，正因如此，在推行过程中遇到的许多挑战，都因不同操作系统和重型客户端对无密码身份验证缺乏支持而被放大。幸运的是，Palantir 的大多数用户在从传统密码向强 FIDO2 支持身份验证过渡的整个过程中，都表现出了极大的支持和耐心。

虽然转向无密码身份验证极大地降低了 Palantir 遭遇账户接管 (ATO) 的风险，但必须铭记于心的是，就像安全领域的所有事物一样，不存在一劳永逸的灵丹妙药。必须将强终端管理、遥测、日志记录等额外的安全控制与 FIDO2 身份验证相结合，才能在组织中构建起连贯且强大的安全态势。

我们希望本系列博客不仅能强调摆脱传统基于密码的身份验证的重要性，同时也能揭示 FIDO2 存在的一些不完美之处。尽管无密码验证的推行需要耗费大量的公司资源和时间，但毫无疑问的是，如果执行得当，FIDO2 能够提供单一最有效的控制手段，以降低当今大多数组织所面临的 ATO 风险。
