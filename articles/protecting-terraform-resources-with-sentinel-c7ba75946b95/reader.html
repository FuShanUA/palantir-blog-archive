Protecting Terraform resources with Sentinel

6 min read

I want to tell you about something cool we’ve recently rolled out in Palantir’s cloud infrastructure: Sentinel policies in Terraform! This blog post demonstrates a simple Sentinel implementation for avoiding accidental deletion of Terraform resources.

We use Terraform to create and maintain most of the cloud resources and infrastructure that our customers and internal development use. Terraform makes it really easy to create and track the many cloud resources we manage, but unfortunately it also makes it really easy to make a mistake (or encounter a Terraform bug) and destroy critical infrastructure.

With the normal workflow of how we make changes, the change author carefully reviews GitHub PRs and the preliminary Terraform plans that tell us what this PR will do if it’s merged. Then, we get a required +1 from someone who also carefully reviews the PR and plans. Once the PR merges, the Terraform backend starts up and actually applies the change. Historically, we haven’t had issues with unexpected destruction when reviewed PRs are merged. But we know of edge cases where changing backend settings of a workspace can trigger a potentially unexpected plan/apply. To avoid any incidents that could be caused by this behavior, some kind of automated sanity checking would be needed.

Enter Sentinel: a framework that Terraform can use to make sure Terraform plans do things we expect them to do. With Sentinel, you can write policies that analyze the state of the workspace and the changes that would be made. Based on that analysis, the policy can choose to pass or fail the change. In our case, we want a backstop that prevents unlikely or assuredly unwanted deletions. The first iteration of the policy that we’ve rolled out does two things:

  • Ensures that certain kinds of crucial resources are never deleted (such as AWS KMS keys or public IPs)
  • Ensures that no more than 10% of all resources are deleted at one time

By the way, Sentinel can currently only be used if you’re running your Terraform from Terraform Cloud.

A view into the process

When a Terraform workspace is configured to use the policy, the policy runs after each plan but before each apply, which gives it the authority to stop the apply:

You can see that after the plan finished, the policy ran. Then it failed, which blocks the apply step that would have happened afterwards. In the case where we would want to actually delete a sensitive resource or a whole pile of resources, we can override the policy, but only with the help of an administrator, who would see the following:

Once the override happens, the apply would continue as normal. Or the apply could be thrown out, in the case of an accidental trigger or reviewing mistake. In this way, unexpected/mistaken applies can never blow up your environment.

Let’s dive into the policy code a bit to see what’s going on.

The heart of the policy

Sentinel policies revolve around a Boolean statement that, when evaluated, tells Sentinel whether the policy passed or failed:

main = rule {
    validate_destroyed_resources(blacklist) and maximum_destroyed_resources(10)
}

The main rule determines if the policy passes or fails. In this case, the rule requires both the validate_destroyed_resources() and the maximum_destroyed_resources() functions to return true in order to allow an apply.

Easy enough so far. Let’s look at the validate_destroyed_resources() function to see how that works:

# Validate that resources of blacklisted types are not being destroyed,
# even if it were to be recreated
validate_destroyed_resources = func(blacklist) {
    valid = true    for blacklist as type {
        found_resources = find_resources_from_plan(type) else {}
        for found_resources as address, r {
            # note that even if something is being destroyed and recreated, it's marked invalid
            if r.destroy {
                print("You are trying to destroy a resource", address, "of blacklisted type", type)
                valid = false
            }
        }
    }
    return valid
}

blacklist is a list of resource names, such as aws_iam_user. All we’re doing in this function is scanning the resources in the plan for any of the blacklisted types. If we find one, we check to see if it’s being destroyed. If so, mark that we will return false and also notify the user that they’re trying to destroy a blacklisted resource. find_resources_from_plan() is a helper function that returns a map of resources with the given type:

# Find all resources of a specific type from all modules using the tfplan import
find_resources_from_plan = func(type) {
    resources = {}    # Iterate over all modules in the tfplan import
    for tfplan.module_paths as path {
        # Iterate over the named resources of desired type in the module
        for tfplan.module(path).resources[type] else {} as name, instances {
            # Iterate over resource instances
            for instances as index, r {                # Get the address of the instance
                if length(path) == 0 {
                    # root module
                    address = type + "." + name + "[" + string(index) + "]"
                } else {
                    # non-root module
                    address = "module." + strings.join(path, ".module.") + "." + type + "." + name + "[" + string(index) + "]"
                }                # Add the instance to resources map, setting the key to the address
                resources[address] = r
            }
        }
    }    return resources
}

tfplan is an object that you get from adding import "tfplan", and that object lets you iterate over the resources that show up in your plan. Finally, we need to actually define blacklist and pass it to validate_destroyed_resources() . Here’s what that looks like:

blacklist = [
    "aws_vpc",
    "aws_s3_bucket",
    "aws_eip",
    "aws_kms_key",
    "azurerm_resource_group",
    "azurerm_public_ip",
    "azurerm_storage_account",
    "azurerm_storage_container",
]

So now we’ve dug into one half of our policy rules. Now let’s see how maximum_destroyed_resources() works:

# Validate that a threshold of destroyed resources hasn't been passed
# percent: number between 1 and 100
maximum_destroyed_resources = func(percent) {
    destroyed = 0    for tfplan.module_paths as path {
        for tfplan.module(path).resources else {} as type, names {
            for names as name, instances {
                for instances as index, r {
                    if r.destroy {
                        destroyed += 1
                    }
                }
            }
        }
    }    num_resources = count_resources()
    threshold = int((float(percent) / 100.0) * float(num_resources))
    print("total number of resources:", num_resources)
    print("number of destroyed resources:", destroyed)
    print("threshold of failure (in destroyed resources):", threshold)    return destroyed < threshold
}

Here, we’re counting up all of the resources marked to be destroyed, then calculate what $percent of the total resources looks like. Counting up all of the resources (using count_resources()) is easy:

count_resources = func() {
    num_resources = 0    for tfstate.module_paths as path {
        for tfstate.module(path).resources else {} as type, names {
            for names as name, instances {
                for instances as index, r {
                    num_resources += 1
                }
            }
        }
    }    return num_resources
}

There is one thing to keep in mind, though. Here, we need to count the number of resources from the tfstate object, attained by import "tfstate".

Conclusion

As you can see, Sentinel is a simple language that can be used to protect your Terraform-managed resources from accidental deletion. And there’s much more you can do with it as well… imagine preventing certain types of resources from being created, or perhaps guaranteeing that certain resources must have tags with a certain name on them. Regardless of what you ultimately decide to do with Sentinel, it’s probably a pretty good idea to apply policies similar to what I’ve shown above. Protect your resources and avoid painful outages!
