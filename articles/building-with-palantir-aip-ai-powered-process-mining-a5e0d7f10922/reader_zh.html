Building with Palantir AIP: AI-Powered Process Mining

2023-12-19 · Original →

作者：Ruben Stroh，Palantir 部署战略师

引言

自 AIP 发布以来，Palantir 已与各行各业的客户合作，部署了以 AI 为中心的流程挖掘与自动化工作流，大幅节省了时间、资金和资源。今天，我们将通过一个概念性的分步示例，详细讲解如何利用 AIP 构建此类解决方案。

但在深入探讨之前，不妨退一步思考：流程挖掘为何如此重要？正如 Palantir 在实践中反复印证的那样，企业的运转依赖于无数流程。流程运转顺畅时，企业能让客户满意，实现降本增效，并达成既定目标。一旦流程出问题，后果可能极为严重——客户不满、收入损失、资源浪费等不一而足。

那么，我们该如何确保流程始终处于最优运转状态？在实现这一目标后，又该如何推行可扩展且可持续的变革？

从根本上说，这涉及以下两点：

  • 认清流程运转的真实业务现状。尽管许多企业自认已全面掌握自身流程的运作方式，但现实中的日常运转往往与流程文档、预想流程及异常管理的演示文稿大相径庭。鲜有公司能以细粒度且近乎实时的方式洞察流程的运转情况。要优化流程，首先必须具备这种可见性，从而理清特定流程如何与企业更广泛的业务现实相互作用。
  • 更快速地做出更优决策。基于掌握的客户全量信息，我们是否该履行该订单？当某款产品短缺时，能否安全地用另一款产品替代？这些选择又会对下游产生何种影响？在许多企业中，这类决策过程迟缓、耗费人力，且陷入运营与信息孤岛。然而，借助以 AI 为中心的逻辑，企业能以更明智的方式更快地做出决策。

本文将逐步讲解如何通过 AIP 实现上述两大目标。

我们将以虚拟公司（Titan Industries）订单到回款流程中的信用冻结为例，演示 AIP 如何助您迅速理解并分析特定流程，将这一认知融入企业以决策为中心的模型（Palantir 本体）中，并部署 AI 驱动的解决方案，通过强化和加速决策来消除瓶颈——此处即为信用冻结。

作为背景，订单到回款流程中设置信用冻结的目的，是在客户极有可能不付款时阻止订单履行。这类冻结可能引发问题，尤其是当它们延误了历来按时付款客户的订单时更是如此，因为识别并解除冻结往往耗时漫长。

让我们开始吧。

《使用 HyperAuto 同步流程数据》

首先，我们需要从订单到回款流程所使用的系统中引入相关源数据，并将其转化为支持流程挖掘的格式。

Palantir 的 HyperAuto——又称软件定义数据集成（SDDI）——不仅能连接常见的 ERP 和 CRM 系统（包括 SAP、Salesforce、Oracle NetSuite 和 Hubspot），还能通过编程方式生成数据管道，快速完成数据集的清洗、规范化和统一化，将其整合为连贯的数据资产。

Titan Industries 的订单到回款流程在 SAP 中运行。鉴于我们已构建好 SAP 连接，可以直接开箱即用地部署 HyperAuto 管道。在配置管道时，我们可以利用诸多省时特性——例如，自动生成连接以及主外键，并将列名和表名翻译为人类可读的格式。

既然已经完成订单到回款数据的清洗、同步，并使其转化为人类可读的格式，接下来需将其转化为适用于流程挖掘的正确格式。

《使用 Pipeline Builder 转换数据》

为了准备用于流程挖掘的 SAP 订单到回款数据，我们需要创建两个数据集：

  • 流程对象数据集：用于描述在特定流程中流转的对象类型。在汽车制造流程中，这指的是车辆；在招聘流程中，则是经历各个面试阶段的候选人。而在本例中，它指的是在订单到回款流程中流转的销售订单行项目。
  • 日志对象数据集：用于描述特定流程对象所经历的步骤。要实现流程挖掘，仅需三列数据：流程对象的主键（本例中为销售订单行项目的引用标识）、当前状态（即流程中的阶段），以及该流程对象进入此阶段的时间戳。

我们可以使用 Pipeline Builder 将同步后的 SAP 数据转化为这两种模式。Pipeline Builder 是 Palantir 的低代码/无代码工具，专为开发生产级数据管道而设计。在 Pipeline Builder 中，我们能快速应用转换操作，从包含数百种函数的库中进行选择以执行几乎任何类型的转换，从而迅速构建起可扩展的管道。

在此，我们能够轻松快捷地将同步的 Titan 订单到回款 SAP 数据套用至我们的模式中，并创建一个整合这些不同数据点的统一日志数据集。至此，我们便构建好了流程日志的底层数据集。

接下来，便可以将其连接至 Titan Industries 的本体了。

《开发流程挖掘本体》

本体是企业的决策中心模型，包含驱动日常决策的数据、逻辑和动作。它是 Titan Industries 数据驱动工作流的基础，承载了公司运营的业务上下文。

为了确保我们的操作融入更广泛的企业级反馈闭环，并充分利用 Titan 本体中已有的交织关系——例如，一张销售订单如何影响特定的生产线——我们必须使订单到回款工作流成为 Titan 本体的一部分。为此，我们将创建一个订单到回款本体，并将其链接到 Titan Industries 现有的本体上。

我们首先在 Pipeline Builder 的数据集预览界面中为订单到回款本体创建本体对象。然后，可以在 Ontology Manager 中定义这些对象（例如，分配名称、配置访问管理、校准属性元数据），并将它们链接到 Titan 本体中的其他对象。这涉及定义对象间的关系；例如，一张销售订单始终与特定客户相关联。

将订单到回款数据链接到更广泛的 Titan 本体，是确保 AI 工具成功落地的关键步骤：这个以决策为中心的模型越能真实反映 Titan 的日常运营，我们就能越有效地发挥 AI 工具的作用。

《使用 Machinery 挖掘流程》

既然已经构建好了这部分 Titan 本体，我们就可以利用 Machinery 中的点选式界面，创建订单到回款流程的本体驱动表示。这条 Machinery 路径将为我们提供易于导航的视图，展示预期行为及相关偏差，并允许我们在单个步骤层级和整体趋势层级进行深入调查。随后，我们便能在整个 Palantir 平台中利用 AI 驱动的工具，基于该模型采取行动以优化此流程。

首先，我们引入从订单到回款流程数据中创建的本体对象，其中包含销售订单行项目的属性数据及其相关日志（这些行项目流转的状态及相关时间戳）。

接着，Machinery 会解读这些数据，创建出 Titan Industries 当前订单到回款流程的交互式表示。在此基础上，我们可以在不同粒度层级调查流程中触发信用冻结的步骤，从而为接下来要构建的以 AI 为中心的解决工作流奠定基础。

《使用 Workshop 构建流程挖掘应用》

既然已经用 Machinery 构建了流程模型，我们希望业务用户和分析师能与它进行交互。为此，我们将把它嵌入到 Workshop 应用程序中。Workshop 赋予应用构建者为业务运营用户创建交互式且高质量应用的能力；它是一种简单的拖放式前端构建方式。

我们正在构建的应用本质上是一组 Workshop 小部件的集合——例如，流程模型只是一个可嵌入最终应用的小部件。

借助 Workshop，我们能迅速构建出一个端到端的流程挖掘应用，为业务运营用户提供操作视图，其中包含了他们针对信用冻结做出决策所需的全部相关信息。

现在，我们要更进一步，构建一个 AI 副驾驶，让决策变得更加高效。

《使用 AIP Logic 搭建决策副驾驶》

至此，我们已经用 Machinery 挖掘了订单到回款流程，并构建了一个基础 Workshop 应用，帮助业务用户管理信用冻结。接下来，我们希望将这个手动操作的应用转化为以 AI 为中心的模式，实现 AI 编排的自动化决策——同时保持人在回路中。

为此，我们将使用 AIP Logic。AIP Logic 是一个无代码开发环境，用于构建、测试和发布由大语言模型（LLM）驱动的函数；它使构建者和架构师能够将 LLM 形式的生成式 AI 嵌入到几乎任何工作流中，且全程利用本体。

在此场景下，我们希望构建一个由 LLM 驱动的函数，用于针对是否维持或解除信用冻结提出决策建议，并展示其建议背后的推理过程。我们可以像指导一名实习生执行任务那样来处理这件事——对方很聪明，但缺乏上下文和背景信息。

我们可以利用 AIP Logic 直观的界面，从 Titan 的本体中选择一个对象供 LLM 处理——此处即销售订单行项目。随后，我们提取该对象的特定属性作为参数，以便在指导 LLM 执行任务时进行引用（例如，信用状态、订单 ID、客户信用额度、12 个月历史订单量）。

接下来，我们可以用纯文本指示 LLM 如何执行任务（无需编写代码）。

我们先提供总体指导原则。接着，需要定义各个属性以向 LLM 解释清楚，例如，告知它“12 个月按时付款率”是一个介于 0 和 1 之间的数值，表示特定客户按时付款的频率（越接近 1 越好）。

然后，我们为 LLM 提供一组使用这些参数进行检查的规则——本质上就是一套用于给出建议的评判标准。例如，我们告诉 LLM，如果相关订单金额明显高于信用额度和历史订单金额，且客户的按时付款率较低，那么它大概率应维持已触发的信用冻结。

一旦 LLM 得出建议，我们希望它能提供做出特定结论的推理过程，我们将其存储在变量中以便日后参考——我们要确保这一决策及其背后的推理都被记录在 Titan 的本体中。

我们可以使用调试器测试该函数，确保其按预期工作。调试器会向我们展示 LLM 是如何理解所有不同指令的，我们可以验证它是否在按照我们期望的方式执行这些指令。

现在，我们希望将其整合到业务用户可部署的应用中，这同样可以在 Workshop 中完成。

现在，我们得到了一款应用，它能针对特定的信用冻结提供解决建议——维持或解除信用冻结——并附上建议背后的推理依据。业务运营用户现在可以在几分钟内做出知情、数据驱动的决策。

《使用 Automate 编排流程决策》

既然已经构建好这个端到端的、AI 驱动的流程挖掘工作流，还有办法进一步提升效率——且不以牺牲安全性为代价。我们可以配置基于触发的自动化操作，随着新数据流入平台，这些操作将调用由 LLM 支撑的功能；如果我们倾向于定期维持或解除信用冻结，也可以将这些自动化操作设定为按计划运行（例如每隔几分钟）。此外，随着我们进一步演进核心工作流，我们还能利用 AIP 全套的健康状况框架和细粒度安全能力，实现稳健的协作。
