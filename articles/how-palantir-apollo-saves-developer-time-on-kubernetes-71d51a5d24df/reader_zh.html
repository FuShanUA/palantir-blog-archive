How Palantir Apollo Saves Developer Time on Kubernetes

2022-09-23 · Original →
*编者按：本文是关于 Palantir Apollo 系列博文的第三篇，前两篇分别为《为何传统的持续部署方法在当今行不通》和《Palantir Apollo 编排：面向现代架构的基于约束的持续部署》。* ![图片](https://miro.medium.com/v2/resize:fit:700/1*UqWbjpGQc6RBFXQMliiOnA.png) 过去十年间，基础设施平台不断发展，以满足将容器作为软件部署基本单元的日益增长的需求。其中最受欢迎的平台是 Kubernetes (K8s)，它支持管理容器化工作负载的大量原语。但同时，它也*极其复杂*。哪怕只是运行一个简单的应用，也需要了解并协调多个 K8s 资源。强迫开发者去应对这些概念，不仅会消耗他们的生产力，还会削弱组织标准化基础设施使用的能力。 在 Palantir，我们意识到让开发者能够在不应对所有这些复杂性的情况下部署到 K8s 上是至关重要的。我们通过定义《Apollo 产品规范》来应对这一挑战，确保遵循该规范能够保证应用程序的可部署性。两个核心的 Apollo 组件巩固了该规范的价值。《Apollo SDK》帮助开发者利用 Apollo 产品规范的声明式 API，而《Apollo 服务管理面》则将这些应用程序转换为一组正常运行的 Kubernetes 原生资源。 这种方法为我们的开发者带来了更简单、可重复的部署体验。同样重要的是，它使我们的部署平台能够随着时间的推移不断演进，既能处理 K8s API 的弃用或移除，也能加速新功能的采纳。在这篇文章中，我们将进一步详细说明组织在 Kubernetes 上部署时面临的问题、Apollo SDK 和服务管理面的作用，以及这为 Palantir 带来的价值示例。 ### 问题陈述 将应用程序部署到 Kubernetes……很难。虽然该平台提供了坚实的原语、出色的扩展点和详尽的文档，但部署哪怕是一个简单的应用程序，都需要面对一系列令人生畏的问题： 我该如何使用 ConfigMap 为我的应用程序提供配置？如果我的应用程序需要 IAM 凭证怎么办？什么是 Pod 控制器，我该使用哪一种？如果我的应用程序需要存储数据，我该如何设置存储？什么是 Pod 中断预算、节点选择器、Pod（反）亲和性和拓扑分布约束，我应该使用它们吗？Kubernetes 的 Pod 网络到底是如何工作的？ 问题清单还在继续……面对所有这些复杂性，开发者常常把时间浪费在陡峭的学习曲线上，或者干脆从其他应用程序的代码仓库中复制一个可用的示例。久而久之，这会导致面向服务架构中的每个部分都有自己定制的部署方案，并伴随一些不良后果： - K8s 资源的使用是*特定于应用程序的*——开发者在孤立中犯错并吸取教训，缺乏部署标准使得调试生产问题变得更加复杂。 - 涉及弃用 API 移除的 K8s 升级，需要组织全员努力去审查*所有*应用程序的部署方法。 - 新的 K8s 功能采纳是一项缓慢的、*逐个团队*推进的工作。 K8s 缺乏高阶资源类型，使得开发者在部署时难以采用简单、一致的方式。在 Palantir，我们意识到需要封装这种复杂性，并向开发者提供一种简单、声明式的方式来定义和部署他们的应用程序。为此，我们定义了《Apollo 产品规范》——应用程序可以满足这一标准，作为回报，不仅能获得可部署性的保证，还能获得一种简单的方式来利用 K8s 的强大功能。我们还构建了《Apollo SDK》，这是一套工具集，旨在促进开发符合该规范的应用程序。最后，我们构建了 Apollo 服务管理面，负责完成将符合规范的应用程序转化为一组正常运行的 Kubernetes 资源的重任。 这些 Apollo 组件既保护了开发者免受部署复杂性的困扰，又使他们能够通过简化的原语利用 Kubernetes 的强大功能。 ### Apollo 产品规范与 SDK 在构建 Apollo 产品规范和 SDK 时，我们有两个基本目标： - 使开发者能够用简单的术语定义应用程序的基本特征（有状态性、对外部网络流量的开放性、IAM 凭证等）。 - 在开发阶段提供可部署性保证，使开发者无需处理软件部署基础设施设置的繁琐细节。 为此，产品规范包含了一系列配置点，供开发者声明应用程序的基本需求或特征。这些配置点简洁且声明式——例如，申请持久化存储只需提供一个名称和所需大小（`my-app-data: 10Gi`）。最关键的是，开发者使用产品规范只声明他们需要*什么*；而决定*如何*满足这些声明的需求，则交由 Apollo 服务管理面处理。 ### WitchcraftServices 与 Apollo 服务管理面 符合 Apollo 产品规范的应用程序直接映射为 WitchcraftServices，这是 Apollo 服务管理面主要的 Kubernetes 自定义资源。服务管理面负责接收一个 WitchcraftService，并生成一组构成正常运行的应用程序的 K8s 资源，并完整实现 WitchcraftService 声明的所有特征。决定在满足 WitchcraftService 声明的需求时使用哪些资源类型，以及协调所有这些资源，均由管理面内的一组控制循环以编程方式处理。这一逻辑包括如何将持久化存储或入口等所需属性赋予应用程序，并跨越多个 K8s 资源。 ![图片](https://miro.medium.com/v2/resize:fit:700/1*0xMmKrhPbgWwhCnZRwWzoQ.png) 上面的图表清楚展示了 Apollo 部署平台封装了多少 K8s 的复杂性。这不仅涉及选择要创建的正确资源，还处理了资源间引用的全面对齐，包括标签选择器（如 Pod 和节点选择器），以及显式对象名称引用（如 ConfigMap 卷挂载）。 至关重要的是，服务管理面将 K8s 资源使用和协调的决策逻辑集中在一个地方进行编码。在应用程序之间应用此标准，使得部署变得可重复，且*无需* *不是* K8s 专家的应用程序开发者*付出努力*。 这种设置可以保护开发者免受部署的繁琐细节困扰，使他们能够将更多时间花在实际构建上。需要与 K8s 直接交互的应用程序仍然可以这样做，同时不会丧失标准化部署方案带来的好处。 ### Apollo 产品规范的影响：示例 接下来，我们看几个示例，展示服务管理面为开发者提供的具体好处。服务管理面作为编码 Kubernetes 用法的单一位置，从而无需处理以下问题： - 在 K8s 中导航复杂的 Pod 调度配置；以及 - 如何处理弃用 API 的移除。 ## 抽象化 Pod 调度 在 Kubernetes 中正确配置应用程序的 Pod 调度至关重要——处于待定或被抢占状态的 Pod 可能意味着可用性降低甚至全面中断。这同样很*困难*。K8s 暴露了大量的 Pod 调度和抢占控制点，包括但不限于： - 节点选择器、亲和性、污点和容忍度； - Pod 亲和性和反亲和性；以及 - Pod 优先级、拓扑分布约束，以及资源请求和限制。 这些概念都需要 Kubernetes 知识才能正确使用。此外，其中一些控制点还需要依赖特定集群的知识才能正确设置： - 集群的节点具有哪些标签、拓扑键和污点？ - 集群修补和重启或替换节点的频率如何？节点故障有多常见？ - 集群中的哪一组节点能容纳我资源密集型的应用程序？ 开发者可能最终为了运行一个 Pod 而去费力处理集群节点配置的细节。实现期望的调度行为可能也有多种途径——在不同应用程序中采用不同方法只会让调试调度问题变得更加困难。当开发者被迫定义这些调度配置时，无论是在单个集群还是跨不同集群中实现可持续部署，都不太现实。 Apollo 产品规范的抽象级别，恰好使我们的服务管理面能够代表开发者拥有调度配置的管理权。服务管理面会推断 WitchcraftServices 的调度意图，并创建合适的调度配置，包括： - 节点亲和性，使得应用程序 Pod 在集群内的专用节点组上运行； - Pod 反亲和性，以减轻非自愿节点中断的影响； - Pod 拓扑分布约束，以缓解可用区故障的影响；以及 - Pod 中断预算，以防止驱逐影响服务可用性所需的法定人数。 您可以在我们新发布的《Apollo 服务管理面 Pod 控制器文档》中查看关于服务管理面对 Pod 调度支持的完整描述。 一旦掌握了这些解决方案，其实并不复杂——真正的成本在于跨越学习曲线去寻找这些解决方案的过程。在服务管理面中*一次性*编码此逻辑，可以让各位开发者跳过这种时间黑洞。 封装这些 Pod 调度配置，也极大提升了我们将软件部署到新环境（例如客户配置的 K8s 集群）的能力。例如，我们不必对构成 Foundry 平台的所有应用程序进行痛苦审查，而是能够在服务管理面本身内部进行严格限定的更改，以根据需要支持不同的调度配置。 服务管理面适应新底层基础设施设置的能力，还延伸到保护开发者免受底层基础设施 API 变更的影响。下面，我们将看看这如何简化对弃用的 K8s API 移除的处理。 ### 代表开发者处理 API 弃用 处理 K8s API 移除是一件痛苦的事。如果没有进行彻底的审查，升级到移除了弃用 API 的 K8s 版本可能会导致服务中断，而且追查违规者需要耗费时间和精力。处理集群升级应该尽可能地限定在极少数工程师的范围内。 Apollo 产品规范提供的 API 边界简化了大量的此类 API 移除问题，因为服务管理面处理了部署服务所需的所有 K8s API 用法。例如，在部署 WitchcraftService 时，服务管理面会代表应用程序创建一个 PodDisruptionBudget。K8s 1.25 版本将停止提供 v1beta1 版本的 PodDisruptionBudgets，转而支持稳定的 v1 API。处理此次 API 移除只需在服务管理面的其中一个控制器中提交一个 PR 即可。如果没有 Apollo 产品规范，PDB 可能会在构成 Palantir Foundry 和 Palantir Gotham 平台的数百个应用程序中分别定义，这将需要进行一次详尽的审查，才能放心地升级我们的 K8s 环境。 ### 回顾 Kubernetes 是一个复杂的平台，学习曲线陡峭，常常让新手感到不知所措。让开发者去弄清楚如何直接将应用程序可重复地部署到 K8s 上，是一个巨大的时间黑洞。我们创建了《Apollo 产品规范》，使开发者能够避开这个陷阱，同时利用 K8s 的强大功能。我们的《Apollo SDK》促进了构建符合规范的应用程序（即 WitchcraftServices），而我们的《服务管理面》则以编程方式处理了将这些 WitchcraftServices 转化为正确 K8s 资源的重任。这种设置不仅保护了开发者的时间，也为我们的服务管理面提供了空间，使其能够透明地增加对不同底层基础设施的支持，将我们的软件带入更具挑战性的环境中。 *了解更多关于 Apollo 的信息：* https://www.palantir.com/platforms/apollo *与 Apollo 团队一起工作：* Palantir Technologies (lever.co)
