Using Envoy for Egress Traffic

2021-08-17 · Original →

在这篇博文中，Palantir 的网络基础设施团队将分享我们近期迁移到 Envoy 的经验，此次迁移旨在为 Palantir 的 Kubernetes 基础设施 Rubix 中的正向代理实现细粒度的出站流量过滤。Envoy 是一款开源、高性能的边缘和服务代理，内置了 L4/L7 过滤、服务发现、动态配置等功能。在本文中，我们将：

  • 解释从 Squid 切换到 Envoy 的动机
  • 描述 Rubix 内部新的 Envoy 正向代理配置
  • 介绍我们如何监控 Envoy 指标

我们基于 Squid 的旧版正向代理

我们最初在 Rubix 中使用 Squid 作为正向代理。这套正向代理的配置基础但可靠：在 Rubix 中搭建 Kubernetes 集群时，Squid 代理的配置是静态定义的，并作为 systemd 单元启动。我们配置了 Rubix 的出站流量，使得所有出站流量都会被定向到路由器实例，并通过 iptables 重定向到 Squid，以进行全局级别的流量过滤。

随着时间的推移，这套方案已无法适应我们不断演进的部署策略。从根本上讲，我们希望将出入站流量过滤配置管理的控制权从 Rubix 转移到 Foundry 平台，以减轻基础设施工程师的手工负担。将这种控制权交到客户手中，也将极大赋能客户的 IT 组织，使其能够自行控制 Foundry 中的网络流量，而无需我们团队提供支持。

为了支持出入站流量过滤配置管理控制权的转移，为我们的 Foundry 平台增加一流的网络支持至关重要。这包括引入安全、细粒度的流量规则，以便我们和客户可以根据需要动态创建。由于 Squid 采用静态配置，且更新规则需要重启整个集群，我们开始研究新的正向代理方案。

使用 Envoy 控制出站流量

Envoy 作为一款现代化、高性能、占用空间小的边缘和服务代理，是我们的自然之选。在将其部署用于 Rubix 中的服务网格和入站流量之后，我们对 Envoy 的优势抱有高度信心。以下两项特性对我们尤为有益：

  • Envoy 可以发现多种动态资源，包括监听器、集群和路由配置。这些资源通过订阅管理服务器来请求，管理服务器通过 gRPC 提供发现服务及相应的 API 更新。这种方法使我们能够在 Foundry 端发生流量规则变更时，无需重启集群即可自动更新代理配置。
  • Envoy 的另一大优势是，它能够通过分布式追踪、日志记录和时间序列指标，生成同类中极为丰富的可观测性数据。这种能力使我们的工程师能够保持清晰的网络透明度，并快速定位问题的根本原因。

这两项极具潜力的特性，加上丰富的配置选项，促使我们急切地希望用 Envoy 替换 Squid。

Rubix 内部的 Envoy 代理设置

在 Rubix 中，我们利用 Envoy 的 go-control-plane 库实现了 xDS 控制面，以提供发现服务 API 服务器和代理配置缓存。xDS 控制面负责监测出站流量过滤规则的变更，进行相应的代理配置更新，并将其推送给已订阅的代理。

离开环境的流量会根据路由表条目，被路由到各可用区的出站主机。当流量到达这些主机时，会通过 iptables 重定向到 Envoy 代理。监听器接受此下游 TCP 连接，并创建一个新的监听器过滤链，其中包含一组预处理过滤器和匹配条件。

处理后，Envoy 使用新提取的数据包信息，将其与定义了全局级别、单 Pod 级别或单工作流级别允许出站目标的过滤链进行匹配。匹配条件可以通过 CIDR 块范围、域名、源地址等的任意组合来设置。如果匹配成功，流量将通过一个 TCP 代理过滤器，该过滤器对上游集群执行单一网络连接代理。由于 iptables 之前已经重定向了数据包并将目的地重写为 Envoy 监听器地址，我们依赖原始目标监听器和集群来恢复原始的出站目标地址，并将流量发送到正确的端点。

在代理配置中设置这些组件的定义时，我们参考了 Envoy 的《将 Envoy 配置为边缘代理》一文，获取了有关边缘代理特定设置的有用建议。

监控 Envoy 指标

如前所述，Envoy 会输出大量下游、上游和服务器统计数据，并支持自定义的可插拔 Sink，这些 Sink 可以聚合并转发这些统计数据到统一的目的地。我们为每个 Envoy 代理部署了 Sidecar，以监听来自配置的 StatsD Sink 的指标，从而利用这一特性。输出的指标会被消费并展示在 Datadog 仪表板上。我们还设置了几个 Datadog 监控器来跟踪关键指标，并在某些指标数值超过预设阈值时向团队发出告警。在向 Envoy 迁移的过程中，我们计划监控几个关键指标，以获取有关迁移健康状况的实用信号：

  • listener.no_filter_chain_match：上游白名单是一个 Envoy 过滤链，因此被拒绝的请求将显示在此指标中。在迁移过程中监控此项，可以让我们及时发现任何配置问题。
  • listener.downstream_cx_destroy/listener.downstream_cx_total：监控此指标可以反映 Rubix Pod 与 Envoy 实例之间连接的健康状况。
  • control_plane.connected_state：我们在所有 Envoy 实例上监控此指标，以确保它们与控制面保持连接并接收到配置更新。

互补的安全特性

在 Palantir，我们使用 Cilium 作为基于 Kubernetes 的防火墙，正如这篇博文中所讨论的。这些防火墙同样允许或拒绝出站流量，因此有必要讨论一下在出站流量的背景下，Cilium 如何与 Envoy 相互配合。

Cilium 策略是应用驱动的，这意味着服务可以声明其希望离开集群前往特定目标。一旦创建了 Cilium 策略，来自 Pod 的流量就可以通过正向代理流出。如果 Pod 尝试在没有针对该目标的 Cilium 策略的情况下流出，那么流量将被 Cilium 阻断。

我们的出站代理还包含另一个目标白名单。出站代理处的白名单包含了集群所有合法 Cilium 策略的并集。如果创建了恶意的 Cilium 策略以向某个目标流出，流量仍会在出站代理处被拦截，因为该目标不会在出站代理的白名单中定义。这种双层方法实现了对出站流量非常细粒度的控制：集群级别的控制在出站代理处定义，而 Pod 级别的控制在防火墙处定义。

结语

将 Envoy 作为新的正向代理，对于在 Rubix 中实现细粒度的出站流量过滤至关重要，同时也免除了基础设施工程师为维护客户白名单请求而必须执行的复杂手工操作链。其提供的海量可观测性指标和日志更是一大助力，使我们能够高效地洞察 Rubix 内部的网络状态。

在网络基础设施方面，我们仍有许多工作要做！我们正在推进的一些出站专项包括：为 Palantir 的客户在 Rubix 中添加高级连接功能，以及为 Spark 计算 Pod 创建 Envoy Sidecar，以实现更严格的出站流量控制。
