Safely Modernize Legacy Systems with Palantir Foundry Container Engine (FCE)

2023-04-10 · Original →

导弹预警。航班状态。卫星观测警报。美国政府大量最关键的数字基础设施，都依赖于冷战时期构建的软件。这些软件使用老旧语言（如 Fortran、COBOL、ADA）编写，且往往仅部署在大型主机上。尽管基础设施陈旧，可能难以满足当今的需求，但其核心逻辑往往运行良好。然而，试图通过重写数十年积累的数百万行代码来实现现代化，根本不切实际。

推出 Foundry 容器引擎（FCE）：FCE 在 Foundry 中运行容器化的遗留代码，使政府机构能够利用现有可用部分，同时安全地摒弃失效部分。打个比方，这就像 AWS Lambda 革命性地改变了工程师运行代码的方式——它抽象掉了所需的硬件基础设施，让人无需再为服务器和集群操心。与之类似，FCE 也正在彻底改变工程师将遗留投资集成并编排到现代软件架构中的方式。FCE 简化了您的现代化历程，让您能够逐步重建数百万行遗留代码，同时持续为组织创造新价值。

挑战：遗留代码的运营化困难重重
几十年前的代码本身并没有问题。相反，它们经过了数十年的实战检验，由在高度专业化领域具有深厚专业知识的人员编写。然而，使老旧软件适应当今不断变化的运营现实，既令人望而生畏，又势在必行。20世纪80年代构建的卫星模型当时旨在探测100颗卫星，如今到2023年却需要探测30000颗卫星，重写并扩展这种模型往往是一项难以维系的工作。

依赖遗留代码和基础设施的客户经常面临以下挑战：

  • 现代化具有破坏性：现代化的选项往往带来高度破坏性——数据湖、代码库全面重构以及耗时数年的路线图。这些做法面临让关键系统下线的风险，却未必能实现必要的运营目标。
  • 无法扩展：扩大环境规模以满足计算需求需要漫长的前置时间，而扩展产品实例的副本往往根本无法实现。想要实时完成这些操作以赶上当今的关键截止日期，更是无从谈起。
  • 孤立的逻辑：复杂的遗留模型一旦与其他数据源集成，价值将大幅提升。以前文提到的卫星为例，这涉及观测数据、监控网络、传感器等。然而，添加新的数据馈送、数据处理流程、输出和接口要么不可行，要么太慢而失去价值。
  • 封闭的生态系统：与遗留代码相关的数据流水线往往是个黑盒。其他平台和开发团队无法安全地协作，这限制了能够与代码交互并提供新颖分析的人员数量，实际上扼杀了潜在的增益。
  • 脱离运营决策：遗留模型能生成有价值的洞察，但其输出缺乏可操作性。没有简便的方法来自动生成直观的可视化或实用的告警逻辑。一个碰撞模型可能会显示卫星即将相撞，但用户无法根据这一信息采取行动，去重新调整这些卫星的飞行轨道。

解决方案：利用 FCE 将逻辑平移至 Foundry，使其价值提升百倍
Foundry 作为一个集中管理的云端 SaaS 平台，可即时访问前沿的现代软件实现，包括流式流水线、实时 API 驱动的推理以及自动扩缩容。如今，FCE 允许容器化代码在 Foundry 中运行，释放遗留系统全部价值变得前所未有的简单。

首日即享的收益包括：

  • 安全且渐进地实现现代化：立即启动现代化历程，且大可放心关键系统将持续运行，旧组件的废弃也不会造成损害。
  • 快速可扩展的基础设施：随着能力的演进和扩展，实现按需扩展计算和存储环境。这提供了弹性和冗余，避免单点故障。在联邦航空局（FAA）的飞行软件中，仅仅因为用一个文件替换另一个文件，绝不应导致全国航班停飞。
  • 灵活性与互操作性：无缝添加未来的数据馈送、数据处理流程、对象、模式和接口。融合异构数据，快速生成新的分析结果。
  • 安全的协作：内置的安全访问控制功能，使平台与开发团队组合起来能够进行安全协作。结合流水线透明性和 DevSecOps 迭代，客户可以通过开放、可扩展的 API 安全地实现输出的普惠化。
  • 现代且动态的用户界面，实现快速自动化决策：用户可轻松配置告警逻辑，并利用低代码/无代码工具构建新应用。将卫星物理模型的复杂输出，转化为与运营密切相关的空间域感知应用。

在孤岛中，遗留软件仍可改进，但这些微小收益的获取，是以牺牲现代化带来的巨大且不断叠加的收益为代价的。FCE 将 FCE 运行的一切与其他 Foundry 产品（如流水线构建器、流处理、工作坊）集成，使机构能够快速推进其软件驱动的目标。借助 Foundry 模块化和互操作性的核心原则，机构可以选择性地弃用遗留软件组件，而不会中断其数据源、本体和操作。在这个导弹上个月还是抛物线轨迹、下个月就变成高超音速的世界里，比特（软件）领域的创新必须超越原子（实体）领域的创新。
