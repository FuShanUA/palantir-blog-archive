Hardening Palantir’s Kubernetes Infrastructure with Cilium

2021-05-06 · Original →

随着工程团队越来越依赖 Docker 或 Kubernetes 等工具来管理、部署和扩展其环境，容器化基础设施已成为全行业的趋势；在这方面，Palantir 也不例外。我们构建了 Rubix——Palantir 的 Kubernetes 基础设施，主要出于两个目标：简化并扩展我们软件平台的部署，同时强化安全态势。

在这篇博文中，Palantir 的信息安全（InfoSec）团队将分享我们近期使用 Cilium 的经验。Cilium 是 Isovalent 的一个开源项目，致力于保护基于容器的基础设施，其提供的可见性与控制能力优于传统防火墙。以 Cilium 作为网络控制的基础，Palantir 的工程师对调节 Rubix 流量的网络规则进行了实质性改进。同样，Palantir 的计算机应急响应团队（CIRT）也将 Cilium 的日志作为遥测数据的重要来源，用于编写针对 Rubix 基础设施可能遭受的理论攻击原语的《告警与检测策略》。在本文中，Palantir 的 InfoSec 团队希望：

  • 介绍 Cilium 的各项功能，正是这些功能使 Palantir 得以进一步强化 Rubix 的攻击面。
  • 探讨我们在告警与检测方面认为极具价值的 Cilium 日志遥测领域。

传统安全工具与控制的局限性

传统的“遗留”信息安全控制手段和工具（主要为静态基础设施设计）在应对容器攻击面时往往存在不足，尤其是当该攻击面具有高度临时性，或是 SaaS 环境中更庞大的微服务生态系统的一部分时。

网络工具与控制

传统网络安全工具的一个常见缺陷在于，在由临时容器构成的环境中，它们相对缺乏针对不同端点、微服务和资源提供细粒度网络控制的支持。传统的三层/四层防火墙可以根据端口和目的地来控制出入站流量，但这种规则是不加区分的，会拦截（或放行）所有符合其规则集参数的流量。一些网络解决方案通过超越基本端口/IP 出入站控制的功能来扩展其特性集（例如基于域名、字节数、时间段或 IP 信誉的控制），但此类控制仍倾向于基于主机名、域名或 IP 地址等标识符运行。

在容器化生态系统中，大部分交互发生在临时微服务和 API 资源之间，若从传统的 TCP/IP 命名法角度出发，要维持对特定端点或资源身份的认知会面临挑战。

端点工具与控制

另一个常见的安全工具缺陷是对端点检测与响应（EDR）源遥测数据的依赖（参见：《osquery 跨企业应用》）。这些源将容器化基础设施视同传统的静态端点，通过 IP 地址或主机名来固化其身份概念。每次启动临时资源时，都必须为其安装 EDR 代理，该代理随后必须向中央服务器注册，或直接将其遥测数据记录到日志目标。一旦 EDR 代理进入临时主机并成功上报遥测数据，它也只会在该资源存续期间继续工作（这可能仅有几秒钟）。或许负责该端点的服务每分钟左右就会启动一个这样的资源；结果是几小时内会有数十个端点启动，它们全部属于同一个服务，执行相同的任务。然而，对于每一个启动的端点，EDR 遥测都会将其视为一个独立的实体（拥有自己的 ID、主机名等）。

基于 Cilium 的容器化网络安全控制

Cilium 对特定环境中资源的概念是基于服务、pod 或容器身份的，这使得在对象可能仅临时存在或生命周期不定的情况下，依然能对特定资源（或资源族）实现持久的可见性与控制。Kubernetes 标签会应用于 Cilium 已知的每个资源上，这些标签通过包含有关资源角色和功能的进一步上下文数据，丰富了资源的名称。与基于主机名或 IP 地址的操作相比，在应对容器化环境的复杂细节时，使用详尽的标签和上下文元数据极大简化了分析师的工作。

为了符合 Palantir 的安全控制要求，必须将 Cilium 部署到我们 Rubix 环境中的每个资源。部署后，Cilium 将根据其《网络策略》中建立的规则运行。Cilium 网络策略被指定为指示特定环境中资源间允许的连接和流量的规则。网络策略规则在三层、四层或七层运行，每条规则都基于与特定资源关联的 Kubernetes 标识符建立控制。例如：

  • 微服务 A 应当只能向 API 资源 X 和 API 资源 Y 发起请求，前提是这些请求符合 条件 Z。
  • 只有带有标签 用途 A 的 pod 才能解析 域名 B。

在 Palantir 使用 Cilium 的控制

在 Palantir，我们的基础设施和安全工程师利用 Cilium 的规则，建立了专为 Rubix 特定功能量身定制的网络控制。通过确保 Rubix 的所有 Kubernetes 资源都适当地标注了能准确反映其在环境中角色和功能的标签，我们为 Cilium 提供了在调节 Rubix 内部流量时解释 Kubernetes 基于身份的概念所需的关键标识符。

在制定了健壮的规则集之后，我们 InfoSec 团队的注意力转向了 Cilium 规则成功拦截 Rubix 中不合规流量的情况。此类活动意味着，可能存在恶意的第三方正试图在环境中执行操作。在这种情况下，我们的团队将转向告警、检测和响应领域。

基于 Cilium 的容器化告警与检测

Hubble 是构建在 Cilium 之上的可观测性/日志平台；它利用 eBPF 实现对端点上发生操作的可见性，进而为各种事件生成独立的日志条目。

Hubble 日志事件类型

截至本文发布时，Hubble 会生成九种事件类型；这些事件可分为两类：

1) 由 Cilium 网络可见性生成的网络流量事件
Cilium 基于网络的日志记录包含来自 Kubernetes 资源间成功与未成功的网络流遥测数据。然而，在像 Cilium 这样具备七层控制能力的网络工具的语境下，我们可以接收到使用传统网络安全解决方案可能无法看到的拦截事件。例如，如果 端点 A 是一个微服务，其唯一流量应当是定期向 HTTP 1 发送 GET 请求，而 Cilium 拦截了发往 HTTP 1 的 POST 请求，我们就可以得出几点结论。首先，如果安全控制仅基于 IP 和端口限制，同样的活动很可能会被放行；其次，是有外部因素导致了该 POST 请求的生成。这并不是说每一个异常的 POST 请求都归咎于恶意行为者，但如果攻击者确实在环境中生成了未经授权的 HTTP 流量，能够对其流量保持可见性将是应对此类事件的重要资源。

2) 由 Hubble 端点上 eBPF 可见性捕获的进程事件
如前所述，虽然 Hubble 的部分事件派生自 Cilium 特定的网络可见性与活动，但其他日志事件仅利用 Hubble 端点上的 eBPF 可见性来记录进程活动。Palantir 的告警与检测工程师发现，Hubble 特定于进程的事件可作为传统企业 EDR 解决方案的替代方案（即使 Cilium 最初部署时的重点在于网络活动而非进程活动）。此类遥测数据包括进程启动事件、套接字连接和进程结束事件。

Hubble 的进程日志与传统 EDR 解决方案的日志之间存在广泛的交集。我们认为，运行 Cilium 和 Hubble 的组织可以通过依赖 Hubble 的进程日志来替代其先前的 EDR 解决方案，从而初步减少代理占用空间、SIEM 摄入量以及企业 EDR 成本。
