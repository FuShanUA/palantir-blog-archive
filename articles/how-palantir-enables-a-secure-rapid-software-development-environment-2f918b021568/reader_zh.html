How Palantir Enables a Secure, Rapid Software Development Environment (Software Supply Chain Security, #1)

2024-11-12 · Original →

编者注：这是系列博文的其中一篇，分享了我们在 Palantir 提升软件供应链安全能力的经验与洞察。本篇重点介绍我们发起软件供应链安全（SSCS）计划的背景与初衷，并深入剖析我们安全控制机制和防御态势背后的威胁模型。

引言

Palantir 的核心竞争力始终在于：面对现实世界中复杂且影响深远的问题，能够快速适应并构建软件解决方案。这要求我们的研发文化具备极高的敏捷性和快速调整的能力。然而，快速迭代必然伴随着风险。鉴于 Palantir 的使命、业务范围以及客户群体，我们对此有着清醒的认识。全球最顶尖、最顽固的攻击者正将矛头对准我们。

那么，如何让开发人员既能保持高效，又能确保安全？

在这系列博文中，我们将拆解自身理念如何从著名的“快速迭代，打破常规”转变为“快速迭代，守护安全”。我们将探讨如何从最初对软件供应链的模糊认知，逐步发展为构建一套前沿的软件供应链安全（SSCS）体系，且该体系原生依托于我们自研的产品。

威胁环境

Palantir 处于极其严峻的威胁环境之中，这要求我们将“安全设计”作为每项重大举措的核心。我们持续遭遇高级且顽固的攻击者侵扰，这些攻击者动机极强，意图攻击西方民主国家、金融体系及其他关键基础设施。

作为一家以处理西方多国敏感数据（涵盖情报与军事机构）而闻名的企业，我们必须针对高级持续性威胁（APT）构建威胁模型。这包括知名的民族国家黑客组织及犯罪团伙。在我们的威胁模型中，我们设定攻击者会利用高级战术、技术和程序（TTP），包括零日漏洞利用、社会工程学、人力资产招募、物理接近、供应链攻击及其他复杂手段。

与众多组织一样，我们面临的最大风险之一是软件供应链攻击。这些攻击可能瞄准软件供应商、我们依赖的上游基础库，或直接将我们视为跳板以触达最终客户。鉴于供应链攻击的频率与严重程度，我们的威胁环境迫使我们部署处于行业最前沿的解决方案。

基础设施

Palantir 全面贯彻零信任、纵深防御以及假设已被渗透的安全理念。我们坚信构建坚固且边界清晰的网络安全环境，能有效遏制攻击者的活动空间。从网络访问层面看，我们严禁从公共互联网直接连接至公司绝大多数内部资源。设备必须首先通过全隧道虚拟专用网络（VPN）接入我们的网络，且该 VPN 需通过我们的 SSO 无密码登录机制进行身份验证。同时，我们深知攻击者总能找到突破口潜入网络，因此在设计安全控制机制时，我们始终假设所有设备均存在恶意或已被渗透。

为提升监控能力、强制执行安全控制机制，并构建当前市场尚无的前瞻性安全解决方案，Palantir 依托内部托管的基础设施来执行安全的软件部署生命周期。这一思路带来三大核心优势：

  • 降低因上游供应商被渗透而引发的入侵风险：我们不依赖软件即服务（SaaS）平台，从而在面对这些平台潜在的渗透风险时具备更强的韧性。
  • 满足安全与合规性要求：运行我们自主掌控的 FedRamp/IL5/IL6 认证环境，确保我们的安全软件开发生命周期严格符合所有安全与合规性标准。
  • 增强安全监控与零信任技术：我们能够为所有基础设施注入高级安全监控能力并部署零信任技术。

核心组件

Palantir 的软件供应链基础设施包含若干核心组件。我们的源代码库托管在 GitHub Enterprise (GHE) 中。软件构建过程在临时的 CircleCI 节点上执行，确保每次构建均处于全新的环境之中。生成的软件构件存放在 Artifactory 中，为所有构建产物提供安全且有序的存储库。最后，我们使用 Palantir Apollo 进行持续部署与应用管理，实现软件的无缝安全发布。

软件供应链威胁建模

在 SLSA、S2C2F 或 P-SSCRM 等现代软件供应链框架出现之前，2020 年发生过著名的 SolarWinds 入侵事件。Palantir 的首席信息安全官（CISO）迅速采取行动，摸清我们开发流程与现有安全控制机制的真实底数。为了制定正确的控制措施，我们必须明确风险所在。遗憾的是，多年高速发展、业务急剧扩张以及产品线不断延伸，导致我们积累了一批各自为政的开发路径，这些路径由不同团队维护，存在大量重复的建设、基础设施、安全控制机制及流程。

为了摸清真实底数，我们进行了大量的会议沟通、访谈、文档修订和架构图更新，但这依然没能让我们完全、准确地掌握开发团队的实际操作情况。
我们从绘制威胁模型图入手，描绘了代码从开发人员流向源码控制、构建系统、打包，最终至生产环境的全过程。然而，现实远比这个看似简单的概念要复杂：为了优化内部工作流和内部高级部署系统，我们自研了大量开发工具。要厘清这些工具和系统，搞清楚它们的实际运行方式与最初设计意图的偏差，是一项浩大的工程。

举例而言，我们在 GitHub 企业版实例中拥有 10,000 个代码库。为此，我们的开发工具团队构建了一套解决方案，通过一个中心化的“基础设施即代码”配置库，实现规模化应用代码库配置。我们无需再通过 GitHub 界面逐个手动配置代码库（例如在受保护分支上启用必需的状态检查），只需在中心库中修改一行 YAML 配置，该变更就会自动同步至任意指定的代码库集合。这款工具极其强大，让我们得以保持高效运转，但同时也引入了巨大的风险——比如可能会在全局范围内意外禁用某项安全控制机制。在开发这些工具时，安全往往不是首要考量，因为它们的设计初衷是不惜一切代价减少阻力。

摸清这些工具的实际使用情况并定义其安全基线，是一个持续迭代的过程。我们的威胁模型聚焦于软件供应链的五个核心领域：

  • 源码控制与软件设计
  • 第三方依赖项
  • 构建与构件发布
  • 构件存储
  • 构件部署

我们针对每个领域界定了风险，并提出了相应的缓解控制措施，旨在防范类似 SolarWinds 规模的入侵事件。软件供应链架构图绘制完成后，我们邀请了各部门的合作伙伴，基于他们的视角共同探讨风险及潜在的安全解决方案。这一过程带来了巨大收益，原因有二：首先，通过建立对供应链安全加固必要性的共识，赢得了各方的认同与支持。其次，引入了来自各业务线的工程师，他们掌握开发活动的一手信息，能够为潜在的解决方案提供实质性的认责。在这个过程中，我们挖掘出诸多不一致与隐患，帮助我们聚焦于高影响力的安全控制机制。这些控制机制构成了我们 SSCS 计划的基石，使我们在默认状态下即可安全地构建软件。

SSCS 计划目标

基于软件供应链威胁模型，我们为 SSCS 计划确立了以下目标：

  • 构建多层防御体系，防范恶意代码注入供应链。
  • 审查所有产出软件的安全架构，确保遵循安全设计原则。
  • 确保所有源码控制提交均采用加密签名，以保障源码控制的真实性、完整性和不可否认性。
  • 针对安全敏感型操作（如 Git 提交签名、SSO 登录），采用硬件级加密签名。
  • 在标准化开发路径上设置防护栏，确保所有开发源码控制库和构建过程均落实安全控制机制。
  • 对所有环境（包括源码控制、构建和部署系统）实施最小权限原则。
  • 充分加固并隔离所有构建环境，力求实现隔离构建。
  • 要求所有产出的软件构件具备端到端来源证明，以保障软件供应链的真实性与完整性。

在本系列的后续博文中，我们将详细拆解这些目标及为实现这些目标而落地的安全解决方案。

结语

鉴于安全资源有限，将精力集中于高影响力、高确定性的项目至关重要。从构建威胁模型入手以量化风险，并全面摸清软件供应链现状，是合理调配资源、补齐软件开发生命周期（SDLC）薄弱环节的必由之路。随着软件供应链攻击日益频繁且复杂，筑牢根基变得尤为关键。此外，培育一支对安全目标共同担责的研发组织，能确保全员目标一致，致力于维护稳固的安全环境。
