How Palantir Foundry’s Ontology Deploys Data Science to the Front Line

2022-12-19 · Original →

编者注：在这篇博文中，Palantir 前线部署架构师 Wes Field 分享了 Foundry 本体如何帮助数据科学家将他们创建的模型与其实际支撑的现实决策联系起来。

在当今的企业中，数据科学家的角色看起来似乎很简单：从数据中提炼洞察，并将其交付给决策者。这个过程就像是一场单程旅行——模型交付出去，业务采取行动，而数据科学家只能在一旁猜测他们的模型是否产生了影响以及如何产生的影响。此外，一个令人清醒的现实是，大量的时间和精力往往消耗在极其缓慢的数据发现、数据获取和模型交付流程上，而不是用于评估模型本身的实际效果。

Palantir Foundry 本体构建了业务操作与支撑决策的模型之间的反馈闭环，提供了一种解决方案。该本体加速了数据科学家识别瓶颈、结合全量相关数据采取行动、交付模型的进程，并最终通过行为改变来量化模型对业务的影响。作为一名机器学习工程师，如今又作为 Palantir 的架构师，我亲眼见证了这项技术如何为数据科学领域带来范式转变。在这篇博文中，我将详细阐述其中的原因。

将数据科学家的角色从被动响应转变为主动驱动

如今，数据科学家通常会从领导层、中央组织或所属业务部门接到宏观指令。例如：“近几个月患者等待时间增加了。我们该如何缩短这一时间？”为了明确具体目标并找出解决问题的方法，数据科学家需要与组织各领域的主題专家（SME）沟通，收集应对这一挑战所需的业务知识。

该本体将组织内所有的实体、关联关系、业务流程和事件智能汇总于一处，清晰呈现业务中的瓶颈与改进机遇。 依托这张“行动导览图”，数据科学家无需依赖主题专家来充当业务流程导师。他们可以直接查看所有系统及其关联，准确确定该向谁咨询以及该构建何种模型。昔日那种极其碎片化的领域知识传递体验得到了极大简化，使数据科学家能够专注于优化解决方案本身。

数分钟内获取数据源，而非数月

一旦确定了模型构建和流程改进的切入点，收集所需数据依然可能困难重重，因为数据资产通常与相关业务部门脱节，散落于企业数据湖或数据仓库中。要获取满足业务需求的正确数据资产，必须对源系统有所了解并经过多重审批。

该本体捕获了真实的业务流程及相关的“操作”，并将其与数据对象关联。这些对象随后作为数据科学目标呈现。以这种方式创建数据科学目标，免去了深入数据湖或数据仓库挖掘相关数据集的麻烦，也省去了数据集确定后冗长的审批链条。

在未使用 Foundry 的企业中，数据获取有时耗时数月；而得益于该本体直观协助数据科学家查找相关数据的特性，这一环节如今已被大幅缩短。

以更快的速度将模型交付到正确的人手中

要将值得信赖、可执行且具备可解释性的模型和预测交付给决策者，数据科学家需要明确谁掌控着某个流程或决策。然而如前所述，数据科学家往往是从组织内部那些最终并不使用这些模型的人员那里接收业务指令。

让情况更加复杂的是，交付过程很少能一帆风顺。受诸多因素影响，模型从开发到最终投入使用，往往需要数月甚至数年时间。独立的机器学习（ML）工程师团队必须接手模型，将其重构为容器和应用层，并配置端点，以实现模型的可扩展性和可操作性。这些额外环节给交付过程徒增了不必要的阻力。

随后，即便模型完成部署（例如作为容器中的微服务），应用开发者仍需修改业务应用，以便在可操作的上下文中请求并调用预测结果。随着业务流程或项目需求可能已经发生变化，这进一步压缩了行动的机会窗口。

该本体通过以下几种方式帮助数据科学家克服这些阻力点：

  • 该本体清晰呈现了究竟是谁在使用数据科学模型，从而免去了数据科学团队对模型受众的盲目猜测。因此，他们能够合理调整模型，避免陷入与决策者之间不必要的反馈循环中。
  • 该本体为机器学习模型打造了真正的敏捷开发周期，使数据科学家能够在真实的业务操作反馈闭环中训练、发布、部署模型并衡量模型效果。无需与机器学习工程师反复沟通，意味着模型交付速度大幅提升。模型交付后，数据科学家可以直接查看结果并相应地微调模型。
  • 该本体允许数据科学团队直接对面向用户的 Foundry 应用进行带版本控制的编辑，或者通过点击操作从零开始构建自己的应用。 由于该本体提供统一 API 和集成工具以支持快速、协作式的应用开发，全栈数据科学工作完全可以交由一个敏捷团队在标准的两周迭代周期内完成——这才是交付数据产品的正确模式。

闭环模型与决策之间的链路

要判断一个模型是否真正发挥了作用，需要决策者持续且一致的反馈，同时还要对已做出的决策结果以及组织行为的其他变化保持可见性。如果模型的预测结果未被采用，数据科学家了解背后的原因同样大有裨益。

该本体的数据回写能力通过其 API 或相关应用对外暴露，实现了数据科学与业务运营团队之间关于模型使用情况、已做决策及业务影响的持续双向沟通。

设想这样一个场景：数据科学团队通过本体记录发现，他们部署在应用中的推荐操作被用户无视。利用本体的数据回写功能，该团队在下游应用中增加了一个“有用度”切换开关和评论框来捕获反馈。瞬间涌入的评论让问题迎刃而解：终端用户不信任这些推荐，因为他们不清楚推荐的依据。

好消息是，新解决方案只需数小时即可部署。数据科学团队将通过本体发布的模型特征重要性与预测结果关联，如今这些信息会透明地显示在每一条推荐旁边。

虽然上述场景是假设性的，但我们已在许多客户那里见证了类似情形反复上演。客户们利用该本体，充分发挥数据科学团队的全面协作能力，从而加速全企业范围的业务变革。

Foundry 本体能够帮助数据科学家充分发挥其潜能。它为数据科学家清晰界定了业务改进的机遇所在，并显著加速了机器学习模型的创建与部署。
