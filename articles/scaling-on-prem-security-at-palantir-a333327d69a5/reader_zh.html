Scaling On-Prem Security at Palantir

Insight、Foundry 与 Apollo 如何管控成千上万台服务器

引言

谈及 Palantir 的本地化部署产品，安全始终是重中之重。我们的生产软件运行在各种环境中——从超大规模公有云，到永远不会与公共互联网交互的客户自建数据中心，再到全球移动的边缘环境——我们会针对每种环境量身定制安全策略。

Palantir 的云安全态势管理（CSPM）解决方案用于保护我们的云服务配置，而本地化部署则通过操作系统（OS）和主机级别的配置来保障安全。DISA STIGs 和 CIS 等行业基准是宝贵的切入点，但 Palantir 的标准更为严格，涵盖了内核参数、日志转发、系统管理模块、备份，以及我们在云端所期望的持续验证。

本地化解决方案带来了不同的风险特征。Palantir 的本地化产品部署在由物理节点组成的环境中——从单台服务器到大型集群——它们要么位于边缘（在移动或半移动技术栈中），要么位于客户数据中心内部。边缘部署通常只涉及一到三个节点，而数据中心的部署规模则可能非常庞大。

物理访问是本地化解决方案需要重点考量的问题。本地网络的安全、底层操作系统的加固以及管理多种操作系统类型都需纳入考量。此外，与云端一样，在本地化环境中使用 Kubernetes 作为部署 Palantir Rubix 等解决方案的底层基础，也会引入其自身的安全要求层。

为应对这些挑战，Palantir 的本地化安全标准包含众多单项安全检查——且随着新威胁的出现，检查数量还在持续增加。关键验证包括数据加密、强制执行特定固件配置、利用可信平台模块（TPM），以及确保账户配置正确。甚至引导加载程序配置等细节也会对安全产生重大影响。

正因如此，我们开发了 Insight，这是一款专门构建的解决方案，旨在帮助 Palantir 环境构建者和客户坚守 Palantir 严格的本地化部署安全标准。借助 Insight，组织可以信任其本地化环境已得到妥善防护，从而将精力集中在最重要的事情上：他们的核心使命。

Palantir Insight：面向本地化部署的主机级基准检测

随着 Palantir 本地化版图的不断扩大，人工审查加固清单很快就变得不可行了。我们需要一种一致且具备云端规模的检测方法，来管理那些远在云端之外的服务器。Palantir Insight 弥补了这一空白。

Insight 是一个单一的 Go 二进制文件，用户可以在 Linux 主机上运行：

$ ./insight scan

Insight 提供了一系列“检查”，每一项用于评估特定的安全配置。这些检查是模块化的，可以根据需要添加、删除或执行。检查范围涵盖从内核参数到必需的灾难恢复服务是否存在。它们有意超越了 STIG 和 CIS 所提供的内容，将我们在每个环境中都期望实施的管控措施进行了编码，而不受外部合规性要求的影响。

通过这些检查，系统会根据 Palantir 的本地化基线安全标准对单台主机进行评估，并输出一个包含以下内容的 JSON 文件：

  • 主机元数据（操作系统、架构、计算基础设施、版本）
  • 检查结果及通过或未通过该项检查的原因
  • 摘要统计（通过/未通过的检查数量）

请注意，所有敏感信息均已脱敏，以确保报告不会意外暴露密钥或密码。

以下是某份报告的摘录。顶部的“detections”包含元数据，为审查主机的安全态势提供了有用的上下文。“results”字段列出了运行的每一项检查，并详细说明了每项检查通过或未通过的原因。最后，摘录底部的“statistics”提供了该主机 Insight 合规性水平的快速概览。

{

        "operating_system": "linux",
        "distribution": "xxx",
        "architecture": "xxx",
        "hostname": "xxx.xxx.xxx",
        "insight_version": "1.0.0",
        "kernel_version": "x.xx.x"
    },
    "results": [
        {
            "check_name": "No credentials in configuration files",
            "reasoning": "There were 0 files with credentials detected",
            "expected_configuration": "SecureBoot should be configured in the BIOS.",

                "credentials_detected": {},
                "known_bad_credential_files_found": {},

构建本体

Insight 能够生成 JSON 报告后，我们的注意力便转向了如何将这些数据转化为可操作的洞察。我们利用 Palantir Foundry 构建了这一解决方案，将 JSON 转换为一等对象。以下是“Insight 本体”，包含四种核心对象类型：

  • 主机：关于机器的元数据信息
  • 扫描报告：JSON 文件和摘要统计
  • 检查：预期的主机配置，包括严重程度和修复文本
  • 结果：检查的输出结果，以及通过或未通过的原因

下图显示，当构建者上传 Insight 扫描报告时，一个轻量级函数会解析 JSON 并创建这些本体对象。这会直接提供血缘关系、访问控制和临时分析能力——无需编写额外代码。

在 Foundry 中进行协同评审

构建者和评审者现在可以在专属的 Foundry Workshop 应用中查看和检查相同的数据，且数据范围限定在他们所拥有和支持的技术栈内：

  • 主机详情：所有受支持技术栈中 Insight 合规性的历史趋势
  • 扫描报告表：每台主机的通过/未通过状态检查
  • 检查数据目录：预期配置列表，包含其重要性说明和修复步骤

构建者和评审者可以点击查看每一项检查，了解通过或未通过的原因，并明确所需的修复步骤。还可以在单项检查结果中添加评论，以促进构建者与评审者之间的沟通。

实施修复后，构建者可以选择再次运行 Insight 或等待自动扫描运行，然后将新报告上传至同一界面。仪表盘会在几秒钟内更新，评审闭环由此完成，全程无需使用电子表格或电子邮件附件。

通过构建问卷捕获上下文

扫描能指出哪些检查通过、哪些未通过，但无法提供主机构建方式或环境范围的信息。对于每次新部署，构建者都需完成一份构建问卷，用于捕获以下信息：

  • 基本情况：构建人员及构建规模
  • 基础设施类型：Kubernetes 发行版及主机操作系统
  • 安装方法：所使用的模块和模板
  • 环境用途：构建该技术栈的业务理由

这份构建问卷在启动新构建之前提供了有用的上下文，确保我们一旦技术栈完全建立就能做好准备进行评审。问卷有专属的 Foundry Workshop 应用，便于相应团队轻松上传和评审。借此，我们能够维护一份构建详情的历史记录，这些信息原本很可能会散失在会议记录中。

在物理隔离环境中运行

部分客户站点处于永久断网状态；数据若要传出，必须经过冗长的向下传输流程，甚至根本无法传出。这一限制带来了额外挑战：虽然可以通过由低至高的软件捆绑方式将 Insight 交付至这些客户站点，但无法在现场对结果进行评审。解决方案是利用 Foundry 的 Marketplace 功能，将整个 Workshop 应用打包并发布为 Foundry Marketplace 捆绑包。环境数据主人现在可以将该捆绑包自行导入到各自的技术栈中，并访问完全相同的仪表盘和修复内容——全程无需互联网连接。

除了通过 Marketplace 捆绑的 Foundry Workshop 应用外，我们还通过标准化的可观测性和监控工具暴露与 Insight 合规性相关的指标。这使得大规模部署无论是否包含 Foundry，都能自动将跨多台主机的 Insight 合规性结果汇聚到其现有的可观测性工具中。

利用 Apollo 打通最后一公里

在 Insight 的早期测试阶段，构建者必须从我们的制品库中获取 Insight 二进制文件，将其复制到每台主机上，执行扫描，取回输出结果，最后上传至 Foundry。面对数百台主机，这一流程成了阻碍，最终变得不可行。

我们用一个 Helm chart 替换了手动步骤，该 chart 通过 Palantir Apollo 将 Insight 作为 DaemonSet 部署在每个节点上。扫描会根据环境设定的周期运行，并将结果写入共享卷。借助 Apollo 的 CI/CD 和打包解决方案，Insight 二进制文件会保持更新并自动滚动发布到各技术栈。

有了 Helm chart，构建者现在只需针对每个环境下载一次汇总报告，并将其上传至 Foundry 即可。不再需要繁琐的 SSH 操作，不再有版本漂移，也不再有 Pod 到主机间的文件传输。对于未使用 Kubernetes 的部署，我们构建了一个 Puppet 模块来实现相同流程。

成效与未来展望

自推出 Insight 以来，信息安全团队在构建流程的每个阶段均有参与，极大提升了技术栈的可见性和安全性。这种方法大幅提高了主机合规性并简化了修复流程，同时将人工操作降至最低。最重要的是，我们与公司各关键利益相关者建立起了稳固的关系。

未来的工作将侧重于让 Insight 更深度地融入构建流程，以减轻技术栈构建者的压力。我们正致力于解决上游构建模板中的问题，让手动修复成为历史。此外，我们计划加大投入，为 Palantir 员工提供更好的教育和资源，帮助他们深入了解本地化安全。

对于那些在大规模环境下——尤其是混合云、本地化和物理隔离环境中——努力应对主机合规性问题的团队，我们希望 Insight 的架构能为您带来启发。
