The Benefits of Running Kubernetes on Ephemeral Compute

2022-09-01 · Original →

编者注：这是我们在博客中以 Kubernetes 为核心重构云架构的系列文章（#1，#2）的第三篇关于 Rubix 的文章。

引言

容器及其编排平台（最典型的是 Kubernetes (K8s)）的出现，让工程师们逐渐熟悉了临时计算的概念：一旦工作负载完成，用于运行它的资源就可以被销毁或回收。诸如 K8s Pod 等临时计算构造，使我们能够做到：

  • 运行源自通用模板的不可变工作负载配置 → 这种不可变性简化了对异常工作负载的调试
  • 通过替换工作负载来实现升级 → 这避免了原地更新带来的复杂性

临时计算的概念也对工程规范产生了积极影响。由于 Pod 可能随时被终止，工程师必须在开发的所有阶段仔细审查应用程序的弹性。

Kubernetes Pod 往往是工程师与临时计算概念交互的唯一层级。在设计 Palantir 基于 Kubernetes 的基础设施平台 Rubix 时，我们问自己：如果将临时计算的概念进一步推向底层技术栈，能获得什么价值？

本文介绍了 Rubix 在管理由不可变、短生命周期的 K8s 节点（即“计算实例”）组成的 K8s 集群方面的实践。我们将讨论对这种方法的坚定信念、随之而来的挑战，以及它如何使 Rubix 的生产环境更具弹性且更易于运维。

为什么选择临时计算实例？

管理一个由短生命周期节点组成的 Kubernetes 集群听起来很困难。那么，何必呢？我们可以通过回顾一个更熟悉的临时计算构造——Kubernetes Pod 的优势，来确信这项工作的价值。这些优势归根结底源于 Pod 的不可变性：

  • Pod 的配置清单（spec）足以让我们了解 Pod 的工作负载
  • 原地升级的问题不复存在——升级通过替换来实现

通过将 K8s 节点视为临时计算实例，我们可以实现类似的收益。源自实例模板的不可变节点，简化了对生产环境中遇到的任何节点问题的调试。销毁并替换节点的能力，使得原地升级实例变得不再必要。

此外，我们增加了一个约束：Rubix 环境中的节点生命周期不能超过 48 小时。这一约束带来了几方面的好处：

  • 减少集群管理员的手动操作： 有问题或过时的实例会被自动替换，且驱动这种替换的逻辑包含了从以往支持工单中提炼的经验。
  • 改善安全态势： 攻击者仅攻破单个节点，不足以获得对环境的持久访问权限。
  • 强化工程规范： Pod 的中断不再只是可能发生，而是必然发生。工程师在开发应用程序时必须将高可用性视为关键。
  • 功能路径化： 在不影响生产环境的情况下，例行启动和终止实例是集群自动扩缩容的基础模块。当工程任务被分解为范围明确的小问题时，往往更容易取得成功。

既然确信运行由临时节点组成的 K8s 集群具有如此多的优势，接下来就谈谈为了实现这一目标我们必须解决的问题。

优雅地排空与终止节点

每两天终止一次实例，给运行在 Rubix 上的应用程序维持可用性带来了挑战。这要求开发人员在编写代码时必须考虑高可用性，但我们的基础设施仍需确保实例终止不会彻底破坏所托管服务的稳定性。

我们通过开发一个包含三个组件的终止流水线来解决这个问题：

  • 基于策略的节点选择，该策略会综合考虑节点年龄和健康状况等标准
  • 通过节点标签和注解记录节点选择决策
  • 终止逻辑，负责处理 K8s 节点及其关联云实例的排空与最终终止

这种分解方式为我们在运维和稳定性上都带来了好处。

在运维方面，我们的节点选择策略抽象允许我们将集群运维中的经验固化下来。一旦在生产环境中识别出某些不良节点状态类别，我们就会定义策略来自动终止和替换此类节点。例如，旧的 AWS EBS 卷插件（后来被废弃，转而采用 EBS CSI 驱动程序）如果检测到节点存在卷卡在附加状态的情况，就会对该节点应用 NoSchedule 污点。Rubix 团队添加了一项终止策略，专门选择带有此类污点的节点，从而免除了在未来遇到相同问题时进行人工干预的需要。

从稳定性的角度来看，我们的终止逻辑通过使用驱逐 API 考虑了应用程序的可用性，并在节点排空期间遵守所有相关的 PodDisruptionBudgets (PDB)。

读者可能已经熟悉 PDB 和驱逐机制；我们终止流水线方案的新颖之处在于对策略进行编码，从而减轻了环境运维人员手动识别和替换无法正常工作的云实例的负担。

升级实例

在容器编排平台出现之前，应用程序的升级既痛苦又容易出错。常见的陷阱包括：

  • 依赖于底层机器状态的某些方面（例如机器上可用的 JDK 版本），而这些状态可能发生变化
  • 在错误的目录下调用新二进制文件，或在其他方面微妙地改变了其调用方式
  • 从前一个进程的工作目录中继承了不需要或难以察觉的状态

经历过这种升级的工程师，定能体会 K8s Deployment 滚动更新的简单之处。旧 Pod 被直接销毁并用新 Pod 替换，而 Pod 模板是每个副本唯一的配置来源。Rubix 对临时计算的承诺，将这种升级语义带到了 K8s 节点层面。

以一次常规的操作系统升级为例。这会为我们的云实例生成一个新模板（无论是 AWS 启动模板还是 GCP 实例模板）。在此基础上，加速升级只需添加一项新的终止策略，优先淘汰源自旧模板的实例即可。随后，云提供商在替换这些被终止的实例时会使用新模板。

这种升级方法并不能覆盖所有类别的实例升级。例如，对启动配置的某些更改会导致云提供商单方面终止实例，这使得前文所述的优雅终止逻辑失去了作用。

我们再次利用不可变性的概念解决了这个问题，不过这一次是在实例组层面。通过将某些实例组配置参数设为不可变，我们能够通过替换来实现实例组升级。

事实证明，将临时性和不可变性扩展应用于计算资源再次体现了其价值。类似于 Deployment 管理的 ReplicaSet，现在的实例组没有配置历史记录。一个实例组的所有成员均源自单一且不可变的模板，使其更易于调试。

此外，当升级在两个截然不同的实例组之间以渐进滚动的方式进行时，我们可以对这两个实例组分别进行评估。我们现在可以追踪每个组的健康状况，这有助于我们确定新实例组配置的正确性，并在遥测系统检测到问题时进行回滚。下面的例子展示了以启动失败率作为实例组健康指标时，升级失败的行为表现。

扩展

随着我们将临时计算的概念广泛应用，我们达到了这样一种状态：实例的升级和回滚可以自动进行。此时，我们问自己：能否利用这个系统来提升我们对云容量问题的抵御能力？

云容量问题可能由多种原因引起，但往往沿着可用区 (AZ) 或实例产品类型（无论是实际的实例类型，如具有特定内存与 CPU 比率的类型，还是在实例之上构建的计算产品类型，如 AWS EC2 竞价实例）的维度发生。将这类与容量相关的特征视为不可变，会产生怎样的效果？

与云容量问题最相关的参数通常具有可枚举的值；生产环境通常跨越三个可用区，而选择竞价实例是一个二元决定。因此，沿着这些新维度拆分现有的实例组，会产生数量有限的实例组，我们可以根据实例组的健康状况将容量路由到这些组中。

虽然拆分实例组带来了新的责任（例如在可用区之间平衡扩缩容），但它通过使控制平面能够自动将容量路由到健康的实例组来增强我们基础设施的健壮性，即使其他实例组遭遇了云提供商的服务中断。

结论

Palantir 将 Rubix 视作一次契机，把在生产中已证明具有价值的临时计算概念，从 Pod 层面扩展到了 K8s 节点层面。在克服了相伴而生的挑战后，我们从这项工作中收获了诸多益处，包括安全性的提升、升级的简化，以及对云提供商服务中断更强的抵御能力。
