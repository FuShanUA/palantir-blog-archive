Ontology-Oriented Software Development

2024-01-22 · Original →

作者：Peter Wilczynski，本体系统产品负责人

“给我看激励，我就能告诉你结果。” —— 查理·芒格

一、

就整个行业而言，我们正以不断加快的速度生产越来越多的软件，然而这些软件对经济生产力的影响却微乎其微。

这种停滞不前是软件行业自身组织方式的必然结果——绝大多数人力资本都集中在开发模块化组件上，而非将这些组件集成为能够真正影响业务结果的、具有凝聚力的企业系统。我们贬低了这种集成工作，反而将组件捧上了“系统”的地位，却没有正确地将其视为构成更大拼图的工具性碎片。数字化转型本身成了目的，而非由熟练从业者践行、以业务自身定义的目标为终点的旅程。

我们迷失了方向，自欺欺人地相信：既然构建单个零件更容易，那总会有人能把它们组装成有价值的东西。

由此产生的软件架构是为风险资本家获取回报而优化的，而非为了实现客户的目标；它奖励单个组件供应商之间的零和竞争，而不是鼓励关注整体系统性能的合作伙伴之间进行正和协作。尽管这些单个组件在过去二十年里有了极大的改进，但矛盾的是，这是以牺牲整个企业系统为代价的。孤立地优化组件，使得跨多个分散组件同步变更变得越来越困难。结果，组件层面的改进未能转化为系统层面的提升。

尽管被推销了模块化的概念，但这个软件工业复合体的客户收到的却是支离破碎的产品。这些投资不仅没有带来敏捷性，反而造就了僵化的企业架构。¹ 这种技术僵化渗透到更广泛的组织文化中，导致了停滞。

在 Palantir，我们的使命是释放技术驱动进步的解放力量——通过赋能组织构建定制化软件工具，催化广泛的企业转型。令人遗憾的是，我们很快意识到，没有任何单一产品能在整个经济领域中实现这一目标；也就是说，这个使命无法通过消化一套通用的需求并向所有客户输出相同的“封装软件”来实现。这种定制化的软件工程必须在现场进行。

为了使其在经济上可行，我们需要构建一种技术，能将定制企业软件开发的边际成本降至零。

支撑这一点的架构是 Palantir 本体，在 Akshay 之前的博文中有所描述。这项技术及其软件开发工具包（我们称之为“Ontology SDK”或“OSDK”），通过将孤立组件集成到一个整体系统中，使企业去碎片化成为可能。通过实施一个以决策为中心的本体，协调来自整个 IT 领域的数据、逻辑和行动要素，组织终于可以开始创造性地思考其技术资产与业务优先级之间的交叉点。

二、

设想一家航空公司试图减少停机时间并消除因机械故障导致的延误。

在常规的航前检查中，一旦警告灯亮起，压力便开始累积，整个组织的人员开始评估各种方案。必须在安全、员工健康和客户影响之间进行权衡，这些涉及实际取舍的复杂决策需要仔细衡量。

这个问题是各行业普遍存在的资源分配问题的一个具体例子。虽然资源可以按照统计上的最优分布进行主动分配，但随着实地情况变化重新分配这些资源，才是衡量企业敏捷性的关键：企业能多快对现实做出反应？

在我们的假设航班示例中，这涉及回答三个问题：

  • 飞机需要停飞吗？
  • 哪些机场有合适的替换零件和机械师来进行维修？
  • 哪些方案能让我们在最小化客户影响的情况下更换飞机？

利用来自飞机的物联网数据（每次飞行可产生超过两百万个数据点），我们需要了解飞机还能飞行多少小时。为此，我们的数据科学家可以加载物联网数据并运行预测性维护模型，以获得剩余服役寿命的初步估计。

假设飞机可以起飞，我们就需要开始规划维修。这意味着我们的运营团队需要找出哪些机场有合适的替换零件库存，并将该数据与维修人员的时间表进行交叉比对，以找到有能力进行维修的持证机械师。我们的目标不是拖延时间，而是立即更换飞机，使其能尽快到达一个有能力尽快完成维修的目的地，最理想的情况是不影响任何客户。

基于这些标准，我们可能会确定两个更换方案，使得飞机降落在 IAD 或 DFW。DFW 满足维修的所有约束条件，但由于机身有不同的座椅配置，我们最终将不得不拒载三名乘客，这会让我们损失 5000 美元，还会导致客户满意度下降。IAD 有需要更换的零件和一个空闲的机库可用于维修，但在飞机到达时，目前没有安排具备特定技能 (Skills) 的机械师；要在一夜之间完成维修，需要我们安排一名机械师加班，这将增加 2000 美元的成本。

第一种方案更简单；第二种最优方案需要更换飞机并与机械师协调——也就是说，需要更多的上下文和更多的自由度。

以编程方式寻找最优解，需要对许多不同的数据、逻辑和行动要素进行协调编排。数据要素来自机场运营数据库、航班舱单和资源管理系统。这些数据要素用作逻辑要素的输入，这些逻辑要素可以预测影响、模拟客户行为并评估风险。最后，这些逻辑要素的输出用于构建行动要素，将决策写回相应的运营系统，并向正确的人员发送实时通知。

将此解决方案转化为当今碎片化企业架构上的一个应用，需要处理来自核心事务型业务系统的数据，将其与 SaaS 工具和内部定制系统相协调，并跨越价值链的多个不同环节编排行动。实施此方案的工程师需要学习每个系统的 API，与每个组织的 IT 团队建立连接，并构建一个直接与每个系统交互的应用。

这些知识——如何与系统交互、如何协调它们不同的数据模型、如何访问它们彼此脱节的逻辑要素以及如何编排它们的行动系统——将存在于又一个碎片化的应用或服务中，组织内其他试图执行类似任务的团队无法访问它。虽然 API 网关在启用 API 发现方面可以发挥重要作用，但程序员仍然不得不在应用层使用胶水代码来协调特定于系统的表示形式。

本体将这些知识集中起来，并封装在一个共享系统中。这个系统连同 OSDK，随后作为一个更高层次的抽象，用于编写业务逻辑，这些逻辑运行在由关键业务概念、运营流程和真实世界任务组成的协调层上。这使得从以组件为中心的表示形式向共享概念模型的转换只需进行一次，而不是每次构建新应用时都要重复转换。

而且这种知识会迅速累积；通过依赖以决策为中心的共享本体，新应用可以利用已有的系统集成工作，而新组件也可以通过将其数据、行动和逻辑要素与现有表示相协调，快速接入系统。²

考虑上面的例子。在本体出现之前的世界里，程序员需要找到合适的组件，然后使用特定于组件的接口、语言和/或方言与每个组件进行交互。因此，代码本身从根本上是以组件为中心的：

const mysql = require('mysql');
const nodemailer = require('nodemailer');
const fetch = require('node-fetch');

// 库存数据库连接配置
const dbConfig = {
  host: 'skyframe-internal',
  user: 'db_admin',
  password: process.env.DB_PASSWORD,
  database: 'skyframe'
};

// 邮件服务器配置（以 Gmail 为例）
const transporter = nodemailer.createTransport({
  service: 'gmail',
  auth: {
    user: 'swapbot@skyframe.com',
    pass: process.env.EM_PASSWORD
  }
});

// 连接到库存数据库并查询有库存的机场
async function findAirportsWithInventory(parts) {
  const connection = mysql.createConnection(dbConfig);
  const query = "SELECT airport_code FROM inventory WHERE part_number IN (?)";
  return new Promise((resolve, reject) => {
    connection.query(query, [parts.map(part => part.partNumber)], (error, results) => {
      connection.end();
      if (error) return reject(error);
      resolve(results.map(row => new Airport(row.airport_code)));
    });
  });
}

// 连接到数据库并查询可更换的航班
async function findSwappableFlights(flightToSwap, aircraft, targetAirports) {
  const connection = mysql.createConnection(dbConfig);
  const query = `SELECT * FROM flights
                 WHERE departure_airport_code = ? AND aircraft_id = ? AND destination_airport_code IN (?)
                 ORDER BY departure_time ASC`;
  return new Promise((resolve, reject) => {
    connection.query(query, [flightToSwap.departureAirportCode, aircraft.id, targetAirports.map(airport => airport.id)], (error, results) => {
      connection.end();
      if (error) return reject(error);
      resolve(results.map(row => new ScheduledFlight(row)));
    });
  });
}

// 通过调用外部 REST 端点计算最佳更换方案的函数
async function calculateOptimalSwaps(flightToSwap, swappableFlights) {
  const requestBody = {
    flightToSwap: flightToSwap,
    swappableFlights: swappableFlights
  };

  const response = await fetch('https://skyframe-internal-gateway/api/calculateOptimalSwaps', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': 'Bearer Ym9zY236Ym9zY28'
    },
    body: JSON.stringify(requestBody)
  });

  if (!response.ok) {
    const error = await response.text();
    throw new Error(`Failed to calculate optimal swaps: ${error}`);
  }

  const swapOptions = await response.json();
  return swapOptions;
}

// 通过调用外部 REST 端点计算维修和客户影响成本的函数
async function calculateImpactCosts(swapOptions) {
  const requestBody = {
    swapOptions: swapOptions
  };

  const response = await fetch('https://skyframe-internal-gateway/api/calculateImpactCosts', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': 'Bearer Ym9zY236Ym9zY28'
    },
    body: JSON.stringify(requestBody)
  });

  if (!response.ok) {
    const error = await response.text();
    throw new Error(`Failed to calculate impact costs: ${error}`);
  }

  const impactCosts = await response.json();
  return impactCosts;
}

// 执行更换飞机操作的函数
async function swapAircraft(targetAircraft, aircraft) {
  const requestBody = {
    targetAircraft: targetAircraft,
    aircraft: aircraft
  };

  const response = await fetch('https://skyframe-internal-gateway/api/swapAircraft', {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
      'Authorization': 'Bearer Ym9zY236Ym9zY28'
    },
    body: JSON.stringify(requestBody)
  });

  if (!response.ok) {
    const error = await response.text();
    throw new Error(`Failed to swap aircraft: ${error}`);
  }


  console.log(`Swapping aircraft ${targetAircraft.id} with ${aircraft.id}`);
}

// 发送邮件请求批准更换方案的函数（使用 nodemailer）
async function emailSwapsForApproval(swapOptions) {
  const mailOptions = {
    from: 'swapbot@skyframe.com',
    to: 'ground-control@skyframe.com',
    subject: 'Flight Swap Approval Required',
    text: `Approval is required for the following swap options: ${JSON.stringify(swapOptions)}`
  };

  return transporter.sendMail(mailOptions);
}

// 安排预防性维修的主函数
async function schedulePreventativeMaintenance(aircraft, parts) {
  try {
    // 假设 aircraft.getSchedule() 和 schedule.getNextFlight() 在别处定义
    const schedule = await aircraft.getSchedule();
    const flightToSwap = await schedule.getNextFlight();

    const targetAirports = await findAirportsWithInventory(parts);
    const swappableFlights = await findSwappableFlights(flightToSwap, aircraft, targetAirports);

    const swapOptions = await calculateOptimalSwaps(flightToSwap, swappableFlights);
    const impactCosts = await calculateImpactCosts(swapOptions);

    const cheapestOption = impactCosts.reduce((cheapest, option) =>
      option.cost < cheapest.cost ? option : cheapest,
      { cost: Number.MAX_VALUE }
    );

    // 方案的护栏
    if (cheapestOption.cost < 1000) {
      await swapAircraft(cheapestOption.swapOption.targetAircraft, aircraft);
    } else {
      await emailSwapsForApproval(swapOptions);
    }
  } catch (error) {
    console.error('Failed to schedule preventative maintenance:', error);
  }
}

这是一个刻意简化的例子——API 请求和响应类型被过度简化，底层逻辑和行动系统通过标准的 API 网关模式暴露，身份验证相对一致，错误处理也极为简单。即便如此，大部分代码都用于编写直接连接底层系统的函数；主要的调度函数只占代码总量的一小部分。

相比之下，在以本体为导向的开发模型中，协调各系统间数据、逻辑和行动所需的去碎片化代码存在于企业的本体中，从而大幅简化了客户端代码。³

使用本体软件开发工具包（OSDK），它以地道的 Python、Typescript 和 Java 暴露本体的数据、逻辑和行动要素，我们能够将大部分复杂性推入本体本身，并暴露出与实际业务概念的直接表示相对应的企业对象：

async function schedulePreventativeMaintenance(aircraft: Aircraft, parts: AircraftComponent[]) {
  const schedule: FlightSchedule = await aircraft.getSchedule();
  const flightToSwap: ScheduledFlight = await schedule.getNextFlight();

  const targetAirports: Airport[] = await OntologyClient.search()
    .airports()
    .filter(a => a.inventory.containsAll(parts))
    .all();

  // 从同一个出发地查找飞往有库存机场的航班
  const swappableFlights: ScheduledFlight[] = await flightToSwap.getDepartureAirport()
    .scheduledFlights()
    .filter(f => f.aircraft.airframe.exactMatch(aircraft.airframe))
    .filter(f => f.destination.isOneOf(targetAirports))
    .orderBy(f => f.departureTime.asc())
    .all();

  const swapOptions: FlightSwapOption[] = await OntologyClient.calculateOptimalSwaps(flightToSwap, swappableFlights);
  const impactCosts: SwapOptionCost[] = await OntologyClient.calculateImpactCosts(swapOptions);

  const cheapestOption = impactCosts.reduce((cheapestOption, option) =>
    option.cost < cheapestOption.cost ? option : cheapestOption,
    { cost: Number.MAX_VALUE },
  );

  // 方案的护栏
  if (cheapestOption.cost < 1000)
    await OntologyClient.action.swapAircraft(cheapestOption.flight, aircraft);
  else
    await OntologyClient.action.emailSwapsForApproval(swapOptions);

正如 Brooks 在其开创性论文中所说，没有银弹：系统集成充满了必须管理且无法消除的固有复杂性。但是，通过强制执行将这种复杂性封装在本体内的设计模式，以本体为导向的客户端代码能够在更高的抽象层次上运行。就像高级语言允许程序员抽象掉组成物理计算机的硬件组件的内部实现细节一样，以本体为导向的方法允许程序员抽象掉组成企业架构的软件组件的内部实现细节。

在这个例子中，程序员使用 OSDK 与这些业务概念进行交互，并用业务的语言编写代码——不是用行和列，而是用飞机、航班时刻表和机场。OSDK 并没有为我们的产品提供通用的 API——它提供的是一个工具包，用于用你的业务语言为你的业务构建 API。

三、

软件中更高层次的抽象总是具有颠覆性的，这恰好是 Clayton Christensen 所描述的意义——它们从解决最简单的程序开始，但由于其潜在的结构优势，这项发展中的技术随后能够实现跨越式发展，超越现有者。

借助本体，这种跨越式发展通过大幅提升我们构建定制应用的能力得以实现。任何使用过低代码可视化编程工具的人都知道，这些应用总会触达天花板。因此，在选择低代码开发工具时，你隐含的决定是限制你所构建解决方案的雄心。

在本体之上建立我们的应用构建平台，从三个方面改变了这种状况。

首先，它使低代码应用变得更加强大。低代码构建者可以在其低代码应用中利用基于代码的数据、逻辑和行动要素，将性能天花板推高至我们在任何其他低代码环境中都未曾见过的水平。这种能力使构建者能够直接在可视化编程环境中开发业务运营线应用。

其次，当构建者达到了低代码开发环境所能构建的极限时，OSDK 允许他们无缝迁移到传统的基于代码的开发环境中，而无需从头开始重新实现整个应用。

第三，本体使得构建旨在相互操作的低代码和专业代码应用变得容易。低代码和专业代码方法都有不同的权衡：低代码工具降低了构建门槛，使应用开发更容易上手，而基于代码的工具提高了上限，使高级用户能够做更强大的事情并构建更复杂的体验。通过本体综合这两种方法，让用户能够为手头的任务选择合适的工具。

最近，我们看到本体提供的相同结构优势在我们的 AI 平台 (AIP) 中得以体现，该平台提供了一套基础的用户端功能，以及一个用于构建定制 AI 系统并将其直接部署到运营工作流中的开发工具链。

为什么我们能如此快速地发布 AIP？

因为从一个关键的角度来看，AI 系统更像应用而不是用户——它们直接与 API 交互！

自计算机诞生以来，界面设计始终以让数字系统更易于人类使用为目标，无论是作为用户还是程序员。在过去的一个世纪里，我们从打孔卡过渡到命令行界面 (CLI)，又从 CLI 过渡到图形用户界面 (GUI)，始终将人类可用性作为北极星。

但当我们步入一个人类直接与功能日益完善的 AI 系统协作以实现其目标的世界时，我们对可用性的定义必须扩展，以允许自主系统承担日益复杂的任务。

基于大语言模型的系统是按照我们的形象创造的；像人类一样，它们可以使用基于文本的界面，并与世界的自然语言表示进行交互。但这种相似性具有欺骗性，且并非对所有 AI 系统都成立；事实上，如今并非所有 AI 系统都是如此——AlphaStar 在玩《星际争霸》时并不与像素或文字交互。⁴

通过将应用和数据系统解构为其构成的数据、逻辑和行动要素，本体暴露了一个单一的、以决策为中心的企业模型。这个共享模型可以作为一个统一表示，通过三种不同类型的接口暴露出来：

  1. 图形用户界面 (GUI)，为用户提供本体的可视化表示。
  2. 应用编程接口 (API)，为程序员和 AI 系统提供对本体的直接访问。
  3. 自然语言接口 (NLI)，允许人类和基于语言的 AI 系统使用自然语言与本 体进行交互。

正如 J. C. R. Licklider 在《人机共生》中所述：“人类语言与计算机语言之间的根本差异，可能是实现真正共生的最严重障碍。” Palantir 本体是应对这一语言挑战的技术解决方案：它不再将每种不同的接口视为一种独立的语言，而是将本体表示为一种单一语言，能够以图形、口头和程序化形式表达。

在企业语境下释放运营 AI 的潜力，这不是一个 AI 问题——而是一个本体问题。
