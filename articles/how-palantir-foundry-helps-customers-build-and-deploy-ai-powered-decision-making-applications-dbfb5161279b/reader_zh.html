How Palantir Foundry Helps Customers Build and Deploy AI-Powered Decision-Making Applications

利用数据做出更优决策，对于推动最佳业务成果至关重要。Palantir 赋能组织，使其能够从最有价值的资产——数据中快速提取最大价值。

如今，各行各业的组织比以往任何时候都更希望借助人工智能 (AI) 和机器学习 (ML) 解决方案，来进一步加速并优化决策过程。然而，将 AI 落地应用的现实过程可能既复杂又昂贵。

事实上，Gartner 依然预测，组织内部开发的《85% 的传统 AI 和 ML 项目》将产生错误的结果。即使模型成功开发出来，它往往也只是一次实验，而无法成为能够持续创造长期业务价值的运维工具。

这可能有以下几种原因：

  • 端到端的模型开发过程孤立存在，与领域专家严重脱节。
  • 基于静态一次性数据拉取的模型很快就会过时并失去准确性。
  • 模型在团队间交接时缺乏文档、流程和结构。
  • 模型以零敲碎打的方式部署在遗留系统和用户界面上。
  • 模型开发很大程度上是一次性过程，旨在获取洞察或进行实验，对实际业务结果不承担责任。

《Palantir Foundry》致力于解决 AI 在真实场景中的应用问题，而非仅仅停留在实验室阶段。如果没有值得信赖的数据基础、对机构决策的记录表示，以及从每次决策中学习的基础设施，就无法实现有效的 AI。

Foundry 提供了一个端到端、模块化且可互操作的平台来满足这些需求。正是这种模块化和可互操作的方式，构成了 Foundry 与 Amazon Web Services (AWS) AI 及 ML 工具最新集成的基础。

在这篇文章中，我们将解释如何利用 Palantir Foundry 和《Amazon SageMaker》做出更佳的 AI 驱动决策。

关于 Palantir Foundry

部署在 AWS 上的 Foundry 在底层利用了多种服务，例如 Amazon Simple Storage Service (Amazon S3)、Amazon Elastic Compute Cloud (Amazon EC2) 和 AWS Key Management Service (AWS KMS)。

Palantir 与 AWS 之间的合作，正助力全球领先组织开展关键业务运营——从《加速临床研究》和《支撑疫苗供应链》，到《提升电网的安全性与可靠性》。

以下是由 AWS 提供支持的 Palantir Foundry 所驱动的一些创新工作实例：

  • 全球领先的制药公司之一 Sanofi 使用 Foundry 提供核心数据基础设施和分析平台，作为其真实世界证据 (RWE) 研究平台的一部分，并因此获得了 2020 年 Gartner 医疗保健与生命科学创新关注奖。
  • 美国国立卫生研究院利用 Foundry 在一个安全的界面中整合了公共与内部研究数据。高通量筛选、基因组学及其他生物学数据被集成到该平台中，以推进国家癌症研究所和国家转化科学促进中心团队的研究工作。
  • bp 与 Palantir 延续了合作伙伴关系，以支持 bp 致力于在 2050 年或更早成为净零公司，并助力全球实现净零排放的目标。由 Palantir 驱动的 bp《数字孪生应用》已经创造了显著价值。如今，有机会应用这些应用来加速 bp 的新目标，优化风力发电场、电动汽车充电网络和太阳能发电，并支持实现 bp 净零目标的其他方面。

随着 Foundry 的不断演进，团队持续了解客户如何利用各个平台内的服务，并在技术栈的更高层发现了许多实现更紧密集成的机会。

近几个月来，Palantir 的软件开发团队与 AWS 密切合作，将这些更紧密的集成推向市场，实现了在应用生命周期的各个层级与 AWS 数据科学工具的深度集成。重点是将 Amazon SageMaker 系列 AI 和 ML 服务与 Foundry 的数据集成及决策模块结合在一起。

方案概述

让我们来探索 Foundry 与 AWS 的最新集成，并展示组织如何利用它们克服 AI 生命周期各个阶段（从数据准备到模型运维）常见的数据与 AI 挑战。

  • 数据准备：利用 Foundry 的数据连接、提取、转换、加载 (ETL) 及数据分支能力快速准备数据，然后通过 Amazon Athena 在《Amazon SageMaker Studio Notebooks》中使用这些数据，以分析数据并开发模型。
  • 模型开发：赋能技术人员和非技术用户，利用 Amazon SageMaker Studio Notebooks 和《Amazon SageMaker Autopilot》进行无代码模型开发，构建有意义的 ML 模型。
  • 模型集成：将来自《Amazon SageMaker Endpoints》、Amazon SageMaker Autopilot 和《AWS AI 服务》的模型集成到 Foundry 机器学习环境中。这得益于 Foundry 丰富的部署基础设施、监控、健康检查和本体集成。
  • 模型运维：在 Foundry 应用构建器和 Foundry 仿真引擎中使用 Amazon SageMaker 模型，以驱动运维应用和工作流，这些应用和工作流受益于反馈循环和模型重训练，并能产生长期的组织价值。

数据准备

在训练新模型之前，必须连接到相关的底层数据源并准备数据以供使用。

数据收集和准备占据了数据科学家的大量时间，导致他们几乎没有剩余带宽去进行算法设计、算法测试和机器学习等高价值活动。

Foundry 的技术和双向开箱即用数据连接器（包括 Amazon S3、Amazon SQS 和 Amazon RDS 等 AWS 服务）从根本上减少了在准备高质量数据基础以开始构建模型上所花费的时间。

Foundry 的数据集成能力帮助组织将源数据转化为企业的数字孪生，这由 Palantir 独特的语义模型“本体”来表示。

本体将所有底层数据呈现为反映业务真实组件（如工厂、产品和客户）的对象。作为一个经过策展、验证、连接且可复用的模型框架，这极大地加速了新模型的 AI 生命周期。

本体将用于支撑运维应用程序，而支撑这些数据对象的集成表将用于训练我们的 ML 模型。

模型开发

Foundry 采用开放且可互操作的模型开发方法，允许用户在 Foundry 的 Code Workbooks 中原生实现其模型，或选择任何第三方应用程序。

Foundry 的核心原则之一是使数据科学家和业务用户（或其他非数据科学专家）能够协同利用数据和模型做出更好的决策。

技术背景较弱的用户通常是领域专家，他们通过 ML 工具包、无代码模型开发应用程序和开箱即用的 AI 服务获得赋能，这些工具对 Foundry 为数据科学家提供的全套端到端 AI 建模工具起到了补充和加速作用。

借助 AWS，希望在《Amazon SageMaker Studio Notebooks》中使用 Foundry 数据的用户，可以通过 Foundry 与 Amazon Athena 之间利用《Amazon Athena 联邦查询》的新集成来实现。

使用原生的《Amazon SageMaker Athena SQL 集成》，用户能够以高效的方式将 Foundry 中经过策展的数据资产直接查询到他们的 Notebook 中，随后进行转换并用于模型开发。Amazon Athena 为众多额外的 AWS《应用程序和服务》提供联邦连接。

这种架构对于用于带有即席或交互式查询的数据科学工作流的数据集特别有利，因为连接可以快速配置，且不需要任何额外的同步或管理。

对于寻求更自动化方法的用户，Amazon SageMaker Autopilot 消除了构建 ML 模型的繁重工作，帮助您根据数据自动构建、训练和调整最佳 ML 模型。

Foundry 与 Autopilot 之间的集成允许用户通过 Foundry 的用户界面直接利用本体中的数据启动 Autopilot 作业。这为用户提供了一种只需点击几下即可训练、评估模型并将其部署到生产环境的方法，无需编写代码。

模型集成

通过 Amazon SageMaker Endpoints 部署的模型（加上预训练的 AWS AI 服务，如《Amazon Textract》、《Amazon Translate》和《Amazon Lookout for Equipment》）现在可以连接到 Foundry ML，并以与原生 Foundry 模型相同的方式绑定到 Palantir 本体对象。

用户只需浏览正在运行中的 Amazon SageMaker Endpoints 并选择其模型。这将在 Foundry 中创建一个代表与该 Amazon SageMaker Endpoint 连接的新模型。

您的 Amazon SageMaker 模型和 AWS AI 服务现已成为 Foundry 中的一等公民，它们可以受益于与原生 Foundry 模型相同的部署基础设施、监控和健康检查。

使该模型投入运维之前的最后一步是将其集成到本体中。只需将模型的输入和输出参数绑定到本体中的数据属性即可。这将生成一个由模型组成的连接网络，这些模型通过属性关系链接在一起。

原生本体集成确保您的 Amazon SageMaker 模型可以被协作用于更广泛的组织工作流、仿真或面向终端用户的应用程序，从而放大其可用性和组织影响力。

模型运维

从这里开始，Amazon SageMaker 模型可以集成到 Palantir 运维应用程序中，用于驱动 Foundry 中的终端用户工作流。

这正是许多 AI 和 ML 项目可能因为我们在本文开头概述的原因而失败的地方。然而在 Foundry 中，一旦模型被集成到本体中，任何用户都可以通过简单地使用模型输出，将其整合到运维应用程序和工作流中。

这些用户可能对模型运行的领域非常熟悉，但不需要是模型开发方面的专家，也不需要是调用模型方面的专家。

Foundry 仿真引擎处理所有的数据加载和模型编排。与 Amazon SageMaker 的集成透明地管理两个平台之间的数据交接和模型调用。

Foundry 仿真引擎允许用户使用假设的输入和输出对这些模型运行“假设”仿真，以为正确的行动方案提供信息。任何决策及其所处的上下文都将被写回到本体中，并在适用时写回到原始数据源。

这种反馈循环支持模型重训练，使平台能够随着时间的推移不断改进决策和建议。

例如，bp 使用 Foundry 的高级仿真引擎自动识别增产机会。工程师提出“假设”问题，例如什么样的油井、设施和路由配置才能实现石油产量最大化。这通过每天增加 30,000 桶的产量，每年创造数亿美元的额外收入。
