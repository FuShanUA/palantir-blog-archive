Production Infrastructure at Palantir: User-minded API design

2022-12-08 · Original →

引言

当下，各类组织热衷于给自己贴上“结果导向”的标签，即要求组织成员对自身工作的最终成效负责。Palantir 的生产基础设施团队致力于围绕用户的最终结果来开展工作——但这究竟如何转化为优秀的软件开发实践？

用户能否取得预期的结果，往往取决于我们提供的 API 复杂程度。以 Kubernetes 为例，其强大且通用的 API 需要使用者具备深厚的经验积累才能有效驾驭。在设计功能特性时，工程师们往往倾向于构建更加通用、“强大”的 API，却在此过程中牺牲了易用性，我们必须警惕这种倾向。本文将探讨 Palantir 如何将工程精力聚焦于赋能用户结果，并通过一个小型案例，展示这种理念如何影响生产基础设施团队的功能规划与 API 设计。

问题陈述

功能需求通常针对特定的用户结果，且会随着时间的推移不断涌现。如果有人认为软件的首个正式发布版就必须功能完备，工程师们会觉得这纯属无稽之谈。因此，产品开发团队理应警惕将功能或 API “过拟合”到当前需求上。为了避开编写定制化、不可复用功能的陷阱，开发者们努力实现更通用、更灵活的解决方案。这听起来无可厚非，但存在一定的局限性：

  • 将用户的问题陈述过度泛化，可能会将其转化为一个用户根本不存在的问题
  • 赋予 API 过多的灵活性/权力，会导致用户为了正确使用它而不得不自行构建专业知识体系

面对这些挑战，我们如何才能避免交付那些过度泛化、却无法真正服务用户的功能特性呢？

倾听用户

Palantir 生产基础设施（PI）团队的内部流程有助于确保我们设计的软件能够切实满足用户的真实需求。在开展每轮功能开发前，我们都会发起“意见征集”，即 RFC。每份 RFC 都包含问题陈述和提议的解决方案。我们鼓励 RFC 审阅者像审视解决方案一样，严格把关问题陈述。这种审查机制有助于回答几个关乎设计决策可持续性的关键问题：

  • 我们是否准确提炼并整合了用户的需求？
  • 如果我们要对用户需求进行泛化，这是否会给用户带来过重的使用负担？
  • 泛化后的问题陈述是否暴露了本应作为内部实现的概念？

如果没有具体的案例，探讨这些问题往往会流于空泛。PI 团队认为，用户的最终结果取决于无数微小设计决策的总和，因此我们将聚焦于近期的一项工程实践。具体而言，我们将审视该实践如何确定一个恰到好处的 API——既能充分表达用户需求，又不会让用户背负过多复杂性。

实践：Spark 应用调度优化

PI 团队今年早些时候致力于优化 Kubernetes 集群中 Spark 应用 Pod 的调度机制，这为我们提供了一个审视系统化 API 设计的契机。在此领域我们已有先前的技术积累：我们曾实现了以先进先出方式进行 Spark 应用 Pod 的成组调度，并基于我们对由临时计算资源组成的 Kubernetes 集群的管理能力，将计算节点仅分配给健康的可用区。

在此基础之上，我们希望支持将特定 Spark 应用的所有 Pod 调度到同一个可用区内。我们的动机如下：Spark 应用的各 Pod 之间需要传输大量数据，将同一应用的所有 Pod 置于同一可用区内可以消除高昂的数据传输成本。

我们提议的解决方案将利用 Spark 调度器与集群自动扩缩容组件之间现有的 API（强烈建议在继续阅读前，先参阅我们关于《Kubernetes 中的 Spark 调度》的博文！）：

  • ResourceReservations，用于在计算节点上为成组调度的 Spark 应用 Pod 预留资源
  • Demands，用于表示未被满足的计算资源需求

为了将应用完全调度在单个可用区内，我们的 Spark 调度器可以利用 K8s 标签按可用区列出可用节点，随后按可用区逐一调用其现有的装箱逻辑：

这种方法适用于至少有一个可用区具备足够空闲计算容量来容纳该应用的情况，但如果情况并非如此呢？Spark 调度器此前一直使用 Demands 来描述此类未被满足的资源需求，但如今我们的自动扩缩容组件必须在单个可用区内满足特定的 Demand。因此，在运行“单可用区”调度模式时，我们的调度器需要一种方式来表达对 Demand 的可用区约束。针对这一特定用例，扩展 Demand API 的最佳方式是什么？

权衡方案

要让调度器请求仅在单个可用区内满足 Demand，有以下几种 API 方案：

  • 在 Demand 规格中新增字符串字段 Zone：调度器使用该字段来指定其希望满足 Demand 的具体可用区
  • 在 Demand 规格中新增布尔值字段 EnforceSingleZoneScheduling：当调度器希望 Demand 仅在单个可用区内被满足时，将其设为 true

第一种方案颇具吸引力；在赋予调度器对调度决策的控制权方面，它显得更加“强大”。然而，它存在一些显著的弊端。

调度器需要负责为每个 Spark 应用挑选可用区，但它如何才能可靠地做到这一点呢？它需要哪些额外信息才能做出决策？事实上，这需要关于各可用区健康状况和可扩展性的信息。此前，这些信息一直被封装在自动扩缩容组件的内部 API 之后。往好的方面看，泄露这些信息会迫使用户（即调度器）重复实现自动扩缩容中已有的逻辑；往坏的方面看，调度器可能实现出存在微妙偏差的逻辑版本，进而导致持续性的调度失败等生产级故障。无论如何，这都使得原先的内部 API 暴露给了调度器；这种暴露引入了依赖关系，从而制约了自动扩缩容组件未来的功能开发。

这种 API 设计陷阱源于对用户（调度器）具体目标的误解：用户的真实意图是获取足以在单个可用区内调度整个应用的计算资源，至于是哪个可用区则无关紧要。我们一味追求更通用、“强大”的 API，反而给用户和集群自动扩缩容的维护者带来了更糟的结果。

第二种方案，即 EnforceSingleZoneScheduling 布尔值，规避了上述弊端，同时提供了恰好足够的表达能力来实现用户期望的结果。集群自动扩缩容组件向调度器屏蔽了更多信息，从而保障了未来开发工作的灵活性。

事后来看，这个决策似乎很简单，但它源于对用户期望结果的严格审视与提炼，并基于该结果构建解决方案。

总结

作为软件工程师，我们天生倾向于“通用”或“强大”的 API，并常常将这些标签与用户成功混为一谈。但在某些情况下，这会对用户和开发者造成损害：用户背负了更重的知识负担，开发者则泄露了内部抽象，进而影响未来的开发效率。API 的真正价值在于它能多有效地实现用户的目标。正如上文所示，Palantir 的 PI 团队致力于在任何规模下提炼并满足用户的期望结果，哪怕是在最微小的细节中也是如此。
