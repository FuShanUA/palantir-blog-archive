Frontend Engineering at Palantir: Plotlines in Three.js

2026-04-07 · Original →
**关于本系列** Palantir 的前端工程远不止是构建标准的 Web 应用。我们的工程师设计用于关键任务决策的界面，构建将洞察转化为行动的操作应用，并创建处理海量数据集的系统——不仅要考虑用户需要什么，还要考虑在网络不稳定、风险高且容错率为零的情况下他们需要什么。 本系列将揭开这项工作的真实面貌：我们解决的技术问题、产生的影响以及采取的方法。无论你是单纯好奇，还是正在寻找加入我们的机会，这些文章都将为你真实展现我们前端团队的工作日常。在这里查看所有前端工程博文。 在这篇博文中，驻纽约的前端工程师 Lasse 将分享 Palantir 工程师负责的各种 3D 工作流概述，以及应对性能挑战的一些独特方法。敬请关注第 5 部分。 去年 9 月是 Stripe 启动《地球仪电影宇宙》在其平台上渲染实时采购订单的五周年。三个半月后，GitHub 也加入战局，推出了《他们自己的地球仪》，展示了发起 PR 的位置与合并位置之间的连线。 事实证明，许多公司都有实时流转的地理空间数据。这往往是一种非常引人入胜的方式，可以展示数据在网络中传输了多远，或者一家公司的业务范围如何远超其服务器托管地。就 Palantir 而言，Gotham 也处于同一领域。这里前端工程师的一大重点是渲染 Gaia 应用中《具有地理空间属性的 Ontology 对象》，以可视化国防和情报行动。或者更简单地说，“渲染一切”。 最近，我们团队一直在开发 Gaia 的 3D 实现。它由 Zodiac 驱动，这是我们使用 Three.js 构建的一个库，以及一组基于 Foundry 的 API 来泛化渲染。简而言之，Foundry Ontology 中存在的任何带有位置数据（点、形状或样式）的“对象”，都会通过 Gotham 的地图渲染 API 提供服务。 去年我刚加入 Palantir 时，很难评估我们将面临哪些针对客户的 3D 挑战。这篇博文详细介绍了我们遇到的一些工作流和性能挑战，以及我们在浏览器中解决这些问题的方法，该浏览器可供国防部（DOD）的任何人访问。为了便于说明，示例中包含了一些过度简化的内容。 ## 3D Gaia 3D 为用户体验地理空间数据引入了新维度。它允许用户像真实世界那样建立对空间的概念理解。渲染高空物体、解决极地畸变问题以及太空中的物体，都需要墨卡托投影无法提供的空间可视化能力。 例如，Starlink 每年贡献数百次卫星发射。险些相撞、轨道投影和太空碎片都属于各种 Gaia 用户需要可视化的跟踪范畴。通常，这些 3D 工作流需要与 2D 相同的工具来进行简报和可视化。 ### Zodiac 与我们的大球体 架构划分十分清晰：后端负责数据存储、过滤和繁重的计算，而前端专注于渲染。后端向下发送简化后的几何图形和样式信息——客户端不需要知道数据是如何计算的，只需知道在哪里绘制以及它应该是什么样子。 有了这个服务层（此处略去达成该层所做的工作），我们拥有了一套组件，可以将物体放置在某个位置并查看它。我们从 API 获取泛化后的可渲染对象，并在每一帧将其渲染在地图上。 ```typescript interface EarthLineItem { id?: string; points: LatLngHeight[]; color?: string; } interface LineVisualProps { color?: string; lineWidth?: number; opacity?: number; dashed?: boolean; } ``` 使用这些接口，绘制一条线非常简单： ```tsx ``` 这些构成了基础构建块，借此我们可以探索非常复杂的项目。高密度线集合可以通过分组几何进行聚合，自定义颜色和样式可以改变线条的样式，并且可以根据时间或其他过滤器切换可见性，而无需重建几何图形。 `EarthLineGroup` 按样式将线条批处理为单次绘制调用。10,000 条具有相同样式的线条变成一次 GPU 操作。 ```tsx {flights.map((flight) => ( picked && setSelectedFlight(flight.id)} /> ))} ``` 在 Zodiac 的底层，上述对象是通过实现 `ZodiacObject` 的类构建的，该类抽象了对地球上位置的管理，包括旋转、坐标和逐帧变化。利用这一点，对象实现可以完全专注于与渲染对象相关的属性，如视觉效果和定位。 ```typescript interface ZodiacObject { // Called on initialization of the object into the scene. onAdd(context: ZodiacContext): void; private currentBatch = 0; private batchSize = 1_000; private mesh: InstancedMesh; public onRender(time: number): void { const batch = this.getSatelliteBatch(this.currentBatch); for (const satellite of batch) { this.updatePosition(satellite, time); } this.currentBatch = this.nextBatch(); } private updatePosition(satellite: string, time: number): void { const position = this.propagateSgp4Position(satellite, time); this.mesh.setPositionAt(satellite, position); } // Later code blocks leverage our satellite mesh to "attach" to // the current position and corresponding TLE used on the map. public getPosition(satellite: string, out: Vector3): void { this.mesh.getPositionAt(satellite, out); } } ``` 在太空中，定位工作也有所不同。地球轨道可以使用双行轨道根数进行推算，这是一种可追溯至 1960 年代的格式，用两行 69 字符的文本编码轨道参数。客户端处理用于渲染的基本推算，但更繁重的轨道数学运算，如碰撞分析、长期预测和机动规划，则在 Foundry 计算模块的服务端运行。 ### 作为轨道路径的线条 当你观察以时间为函数进行定位的物体时，查看物体在 `T` 时间单位后的位置通常很有帮助。对于我们的可视化，我们允许选定的卫星显示其在地球表面及周围的位置，以估算其路径。然而，这种做法开销很大。 我们最初在选中时计算约 2 天的路径，但每一帧都会导致一个全新的向前推算。这在单一用例中可行，但在性能较弱的硬件上或选中多个对象时就会捉襟见肘。 因为我们观察的是时间函数，所以大部分推算结果可以重用。通过一些抽象，我们可以在任何需要它的对象之间共享轨道位置，并随着时间的推移计算新的区段。 ```typescript abstract class OrbitPath extends ZodiacObject { private.geometry: LineGeometry; private material: LineMaterial; protected points: CircularBuffer; protected timePerPoint: number; public onRender(time: number): void { if (this.needsFullRecompute(time)) { this.computeFullOrbit(time); return; } this.advanceQueue(time); } private advanceQueue(time: number): void { while (this.points.head().timestamp < time) { const point = this.points.shift(); point.timestamp = this.lastTimestamp() + this.timePerPoint; point.position = this.propagatePosition(point.timestamp); this.points.push(point); } } } ``` 队列随时间向前滑动。当一个点从队列前端移出时，我们会为后端的下一个区块重新计算它。使用旧方法，我们的地图每秒需要执行数千次 SGP4 推算。借助队列，我们可以减少为每个选定对象所做的 95% 的计算。 这使得可以同时渲染更多的轨道路径。与我们的卫星网格并排，我们可以监听用户拖动时间轴时的 TLE 更新，以确保使用正确推算的轨道根数。 ### 视线 展示卫星集合与地面物体之间的视线也很有帮助。卫星通常在系统中协同工作，可视化它们之间的“链路”非常有用。某些卫星配置只会在它们之间没有被地球遮挡时才渲染配对线。 使用之前相同的 Satellite 辅助对象，现在更容易转化出类似的连线。 ```typescript class SatelliteConnectionLine extends ZodiacObject { private.geometry: LineGeometry; private material: LineMaterial; constructor( private primarySatellite: string, private secondarySatellite: string, private getPosition: (id: string) => Vector3 ) {} public onRender(): void { const posA = this.getPosition(this.primarySatellite); const posB = this.getPosition(this.secondarySatellite); const visible = this.hasLineOfSight(posA, posB); if (visible) { this.geometry.setPositions([posA, posB]); } this.material.opacity = visible ? 1.0 : 0.0; } } ``` 每一帧，我们从共享上下文中获取当前位置并检查视线。这是一种射线与椭球体相交测试：如果两颗卫星之间的射线与地球相交，我们就隐藏该连线。 ### 带角度的传感器可视化 我们为卫星工作绘制的最后一种线条涉及传感器可视化。可视化卫星能力需要一套更复杂的线条。我们创建了一个与卫星运动匹配的 3D 多边形，并具有各种规格以匹配方向、尺寸和能力。通常，更“复杂”的形状是多个基本形状的聚合。对于传感器，我们构建了一个线条框架和一些不透明的平面来展示覆盖区域。 由于卫星绕地球运行，我们的地球扫描带应匹配传感器在整个轨道上可以扫描的位置。正确的路径是聚合卫星位置、地球自转、对地球的瞄准以及传感器旋转的任何属性。将这些组合成一个对象后，得到的是一条更符合传感器“能力”的轨迹，而不是粗略的飞越路径。 上面是我附在 PR 中的图片，用于解释扫描带旋转背后的数学原理。通常，这方面最快的工作流是在笔记本上头脑风暴步骤，然后将数学公式交给带有一些测试用例的大语言模型（LLM）。 在计算了最大宽度、在射线投射点之间构建多边形并进行推算后，你就得到了一个相对准确的卫星视野表现。随着时间戳的变化，卫星将更紧密地贴合路径。 上述可视化基于完全标称的传感器规格。 ## 回到地球 Gotham 作为一个平台，为常见模式和数据形状构建了许多抽象。随着我们的 3D 库复杂度增加，它需要引入与 Foundry 数据源相匹配并在我们 2D 场景中已经渲染的 API。 这可能涉及一套基于时间和属性的数据，但由于在父集合中渲染大量线条，渲染变更的开销更大。这促使我们采取了几种不同的实现方法，以确保我们的地球仪感觉响应迅速。 ### 基于时间过滤的线条 线条通常带有时间数据。飞行路径有出发和到达时间，船舶航迹有观测窗口，传感器覆盖范围有可见时段。当用户在时间轴上拖动时，线条需要相应地出现和消失。 一种初步方法可能涉及每一帧剔除并重建几何图形。对于 60fps 下的数千条线，客户端在重新分配数组时会产生大量开销。相反，我们选择附加一个简单的过滤函数、一些时间过滤中间件和一个着色器，以更高效地管理可见性状态。对于原生应用了时间接口的 Foundry 数据，客户端可以按每条线自动配置这些内容。 ```typescript // Most commonly a range function coming from Foundry. type TimeFilter = (time: number) => boolean; interface TimeFilterableLine { points: LatLngHeight[]; isVisibleAt?: TimeFilter; } const flightPath: TimeFilterableLine = { points: waypoints, isVisibleAt: (time) => departureTime < time && time < arrivalTime, }; ``` 几何图形只构建一次。当时间发生变化时，我们评估每个过滤器并更新可见性，而无需触碰顶点数据。过滤器可以每一帧运行，但我们只在可见性实际发生变化时才触碰 GPU 缓冲区： ```typescript class TimeFilterableLineCollection { private geometry: TimeFilterableGeometry; private material: TimeFilterableMaterial; // Includes opacity shader. private filters: Map = new Map(); private visibility: Map = new Map(); applyFilters(time: number): void { for (const [index, filter] of this.filters) { const isVisible = filter(time); const wasVisible = this.visibility.get(index) ?? true; if (isVisible !== wasVisible) { this.geometry.setVisibility(index, isVisible); this.visibility.set(index, isVisible); } } } } // A custom shader reads opacity and discards invisible fragments. const fragmentShader = ` if (opacity < 0.01) discard; // Gone, not rendered gl_FragColor.a *= opacity; `; ``` 当可见性确实改变时，我们更新 GPU 上的每段不透明度缓冲区。每条线映射到一个线段范围，我们可以在 0 和 1 之间切换它们的不透明度： ```typescript // The geometry stores an opacity value per line in a GPU buffer. class TimeFilterableGeometry { private opacities: Float32Array; setVisibility(lineIndex: number, visible: boolean): void { this.opacities[lineIndex] = visible ? 1.0 : 0.0; this.opacityBuffer.needsUpdate = true; } } ``` ### 动画线条 过去一年中我最喜欢的迭代周期涉及向底层 Gaia 地图暴露 API 以操作 Three.JS 对象。我们在动画方面遇到了一系列类似的问题。与太空中的视线类似，在许多场景下，用户可以在其工作流中看到不同的视觉变化。 在我们的一个用例中，提供转换状态的实时动画对于保持客户对态势的顶层域感知至关重要。当地图上管理的单元数量过多时，使用地图的传统可视化比卡片、表格和提示框更有效。 我们最终确定的 API 传递了一个回调函数，该函数接收当前时间和动画开始后的经过时间。你返回任何需要更改的样式属性： ```typescript const pulseCallback = (currentTime: number, elapsedTime: number) => { const progress = (elapsedTime % 2000) / 2000; const pulse = Math.sin(progress * Math.PI * 2) * 0.5 + 0.5; return { color: baseColor.clone().lerp(targetColor, pulse) }; }; ; ``` 回调通过同步到 3D 渲染循环的 `onPrerender` 钩子在每一帧运行。只更新材质属性，不重建几何图形。 对于状态驱动的动画，回调可以从 React 状态派生其行为。代表一项任务的线条可能会在等待确认时脉冲闪烁，然后在确认后渐变为纯色： ```typescript const animationCallback = useCallback( (currentTime: number) => { if (phase === "awaiting") { return { color: createPulsingColor(baseColor, white, currentTime, 1000) }; } return { color: confirmedColor }; }, [phase] ); ``` 这使得 3D 渲染与 UI 状态解耦，同时仍允许状态变化驱动视觉转换。用户可以利用颜色变化、闪烁、不透明度更新和方向虚线变化。将这些动画聚焦于底层的视觉变化，并允许平台利用属性映射到状态，使 3D 团队能够专注于底层实现，而无需破译特定的工作流。 ### 展望未来 最终，3D 领域有许多挑战需要解决，我们一直在迭代 API，以达到用户友好性、与更广泛平台集成以及性能的正确平衡。这些类型的功能通常需要监控和多次反复才能完善，随着我们接入客户的新数据源和工作流，这也变得更有趣。 展望未来，我们对 Foundry 在 3D 中支持渲染属性感到兴奋。2D 传感器范围的历史视觉现在可以作为体积或点云呈现，并且在极地等投影区域的工作流也更容易获得。我们将继续支持平台拥有的所有类型的数据源，目标是让用户为其需求选择最有效的可视化方式。这可以是从地面上的对象模型到跨越太阳系的地图的任何事物。 回首过去，我在考虑哪种教育对我的工作影响最大。鉴于 Palantir 专注于地理空间工作，毫不意外，我初中几何课每周带来的回报最大。 *在 Palantir 文档中阅读更多关于 Gotham 平台的信息：*https://www.palantir.com/offerings/defense/secure-collaboration/capabilities/ *如果这听起来像你感兴趣的项目和影响力，请查看我们今天的空缺职位：*https://www.palantir.com/careers/open-positions/*。我们最相关的前端职位是“Web 应用程序开发者”岗位。我们目前还在招聘以下两个特定职位：*《软件工程师 — 核心接口》*（帕洛阿尔托）和*《软件工程师 — 国防应用》*（华盛顿特区）。*
