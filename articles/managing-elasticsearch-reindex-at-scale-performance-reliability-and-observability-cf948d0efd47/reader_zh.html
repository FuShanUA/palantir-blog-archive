以下内容来自 Gotham 核心平台团队针对“基础”系列文章的客座投稿，该团队负责构建和维护 Gotham 生态系统内关键任务应用的基石。这篇博文由驻加州后端开发工程师 Kevin Liang 撰写，重点介绍了针对 Elasticsearch 重建索引机制的设计考量与改进——该机制旨在提供一种易用、高性能、可靠且可观测的方式，用于修复和重建后端应用的搜索索引。本文的目标有二：一是与广大技术社区分享我们的设计决策和经验教训；二是揭示在 Palantir 的 Gotham 核心平台团队中，一名侧重系统层面的后端工程师所负责的典型项目面貌。

在 Palantir Gotham 中，搜索是使用频率最高的工作流之一。用户不断查询、过滤并深挖大型数据集，以发掘实体、追踪关联，并整合驱动关键任务运营决策的分析结果。在整个工作流的任何环节，无论底层数据图谱的规模或复杂度如何，他们都期望结果的呈现是秒出的。

许多此类搜索工作流都借助了 Elasticsearch——一款开源的分布式搜索与分析引擎，能够在大规模场景下提供稳健的全文搜索能力。我们此前已阐述过 Elasticsearch 在更广泛的生态系统与架构中扮演的角色，如欲深入了解，可阅读《防御性数据库：优化索引刷新语义》一文。

正如那篇博文所述，多数 Gotham 应用通过一个内部构建的数据库客户端服务与 Elasticsearch 交互，该服务充当后端服务与存储基础设施之间的中间层。宏观来看，该服务类似于一个文档存储库，并提供以下能力：

  • 由事务型数据库支撑的强一致性键值 CRUD 操作，该数据库仍是唯一权威的事实来源
  • 借助 Elasticsearch 对文档字段进行全文搜索，Elasticsearch 在此充当“外部”的辅助索引
  • 内置于核心读取、搜索和写入路径的一流安全策略执行机制，而非事后补加的附属品

此后，我们将这个内部数据库客户端服务称为“文档存储库”。文档存储库服务于众多不同的客户端，每个客户端都有自己的一组模式（类似于数据库表）。每个模式声明了字段名称、类型、安全设置、文档结构布局，最关键的是，声明了字段是否以及如何通过所谓的映射被索引到 Elasticsearch 中。有些模式完全不建立索引；另一些则带有丰富的映射，以支持分面搜索、地理查询和日期范围过滤。你可以在其官方文档中阅读更多关于 Elasticsearch 映射的内容。

下图的架构图扩充了我们在《防御性数据库》博文中使用的图表，展示了从各个应用后端到数据库、Elasticsearch 以及认证服务的数据流向。

尽管 Palantir 的 Foundations 团队负责在我们多样化的部署集群中运维和交付 Elasticsearch 集群，但与该基础设施交互的服务也承担着高效、负责地使用集群并维护自身索引的责任。

有时，服务也需要修复或重建 Elasticsearch 索引。当索引因映射变更或运维故障而过时或陷入不一致状态时，通常需要通过一个称为“重建索引”的流程来修复。

  • 例如，一个物流应用若需新增一个字段供用户搜索供应链中断数据，或者需要在访问策略中支持额外的安全元数据字段，就需要进行重建索引。
  • 重建索引也可应用于集群层面。例如，在从 Elasticsearch 8.x 迁移至 9.x 时，我们倾向于选择重建索引而非原地升级，因为一旦出现问题，重建索引能让我们快速回滚。

本文探讨了内嵌于文档存储库中的 Elasticsearch 重建索引机制的设计与演进。该机制使我们能够可靠、高性能地执行重建索引，且无需使服务下线或投入过多人工监控，这一切跨越了无数的客户端应用。

我们的需求是什么

如前所述，重建索引可能因多种原因而变得必要。随着产品迭代，模式及其映射会发生演变，索引设置（包括分片数量）可能改变，且索引可能因意外故障或软件漏洞而损坏。一旦发生这些情况，就必须创建一个带有正确映射和更新后设置的新索引，填入数据并替换原索引。此外，当我们需要在集群间迁移时，例如进行版本升级（如从 Elasticsearch 8.x 升级到 9.x），或者在吞吐量足够时迁移至独立集群而非与其他服务共享集群时，我们需要执行跨集群重建索引。

Elasticsearch 确实提供了一个 _reindex API，用于在同一集群内将文档从一个索引复制到另一个索引，同时也支持在 API 中从远程集群获取重建索引的数据源。然而，由于在我们的架构中，数据库是具备强一致性的主要事实来源，我们进行的重建索引将从数据库加载文档来重新填充新索引，这意味着我们的文档存储库服务必须编排并监控整个重建索引过程，而不能仅仅派发一条命令后便交由 Elasticsearch 集群处理剩下的事务。

我们的 Elasticsearch 重建索引机制需满足以下要求：

  1. 感知模式的重建：系统必须知晓如何构建每个客户端模式的索引——字段类型、分析器、分片数量等。
  2. 低人工干预：运维人员不应手把手引导每个模式走完流程并推进各个阶段。在数百个部署中管理重建索引，且每个部署包含 100 多个模式，靠人工处理工作量过于庞大。
  3. 自动检测并修复不一致：系统应能检测特定模式何时需要重建索引，并自主执行。
  4. 受限的资源使用：重建索引绝不能拖垮服务器或妨碍 API 服务。它不能独占内存、耗尽线程池或使数据库连接池饱和。
  5. 在线实时重建索引：服务在整个期间（包括搜索）都在处理实时流量。以我们的物流应用为例，现有的搜索功能必须持续可用。

第一代重建索引机制

明确需求后，我们在文档存储库中实现了首个 Elasticsearch 重建索引机制。下图展示了针对特定模式进行重建索引的流程。

影子索引

我们的示例应用必须在重建索引期间及之后持续处理搜索请求。维持重建索引期间服务活性的关键在于影子索引，它是同一 Elasticsearch 集群上针对同一模式的一个独立物理索引，具备所需的映射和设置。它一方面接收来自文档存储库客户端 API 的全部索引/取消索引请求，同时由重建索引机制进行数据回填。关键在于，影子索引此时还不提供读取服务。重建索引机制将创建一个影子索引：若当前索引以 _a 结尾，则新索引以 _b 为后缀（反之亦然），并在创建之初便设定好预期的映射与设置，随后从数据库回填文档。完成后，机制会将影子索引设为主索引，并丢弃先前的索引。

搜索始终由主索引提供支持，而来自 API 的写入请求则同时双写到主索引和影子索引，以免丢失最新编辑。我们利用 Elasticsearch 提供的乐观并发控制原语（version_type.external_gte）来消除 API 与重建索引机制并发写入的冲突。

至于读写请求的路由，我们没有直接引用物理索引的名称，而是在 read、write 和 reindex 代码路径中使用了 Elasticsearch 索引别名。别名到物理索引的映射仅在索引创建期间及重建索引操作后才进行设置或更改。

下表说明了在重建索引的关键阶段，涉及影子索引 _b 和现有主索引 _a 的别名变更情况。在此示例中，_b 在重建索引开始时被创建并打上重建索引别名标签。_a 保留读取别名，而写入别名被移除。当重建索引的回填阶段完成后，_b 通过接管读写别名成为主索引，而 _a 则被删除。

编排重建索引

重建索引机制维护着一张模式级别的索引元数据表，用于追踪与重建索引作业运行相关的信息，例如某模式最后一次开始/成功/失败的时间，以及该模式是否处于排队、正在重建索引、正在中止或无需操作的状态。模式级别的索引元数据是重建索引机制内部协调的唯一事实来源。

作为服务器上的后台任务运行，重建索引机制会定期扫描所有模式定义，将数据库中持久化的元数据与 Elasticsearch 集群上的实际索引状态进行比对，从而确定哪些模式需要处理：继续进行中的重建索引、启动下一个排队任务，或者在排队进行完整重建索引之前，先尝试修复不一致的模式索引（若修复不足以解决问题）。

除了为需要修复索引的模式主动进行重建索引外，另一个负责处理配置 supplied 请求的后台任务，以及若干管理 API 端点，允许运维人员和客户端（包括示例中的物流应用）请求或中止重建索引。

集群无关

跨文档存储库服务的一项关键设计选择是集群无关。Elasticsearch 集群会在代码中被发现并映射到逻辑组件。在大多数情况下只有一个集群，但在跨集群重建索引期间，会挂载一个额外的迁移集群并被识别。通过一个动态层，读取操作仅在主集群上进行，而写入操作——包括重建索引——会扇出到所有已连接的集群，从而以极少的额外代码支持单集群和跨集群操作。

基于这一基础设计，我们能够对文档存储库中的任何模式执行重建索引。我们那以物流为核心的示例应用可以在其模式升级中包含重建索引步骤，并以重建索引的完成为条件来开启新的搜索功能。

重建索引机制的演进

随着产品在功能和用量上的扩展，以及存储基础设施的演进，重建索引机制面临着日益增长的需求。在文档存储库服务持续处理实时流量的同时，对包含数百万文档的 Elasticsearch 索引进行全量重建，是一项不断演进的挑战。随着我们增加新功能并提升稳定性，重建索引变得更加频繁且关键。为此，我们不断强化重建索引机制，以提升性能、稳定性和可观测性。在此，我们重点介绍几项显著的改进。

并行化与异步处理

并行化在早期便被引入重建索引机制，因为从数据库读取文档分页和将其索引到搜索引擎都涉及显著的 I/O 等待。解决方案是一个由可配置线程池和有界暂存队列支撑的并行作业运行器。工作线程在每次迭代中动态承担两种角色之一：获取分页锁的线程将下一批记录提取到队列中，而发现锁被持有的线程则转而认领队列中的批次并执行索引工作。一个独立的守护线程会定期记录最早未完成批次的位置来快照进度，使作业在故障后能从断点恢复。这种设计构建了一条流水线，使得多个批次同时处于处理中——一些正从数据库读取，另一些正被索引——且无需显式的角色分配或阻塞重建索引循环。

批次限制

为防止重建索引压垮文档存储库，我们在重建索引机制周围加入了多种速率和批次限制能力。单次重建索引运行在三个维度上受到限制：

  • 每批次文档数量：限制单次事务中从数据库获取的对象数量，缩短数据库连接占用时间。
  • 每批次内存：对单批次内存占用设的软上限，防止因死板的批次大小限制而导致大文档引发内存骤增。
  • 每次运行总批次数：限制重建索引循环单次调用在退让前处理的批次数量。结合可配置的运行间休息间隔，为 API 服务线程和垃圾回收创造了自然的喘息空间。

这三个调节参数相互作用，防止单一维度成为瓶颈。包含大量小文档的模式会触及文档数量限制；文档较少但体量较大的模式会触及内存限制；而每次运行批次数限制则确保重建索引机制无论如何都会周期性暂停。

故障安全

设想一下，如果文档存储库服务器在修改索引和别名的操作中途崩溃——比如在为物流应用启动或完成重建索引时——索引可能陷入破损状态，若不从零开始重启整个重建索引，便无法进行确定性恢复。为防范此情况，每个索引生命周期操作都被建模为细粒度的状态机：在向搜索引擎派发任何变更操作前，当前状态会持久化到数据库，且每个搜索引擎操作都设计为幂等的。若服务器在任意节点崩溃，它只需从最后持久化的状态恢复，并安全地重新执行待处理操作。

下图所示的状态会被持久化在一张按集群、按模式划分的独立元数据表中，我们同时在该表中持久化当前主索引等关键参数以确保故障安全。

日志？指标？还是诊断报告？全都要。

当后台进程悄无声息地重建包含数百万文档的索引时，可观测性是不可妥协的。Palantir 产品部署的多样化且往往受限的环境进一步意味着，不同形式的可观测性在各部署中的可行性和访问权限差异巨大。你无法在出问题时才去追溯祈求本来该有的可观测性。更不用说不同形式的可观测性各有优劣，而我们想要兼收并蓄。

日志：

  • 优点：展示带有丰富参数的时间戳状态快照，极利于追踪确切步骤和检查每一步的详细状态。
  • 缺点：若参数过于详尽，可能过于冗余并占用过多磁盘空间；尽管日志有助于事后调查且在根因分析时我们希望有详尽日志，但多数时候日志及其参数根本无人问津。

指标：

  • 优点：追踪状态和进度随时间变化的理想、轻量级工具，可转化为用户友好的仪表盘可视化，辅助快速洞察收集和聚合分析。
  • 缺点：除了时间序列和标签本身外，不包含任何额外信息，即无附加参数。

诊断报告（按需生成的一次性报告，常在 Palantir Apollo 上查看）：

  • 优点：可以在合理范围内做到我们需要的详细程度，且鉴于按需生成，无需担心刷屏或磁盘空间问题。
  • 缺点：被请求时通常只是一个快照（除非专门设计用来重述历史事件），这使得它们在事后分析中用处不大。

就我们而言，我们选择实现所有三种可用的可观测性形式，事实证明这极具价值。

处理跨集群重建索引

回顾前文的示例，触发重建索引的原因可能不仅限于应用层面的变更。跨集群迁移——将服务的搜索流量从一个 Elasticsearch 集群无损转移到另一个——是大规模平台中不可避免的事实。例如，如果我们想在从 Elasticsearch 8.x 迁移至 9.x 的同时保留随时回滚的能力，就需要通过重建索引来实施迁移。

在我们迄今探讨的重建索引机制设计中（旨在修复提供服务请求的主 Elasticsearch 集群），我们在同一集群上创建一个新索引作为影子索引，随后从主存储库对其进行重建。回填完成后，影子索引被提升用于搜索。尽管在概念上与常规重建索引相似，跨集群重建索引确实需要略作不同的处理。具体而言，我们需确保新索引位于与托管旧索引的集群不同的集群上，并且在重建索引完成后跳过删除旧索引的操作，因为新集群上无内容可删，且我们希望保留旧集群上的索引以备回滚之需。

用于迁移的独立重建索引机制

我们决定将两个重建索引机制分开，各自基于与相关 Elasticsearch 集群绑定的元数据来编排重建索引。我们迄今为止在本文中探讨的主重建索引机制，与主集群协同工作，负责修复处理 API 请求的活跃索引。而新的迁移重建索引机制则对迁移集群进行回填，并为其成为新主集群做准备。与一个充斥着集群特定分支逻辑的单一重建索引机制相比，创建两个独立的机制使得配置、元数据管理、代码可读性和可观测性都更为简易。每个重建索引机制将拥有自身的配置：线程数、批次大小、内存限制、调度间隔等，并作为独立的后台任务运行，仅与各自的集群协同而非所有已连接的集群。
