利用 Foundry 扩展数据运营

编者注：本篇客座文章由我们的客户瑞士再保险撰写。作者 Lukasz Lewandowski、Marco Lotz 和 Jarek Sobanski 领导着负责在瑞士再保险公司实施 Palantir Foundry 的核心技术团队。多年来，他们一直负责管理整体平台运营、核心架构原则、站点可靠性、成本管理和数据治理。

引言

作为全球领先的再保险公司之一，瑞士再保险在风险认知领域保持领先已有 160 余年。面对气候变化、网络攻击和人口增长等全球性挑战，公司在其让世界更具韧性的使命中，正应对着日益交织的复杂风险。

瑞士再保险拥有超过 14,000 名员工，他们既是分析数据的使用者，也是生产者。该组织从各种商业渠道收集并生成关于自然灾害、寿险和健康险以及商业与信用风险的数据。

为了保持敏捷性并实现快速、数据驱动的决策，瑞士再保险必须大规模管理复杂数据，识别看似不相关的趋势与事件之间的联系，并将这些洞察转化为业务功能和客户服务的实际运营能力。这一切都必须在符合适用法律、规则和法规的前提下完成。

2018 年，我们选择与 Palantir Technologies 合作，升级分散的基础架构，并在 Palantir Foundry 上构建数据平台。

面向未来的数据平台

瑞士再保险运营着各种数据密集型应用，涵盖从初始承保、核保、定价到条约与理赔管理。这些应用还进一步延伸到精算分析、偿付能力压力测试、专注于财务的资产管理和资产管理系统，以及法定报告。这些系统包含众多技术和模式，包括操作型数据存储、数据仓库、数据代理和信息中心。

传统上，数据和分析平台是分散的，每个功能从其核心系统获取输入，并在各自选择的平台上进行分析。虽然这种方法提供了灵活性，但并未产生显著的跨公司协同效应。

为了使我们的组织面向未来，我们设想了一系列目标，旨在实现透明度、稳健的治理和高效的数据管理，同时为子组织保持高度的自由度。这些目标包括：

  • 跨流程和服务的可观测性与溯源追踪： 确保数据资产和业务逻辑的数据权属，从而在处理的每个步骤实现目的清晰明确。
  • 独特且专有的能力： 构建一套连贯、明确的技术标准和指南，覆盖整个数据处理范畴，减少对补充性和小众软件的需求。
  • 成本与价值之间的清晰联系： 确保存储、处理和业务逻辑可观测性方面的成本透明，允许在整个组织范围内进行持续优化。
  • 志同道合的社区： 制定统一的方法，涵盖培训与技能提升、倡导、定义最佳实践、理念推广以及跨业务部门方法的通用性。

为实现这些目标，我们采用了受数据网格启发的架构和平台。该平台的核心重点是分析和决策支持应用程序。在达到一定的成熟度后，它还将作为数据共享平台，促进分析结果的受控共享，同时遵守来自客户和监管机构的数据使用限制。为了平衡有效性与效率，瑞士再保险设计了管理复杂性和成本的流程，并在设计和治理上采用了网格原则。

管理复杂性：分而治之

用运行联邦工作负载的单一平台取代分散的异构平台，大大降低了技术复杂性并避免了技术债务，从而将更多精力集中在真正的增值开发上。然而，降低新软件开发的准入门槛也可能带来治理挑战。

作为应对，瑞士再保险建立了特定的治理边界。每一捆数据和逻辑实际上都被视为一个应用程序——一个具有指定业务和 IT 数据权属的半自治逻辑单元。这些应用程序必须预先注册，并在多个维度进行评估：目的、架构、访问控制和数据治理。

然后，应用程序被分组为领域，它们充当业务部门和企业职能部门的数字孪生。每个领域承担数据权属，并促进应用程序和数据资产的可重用性和协调统一。标准的 Foundry 项目创建和访问控制界面得到了增强，以强制执行瑞士再保险自己的命名规范、数据权属和审批链，所有这些都链接到公司范围的治理工具集。这种增强是通过 Workshop 应用程序和定时转换的组合实现的，它们遍历元数据目录以生成统计数据、识别异常并阻止不合规的资源（如项目或数据集）。

这种方法强制执行了数据网格架构的两个最关键原则：领域数据权属和自助服务数据平台。

实施数据网格原则以实现联邦化与可扩展性

在单一平台上集中管理数据大大提高了透明度，但并不能保证无摩擦增长。虽然通过数据和应用程序的联邦化数据权属确保了治理，但数据流和依赖关系日益增长的复杂性需要额外的架构构建来控制熵。

跨领域的一致性和合规性由一系列指南来保证，这些指南限制了访问管理和允许的数据流。这些指南通过项目原型强制执行，并由中央平台团队进行持续监控，该团队识别异常并尽早与数据主人一起解决。领域内跨应用程序的数据流通常允许不受限制，而跨领域的数据流则需要额外的审查。

在实施时会建立隐式数据交换协议，允许平台上的任何两个应用程序交换数据。对于跨领域传输，这些交换协议变得更加显式：瑞士再保险要求数据交换的物理实现必须遵循基本的治理流程，例如识别数据主人并明确确认交换的参数，包括内容、频率和质量。

此外，我们的运营模型通过数据结构对齐来关注可重用性。每个领域在其应用网格的核心维护着关键的数据资产，称为分析数据模型——这是一类特殊的数据产品。

为了鼓励协作和数据共享同时防止孤岛，我们依赖访问管理。默认的“发现者”角色允许平台用户查看所有资产的元数据，而无需访问实际内容。我们使用单一组织和命名空间，通过访问产品分组结合不同的项目原型来实现领域隔离。跨领域数据共享是通过创建包含来自不同领域的访问产品的项目来促进的。

这种方法平衡了清晰的数据权属责任、合规的数据访问，以及对透明度和共享的需求，有效地防止了数据孤岛。

注重成本的 PB 级规模处理

Palantir Foundry 的快速应用程序创建能力需要可扩展的平台管理以跟上其速度。该平台支持二十多个业务领域、集团功能和专门部门，管理着数千个入站数据源和 PB 级的数据。由于瑞士再保险近一半的员工每季度使用该平台，且该平台的基础设施最初每年翻一番，我们需要一种可持续的方法来观察和管理所有托管和处理成本。

Foundry 数据平台主要通过三种方式消耗资源：

  • 磁盘存储： 数据集文件使用的存储量，通常是经典的“数据湖” blob 存储。
  • 计算（批处理和交互式）： 衡量计算任务（如 Spark 作业或索引）中使用的 vCPU 数量和内存量。
  • 索引存储： 衡量用于本体驱动工作流的基础设施。

瑞士再保险通过三种方式解决这些资源产生的成本：

提高可观测性： 瑞士再保险消耗大量 Foundry API，并构建包含使用信息和事务图的补充性 Foundry 数据集。这些信息按使用类型、分支、前驱和后继、存储使用情况、查询计划复杂性、数据集大小与所请求的计算资源等进行分析。这种提高的可观测性使我们能够微调数据保留策略，并将归档大小减少了 50% 以上（约 3.6 PB），而不会造成任何功能或可审计性的实质性损失。

限制索引存储： 在瑞士再保险的实施中，索引存储是最昂贵的单一资源类型。虽然对象层对于加速需要在线处理的应用程序构建非常有用，但它带来了开销：批处理计算将每一行和每一列同步到索引系统，并在块存储和内存中进行维护以供查询，通常具有多重冗余、负载均衡的 REST API 等。重要的是，只索引需要的内容，而不是所有数据集、列和列的搜索类型，并有意识地决定哪些可以在 Contour 或 Workbooks 中分析，哪些可以在 Quiver 和 Workshop 中分析。对优化候选者的初步审查及后续行动已使基础设施成本降低或避免了高达 40%。开发人员培训课程和对“超重”对象的主动监控现在已成为持续的过程。向 Object Storage V2 的迁移通过仅同步映射的列，进一步减少了开销。

持续的动态优化，特别是批处理计算： 警报和预警机制对于识别冗余或低效的流程是必要的，例如过度调度、在主分支之外构建大型数据集、不常使用的资产，或强制构建未更改的数据集。我们还通过“配置文件调整活动”跟踪具有不成比例 vCPU 或内存需求的过度配置作业。

我们分析不必要的列传播，即父数据集中的列在管道中向下传递而没有任何后续使用。为了解决这个问题，我们收集并分析 Spark 查询计划，并重建基本的列血缘关系。消除冗余列对存储和处理产生重大影响，因为更窄的数据集加载更快，在执行器中需要更少的内存，并最终占用更少的空间。

我们还监控高增长率而不仅仅是高绝对成本的项目，以便及早识别潜在问题。仅此监控过程就产生了 10-15% 范围内的成本节约和成本规避。

为了确保持续的影响，我们使用一套基于 Foundry 的工具来显示指标并向相关用户突出显示异常：

  • 成本仪表板： 指示每种类型和每种资源的总成本，并汇总到应用程序和领域。
  • 平台仪表板： 包含多个平台指标，并近乎实时地计算其统计数据（平均值、p95、p99）。它还跟踪问题处理性能（响应时间、解决时间等）、平台服务的可用性和队列性能。
  • 启发式仪表板： 定期监控 40 多种条件，以控制数据流及其效率的熵。这包括识别与企业架构指南相矛盾的数据流、意外的访问授权和异常的使用模式（例如，随着时间的推移对某些资源的读写访问，执行这些操作的用户数量）。

自两年前实施这些工具以来，我们产生了大量的年度节约，同时我们的用户群和流程增加了一倍多。这些工具对于维持精简的平台团队至关重要，使运营从被动响应转向主动预防。这种转变使团队能够专注于增值服务，例如工程指导、治理和架构蓝图。

无法回避的问题：AI 革命

与许多其他机构一样，瑞士再保险也开始探索生成式 AI、LLM 甚至 LMM 在改善保险和再保险核心业务运营方面所能带来的机遇。

我们的本体将我们的组织表示为一组数字孪生，传统上它依赖于结构化数据，但现在也正迅速扩展并融入文本信息。我们早就认识到，用机器学习流程扩展我们的数据网格，以及现在的生成式 AI，是一项战略上的当务之急。在许多“远超聊天”的运营决策场景中，对来自保险投保单、保单和理赔等大量文档语料库的信息进行协调、分类和提取的需求至关重要；为了以合理的成本实现高度自动化，采用现代 AI 是顺理成章的。

鉴于我们在过去几年中在数据权属、可观测性、可追溯性、成本控制和优化方面投入的精力，我们正处于一个相对有利的位置，准备在这个蓬勃发展且快速演变的行业中航行。我们的数据网格架构正迅速向数据和 AI 网格架构演进，而 Palantir Foundry 和 AIP 平台使我们能够在管理架构的同时，保持同样的前沿安全和治理标准。平台提供的高透明度和灵活性帮助我们应对 AI 运动的无情步伐，并应对不断出现的监管环境。我们乐观地认为，随着 AI 革命的继续加速，我们在过去几年中做出的选择将变得同样重要，甚至更加重要。

结论

单一的、可寻址的数据空间为瑞士再保险带来了诸多好处，包括：

  • 领域驱动设计。 这保留了业务部门的自主性，实现了短周期的开发和对需求变化的敏捷响应。几周内就孵化出了接近生产级的完整解决方案，这得益于数据的可用性，有时完全由信息消费者自己编写，仅需技术团队的最少支持。
  • 用户体验一致性： 对视觉层和底层解决方案机制的熟悉度在领域之间和用户之间传递，鼓励了更多的自助服务和应用程序之间的连接。
  • 架构一致性： 这使得瑞士再保险能够在工业规模上进一步推广和重构，不断缩短上市时间和降低单个解决方案的成本。
  • 提高数据治理流程的有效性： 通过灌输强烈的数据权属意识，防止滥用和违规行为，这是持续、可扩展增长所必需的。
  • 降低总拥有成本： 正在进行的优化有效地为平台增长提供了自我融资。

这些变革性的好处凸显了统一数据战略的力量，使瑞士再保险不仅能满足当前需求，还能无缝扩展并创新以应对未来挑战。通过利用 Palantir Foundry 平台，瑞士再保险展示了前瞻性的 IT 解决方案如何推动整个保险行业的可持续增长和效率。
