Testing in Depth. (Also, please don’t use undeclared dependencies.)

2021-05-27 · Original →

今天早上，我在内部聊天中看到了这样一段讨论（略有改写）：

Alice：我们在启动日志中发现了 java.lang.ClassNotFoundException；这是在升级数据库客户端依赖之后发生的。

Bob：数据库团队需要更新他们的 RPC 库依赖。数据库客户端之前使用的是从 RPC 库传递过来的 OkHttp 依赖。RPC 库移除了对 OkHttp 的支持，因此在升级后，数据库客户端就找不到 OkHttp 类了。

Alice：嗯，但是我们的构建没有失败？啊，我猜我们在集成测试中用的是内存数据库。它们不会走到 OkHttp 的代码路径。

Bob。没错。幸运的是，这对任何人来说都不会是 P0 级故障，因为 Apollo 已经自动回滚并召回了受影响的版本。

Alice：确实。不过，我们还是尽快修复数据库依赖配置吧？

这个 bug 是怎么逃过测试的？为什么它没在半夜把工程师叫醒？我们的持续交付系统 Apollo 又是如何帮到我们的？这里面有很多值得深挖的内容，这篇博文试图一探究竟。

不要使用未声明的依赖：也就是这个 bug

上面讨论的产品，其调用过程大致如下所示。

该产品使用 RPC 库（我们使用 Conjure 进行 RPC）来调用多个 Web 服务。它还依赖于一个 数据库客户端，该客户端有两种不同的实现：

  • 对于发布前的集成测试（“体外”，蓝色），我们使用内存中的伪实现（下文详述）。
  • 生产环境的实现（“体内”，红色）使用 RPC 库来调用远程数据库服务。（即 AtlasDb/Cassandra）。

你可能已经嗅到了菱形依赖问题的味道；没错，它来了。RPC 库最初在内部使用 OkHttp 来发起 HTTP 调用，因此产生了如下的依赖结构：

Product @1.0.0
    → RPC @2.0.0
        → OkHttp
    → DatabaseClient
        → RPC @2.0.0
            → OkHttp

后来，RPC 库迁移到了 Apache HTTP 客户端并移除了 OkHttp 依赖；当产品升级到 2.1 版本的新 RPC 库时，依赖图变成了这样：

Product @1.1.0
    → RPC @2.1.0
    → DatabaseClient
        → RPC @2.0.0
            → OkHttp

注意，RPC 依赖在依赖图中出现了两次——一次是版本号为 2.1.0 的直接依赖（使用 Apache HTTP 客户端），另一次是 DatabaseClient 版本号为 2.0.0 的传递依赖（使用 OkHttp）。我们的构建系统 Gradle 使用锁文件来管理依赖（顺便推荐一下我们的 gradle-consistent-versions 插件，非常好用）。通常，产品开发者会将 RPC 依赖锁定在 2.1.0，从而有效地覆盖并升级 DatabaseClient 对 RPC@2.0.0 的依赖。

到目前为止，我们并没有做错什么：RPC 升级了一个内部库，其行为或 API 没有任何改变（因此从 2.0.0 升级到 2.1.0 的次版本号变动是合理的），并且构建系统也正确地拾取了新的 RPC 依赖并进行了统一应用。注意，OkHttp 不再是一个依赖项，但这没问题，因为 RPC@2.1.0 已经不再使用 OkHttp 了。

那么，既然我们没做错什么，怎么就出问题了呢？原来，DatabaseClient 直接使用了 OkHttp 的类（即在 RPC 库之外）。但是，DatabaseClient 并没有声明对 OkHttp 的依赖。换句话说，DatabaseClient 使用了传递依赖却没有声明它，实际上是依赖了 RPC 库的私有实现细节。这是一个构建系统的 bug，只需向 DatabaseClient 库添加一个显式的 OkHttp 依赖即可轻松修复。

我们能预防这个 bug 吗？这取决于构建系统的工具支持。在 Palantir，我们使用许多 Gradle 插件来协助管理依赖卫生：

  • gradle-consistent-versions 允许开发者为项目声明依赖版本。（它与 nebula.dependency-recommender 以及原生的 Gradle 4.8+ 锁文件有细微差别，详情请参阅 README。）
  • checkClassUniqueness 防止类路径中出现重复的类。请注意，仅通过 Maven 坐标对依赖进行去重是不够的，例如当内部或开源依赖被重命名时。重复的类是引发隐蔽 bug 的一个棘手源头，因为运行时的行为（即实际加载哪个类）取决于诸如依赖 JAR 文件的文件系统顺序等难以观察的属性。
  • checkImplicitDependencies 竭力防止的正是导致上述 bug 的原因：被使用却未被声明的依赖。目前，准确判断一个类何时被使用但未被声明并不是一个完全解决的问题，特别是当库使用反射时。那么，为什么 checkImplicitDependencies 没有在这个案例中抓住问题呢？遗憾的是，这项检查在 CI 阶段没有被强制执行；我们当然正在通过添加该检查来修复根本原因。
  • checkUnusedDependencies 会在开发者声明了依赖却不使用时发出提醒。虽然这看起来主要是一个代码卫生问题，但最小化依赖足迹可以降低发布漏洞的风险，并有助于保持较小的发布体积。
  • gradle-dependency-control（内部工具）管理依赖的核心白名单和黑名单

Java 的 Jigsaw 项目以及 Maven 的依赖作用域（通过 Gradle 的 api 与 implementation 配置可以清晰地表达）以更严谨的方式解决了这些问题，它们允许开发者对公共和私有 API 及依赖进行明确的（且可强制的！）划分。我希望 Jigsaw 模块的采用能逐渐渗透到开源和 Palantir 的代码库中，帮助提升代码质量并防止依赖 bug。关于该主题的更多阅读，请查阅 Google Bazel 构建系统的文档，其中描述了完全相同的问题。

深度测试

到目前为止，我们讨论了 Alice 和 Bob 的 bug 以及某些构建系统和插件如何帮助管理依赖卫生。现在，让我们继续讨论 Alice 的“嗯，但是我们的构建没有失败？”以及 Bob 的“这对任何人来说都不会是 P0 级故障”的说法。为什么这个 bug 没有在 Palantir 引发大范围的宕机？答案就是我所说的深度测试。类比信息安全中的纵深防御概念，深度测试是一种将不同的测试和监控技术进行分层的实践，每一层都关注整体软件正确性的不同方面。

标准的测试层分为单元测试、集成测试和端到端测试。上面讨论的产品拥有全面的单元测试覆盖，并采用合并前的集成测试来演练产品与数据库的交互。为了追求极快的 CI 耗时，这些测试是针对数据库服务的内存伪实现编写的。这些测试都没有失败，因为它们没有触及依赖于（缺失的）OkHttp 库的 RPC 代码路径。

显而易见的选择是增加端到端测试来演练 RPC 代码路径，例如针对开发数据库栈或 Docker 容器进行测试。但显而易见的答案也是正确的答案吗？这很难说，且一如既往地需要权衡。端到端测试运行缓慢且出了名地难以维护（可以参考《Software Engineering at Google》中的不同观点）。换句话说，它们不利于开发速度的提升，因此在开发者时间和最终成本上都是昂贵的。如果它们只是偶尔捕获到问题，那么成本效益分析就划不来了。

在 Palantir，我们倾向于将体外（即发布前）的单元和集成测试与体内（即发布后）的监控分层结合。我之前写过关于 Apollo 的文章，它是我们 SaaS 平台的持续交付系统。正如 Alice 和 Bob 的对话中所暗示的，Apollo 的几项功能合力，以低成本且有效的方式捕获了这个依赖 bug。首先，所有由 Apollo 管理的应用都会产生健康检查指标，而 Apollo 会监控这些指标。健康检查可以对性能下降、异常行为、RPC 问题、内存占用等发出告警。Apollo 持续监控所有的健康检查，并利用它们采取纠正措施（见下文）。

其次，Apollo 以一种可控的方式推出新的产品版本。Apollo 会先升级开发栈和金丝雀栈，然后再推向内部栈，最后才是面向客户的生产栈。在给定的栈内，Apollo 使用蓝绿部署的变体，逐个节点地升级同一应用程序的不同节点。每次节点升级后，Apollo 都会检查健康检查指标（我们称之为裁定）。当健康检查失败时，会发生两件事：升级立即回滚到最近已知的良好版本，并且该版本会被召回。被召回的版本会在所有栈上（不仅仅是报告问题的那个栈）进行回滚，而且 Apollo 绝不会再次安装它。受影响的开发团队会自动收到回滚和召回的告警，以便他们进行调查并修复产品。

整个过程是自动化的，由 Apollo 在数百个包含数百个服务、应用和数据库的栈上进行管理。在 Alice 和 Bob 的案例中，Apollo 通过捕获库依赖声明中的一个小疏忽，阻止了一次全系统范围的宕机和 P0 级故障。更广泛地说，我们发现体内监控比体外的端到端测试能产生更好、更廉价的错误信号，因此它是 Palantir 深度测试战略中不可或缺的组成部分。
