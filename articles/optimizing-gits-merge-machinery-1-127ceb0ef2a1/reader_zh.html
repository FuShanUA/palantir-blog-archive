Optimizing Git’s Merge Machinery, #1

2021-03-08 · Original →

编者注：这是一位 Palantir 软件工程师关于优化 git 合并和重命名检测机制系列文章的第一篇。点击阅读第二篇。

过去，我曾讨论过我在《Git 中的重命名与深层目录层级》方面的工作。当时我提到，将对 git 的合并机制和重命名检测能力进行一些性能改进。在这篇及后续的文章中，我想谈谈这些改进，它们是我为 git 贡献新合并算法工作的一部分[1]。在这个系列博文中，我将描述我对 git 所做的优化，我在《Git Merge 2020》的演讲中曾开玩笑地将它们总结为：

  • 不做不必要的工作
  • 不重复做工作
  • 不重复做不必要的工作
  • 变通界定“不必要”

但如果稍微更准确地总结，应该是：

  • 找出那些即使检测重命名也不会改变结果的路径，并跳过这些路径的重命名检测
  • 不要去寻找比完美匹配更好的匹配
  • 在变基（或拣选）一系列提交时，不要重复检测上游分支已完成的重命名（尤其是不要重复检测那些本就不必要的重命名）
  • 寻找有用的启发式方法来指导重命名检测，以避免（或至少减少）暴力的穷举式两两比较

自《Git Merge 2020》演讲以来，我已经完成了实现，对上述优化进行了多项改进，并新增了两组优化：

  • 树级合并（看起来与重命名检测格格不入，但一些技巧往往能让它们协同工作）
  • 传统且扎实的优化工作——测量性能，找出缓慢的代码路径，并逐一修复

本系列的第一篇博文将首先介绍我们在 Palantir 如何使用 git 以及我们为何关注这些优化，接着提供理解这些优化所需的背景知识，最后展示第一个极其简单的优化：不要去寻找比完美匹配更好的匹配（这本身就是对现有优化的一项改进，即通过基于文件哈希而不是查看文件内容来检测重命名，从而加速 git 合并）。

合并算法仅仅关乎 git merge 吗？

git 中的合并机制（称为 merge-recursive）驱动着 git 的多个方面：

  • merge
  • cherry-pick
  • rebase
  • revert
  • am -3
  • stash
  • checkout -m

所以我们讨论的远不止 git merge。

为什么 git 的合并算法需要改进？

简而言之：

  • 我们需要高效地重构大型代码库
  • 我们希望向稀疏操作迈进并提升其效能
  • 现有代码已达到其设计极限

高效重构大型代码库

在 Palantir，我们有几个规模与 Linux 内核相当的大型单一仓库。在这些仓库中，我们有时会进行深度的重构，包括重命名高层目录。我们希望 git 能高效处理这类工作。如果因为系统限制或耗时过长，导致无法通过拣选补丁将其移植到较旧的分支，大家就会诉诸手动打补丁，从而面临遗漏重要细节的风险。我们曾遇到过这样一个案例：一个团队因为涉及大量重命名重构后补丁回溯困难，浪费了大量时间，导致他们的工作进度停滞了数月。这导致他们召开了一次会议，要求达成一致意见，以后永远不再重构代码；这是一种非常不理想的状态，如果具备更好的合并能力，本是可以避免的。

稀疏性

除了重构工作流，我们还希望开发者能够像在使用一个小得多的仓库一样，专注于自己感兴趣的领域。在 git sparse-checkout 出现之前，我们就创建了允许稀疏检出的脚本（因为稀疏检出本身是基于 git 中存在已久的 SKIP_WORKTREE 能力，但当时没有易于使用的界面），随后我们审查了由 Microsoft 贡献的 git sparse-checkout 补丁，并为该功能贡献了几个自己的补丁。

该领域持续进行的工作涉及创建稀疏索引（也由 Microsoft 牵头），这不仅能让工作树变得稀疏，还能让 git 的索引（或暂存区）变得稀疏。由于索引在 git 中是一种无处不在的数据结构，这种改变将需要对整个 git 代码库进行重大修改。现有的合并机制与索引的联系比大多数代码领域都要复杂得多，修改它实际上等同于完全重写。因此，稀疏索引工作将需要一种新的合并算法，这正是我们正在贡献的。

与稀疏检出和稀疏索引理念相关联的，是部分克隆——即仅下载感兴趣路径的历史记录。我长期以来一直对这项能力很感兴趣，并且我相信我是第一个发布带有非常基础的稀疏克隆实现的补丁的人。我从未完成那个实现，但其他人最终创建了类似的能力。部分克隆会下载部分历史记录，然后在你访问时按需延迟下载额外的对象。自然，为了使部分克隆发挥效用，它们需要避免过于频繁地、以非优化的方式下载额外对象，或下载不必要的对象。遗憾的是，重命名检测可能需要下载数量极多的对象，尽管在某些情况下已知有其他方法有时能在不使用重命名检测的情况下解决问题，并避免下载那么多对象。因此，寻找方法改进合并的重命名检测也是部分克隆性能提升的一大驱动力。

设计极限

实现背后的设计大致是“另一段代码几乎实现了我们所需的功能，我们只需稍加修补即可”。（如果你非要知道的话，那另一小段代码是 unpack-trees，但除非你熟悉 git 的代码，否则这对你来说毫无意义。）这种“稍加修补”越积越多，直到这些修补变得无法维护。不幸的是，merge-recursive 是代码库中那些阴暗、几乎带有神秘色彩、几乎所有人都避之不及的区域之一；最高产的两位 git 作者曾表示：

  • “[这代码]相当地棘手。每次我开始查看它时，我都会感到困惑，想不起我之前以为自己即将取得什么突破。” (Jeff King)
  • “很久以前我就认为这段代码基本上是无法挽救的了。” (Junio Hamano)

我在它编写完成约 3-4 年后才加入，但我提交了足够多的修复，以至于在某个时候我成了该领域的专家。我最初的大部分工作实际上是为了解决各种疯狂的边缘和角落情况下的正确性问题，而且由于代码结构的原因，有些边缘情况非常难以修复。

为什么要开发新的合并算法？

我最初开始只是重构现有的合并算法，就像我长期以来所做的那样。当我提议对设计中一些较大的部分进行修改时，Junio Hamano（git 的维护者）建议干脆重写它。他指出，git 的合并后端采用可插拔设计，允许某人编写能与现有算法并存的新算法。他认为这在用户稳定性方面实际上更可取，因为这能让我们同时让一些人尝试新算法，而另一些人使用旧算法（甚至允许用户通过简单的配置开关在两者之间来回切换）。

这也让我得以自由地重新思考一些基本的数据结构和假设，着眼于提升性能并修复那些在旧设计中太难修复的恼人的角落情况。

背景：合并/变基/拣选工作原理的基础

我的优化工作需要了解关于 git 的几件事：

  • 合并两个分支是一个三方操作
  • Git 除了存储提交的哈希外，还存储文件和树的哈希
  • Git 会即时进行重命名检测；重命名不被存储在任何地方。合并出于两种不同的原因需要重命名检测，并且每次合并需要两组重命名检测。遗憾的是，重命名检测有时会很慢。
  • 变基和拣选对于每个被移植的提交来说也是一次合并操作（使用一个看起来滑稽的合并基），但生成的提交被记录为只有一个父节点，而不是作为合并提交。

如果你已经了解并理解了这些，完全可以跳过这个背景部分。如果其中任何一点让你感到惊讶或新鲜，请阅读下面相关的章节。

三方内容合并

假设存在以下历史记录，且当前分支为“main”：

     A---B---C topic
    /
D---E---F---G main

进一步假设在“main”分支的一个文件中，有一段代码如下

line 1
line 2
line 3

然而，在“topic”分支的同一文件中并没有第二行。合并机制在这里应该怎么做？是“main”添加了中间那行，还是“topic”删除了它？要知道答案，唯一的办法是查阅历史的共同点；最近的共同提交（在上图中标记为“E”的提交）就是 git 所说的“合并基”[2]。如果合并基包含该行，那么显然是“topic”删除了它，合并结果应该将其删除。如果合并基不包含该行，那么显然是“main”添加了它，合并结果应该将其添加。如果合并基在上下文区域与任意一侧的历史都不匹配，就会产生合并冲突。

因此，合并不是一种两方操作，而是一种三方操作，git 会查阅合并基（提交 E）、你当前的提交（提交 G，“main”指向的位置）以及另一分支末端的提交（提交 C，“topic”指向的位置）。[3]。（作为旁注，你可以通过运行 git config --global merge.conflictStyle diff3 来利用这一点；这将在你解决合并冲突时，为你提供关于原始版本的信息。）

合并算法背后的基本思想是，使用来自这三个提交中每个文件的版本，对仓库中的每个文件进行三方内容合并。[4]

Merkle 树

许多用户熟知 git 为每个提交都有一个哈希。然而，git 也为提交中的每个文件（“blob”）和子目录（“tree”）分配一个哈希，树的哈希是从其中的 blob 和子树的哈希派生而来，而提交的哈希则是从顶层树以及提交的作者/日期/消息等派生而来。这就是被称为默克尔树的数据结构。你可以通过运行以下命令自己查看这些哈希

git ls-tree -rt $COMMIT

或者重复运行

git cat-file -p $HASH

从某个提交的哈希开始，然后对在先前命令输出中发现的任何哈希重复此操作。Git 在内部通过哈希来存储和查找所有内容。树对象记录了 git 内部数据存储中的哈希与应写入工作目录的文件名之间的映射。（基于哈希的存储还提供了一种自动去重形式。）

重命名检测与合并

如前一节所述，git 不按文件存储修订版本。它也没有任何形式的元数据来跟踪各个提交中完成的文件重命名。相反，它会在执行相关命令时即时检测重命名。

合并需要重命名检测有两个原因

  • 用于三方内容合并：我们需要将相关的三个文件配对（例如 merge-base:src/oldname.java, side1:src/oldname.java, side2:src/subdir/newname.java），以便合并它们的内容
  • 用于目录重命名检测：如果历史的一侧在某个目录中添加了文件，而另一侧重命名了该目录，我们可以通过目录内文件的重命名检测到目录本身已被重命名，并将新文件移动到新目录中（或者至少通知用户他们存在冲突，需要决定是将新文件留在旧目录中还是移至新目录中）

因为合并有两方以及一个合并基，我们必须进行两组重命名检测——从合并基到 side1，以及从合并基到 side2。

遗憾的是，重命名检测是一个二次复杂度的算法。它首先获取合并基特有的所有 N 个文件名的集合，然后获取给定侧特有的所有 M 个文件名的集合，接着对所有 N x M 个文件的内容进行相似度比较，并将足够相似的最佳匹配标记为重命名。即使只有很少的重命名，也足以让重命名检测成为合并中最慢的部分。

变基与拣选

合并基、side 1 和 side 2 之间的三方合并这个概念，也可以从另外两个角度来看。你同样可以将其视为将合并基与 side 1 之间的差异应用到 side 2，或者将合并基与 side 2 之间的差异应用到 side 1。

这些看待合并的其他方式让我们能够理解变基或拣选是如何工作的。从宏观层面来看，变基或拣选意味着我们需要选取一个提交，并将该提交的更改应用到某个新的基线提交上。换言之，我们需要获取正在拣选的提交与其父提交之间的更改，并将其应用到新的基线提交上。但如上所述，这等同于说我们希望在父提交、正在拣选的提交和新的基线提交之间进行三方合并。因此，变基或拣选是作为带有一个看起来滑稽的合并基的三方合并来实现的[5]……只是在最后，合并结果只被记录为具有一个父节点（新的基线提交）。

对于多个提交的变基和拣选，我们会执行一系列合并，其中第 N 次三方合并的新基线提交是序列中前一个提交变基/拣选后的结果。

优化：更全面地利用精确重命名

如上所述，重命名检测的工作原理是首先获取合并基特有的所有 N 个文件名的集合，然后获取给定侧特有的所有 M 个文件名的集合，接着对所有 N x M 个文件的内容进行相似度比较，并将足够相似的最佳匹配标记为重命名。我们能做些什么来减小 N 和 M 吗？有可以在预处理步骤中检测到的特殊情况吗？

如果文件被重命名，但其内容未被修改呢？这样的文件将具有完全相同的哈希。由于 git 在仓库中存储了每个文件的哈希，我们甚至不需要做任何额外的工作来计算它们。我们只需检查 N 和 M 这两组文件，看看两组中是否有共享哈希的文件。例如，如果我们的旧文件及其（缩写的）哈希为：

c01055a1   hex.java
5eac0a57   word.java
acc057ed   fun.java

新文件及其（缩写的）哈希为

c01055a1   hexadecimal.java
0b57ac1e   phrase.java
acc057ed   games.java
decea5ed   copy.java

那么我们可以检测出 hex.java 和 hexadecimal.java 是重命名，而无需打开任何一个文件，因为这两个文件具有相同的哈希。fun.java 和 games.java 也是如此。这意味着在进行穷举的两两比较时，不需要包含 hex.java 和 hexadecimal.java，fun.java 和 games.java 也不需要。所以，最终我们只需检查以下文件对的相似度：, copy.java)。这让我们从 12 对降到了 2 对。

这个简单的想法几年前就实现了……但有个问题。执行重命名检测的代码也执行复制检测。在复制检测中，单个源文件可能与多个目标文件配对。因此，当找到共同的哈希时，你可以移除一个目标文件（使 M 减一），但必须保持源文件数量不变。这意味着在上面这种情况中，我们可以移除 hexadecimal.java 和 games.java，但不能移除 hex.java 或 fun.java——基于 copy.java 可能是这两个文件中任意一个的副本的事实。因此我们最终需要进行总共 3*2 对文件的比较。6 对少于最初的 12 对，但仍然比仅仅 2 对的比较多得多。

当然，只有在我们需要进行复制检测时，才需要进行这六次比较。如果只需要重命名检测（如合并算法的情况），那么我们只需要两次。遗憾的是，实现重命名和复制检测的代码被编写为总是执行较大数量的比较，这对于重命名检测来说，基本上意味着它花费时间去寻找比已为源文件找到的完美匹配更好的匹配。

诚然，这种优化仅适用于在合并基与移动/重命名它的分支末端之间，文件被移动/重命名而未进行任何编辑的情况。因此，结果很大程度上取决于你查看的特定仓库和特定提交。然而，让重命名检测不去寻找比完美匹配更好的匹配，在一个涉及将 35 个补丁变基到一个重命名了约 26K 个文件的分支的特定测试用例中，提供了约 3 倍的加速：

                       Before                   After
mega-renames:   5504.231 s ±  5.150 s   1799.937 s ±  0.493 s

我尝试过该优化的第一个包含大量重命名的真实测试用例（一次简单的单提交拣选）见证了 2 倍的加速。这是迄今为止最简单的优化，但它仍然提供了相当可观的加速。有关此优化的更多细节，请参阅上游讨论。

这项优化将出现在 git-2.31 中，预计于 3 月中旬发布。
