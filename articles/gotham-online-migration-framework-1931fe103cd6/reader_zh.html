为什么 Palantir 选择在线升级而非离线升级

业界有两种常见的策略来变更存储数据的结构：离线升级和在线升级。

大家对离线升级应该不陌生。你正在用来阅读这篇文章的设备，极有可能就是采用这种方式升级的。当设备软件有新版本可用时，系统会提示你保存当前工作并关机以执行升级。升级完成后，你重新启动设备，就可以使用最新版本的软件了。

从软件开发的角度来看，离线升级非常理想。它实现起来相对简单，且易于理解。正因如此，如果条件允许，离线升级往往是首选策略。

Palantir 没有这种条件。我们的软件支撑着全球全天候的关键任务，且经常处于生死攸关的情境中。系统必须时刻保持可用，因此我们需要执行在线数据迁移。

在线升级的挑战

在线数据迁移比离线迁移复杂得多。如果软件需要实时升级，就必须在停用旧版本之前，确保新版本已启动并可用（Apollo 支持滚动蓝绿部署升级）。旧代码要使用旧格式的数据，而新代码则要使用新格式的数据。

举个非技术性的例子，假设你要增加一张桌子的高度。离线迁移会非常直接：拆下桌腿，换上更长的桌腿，再把桌子立起来。但如果在操作过程中桌子仍需正常使用呢？你需要一次小心地更换一根桌腿。然而，在短桌腿仍在支撑桌子的情况下，你究竟该如何将更长的新桌腿装上去呢？

当然，对软件有高可用性要求的并非只有 Palantir 一家；许多公司都需要对海量数据执行在线迁移。Stripe 技术博客 中对此的标准策略进行了很好的阐述。我们在升级 Gotham 并采用 AtlasDB 作为数据的事务层时，使用了非常相似的策略。简而言之，该策略如下：

  • 为待迁移数据创建目标位置（例如，在关系型数据库中新增一列或一张表）。
  • 将数据从旧格式迁移至新格式。在迁移过程中，代码应读取旧格式，同时写入新旧两种格式。
  • 迁移完成后，代码可仅对新格式进行读写，旧格式则可删除（例如，删除不再使用的列或表）。

这种方法从宏观层面看相当直接。然而，就像桌子比喻中的类似方案（即制作可伸缩桌腿，在逐一更换每根桌腿后再将其拉长）一样，要把细节处理好所带来的技术挑战，远比离线迁移大得多。工程师可能要花费数周甚至数月的时间，为生产环境规划、开发并谨慎执行此类迁移。如果仅需在一个或少数几个生产环境中运行此迁移，且工程师能够直接访问这些环境，这通常是一个可控的项目。

Palantir 再次失去了这种便利条件。我们的软件部署在数百个环境中，包括水上、水下、空中以及远离本土的偏远哨所。即便我们能设法使用标准策略升级这些环境，这种方式也无法规模化。我们必须实现迁移执行的自动化。

此外，Palantir 的迭代速度极快。我们不断改进软件并开发新功能，这就要求数据迁移必须频繁进行。我们一些迭代最快的项目几乎每月都需要进行重大的数据迁移。如果每个月都要花一个月时间来开发数据迁移，对研发进度将是毁灭性的打击。

在线迁移框架的益处

在线迁移框架是我们应对实时数据迁移挑战的解决方案。该框架行之有效，因为它在数据、部署与执行以及开发三个层面都具备出色的扩展性。

为了更具体地说明这个框架，我将使用一个经典的迁移示例：将姓名从单一的 last, first 列更改为 last_name 和 first_name 两个独立的列进行存储，如下图所示。我将使用伪代码来展示此类迁移的开发过程大概是什么样。

编写迁移

在线迁移框架首先为开发者提供了一种编写迁移的方法。这分为三个部分，看起来应该很眼熟：

  • 迁移前模式新增：开发者创建表、列以及任何其他必要的模式元素，以支持新的数据格式。
CREATE TABLE Users_2 (
   user_id VARCHAR2(128),
   last_name VARCHAR2(128),
   first_name VARCHAR2(128),
   CONSTRAINT pk_users_2 PRIMARY KEY (user_id)
);
  • 迁移批次：开发者编写代码，将数据分批从旧格式迁移至新格式。框架还为迁移提供了数据存储区，用于在此阶段记录进度，因为运行迁移的进程随时可能发生故障。我们不希望让一个耗时的迁移从头再来。

List<Users2Record> userBatch = sql.select(Users.USER_ID, Users.FULL_NAME)
    .from(Users)
    .where(Users.USER_ID.greaterThan(getLastProcessedId()))
    .orderBy(Users.USER_ID.asc())
    .limit(batchSize)
    .fetch(row -> createUsers2Record(row.get(Users.USER_ID), row.get(Users.FULL_NAME)));
if (userBatch.size() > 0) {
    List<Query> inserts = userBatch.stream()
        .map(record -> sql.insertInto(Users_2).set(record).onDuplicateKeyIgnore())
        .collect(Collectors.toList();
    sql.batch(inserts).execute();
    updateLastProcessedId(userBatch.get(userBatch.size() - 1).getUserId());
} else {
    markMigrationCompleted();
}

  • 迁移后模式删除：开发者删除不再需要的模式元素及其包含的数据。
DROP TABLE Users;

如前所述，由于运行这些迁移的进程随时可能发生故障，因此每个部分都必须是幂等的，这意味着多次运行是安全的，且效果与运行一次相同（对于 DDL 语句，只需捕获数据存储区返回的任何异常并检查其是否为预期类型即可，例如表已存在）。

支持回滚

这些迁移还需要支持回滚。缓解问题最快的方法是恢复到上一个已知正常的运行状态。这通常意味着将软件降级到较早的版本。但如果此时正在进行迁移呢？旧版本的代码可能不知道如何处理这种状态，因此我们需要将迁移回滚。由于回滚迁移的唯一目的就是为了缓解问题，我们要求回滚必须迅速。事实上，唯一应该实现它的地方就是上述步骤一，这暗示着回滚代码的执行速度应该与删除新建的列或表一样快。

更新代码以处理进行中的迁移

开发者还需要更新访问此数据的代码，以确保其在迁移前、迁移中和迁移后都能正常工作。为此，我们构建了迁移代理。开发者可以将数据访问代码置于接口之后，并用 @Read 或 @Write 注解接口的方法。然后，他们可以针对数据的新状态和旧状态分别实现一次该接口。通过 Write 注解，开发者还可以指定迁移进行期间的具体行为：

  • 写入两种实现（默认）。
  • 仅写入新实现——例如，在表中新增一列，且新写入的内容是旧写入内容的超集。
  • 仅写入旧实现——例如，当迁移仅删除一列，且旧写入内容是新写入内容的超集时。
  • （极少见）写入一个“中间”实现，在迁移进行期间提供自定义处理。

在示例中：

public interface UserStore {
  @Read
  User loadUser(String userId);  
  @Write
  void storeUser(User);
}

这听起来可能工作量巨大，但我们已经开发出一套流程：开发者维护一个与最新数据格式保持同步的主实现，并通过继承该实现，仅重写为向后兼容每次迁移所需重写的方法。正如这所暗示的，这个过程可以对影响同一数据的多次迁移重复进行，迁移代理将接收一个实现列表以及它们之间的迁移路径。基于此列表，它将根据每次迁移的状态选择正确的实现。在我们的示例中，UserStore 的迁移代理将如下创建：

UserStore proxiedUserStore = SimpleMigrationAwareProxy.wrap(
    UserStore.class,
    new LegacyUserStoreImpl(dbSession),
    new UserStoreImpl(dbSession),
    USER_STORE_MIGRATION_TASK_ID,
    dbSession);

通过迁移运行器协调迁移

这些迁移在提交时都会获得一个单调递增的 ID（例如 USER_STORE_MIGRATION_TASK_ID），并由我们的迁移运行器按顺序执行。通过让迁移在每个环境中以相同顺序运行，我们能够将数据存储区可能的状态数量控制在可控范围内，同时还能查看任何服务的迁移状态，无需直接检查即可立即知晓数据存储区的状态。

在运行迁移时，运行器会在每次更改迁移状态时获取该状态的写锁。同时，代理在执行操作期间会获取该状态的读锁。关键在于，这使得大规模迁移能够安全运行，且对代码其余部分的影响降至最低，因为多个线程可以在执行数据访问操作时持有读锁。运行器只需在数据存储区中更改迁移状态的短暂瞬间持有写锁，而非在整个迁移执行过程中持有。

运行器启动时，会与该服务的其他节点执行共识协议，以确定所有节点都知道哪些迁移。这一点至关重要，因为随着我们在不断缩短的开发周期中持续交付新软件，服务中出现多个节点部署异构软件版本的情况将越来越频繁。运行器需要确定所有节点都支持哪些迁移，以便即使在服务通过 Apollo 接收频繁滚动升级时，也能继续推进一系列迁移工作。

采用预热时间与分阶段迁移

为了加速回滚，我们在迁移可用后、开始运行前，设置了一段可配置的“预热时间”（默认为四天）。迁移运行器会跟踪每项迁移最初可用的时间。预热结束后，运行器开始工作。运行迁移分为多个阶段，比前面提到的三个部分要多，以应对运行器在执行任何节点发生故障的情况。

  • 未初始化：迁移尚未开始，所有代理均读写旧数据格式。
  • 回滚中：迁移正在回滚至未初始化状态，但可能尚未完成。
  • 初始化中：运行器正在执行迁移前模式新增，但可能尚未完成。如果需要降级，则需要运行回滚。代理仍读写旧格式。
  • 运行中：迁移前模式新增已完成，迁移已（可能）开始运行其批次。代理读取旧格式并写入两种格式（或根据开发者指定的行为仅写旧或新格式）。
  • 等待额外操作（可选）：迁移已完成，但需要在框架控制之外采取进一步操作。例如，可能需要将新数据索引到 Elasticsearch 或其他某个二级索引中，迁移才算真正完成。代理现在开始读取新格式，但仍向两者写入以支持回滚。
  • 等待完成：迁移已完成。代理读取新格式，但仍向两者写入以支持回滚。
  • 结束中：运行器正在执行迁移后模式删除，但可能尚未完成。迁移无法再回滚。代理仅读写新格式。
  • 已结束：迁移完成，所有旧模式和数据均已删除。

开发者必须编写幂等的迁移行为，这样即使运行器在任何环节发生故障，我们也能重启该阶段并继续推进而不出问题。我们还在“等待完成”与“结束中”之间设置了额外的预热时间。这使我们能够发现那些只有在新数据格式可用后才会暴露的问题，并在无法回滚到原始状态之前予以解决。

管理迁移生命周期

迁移完成后，我们通常希望移除迁移代码。但迁移究竟何时算完成？它可能需要在数百个独立的环境中运行。有些迁移耗时不到一秒，而有些则可能需要数周。

为了明确何时可以移除迁移代码，我们将特定版本指定为“检查点”。要安装检查点之后的升级，服务必须完成直至该检查点的所有迁移。Apollo 会记录（1）每个版本支持的模式版本，以及（2）基于已运行的迁移得出的当前线上服务的模式版本。对于检查点之后的版本，我们将最低模式版本更改为该检查点中最后一次迁移的版本，以便 Apollo 强制执行此要求。Apollo 还可以要求在降级前先回滚迁移，因为旧二进制文件上报的最高模式版本只会是它所知晓的迁移版本。

得益于前述支持新旧数据格式的多种实现，检查点后的清理工作通常相对简单。由于我们已经有针对所有已完成迁移的完整数据访问代码实现，我们可以删除支持旧版本的旧子类，并直接内联最新的实现，而无需将其包装在代理中。这让我们能够在保持代码库整洁的同时，扩展开发工作。开发者在检查点后只需几分钟即可完成清理。

全平台迁移测试

我们还为在线迁移构建了全面的测试框架。迁移开发者可以利用一个简单的 JUnit 测试扩展，该扩展为被测迁移提供自定义的迁移运行器，并支持轻松的滚进和滚退测试。我们使用同一套扩展框架，在整个产品线中进行更全面的测试。当提交新的迁移时，我们的构建系统会为我们支持的每种数据存储组合生成测试。它还会针对四种迁移状态（未初始化、运行中、等待完成和已结束）生成测试，并在所有这些组合中运行我们的测试，以确保产品在整个迁移周期的所有阶段都能正常工作。

最后，在线迁移框架与服务和数据存储无关，已在我们多个服务和几种不同的数据存储中实现。我们开发了一套抽象测试类，可以通过服务对该框架的实现进行实例化，以快速测试该实现是否正常工作。这使我们能够快速扩展能力，在整个产品生态系统中安全、可持续地迁移数据。

规模化应用在线迁移框架

在线迁移框架极大地提升了我们持续向客户交付改进和新功能的能力。过去几年中，数十名开发者编写了数百个迁移。总体而言，我们已在整个集群中运行了约 15,000 次在线数据迁移，未对用户造成任何中断。在线迁移框架甚至已被扩展到数据迁移以外的用途，并内置了几种常见的迁移类型，因此开发者可能只需编写十行代码即可完成一次迁移，从而让他们能够专注于为客户交付最优秀的技术。
