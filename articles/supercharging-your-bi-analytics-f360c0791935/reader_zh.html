Palantir Foundry: Supercharge Your BI & Analytics

2022-05-25 · Original →

作者：Akshay Krishnaswamy，Palantir 首席架构师

实现互联化运营意味着要超越数据和数据分析的流水线方法。它要求在数据、分析与运营工作流之间建立全保真度的反馈闭环。Palantir Foundry 能够助您实现这一目标：首先，丰富您现有的 BI 与分析工具栈；随后，将其扩展至更动态的决策工作流中。

第一步：利用 Foundry 的数据集成与本体，增强现有 BI 工具栈

Foundry 凭借一流的数据集成与本体能力，为现有的 BI 和分析工具栈赋能。

Foundry 能够集成任何现有的数据资产和模型——从大规模传感器与物联网源、遥测和视频数据、授权的第三方数据，到所有内部数据源。《Foundry 本体》构建于这些集成的数字产物之上，利用它们为构成您业务的真实事物（如工厂、客户、交易）建立一个“数字孪生”。本体作为一个具备多模态存储能力的通用数据层，支持从搜索到时间序列分析再到地理空间可视化等各种各样的终端用户功能。

综合来看，这些能力在以下几个方面强化了传统的 BI/分析能力：

  • 提升数据的可读性与可信度。 Foundry 为您的数据和模型管道提供了交互式的可视化能力，允许您追溯特定数据集或某段逻辑的生成历史。平台会自动追踪从原始数据到使用该数据的应用程序之间的血缘关系，从而无需编写那些一旦发生变化就会过时的管道血缘文档。当团队在运营前线做出决策并生成新数据时，本体会协调技术栈各层之间的交互，确保您的用户应用层在扩展时不会破坏核心数据与模型基础。
  • 内置强大的安全与数据治理能力。 许多 BI 和分析工具缺乏集成的安全机制，而 Foundry 提供了从源头到应用的统一安全、血缘关系和数据治理模型。Foundry 中的每个数据产物都可以关联相应的权限和留存策略。数据管理员可以将治理策略应用于单个数据集、数据集内的行、文件夹、项目等，减少了对管道进行分叉处理的需求，并为您的 BI 应用提供了一个可扩展、安全的基础。
  • 将读写能力嵌入工作流。 本体负责协调数据、模型和操作在异构工具与系统间的流转，从而实现持续学习（下文将详细讨论）。

Foundry 旨在与您现有的 BI 和分析工具深度融合。这通过 REST API、ODBC 和 JDBC 驱动、用于连接数据科学工具的 Python 和 R SDK，以及针对 Power BI 和 Tableau 等常见工具的开箱即用连接器来实现。

第二步：通过交互式决策应用扩展 BI 生态系统

瞬息万变的世界要求我们具备从“被动响应”转向“主动出击”的决策能力。在技术层面，这意味着要赋予技术和非技术用户不仅消费洞察的能力，还要让他们能在动态、高交互性的应用中操作、分析异构数据并采取行动。

因此，在设计 Foundry 时，我们赋予了它拓展受众的能力，让更多人能够转换数据、进行分析、构建模型和应用程序——从而减少对工程资源的依赖瓶颈。

  • 《Quiver》是一个分析套件，允许您对大规模时间序列数据和对象进行绘图、转换和分析。Quiver 依托的时间序列基础设施在卓越性能与可持续架构之间取得了平衡，能够随着数据量和用户规模的增长而扩展，使 Foundry 既是高效的历史数据存储库，又是处理时间序列数据的实时分析工具。Quiver 还提供对象分析功能，使用户能够灵活地创建和比较群组、跨对象群体执行聚合分析，并分析对象之间的链接与关联。
  • 《Contour》是一款点击式分析、转换和报告工具，非常擅长为具有临时自上而下工作流的非技术用户赋能。Contour 允许您快速访问数据集，按顺序执行常见的分析和逻辑操作以探索数据、排查数据质量，并对数据进行清洗和转换。所有这些都可以大规模进行——例如，它允许对包含数亿行的大型数据集进行分析。
  • 《Foundry Reports》、《Object Explorer》、《Fusion》以及其他工具为用户提供了无尽的选择，以进一步分析集成的数据并采取行动。

除了这些开箱即用的功能外，Foundry 意识到许多高价值的工作流高度特定于您的组织。为此，我们设计了《Workshop》，这是一个模块化应用构建器，赋能多元化的构建者群体来开发和管理复杂的读写应用程序。

Workshop 让您的团队能够使用高质量的布局和动态组件来构建应用程序，构建者可以在其中阐明与其用例相关的输入、输出和操作。以下是使用 Foundry 模块化构建组件配置的交互式应用程序的几个示例：

  • 制造业物料清单应用 使组织的原材料采购团队能够对现货采购机会进行计算，并计算出更廉价的材料如何在价值链中被利用——所有这些都可以在调整配方限制、现有库存和预测需求等参数的同时进行。
  • 零售通用作战图 允许组织的门店经理将 COVID 病例信息叠加在零售门店的销售和运营数据之上，然后根据本地病例数的变化调整库存和其他计划。
  • 航空飞行告警收件箱 向飞机调配经理展示一组清晰的告警，指出需要进行分类、审查和处理的潜在中断事件。该收件箱包含每个告警的详细视图，操作路径允许用户就飞机调换做出决策，并直接从应用程序中触发运营系统的变更。

通常，这些应用程序在数小时内即可部署完成，极大地推动了概念的快速迭代，与完全定制的解决方案相比，节省了数周或数月的时间。通过抽象掉托管、身份验证、授权、数据集成、监控等技术障碍，Workshop 大幅降低了开发成本，让您能够专注于构建本身。

第三步：驱动组织级决策流程

虽然每个 BI/分析应用的质量至关重要，但整个系统作为一个整体的互联性同样重要。如果缺乏跨工具的智能网络，组织可能会在数据、逻辑和流程上出现割裂，进而拖慢其适应新环境和实现改进的步伐。

为了在您的 BI、分析和运营工具之间实现深度的运营互联，Foundry 采用了一种称为决策编排的方法。传统的 BI 和分析缺乏基于决策的动态工作流，而 Foundry 的决策编排层会捕获面向用户的应用程序中做出的决策，并将结果写回组织的本体中，从而创建高速组织学习所需的反馈闭环。

重要的是，我们在设计 Foundry 时，使其不仅能够记录过去的决策，还能让您模拟未来可能出现的状态。通过利用基于本体的应用程序，决策者可以进行“假设”分析，并对行动的下游后果进行场景测试，从而更好地确定这些行动如何为全局业务目标做出贡献。

为了运行这些模拟，Foundry 本体融合了来自集成数据和模型、操作行动系统等多个来源的输入。随后，平台将所有决策、结果和相关上下文捕获回本体，并同步到底层的操作行动系统（如 ERP、CRM、MES、资产配置、边缘端）。这种运营反馈会为未来的模拟提供信息，改进数据和 AI/ML 模型，驱动更高层级的决策，并加速组织内部的学习。
