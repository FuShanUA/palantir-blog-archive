Securing Agents in Production (Agentic Runtime, #1)

2026-01-22 · Original →

编者注：这是探讨 Palantir AIP 智能体运行时的系列博文的第一篇。该运行时是一套集成的工具链，用于在关键任务环境中构建、部署和管理智能体。

从第一天起，Palantir 的客户就要求严格的安全和数据治理能力，这些能力远非针对数据池的常规基于角色的策略所能及。这包括一套能够融合基于标记、基于用途和基于角色的策略的安全架构；跨越数据、逻辑、动作和应用产物的动态血缘关系；以及一整套适用于人类驱动和智能体工作流的集成变更与发布管理工具。在这篇博文中，我们将探讨 AIP 如何在整个运营生命周期内保障智能体的安全。

设计运营型 AI 智能体时，有几个安全“维度”是任何交付的核心：

  • 安全且具备弹性的推理核心访问机制
  • 智能体执行器的隔离编排
  • 跨越所有形式记忆的细粒度策略执行
  • 受治理的多模态工具访问与调用
  • 智能体活动的实时可观测性与事后审计

安全且具备弹性的推理核心访问机制

智能体设计的起点是选择大语言模型。有趣的是，随着前沿模型能力的提升，它们在关键基准测试上的资质和表现也日趋同质化；我们正处于一个“商品化认知”的时代。因此，Palantir AIP 通过区域性分布的模型枢纽，提供对广泛商业和开源模型的可扩展访问。与 LLM 打过交道的开发者都知道，建立模型端点相当简单，真正的复杂性在于确保在作用范围可能发生剧烈变化的异构工作负载中保持性能稳定。

对于绝大多数客户，Palantir 采用了共享安全模型。这意味着 Palantir Foundry 和 AIP 的核心基础设施安全——包括防火墙配置、网络流量监控、对所有静态和传输中数据的强制加密、临时计算环境的管理，以及新服务更新和安全补丁的持续交付——为模型枢纽、数据目录和管理服务提供了坚实的基础。对 LLM 的访问可以基于企业组织或用户组进行控制，并视地理可用性或管理员设定的分类策略而定。此外，在常规工作流中（例如提供平台内指导和编码辅助的 AIP Assist）使用 LLM 的权限，可以与将这些 LLM 用于智能体构建和深度开发的权限清晰地隔离开来。

通过 Palantir AIP 模型枢纽提供的每个模型都提供严格的保障：第三方模型提供商不会保留任何提示词或补全内容，也不会将交换的数据用于模型训练。所有模型使用情况都可以作为统一资源管理的一部分进行追踪，该功能会自动追踪特定项目、工作流和用户的 Token 使用量。团队也被允许接入并注册自己的模型，其中可能包括现有的订阅模型或微调模型。这使得开发者在迭代过程中能够自由切换模型，无需担心合规性、可用性或规模扩展问题。

智能体执行器的隔离编排

智能体可以被定义为一个有状态的控制循环，它不断调用无状态的推理核心（如前沿语言模型），解释其输出，执行工具和记忆操作，并将结果反馈回去，直到满足终止条件。这一定义突显了对稳健计算底座的需求，在此底座中，编排过程除了受开发者自行配置的控制约束外，还受到基础设施和平台级护栏的保障。Palantir 强化的 Kubernetes 基础设施 Rubix 为所有智能体操作提供了基础保障：每个工作负载都根据必要需求进行安全隔离，从而能够安全执行具有精确治理权限的特定应用运行时。环境中的每个元素都严格执行加密，工作负载之间的每次交互都必须根据不可变配置进行身份验证、授权和记录。

值得信赖的基础设施为控制循环的构建带来了极大的灵活性。智能体可以根据企业上下文的细粒度更新、预定义的时间表或来自外部的触发条件来执行。这些条件通常被编纂到《本体论》中——这是支撑 Palantir Foundry 和 AIP 中人类与 AI 活动的数据、逻辑和动作的通用模型。自动化运行时、本体论后端服务以及所有其他服务都受益于 Rubix 提供的临时性、自动伸缩的基础设施，并被设计为可支持数以万计的并发智能体编排规模。单个编排可以配置回退效应，对错误信息进行打包，并利用不同的重试策略（如恒定退避或指数退避）。关键在于，高可用性与 Rubix 的临时性理念交织在一起；Rubix 环境中的节点存活时间不能超过 48 小时。这确保了每一个智能体编排，无论是通过 Code Workspaces 这样的纯代码环境构建，还是通过 AIP Logic 这样的低代码环境构建，都是为应对中断和弹性故障转移而设计的。

无论智能体的控制循环如何配置，编排本身都是平台级的原语，遵循与系统连接、本体论上下文、分析产物和应用相同的安全控制。特定智能体的运行权限取决于以下三者的综合作用：一是配置并“拥有”它的用户；二是智能体服务用户和/或智能体 OSDK 作用域；三是智能体在特定时刻代表其执行操作的用户。生产用例通常需要这些权限方案的某种组合，并需要穿越复杂的策略——例如，必须尊重源自细粒度数据策略的个体用户权限的条件效应。以一个评估销售机会的智能体为例，这些机会包含必须进行地理限制以仅对特定用户可见的详细信息。当智能体处理来自欧洲的数据并执行下游分析时，只有已具备欧洲数据访问权限（通过基于角色或基于标记的控制）的用户才会收到新销售产出的通知。其他用户不仅不会收到通知，甚至在 AIP 的编排界面中根本无法看到该活动的记录。

跨越所有形式记忆的细粒度策略执行

在智能体架构中涌现的众多记忆模式中，有四个常见类别：工作记忆、情景记忆、语义记忆和程序记忆。工作记忆与当前循环中智能体可支配的信息有关；即明确用于反馈后续调用并达成完成标准的提示词信息和工作变量。情景记忆跨执行会话存储相关信息，通常侧重于为后续操作提供参考的时间标记。语义记忆代表已习得的知识和技能集合，在取向上往往更多是分类性的而非时间性的。最后，程序记忆通常是旨在补充模型参数权重中隐含知识的代码，用于帮助驱动稳定执行和可靠地使用工具。

AIP 中的本体论系统旨在服务于上述每一种记忆模式——提供灵活的读写接口，同时确保对安全和数据治理策略的一致遵循。以物流支持智能体为例：智能体通过程序记忆进行初始化，程序记忆可以作为存储在本体论对象中的字面代码存在，也可以通过编排应用（如 Agent Studio、AIP Logic）来促成。在初始化过程中，系统和任务提示词都可以通过本体论 SDK 调用，从指定工作记忆的策展本体论对象（“知识节点”）中加载。此外，语义记忆也通过关联的知识节点加载；可能包括相关客户问题的详细情况、该客户过去提供的反馈，以及关于更广泛上下文（包括全规模生产、履约、支持数据）的指导，以便查询供应链的当前状态。随着执行的推进，情景记忆可以通过对向量化的本体论属性进行相似性搜索来视情况加载，使先前的经验能够引导控制循环。在整个执行过程中，工作记忆的变量和其他元素可以通过《本体论动作》进行更新——这不仅提供了一个临时的草稿本，还形成了一个丰富的会话账本，可以为反馈驱动的学习提供动力。

细粒度策略可以附加在本体论各处，以限制智能体和人类对敏感或上下文相关信息的访问。这些策略在运行时针对每次交互进行动态计算，并可结合应用于底层数据集的行级和列级限制、特定用户组的属性（包括通过 SSO 传递的属性）、在底层数据管道中传播的安全标记等等。在物流案例中：可能存在一组限制访问包含 PII 的用户数据的安全标记，以及一组只有在智能体作为具备特定属性的人类用户代理运行时才能查询的客户详细信息。当管理员随时间推移管理策略时，每一次变更都会被自动捕获到审计日志中。

本体论的模块化架构反映出这样一个事实：在管理和使用数据方面，不存在放之四海而皆准的方案。针对对象和向量数据、流数据、地理空间数据和媒体数据等不同模式，系统采用了专门的存储。跨越这些数据类型的通用治理是缓解攻击向量的关键，包括提示词注入风险。智能体不仅要检索相关的功能知识，还要编排利用企业核心运营信息的查询，因此维持一致且严密的安全防护至关重要。通过将本体论作为多用途记忆系统，AIP 确保了工作、情景、语义和程序结构具有统一的交互面——无论对于人类用户还是智能体用户皆是如此。

受治理的多模态工具访问与调用

在智能体范式中，任何带有接口的信息源、系统或执行器都成为了可被调用的潜在工具。最常见的工具类型是面向数据的，涉及对特定信息源的查询。此外还有面向逻辑的工具，用于执行某种计算——无论是简单的矩阵乘法例程，还是利用多种计算模式的复杂模拟。与这些工具相辅相成的是面向动作的工具，它们以某种方式改变世界的状态；无论是 ERP 中的事务性更新、数字应用中的状态更新，还是传输给工业机器的新指令。保障这样一个异构且不断演进的工具环境，需要在系统层面进行强制执行。

工具的使用通过治理记忆的同一安全架构进行动态强制执行。这至少确保了任何工具调用都取决于对底层本体论中对象、属性和链接的访问权限。此外，工具可以包含依赖于细粒度提交标准的运行时验证。在前述物流案例中，智能体更新客户订单的能力可能取决于智能体所属群组、客户的实时状态以及预估交付窗口的综合情况。每一个智能体动作都依赖于明确的授权许可，这些许可明确规定了允许的操作集合，从而防范意外调用（例如查询跨组织边界的数据，或连接到未指定的外部系统的工具）和其他形式的权限提升。

在与外部系统管理连接时，AIP 为开发者提供了分层防御。除了由 Palantir 信息安全团队管理的基础设施级网络和防火墙保护外，开发者还可以管理与人类和智能体工作负载执行相绑定的出口策略。这些策略治理着所有指向外部数据源、Webhook 或 API 端点的出站连接。任何向外部系统的导出都必须明确配置安全标记，以定义可传输的数据类型。

正如在 DevCon 4 大会上所展示的，管理跨智能体集群工具使用的最有力抓手是基于来源的安全机制。基于来源的安全机制使开发者能够构建工具（或任何类型的函数），在运行时尊重附加在数据源和其他原语上的强制控制。关键在于，随着简单函数被嵌套到更复杂的工具中，这种机制也能随之扩展；整个调用链在运行时被解析，例如防止标记数据在未经批准的执行（包括导出任务）中被下游使用。在“编译期”，基于来源的控制还会阻止违反策略的新工具版本发布，并向开发者提供清晰的错误信息。

智能体活动的实时可观测性与事后审计

没有可观测性就没有可靠性，对于非确定性的智能体尤为如此。虽然智能体的控制循环定义了相关的交互面（例如潜在的推理调用、记忆模式、可能的工具调用），但穿越“决策流形”的可能路径往往数不胜数，且在功能深度上可能存在巨大差异。因此，开发者必须拥有一个能够收集链式执行中所有相关遥测数据、可视化完整流程并实现更有针对性的下游工作流的系统——包括性能和合规性驱动的监控。

AIP 采用了一种集成的可观测性方法，实现了从“数据到决策”的监控。对于给定的分布式追踪，可以审查和探究各个 LLM 查询、函数调用、数据转换和其他离散操作。与常规平台不同，AIP 中的每一次追踪都是深入分析的起点。每一个数据查询都可以关联到给定数据源的完整版本历史——包括用于生成该特定数据版本的转换逻辑。每一个函数调用都可以关联到语义版本，以及所采用的 Webhook 或源连接的版本化配置。对于 LLM 驱动的函数（如构成子智能体的函数），相关的 Evals 套件会被自动追踪；这提供了对特定调用和响应如何随时间推移发生变化的细粒度记录，所有这些都在定性或定量标准定义的允许范围内。

随着智能体生成详细的遥测数据，日志的安全和传输成为了关键的“最后一公里”问题。AIP 允许管理员控制如何跨特定项目、工作流和智能体访问日志。数据标记和其他活动的安全原语对日志访问进行治理，其方式与治理对底层数据、逻辑和动作原语的访问完全相同。管理员可以选择允许将智能体遥测数据实时导出到平台中的流式数据集，或通过授权的导出任务导出到外部系统。Foundry 的内置应用（如 Quiver 和 Contour）提供了用于分析大规模遥测数据并生成合成下游数据集的强大工具，从而为闭环工作流提供动力。
