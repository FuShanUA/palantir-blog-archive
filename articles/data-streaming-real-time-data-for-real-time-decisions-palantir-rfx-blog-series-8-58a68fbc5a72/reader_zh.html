Data Streaming: Real-time data for real-time decisions (Palantir RFx Blog Series, #8)

最关键的业务决策往往也最具有时间敏感性。数据流技术让组织几乎能在信息涌入的瞬间就采取行动。

编者按：本文是《Palantir RFx 博客系列》的第八篇，该系列探讨组织如何更好地撰写 RFI 和 RFP，以评估数字化转型软件。每篇文章聚焦数据生态系统中的一个关键能力领域，旨在帮助企业提出正确的问题，从而更好地评估技术。此前发布的章节包括探讨《本体论》、《数据连接》、《版本控制》、《互操作性》、《运营安全》、《隐私增强技术》以及《建模目标》的文章。

引言

“实时数据”改变了人们的生活方式和企业的运营模式。数据采集后即刻送达的能力，触及并改善了我们诸多最基础的功能——从查看天气、点餐到汇款——仿佛在一夜之间，改变了人们对信息、产品和服务的交付与消费速度的传统预期。在本文中，我们将实时数据定义为能够以低延迟摄取、处理并采取行动的任何数据，而数据流则是将实时数据纳入企业数据生态系统所需的一组能力。

对于企业而言，实时数据技术虽然耗时更久才充分发挥作用，但对特定业务的整体效能同样具有变革性影响。随着数据生态系统处理数据的速度越来越快，组织对这些数据流如何支撑关键决策的期望也随之提升。具体而言，食品饮料制造商现在可以在发现缺陷后几分钟内暂停特定产品的生产；航空公司可以在发现不利天气模式后几分钟内更换飞机并调整机组人员日程；银行可以在标记可疑交易后几秒内冻结账户以待审查。通过将数据采集与决策之间的间隔缩短至近乎为零，实时数据为无数业务流程带来了实质性的改进，在某些情况下甚至催生了全新的能力。

与许多其他变革性技术一样，实时数据的承诺与实现之间的鸿沟可能十分巨大。在 RFP 的语境下尤为如此，“千兆级吞吐量”和“毫秒级延迟”的承诺屡见不鲜，却无法在纸面上得到验证。在本文中，我们将基于协助众多行业和政府机构从实时数据中创造价值的经验，探讨实时数据及其支撑流技术的一些重要考量因素。

什么是数据流？

数据流是指在数据生态系统内处理实时信息所需的一组能力。¹ 流可以指代任意数量的操作，包括数据采集、摄取、转换、建模、情境化、回写、分析和行动——对于组织而言，在寻找可能的解决方案之前，明确界定他们所说的数据流究竟意味着什么至关重要。毕竟，“数据流”和“实时数据”在不同语境下的含义可能截然不同。点午餐或给朋友转账的“可接受延迟”，与派遣急救人员或标记潜艇 GPS 系统故障的可接受延迟有着天壤之别。在每一个用例中，数据源、系统架构和消费者期望都大相径庭，这进而改变了数据流应用于这些语境时的确切定义。

流数据源通常与“批处理”数据源区分开来，后者顾名思义，是指按照特定计划以离散分组形式处理的数据。² 广义上讲，数据流的决定性要素是速度和连续性。数据流管道以低延迟（通常在几秒或更短时间）和快速、连续的间隔处理数据。流数据通常以连续的时间戳“事件”序列形式进入数据生态系统，因此常被称为“事件流处理”和“实时数据流”。

从这个意义上讲，数据流的定义很简单——在数据生态系统中快速、连续地传输信息。但是，当考虑到构成流管道的所有操作，以及更广泛的数据生态系统所执行的所有任务时，在真实世界数据生态系统的语境下定义数据流可能会变得困难。截至目前，在这个《RFx 博客系列》中，我们已经讨论了数据生态系统的一些关键组件——《本体论》、《数据集成》、《数据管道版本控制》、《互操作性》、《运营安全》、《隐私》、《AI/ML 建模》——而要让一个数据生态系统真正纳入流数据，它必须能够将上述每一种能力（乃至更多）应用于高吞吐量、低延迟的数据。下表突出显示了功能性数据流架构的一些关键能力，分为数据摄取、数据转换和数据消费三个阶段。

数据流 — 关键能力

从平台架构的角度来看，核心问题不在于如何为流数据设计一个独立系统，而在于如何设计一个系统，让最终用户能够与流数据源及传统批处理数据源进行同等交互。换言之，最有效的数据流解决方案是，系统用户能够在相同的治理策略下，以相同的方式与所有集成的数据源进行交互，而无需顾及数据究竟来自哪种数据源。这之所以具有挑战性，是因为流数据和批处理数据所需的后端技术大相径庭。你无法将流数据直接导入专为传统数据源设计的平台，正如你不能给波音 737 绑上更大的引擎，就指望它能安全地超音速飞行一样。

数据流 — 功能架构

鉴于与流相关的用例和数据源范围广泛，构建有效的数据流架构有多种方式。但各系统之间的主要功能组件往往相似。上图展示了其中一些关键组件以及它们是如何协同运作的。

  • 数据摄取将流数据从源系统引入数据生态系统。数据连接 API 执行长时间的摄取任务，而流代理则暴露端点以接收来自外部系统的新记录，随后对信息进行验证并摄取到平台中以供进一步处理。
  • 数据转换对摄取的数据应用多个处理流程。数据存储在诸如 Kafka 这样具备集群和高可用性的数据存储解决方案中。诸如 Apache Flink 之类的计算引擎用于对存储的数据运行分布式流转换和计算。该引擎由一个负责协调集群内工作任务的作业管理器和一个直接对数据执行转换的任务管理器组成。集群管理器负责管理生命周期任务，例如保持 Flink 版本更新和下载用户代码 jar 包，而流工作者则协助启动新集群。流归档器将流数据读取到冷存储中，将信息组织到已知主题及其对应的数据集中。系统如何平衡存储考量（低延迟但昂贵的热存储与慢速但高性价比的冷存储）是任何流解决方案的关键考量因素，正如下方要求部分最后一条所述。流数据目录维护流（热）存储与归档（冷）存储之间的数据映射，以确保高效的存储操作。
  • 数据消费包含让用户根据其特定需求和工作流与流数据进行交互的机制。可插拔接收器为超低延迟数据流提供了一个带有倾向性的框架，从计算引擎接收数据并将其发送到指定目的地，以供下游应用使用。本体论回写将流数据连接到本体论的对象和动作上，让最终用户能够在上下文中理解流数据。规则与警报是流行的流处理工作流，组织会建立逻辑来监控系统或资产，并在达到某个阈值时向用户发出警报。用户可以设置接收器，将流数据直接写入目标数据库，例如时序数据库和其他客户数据存储。

事实证明，定义数据流相当困难且涵盖广泛，因为它涉及数据生态系统所执行的所有操作，只不过是针对流数据罢了。由于大多数数据生态系统都是为更常见的批处理数据源设计的，因此无缝纳入流数据的能力对许多企业来说是一项重大挑战。正因如此，评估流技术时，不仅要看它能否满足与延迟、可用性和可扩展性相关的严格标准，还要看它与其他数据资产和平台组件的交互程度如何。

数据流为何重要？

当分秒必争时，数据流就至关重要。企业处理实时数据源的速度越快，它就能越快地针对该数据源传达的信息做出响应。

当然，实时数据流究竟如何集成、转换并采取行动，在不同组织和行业间存在差异。航运公司可以利用突发的天气数据来调整特定集装箱船队的航线和日程；金融服务公司可以利用实时交易数据来评估特定账户或交易存在欺诈或洗钱的可能性；汽车制造商可以利用工厂车间焊接机器人的传感器数据来识别特定缺陷并相应地暂停生产。对于我们的许多客户而言，数据流显著缩短了关键工作流的时间周期。对于其他客户，它则催生了以前根本无法实现的全新工作流。

以海上石油平台的建设为例。启动石油钻井平台是一项危险且资本密集的工程。它需要多支工程师和建筑工人团队来建立油井本身以及其上方的管道、起重机、仓储和生活区生态系统。现代平台在主井管道和平台本身沿线安装了传感器，这些传感器会流式传输大量与温度、压力和操作限制相关的时间戳数据。数据流技术允许石油天然气公司摄取、协调并对这些数据执行复杂的计算任务，提供实时监控能力以即时识别油井性能问题。结合强大的仪表板和警报基础设施，数据流通过防止人员伤害、环境事故和油井关停，使石油平台变得显著更安全、更高效。

更广泛地说，各组织已经开始期望获得实时数据处理能力。正如个人消费者已经在新闻、餐饮和网约车等领域拥抱了“即时经济”，企业也开始要求立即获取有关其业务的所有最新信息，而不受规模或复杂性的限制。沿用上文的例子，石油天然气行业的客户现在依赖流式传感器数据来确保其管网沿线的安全与性能，因为这些能力代表着他们在保障工人安全和维持油井在线方面实现了变革性跨越。同样，其他行业的客户现在也已将数据流的期望融入到其最关键的工作流中，因为这些工作流能实现被视为不可或缺的流程和结果。

数据流可视为现代数据生态系统必然且必不可少的前进步伐。技术不断进步；计算机速度越来越快；数据变得越发繁杂，这就要求更即时的计算和交付。

要求

首先，谈一谈关于延迟要求的注意事项。与数据流相关的最常见（有时也是唯一的）技术要求涉及延迟 SLA（服务等级协议）。例如，RFP 中通常包含类似如下的流处理要求：“该解决方案对于流数据源必须具备低于两秒的端到端延迟。”虽然这种“对速度的需求”真实且重要，但它忽略了一些影响流解决方案有效性的关键考量。更具体地说，延迟 SLA 必须明确且切合实际地界定在给定时间范围内预期执行哪些操作。这两秒的 SLA 是否包括读取、摄取数据并将其协调至本体论？是否包括转换、连接以及向本体论的回写？随着数据管道的增长以及操作预期列表的同步增加，亚秒级 SLA 在最好情况下可能缺乏实用性，在最坏情况下则可能变得具有限制性，从而挤占那些在现实中无法塞入任意时间窗口的重要数据管道功能的空间。

该流解决方案必须支持与结构化/批处理数据源的全面集成。如果数据生态系统的主要目标之一是统一企业内部的所有数据资产，并为系统用户提供统一的访问入口，那么该系统就必须能够将流数据与批处理数据集成。这种数据源的融合有助于用户结合其他数据源探索实时数据，为后续的行动或决策提供尽可能完整的图景。

该解决方案必须提供对流数据完整历史的即时访问。流数据面临的一个主要挑战是存储成本；数据量和更新频率如此之高，以至于将整个数据历史提供给系统用户往往在成本上是不可承受的。尽管如此，组织仍需要对流数据源历史版本的可靠且持久的访问，否则流系统将以数据生态系统当初设立旨在防止的方式走向孤岛。为了持久地提供流数据的历史版本，流解决方案应包含将数据从热存储转移到冷存储的自动化流程，以及一个定义完善的库，允许在两者之间无缝读取数据。

该解决方案必须对流数据启用灵活的保留和删除策略。流数据的本质——海量、嘈杂且往往只在很短的时间内具有相关性——加剧了对强大保留服务的需求。正因如此，流解决方案必须包含基于规则的保留服务，该服务可配置为根据特定客户要求自动识别并（如果需要）删除信息。此服务必须支持细粒度策略（即指定哪些策略适用于哪些数据源），以及完全可定制的数据保留/删除策略范围。例如，用户应该能够实施一项自动数据保留策略，该策略可识别具有特定元数据特征的流数据子集，在规定时间段内让特定用户组无法访问该子集，然后从系统中永久清除该数据。

该解决方案必须将数据流连接到本体论对象。将实时数据置于上下文中需要的不仅仅是将数据流与其他数据源结合；数据流还必须通过本体论系统地映射到有意义的语义概念。在《上一篇博文》中，我们讨论了本体论的概念——一种通过定义对组织有意义的对象、属性和链接来将数据与含义联系起来的地图——它是任何有效数据生态系统的基础要素。推而广之，在更广泛的生态系统内将流数据置于上下文的过程，涉及将实时数据绑定到其相关的本体论对象——例如石油钻井平台、飞机或人员。只有做到这一点，组织才能对其实时数据拥有完整的上下文视图。

该解决方案必须支持流数据的版本控制和分支，包括对定时事件的细粒度重放。在数据管道的语境下，版本控制是指帮助跟踪和管理数据变更的系统，而分支是指一个独立（或“分支”）的环境，开发者可以在代码获准投入生产并合并回主分支之前，在其中编写、测试和修改代码。在《上一篇博文》中，我们讨论了这些技术对数据生态系统的重要性，它不仅能实现协作、实验和所有变更的细粒度历史记录，还能培养对生态系统本身的集体信任。所有关于版本控制和分支能力为何至关重要的论点同样适用于流数据源——也许尤其如此，因为流数据往往以时间序列形式呈现，这使得定时事件的重放变得更加有用和重要。

该解决方案必须支持对流数据进行实时数据转换，实现快速集成和持续构建。数据管道涉及多重转换，即对原始数据应用一系列增量变更，以使其对下游系统用户有用。由于传入数据更新的量和频率，流数据带来了多重数据转换挑战。要使流架构有效，它必须支持复杂且持久的数据转换，将一个或多个流数据集作为输入，并生成一个或多个流数据集作为输出。开发者还需要一个强大的创作环境来定义这些持续构建并创建用户定义函数（UDF）。

该解决方案必须包含预构建的、开箱即用的常见流数据源连接器。数据流来自许多不同的源头。常见的系统类型包括 Apache Kafka、Google Pub/Sub、OSI PI、Amazon Kinesis 和 Azure Event Hub。将流数据源集成到数据生态系统中的主要成本之一，是建立数据连接所需的自定义开发。如果系统具备预构建的常见流数据源连接器，以及引导用户完成集成过程的向导式 UI，这些成本就可以被减少或消除。

结论

鉴于稳健的解决方案需要涵盖广泛的组件，且在纸面上评估这些组件颇具难度，因此评估流技术可能会非常困难。有效的解决方案既需要专门的流架构来运行摄取、转换和消费任务，也需要一种将该数据集成到更广泛生态系统中的机制。这种集成组件——组织必须通过定义好的数据模型将流数据与其他非流数据源连接起来——是流解决方案的核心，却往往在 RFP 中被忽视，因为 RFP 往往只关注时间延迟和数据源兼容性。评估者若能深入理解流系统如何在更广泛的数据生态系统内将实时数据置于上下文中，以及这些流技术如何在真实世界环境中产生影响，必将从中大获裨益。

[1] “实时数据”当然是一个误称，因为不可能以零延迟处理数据。我们撰写本文的目标之一就是更精确地定义这个被广泛接受的术语。“近实时数据”或“低延迟数据”是更准确的表述（尽管不那么朗朗上口），因为它们暗示了数据采集时间与使用时间之间存在极小的延迟。

[2] 流数据与批处理数据之间的这种区分并不总是准确或有用的，因为流数据也是按批次和定义好的计划交付的——只是这些批次通常更小且频率更高。因此，将流数据和批处理数据视为一个连续体上的不同点，而不是种类的差异，通常会更有帮助。
