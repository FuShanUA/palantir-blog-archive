How Palantir Secures Source Control (Software Supply Chain Security Series, #3)

2025-01-24 · Original →

编者按：这是系列文章的第三篇，分享我们在改善 Palantir 软件供应链安全方面的经验。本篇重点介绍我们在源代码控制风险方面的考量与应对措施。

零信任开发

软件开发者编写代码的工作流多种多样。他们可能在本地终端上工作，也可能在远程托管的开发机上工作，或者在测试/开发环境中工作，抑或在使用其他众多平台。随着构建和开发工具的攻击面不断扩大，源代码遭到破坏的风险也随之增加。攻击者一旦获取了开发者的浏览器会话、SSH 密钥或 API 密钥的访问权限，就能植入恶意代码，绕过安全控制，窃取数据，或者植入漏洞以便日后利用。怀有恶意的内部人员同样可能蓄意实施此类攻击。

要防范这些及其他针对源代码控制的攻击，需要采取分层且纵深防御的策略。

提交签名

将代码提交并推送到我们的 GitHub Enterprise (GHE) 服务器有两种常见的访问模式：通过安全外壳协议 (SSH) 密钥和通过浏览器用户界面。另一种可用但属于反模式的做法是使用 GHE 访问令牌，我们强烈反对这种做法，但并未完全封禁。一旦这些凭证材料泄露或被攻破，攻击者就会有机会冒充该用户提交代码。

我们的第一道防线是 git 提交签名，这是一个为存储的 git 对象创建独立加密签名的过程。我们对提交签名提出了更高要求：所有提交都必须由需要物理在场的受信任硬件密钥进行加密证明。虽然这并非万灵药，但确实提高了攻击者的门槛，因为他们再也不能仅仅通过窃取开发者的凭证或会话来作恶。它还使 git 树具备加密可验证性，从而保护了源代码控制设备上静止状态的提交。此外，在提交从开发者工作站传输到远程仓库的过程中，还能防范中间人 (MitM) 攻击。

基于硬件的签名密钥降低了密钥被窃取的风险。作为额外措施，我们要求必须物理触摸硬件密钥才能启动签名过程。即使攻击者以某种方式在本地开发机上获得了立足点并想自己创建提交，只要用户不触摸 YubiKey，签名操作就不会发生。我们要求用户通过注册流程注册其提交签名密钥，该流程确保硬件及其配置符合多项特定属性。

  1. 用户希望注册的公钥所对应的私钥确实存储在 YubiKey 上，且无法导出。
  2. YubiKey 必须是符合 FIPS 标准的特定型号。
  3. 必须启用签名操作的触摸策略。

我们利用 YubiKey 生成的证明来实现这一点，要求用户在注册签名密钥时，将其公钥连同证明一起上传到我们的自定义密钥服务器。即使用户已经拥有有效的单点登录 (SSO) 会话，密钥服务器也要求他们通过我们的 AD 无密码基础设施进行身份验证。这确保了用户在上传受信任的提交签名密钥之前，还必须通过 YubiKey FIDO 身份验证流程证明其身份。（关于我们通过硬件认证器推行 FIDO2 身份验证的历程，请参阅我们的《无密码身份验证系列》。）

为了强制执行提交签名，我们构建了自己的拉取请求 (PR) 状态检查机制，该机制会从密钥服务器拉取受信任的密钥，并用它来验证每个提交签名。如果所有提交签名均有效且由用户预期的密钥签署，则检查通过，允许合并 PR。这项检查还解决了验证用户通过 GitHub UI 进行提交的问题。由于 GitHub 设备会自动使用自己的密钥对这些提交进行签名，而用户无法通过浏览器对其进行签名，因此我们的检查要求用户手动审查每个 UI 提交，然后通过我们的无密码身份验证流程重新进行身份验证，该流程需要 YubiKey FIDO 身份验证。通过这种方式，我们强制开发者承认归因于他们的基于 UI 的更改，从而确保了不可否认性。

GitHub 确实有一项内置功能，可以强制对分支上的所有提交进行签名。然而，经过测试，我们发现它不够灵活，也不够安全。它有几个缺点，包括要求源分支和目标分支上的所有提交都必须被签名，这意味着必须追溯对仓库中的所有提交进行签名，这对于旧仓库来说非常痛苦。它还允许用户将任意公钥上传到其个人资料中以示信任。这无法保证密钥存储在受信任且配置正确的硬件设备上，也无法保证攻击者不能通过攻破用户凭证来上传有效密钥。最后，基于 UI 的提交将始终有效，我们无法通过基于硬件的认证器来强制执行额外的审查或身份验证。

代码审查强制执行

虽然提交签名大大增加了篡改提交或实质性攻破用户源代码控制帐户的难度，但它无法缓解内部人员或高级威胁行为者植入恶意代码的其他场景。代码审查作为行业标准是有原因的；强身份验证的受信任审查者的眼睛，是捕获各类脆弱代码最稳健的方法之一，无论代码作者的意图如何。

我们通过 GHE 中另一个必需的 PR 状态检查——policy bot 来强制执行代码审查。Policy bot 是一个自定义的 GitHub 应用，旨在提供更灵活的 +1 策略，包括以下功能：

  • 要求特定用户、组织或团队进行审查。
  • 根据拉取请求涉及的文件、作者或分支应用规则。
  • 使用 AND 和 OR 条件组合多个批准规则。

其中一项规则是，我们要求所有非人类提交者和标签创建者必须通过 KMS 支持的密钥对其操作进行签名，并且公钥 ID 须与其在 policy bot 策略中的身份相关联。这确保了攻击者无法伪造任何自动提交者的身份，并帮助我们进一步巩固提交签名的保证。

扩展 policy bot 策略并确保每个仓库都使用经过安全审查的策略是一项挑战。我们目前的实现方式是将所有 policy bot 策略存储在一个中央仓库中，以确保策略经过审查且透明，修改需要多名开发主管的 +1 批准，并且任何不正确的策略都可以在一个位置进行修复，而无需逐个仓库修改。在此之前，每个代码仓库都有自己的策略，这存在几个安全缺陷：

  • 开发者可以定义任意策略，包括那些对更改不需要任何审查的策略。
  • 无法审计每个策略或强制执行标准模板。
  • 如果存在常见的拼写错误、失误或需要更改的地方，就必须在我们的仓库中的数千个策略文件中进行修改。

GHE 权限管理

GitHub 使用基于角色的访问控制模型，包含几个预定义角色。权限最小的“reader”角色允许用户查看代码、问题、PR 和其他仓库元数据。权限最大的“owner”角色授予完全的仓库控制权，包括启用或禁用所有安全设置的能力，例如分支保护规则、机密扫描和静态代码分析。为了维护安全，我们明确禁止开发者持有 owner 角色，因为这将使他们能够通过禁用或覆盖安全控制来绕过它们，例如强制合并未通过所有必需状态检查的 PR。相反，开发者被授予“maintainer”权限，允许他们批准由 policy bot 定义的 PR，查看相关元数据和查看安全扫描结果，但不能编辑任何安全设置。

开发主管和我们的开发者工具团队通过特殊的服务管理员帐户拥有管理员权限，这些帐户与其普通用户帐户是分开的。访问这些帐户需要登录到具有多因素身份验证 (MFA) 的安全、加固且临时的堡垒机。这大大降低了被攻破的普通帐户导致我们的源代码控制安全配置降级的风险。

此外，我们的 GHE 实例默认将仓库设置为私有。这意味着即使拥有有效的 GHE 登录凭证，用户也没有自动查看所有仓库的权限。广泛的查看权限通过我们的入职和离职流程进行谨慎管理，以确保承包商或其他非开发员工不会获得对我们源代码的不必要访问权。

静态代码分析

无论是无意还是有意引入的安全相关漏洞，静态代码分析是识别它们最具可扩展性的方法之一。我们使用 CodeQL 扫描每个 PR 和每个版本以识别漏洞。我们的规则集是标准规则和自定义规则的组合，经过调整以优化构建时间性能，同时保持较低的误报率。我们投入了大量时间开发指标管道来衡量这些性能指标，并根据这些衡量结果管理各种权衡因素。

从静态代码分析中获取价值极具挑战性，尤其是在处理数千个仓库以及广泛的语言和生态系统时。误报、构建缓慢、不必要的 PR 合并阻断、荒谬的发现以及其他诸多问题，都会导致开发者对工具和结果失去信任。此外，编写高信号、高性能的自定义规则需要丰富的经验。我们有多名应用安全 工程师专职负责维护规则、响应错误并监控指标。在以后的文章中，我们将深入探讨有关静态分析实现的更多细节，但目前，将其作为我们代码和源代码控制安全纵深防御方法的一部分来提及就足够了。

安全发布流程

我们使用名为 Autorelease 的内部工具来安全地标记和发布代码。Autorelease 是唯一能够创建标签并触发发布构建的工具，我们通过 pre-receive Git 钩子来强制执行此限制。Autorelease 还强制执行了一组规则，规定哪些分支可以被标记以及哪些用户可以触发标记和发布操作。这对于区分哪些构建可以进入生产环境而哪些不能至关重要。以下是我们如何划分生产构建与开发构建的详细说明。

在 Artifactory 中，我们为内部软件生产版本维护了专门的仓库，我们的部署系统 Apollo 会使用这些仓库拉取软件进行部署。关于如何在我们部署系统中使用这些仓库的强制执行机制虽然属于另一个话题，但涉及确保合规性的内部控制。另外，我们还有预发布和开发构建仓库，这些只能安装在非生产环境中，例如开发云实例或模拟生产环境的开发和测试环境。为了确保代码的完整性和安全性，我们必须为软件进入这些不同仓库的方式设置防护栏。这些防护栏确保只有满足我们安全控制的代码才能进入生产环境。

Autorelease 作为第一道防护栏，确保只有受保护的分支才能被标记为生产版本。然后，构建系统会获取这些标签，并将其发布到 Artifactory 的生产版本仓库中。我们还允许开发者请求 Autorelease 生成预发布标签，构建系统会构建该标签并将其发布到我们指定的 Artifactory 预发布仓库。Autorelease 还对谁可以请求生产版本标签执行权限检查，将此能力限制在仓库维护者范围内，而不是任何具有读或写权限的用户。最后，Autorelease 会对它创建的标签进行签名，以保护标签的完整性并保证其真实性，使标签始终可审计，例如供构建系统审计。

从历史上看，GitHub 围绕标签（定义源代码控制中的发布点）的规则不够灵活且保护薄弱。通过使用 Autorelease，我们建立了高度精细的控制和自定义逻辑，以确保标签创建是可审计、安全的，并且符合我们的构建和制品存储系统对允许进入生产环境的内容的预期。

我们未缓解的风险是什么？

提交签名是我们保护代码仓库和防止未经授权代码引入的纵深防御方案的一部分。其有效性通常是一个有争议的话题，重要的是要承认它无法做到的事情。它不是万能药，有些特定风险它无法缓解。提交签名不能防止提交恶意代码。如果内部人员故意引入恶意代码，或者持久潜伏在开发者机器上的攻击者添加了恶意代码，而开发者在正常更改中不知情地提交了这些代码，就会发生这种情况。提交签名只能确保 Palantir 员工物理在场，并且确实使用其硬件密钥执行了提交操作。为了降低引入恶意代码的风险，我们依赖其他控制措施，例如代码审查和静态分析。

此外，只有在检查签名并使用它来限制该标签的发布或构建时，标签签名才有价值。仅靠签名不能保证任何东西，因此，调整您的构建、发布或部署系统以检查有效的提交和标签签名至关重要。同样至关重要的还有，保护引入受信任密钥的路径以及这些密钥的存储位置。

结论

源代码控制安全需要一种多层方法，该方法能对代码作者进行强身份验证，确保源代码的完整性，防止未经授权的访问，并通过人工和自动化审查者识别脆弱代码。即使拥有这些先进且强大的控制措施，仍然存在坚定的威胁行为者引入未经授权代码的风险。作为我们更广泛的软件供应链安全计划的一部分，我们不断测试、验证和改进我们的控制措施。在下一篇文章中，我们将深入探讨如何使用 in-toto 和证明来保证软件的完整性和真实性。
