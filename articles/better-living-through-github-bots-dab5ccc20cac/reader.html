Better Living Through Github Bots

2026-03-26 · Original →

Editor’s Note: This is the third post in the Palantir Foundations series exploring how we customize infrastructure software for reliable operation at scale.

Palantir maintains approximately 6,000 repositories that touch production, spanning microservices, configuration-as-code, infrastructure-as-code, and other combinations. Each service owns its own repository, which teams independently operate. We refer to this layout as the “polyrepo.”

This architecture enables team autonomy, but it introduces a coordination problem when a developer needs to apply a cross-cutting change across thousands of repositories: a widely used dependency upgrade; a CI configuration change; or a wide migration. The cost scales linearly with the number of repos. The same class of problem appears in monorepos as they grow, though the failure modes differ.

The Developer Tools team and its parent Foundations organization are responsible for solving these cross-cutting problems. This post describes the GitHub applications we’ve built to do so.

Our Solution

Our approach is not a single tool or a platform rewrite. It is a set of focused GitHub applications, built incrementally over several years, that compose through GitHub’s native primitives of webhooks, status, checks, labels and pull requests.

Each application is independent and follows a Unix-like design philosophy: do one thing well. GitHub is the connective tissue between the workflows as bots subscribe to webhook events, evaluate conditions, and take action. Workflows are a side effect of these applications interacting through GitHub’s event system. Standardized build tasks and release processes across all repositories reduces overall frictions.

Core Applications

Policy-Bot
(open-source) — GitHub’s native branch protection rules support basic approval requirements, but lack the expressiveness needed for conditional policies at scale. Policy-bot enables complex, conditional PR approval rules beyond GitHub’s native options and supports path-based, author-based, and team-based predicates. Policies are defined declaratively in each repository or loaded from a shared remote configuration. Internally, policy-bot has evaluated 270k PRs across 4.5k repos in the last 30 days.

Bulldozer
(open-source) — Merging pull requests manually after all checks pass is a small but persistent source of latency and context-switching. Bulldozer subscribes to status and review webhook events and automatically merges pull requests once all required status checks pass and the configured merge criteria (typically the presence of a merge-when-ready label) are met. Bulldozer is stateless and evaluates merge eligibility on every event, taking no action until all conditions are satisfied. Bulldozer merged 250k PRs across 4.2k repos in 30 days.

Excavator
(internal) - A distributed system for applying changes across the polyrepo at scale. Authors define a program that runs in a sandboxed context against each target repository; Excavator executes it, computes the resulting diff, and opens a pull request. Use cases include dependency upgrades, CI configuration rollouts, and large-scale code refactoring. In the last 30 days, Excavator created or updated 300k pull requests across 5k repositories, running approximately 900 unique checks.

Label Maker
(internal) - Label Maker automatically adds labels to pull requests based on configurable predicates: changed file paths, PR author, base branch, and other conditions. Labels are triggers for other bots (like Bulldozer, Nit, and Autorelease), so automation can enable additional behavior on PRs rather than requiring a human to context switch between tasks. Label Maker added labels 235k times across 160k pull requests in the last 30 days.

Octo-Correct
(internal) - GitHub organization and repository settings are configurable through the UI, but manual changes are difficult to audit and easy to drift. Octo-Correct enforces repository and organization settings from a centralized declarative configuration stored in a single GitHub repository. It continuously reconciles actual state against desired state, reverting manual UI changes that fall outside the defined configuration. This provides a single point of visibility into all managed settings across the fleet. Octo-correct continuously managed 6200 repos in the last 30 days.

Nit
(internal) - CI failures caused by trivial issues such as lockfile inconsistencies and formatting violations create unnecessary review cycles. Nit subscribes to CI status events and, when it detects a class of failure it knows how to resolve, pushes a fix commit directly to the pull request branch. This keeps automated and human-authored PRs moving through the pipeline without requiring manual intervention for mechanical fixes. Nit has fixed 14k issues across 1k repos in the last 30 days.

Backport
(internal) - When a fix needs to land on older release branches, manually cherry-picking and resolving conflicts is error-prone. Backport automates this by replaying pull requests onto target branches in GitHub. Backport also integrates with Apollo release channels (https://www.palantir.com/docs/apollo/core/release-channels) to allow users to automatically select versions based on what is deployed to the field. Backport has created 3k backports across 200 repos in the last 30 days.

Autorelease
(internal) - Releasing software is often a chore and should not usually require human coordination. Autorelease monitors merged pull requests for the presence of a configured label (typically autorelease) and, when detected, creates a versioned release following our internal version schema. It supports multiple language ecosystems: Java, Go, Rust, Python, PNPM, and produces releases in a consistent, structured pattern. By supporting regular and non-toil releases, Autorelease promotes continuous release and deployment flows through Apollo (https://www.palantir.com/docs/apollo/core/products-releases-versions/). Autorelease has created 28k releases across 2k repos in the last 30 days.

Highline
(internal) - New repositories should start with current best practices by default, not accumulate them over time. Highline is a self-service application that provisions repositories from maintained templates, applying standard CI configuration, bot enrollment, and toolchain setup at creation time. Highline has created 100 repos in the last 30 days.

Go-Githubapp
(open-source) A Go library for building production-quality GitHub applications. It encapsulates common operational concerns: GitHub App authentication, installation token management, rate limit handling, and multi-organization operation. All of the above applications are built on go-githubapp.

Standardization

The applications described above are only useful at scale if repositories don't need to be configured individually. A system that requires each team to hand-author configuration for every bot in every repository would reintroduce the coordination overhead we set out to eliminate.

To solve this, all of our GitHub applications support remote configuration loading. Rather than reading configuration exclusively from the repository they're operating on, each bot can resolve its configuration from a shared source. We maintain a shared github org called standards that contains canonical, reusable configuration files for each application. When a bot evaluates a repository, it first checks for local overrides, then falls back to the appropriate configuration in standards.

Excavator then helps close the loop on application rollout. When a new bot or configuration is introduced, Excavator opens pull requests across target repositories to add the necessary local configuration or to migrate repositories to the standard remote configuration. Each new application benefits from the enrollment infrastructure that already exists, and each addition to standards increases the value of that shared configuration layer.

Composing Workflows — End to End Examples

Each application solves a discrete problem and they compound together. Consider the following change that resolves a CVE by upgrading a library and modifying code.

  1. PR Creation: Excavator runs continuously in the background and detects that new library version is available and opens a pull request on each affected repository. As part of PR creation, Excavator runs repository-local tooling to regenerate lockfiles (e.g., GCV for Gradle, go mod tidy for Go modules) so that the dependency graph is consistent at the point the PR is opened.
    When the pull request is opened, Label Maker evaluates the PR against its configured predicates and applies labels. Typically, merge-when-ready , autorelease, and fix nits. These labels will be used later on to trigger additional behavior.
  2. CI and Automated Remediation: GitHub triggers CI, which runs the repository’s test suite and reports back via status checks. If CI fails due to a mechanically fixable issue, such as a formatting issue or a lockfile inconsistency, then Nit detects the failure class from the CI provider and pushes a fix commit directly to the PR branch. This new commit triggers a new CI run. If CI fails for reasons Nit cannot resolve, the PR stalls and surfaces for human attention.
  3. Policy Evaluation: Policy-Bot evaluates the PR on each relevant event. For bot-authored PRs, policies are typically configured to approve without human review, conditional on predicates such as: the modified files fall within an expected set, the commit author matches the bot’s identity, and the commit is signed with a known key. If all predicates are satisfied, Policy-Bot sets its status check to success. Octo-Correct has already configured the required status checks for the repository from the centralized settings definition, ensuring that Policy-Bot’s check is among those required for merge.
  4. Merge: Bulldozer evaluates merge eligibility on every status event. Its conditions are: all required status checks passing, and the merge-when-ready label present. When both conditions are met, Bulldozer merges the PR. Bulldozer is stateless, so its evaluation is idempotent and it can safely process duplicate or redelivered webhooks without side effects.
  5. Release and Deployment: On merge, Autorelease detects the autorelease label and creates a versioned release following the internal version schema. The release is adjudicated through Apollo onto pre-stable development stacks. Vulnerability scanning then validates the product's CVEs and whether it can land on production stacks (see our previous blog post for details).

No human is required once the flow is set up. The dependency or change gets automatically updated, tested, checked for compliance, and deployed to the fleet.

Human-Authored Pull Requests

The event chain is identical with one key difference: Policy-Bot requires a human review.

When a developer opens a PR, Label Maker applies the same labels and CI runs the same checks. Nit will remediate mechanical failures, which are made by AI agents and humans more often than deterministic processes like Excavator. However, Policy-Bot’s predicates for human-authored PRs require at least one approving review from a teammate before setting its status to success. This is the gate that prevents auto-merge until a human has reviewed the change.

Once approved, the remainder of the flow is unchanged. Bulldozer merges, Autorelease cuts a release, and Apollo adjudicates deployment. If the change fixes a critical bug, the developer can invoke Backport to replay the PR onto older release branches, selecting target versions based on what Apollo reports as currently deployed to the fleet.

The bots handle the tedious work so developers can focus on what is interesting and hard.

Other applications can replace some of the above workflow, but the key principle is that each application can operate independently and do one thing right. For example, both dependabot and renovate are possible open source replacements for Excavator to open Pull Requests.

Automation Enables Failure At Scale

These workflows aren’t always perfect, and it is prudent to be thoughtful of the potential failure modes. For example, while backporting might seem like a simple operation against supported release branches, a developer once invoked backport on a monorepo across every version in the last year. The resulting load from CI saturation temporarily blocked other developers from testing their changes on different repos. The volume of large, uncached monorepo CI builds then saturated our artifact hosting, which further slowed builds and local development.

We added upper bounds on the number of PRs Backport can create in a single invocation. PR Rate limits are something we think about for other systems as well. Excavator applies similar constraints by default as it rate-limits PR creation to bound the load it places on GitHub’s webhook infrastructure.

Lessons Learned

We did not set out to build a developer platform. Each tool filled a specific gap, and the system grew incrementally around them.

Where possible, we’ve built stateless apps. Policy-Bot and Bulldozer are stateless, responding to GitHub webhooks. Excavator is smarter and needs state, but we accept that complexity for the value it provides.

We have learned the importance of measuring friction: Developer productivity is difficult to quantify reliably, even with frameworks such as SPACE<https://queue.acm.org/detail.cfm?id=3454124> and DORA<https://dl.acm.org/doi/10.5555/3235404>. Friction is more observable: where do developers file workaround scripts? Where do manual steps accumulate? Where do teams diverge from standard configuration?

We did not build these bots overnight, and no single bot was a massive step change on its own. We built Policy-Bot to improve GitHub reviews, Excavator to manage repos at scale, and Octo-correct to handle repo settings. The impact comes from sustained investment and the tools composing together. Over time, they’ve shifted the baseline so that developers spend less time on operational overhead.
