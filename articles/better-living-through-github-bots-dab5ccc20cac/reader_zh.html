Better Living Through Github Bots

2026-03-26 · Original →

编者注：这是 Palantir 基础设施系列文章的第三篇，探讨我们如何定制基础设施工件以实现大规模的可靠运行。

Palantir 维护着约 6000 个涉及生产环境的代码库，涵盖微服务、配置即代码、基础设施即代码以及其他组合形式。每个服务都拥有独立的代码库，由各团队自行运营。我们将这种布局称为“多代码库”。

这种架构赋予了团队自主权，但当开发者需要在数千个代码库中推行一项横切变更时——比如广泛使用的依赖项升级、CI 配置变更或大规模迁移——就会引发协同问题。成本会随着代码库数量的增加呈线性增长。随着单体代码库规模不断扩大，同样的问题也会出现，尽管故障模式有所不同。

开发者工具团队及其上级基础架构部门负责解决这类横切问题。本文将介绍我们为此构建的 GitHub 应用程序。

我们的解决方案

我们的方法并非单一工具或平台重构，而是一组专注的 GitHub 应用程序。这些应用是几年来逐步构建起来的，通过 GitHub 原生的 Webhook、状态、检查、标签和拉取请求等原语组合而成。

每个应用都是独立的，并遵循类似 Unix 的设计哲学：做好一件事。GitHub 是工作流之间的连接组织，机器人订阅 Webhook 事件，评估条件并采取行动。工作流正是这些应用通过 GitHub 事件系统交互而产生的副作用。所有代码库中标准化的构建任务和发布流程减少了整体阻力。

核心应用

Policy-Bot
(开源) — GitHub 原生的分支保护规则支持基本的批准要求，但缺乏大规模条件策略所需的表达能力。Policy-bot 支持超越 GitHub 原生选项的复杂、有条件的拉取请求批准规则，并支持基于路径、作者和团队的判定逻辑。策略以声明式方式在每个代码库中定义，或从共享的远程配置中加载。在内部，Policy-bot 在过去 30 天内对 4500 个代码库中的 27 万个拉取请求进行了评估。

Bulldozer
(开源) — 在所有检查通过后手动合并拉取请求，虽然耗时不多，但却是造成延迟和上下文切换的持续性源头。Bulldozer 订阅状态和审查 Webhook 事件，并在所有必需的状态检查通过且满足配置的合并条件（通常是存在 merge-when-ready 标签）后，自动合并拉取请求。Bulldozer 是无状态的，会在每次事件时评估合并资格，在所有条件满足前不采取任何行动。在 30 天内，Bulldozer 在 4200 个代码库中合并了 25 万个拉取请求。

Excavator
(内部) — 用于在多代码库环境中大规模应用变更的分布式系统。开发者定义一个程序，在沙盒环境中针对每个目标代码库运行；Excavator 执行该程序，计算产生的差异，并开启拉取请求。使用场景包括依赖项升级、CI 配置推出和大规模代码重构。在过去 30 天里，Excavator 在 5000 个代码库中创建或更新了 30 万个拉取请求，运行了约 900 项独立检查。

Label Maker
(内部) - Label Maker 根据可配置的判定条件自动为拉取请求添加标签：更改的文件路径、拉取请求作者、基础分支以及其他条件。标签是其他机器人（如 Bulldozer、Nit 和 Autorelease）的触发器，因此自动化可以在拉取请求上启用额外行为，而无需人工在不同任务间切换上下文。在过去 30 天内，Label Maker 在 16 万个拉取请求中添加了 23.5 万次标签。

Octo-Correct
(内部) - GitHub 的组织和代码库设置可通过 UI 进行配置，但手动更改难以审计且容易发生配置漂移。Octo-Correct 根据存储在单一 GitHub 代码库中的集中式声明配置来强制执行代码库和组织设置。它会持续比对实际状态与期望状态，将超出定义配置范围的手动 UI 更改回滚。这为整个代码库群中所有受管设置提供了单一可见性视图。在过去 30 天内，Octo-correct 持续管理了 6200 个代码库。

Nit
(内部) - 由锁文件不一致和格式违规等琐碎问题导致的 CI 失败会制造不必要的审查周期。Nit 订阅 CI 状态事件，当检测到其已知如何解决的故障类别时，会直接将修复提交推送到拉取请求分支。这能让自动化和人工编写的拉取请求顺利通过流水线，无需人工干预即可解决机械性修复问题。在过去 30 天内，Nit 在 1000 个代码库中修复了 1.4 万个问题。

Backport
(内部) - 当修复程序需要合入较旧的发布分支时，手动执行 cherry-pick 和解决冲突极易出错。Backport 通过在 GitHub 上将拉取请求重放到目标分支来自动化这一过程。Backport 还与 Apollo 发布频道（https://www.palantir.com/docs/apollo/core/release-channels）集成，允许用户根据部署到现场的内容自动选择版本。在过去 30 天内，Backport 在 200 个代码库中创建了 3000 个反向移植。

Autorelease
(内部) - 发布软件通常是一项繁杂的工作，通常不需要人工协同。Autorelease 监控已合并的拉取请求是否存在已配置的标签（通常是 autorelease），一旦检测到，就会按照我们内部的版本架构创建带有版本号的发布。它支持多种语言生态系统：Java、Go、Rust、Python、PNPM，并以一致、结构化的模式生成发布。通过支持常规和非繁杂的发布，Autorelease 通过 Apollo 促进持续的发布和部署流（https://www.palantir.com/docs/apollo/core/products-releases-versions/）。在过去 30 天内，Autorelease 在 2000 个代码库中创建了 2.8 万次发布。

Highline
(内部) - 新代码库应默认遵循当前的最佳实践，而不是随时间推移慢慢积累。Highline 是一个自服务应用，从维护的模板中配置代码库，在创建时应用标准的 CI 配置、机器人注册和工具链设置。在过去 30 天内，Highline 创建了 100 个代码库。

Go-Githubapp
(开源) 一个用于构建生产级 GitHub 应用程序的 Go 库。它封装了常见的运营关注点：GitHub App 身份验证、安装令牌管理、速率限制处理和多组织运营。上述所有应用程序均基于 go-githubapp 构建。

标准化

只有不需要逐个配置代码库，上述应用程序才能在大规模场景下发挥作用。如果系统要求每个团队为每个代码库中的每个机器人手动编写配置，就会重新引入我们原本要消除的协同开销。

为解决此问题，我们所有的 GitHub 应用程序都支持远程配置加载。每个机器人不仅从其操作的代码库中读取配置，还可以从共享源解析配置。我们维护着一个名为 standards 的共享 GitHub 组织，其中包含每个应用程序的规范、可重用配置文件。当机器人评估代码库时，它会首先检查本地覆盖配置，然后回退到 standards 中的相应配置。

随后，Excavator 帮助实现应用推出的闭环。当引入新的机器人或配置时，Excavator 会在目标代码库中开启拉取请求，以添加必要的本地配置或将代码库迁移到标准的远程配置。每个新应用都会受益于已有的注册基础设施，而 standards 中的每一次添加都会提升该共享配置层的价值。

组合工作流 — 端到端示例

每个应用程序都解决了一个独立的问题，它们组合在一起会产生协同效应。考虑以下通过升级库和修改代码来解决 CVE 漏洞的变更。

  1. 拉取请求创建：Excavator 在后台持续运行，检测到有新库版本可用，并在每个受影响的代码库中开启拉取请求。作为创建拉取请求的一部分，Excavator 运行代码库本地工具以重新生成锁文件（例如，Gradle 的 GCV，Go 模块的 go mod tidy），以便在开启拉取请求时依赖图保持一致。
    当开启拉取请求时，Label Maker 根据其配置的判定条件评估拉取请求并应用标签。通常是 merge-when-ready 、autorelease 和 fix nits。这些标签稍后将用于触发额外行为。
  2. CI 和自动修复：GitHub 触发 CI，运行代码库的测试套件并通过状态检查报告结果。如果 CI 因机械性可修复的问题（如格式问题或锁文件不一致）而失败，Nit 会从 CI 提供商处检测到该故障类别，并直接将修复提交推送到拉取请求分支。此新提交会触发新的 CI 运行。如果 CI 因 Nit 无法解决的原因而失败，拉取请求将停滞并提交人工处理。
  3. 策略评估：Policy-Bot 在每个相关事件上对拉取请求进行评估。对于机器人编写的拉取请求，策略通常配置为无需人工审查即可批准，其条件包括：修改的文件在预期集合内、提交作者与机器人身份匹配，以及提交使用已知密钥签名。如果所有判定条件都满足，Policy-Bot 会将其状态检查设置为成功。Octo-Correct 已经从集中式设置定义中为代码库配置了必需的状态检查，确保 Policy-Bot 的检查属于合并所必需的检查之一。
  4. 合并：Bulldozer 在每次 status 事件时评估合并资格。其条件为：所有必需的状态检查均通过，且存在 merge-when-ready 标签。当这两个条件都满足时，Bulldozer 合并拉取请求。Bulldozer 是无状态的，因此其评估是幂等的，可以安全地处理重复或重新传递的 Webhook 而不会产生副作用。
  5. 发布和部署：合并时，Autorelease 检测到 autorelease 标签，并按照内部版本架构创建带有版本号的发布。该发布通过 Apollo 裁定到预稳定开发栈。然后，漏洞扫描会验证产品的 CVE 并确定其是否可以部署到生产栈（详情请参阅我们之前的博文）。

一旦流程设置好，就无需人工介入。依赖项或变更会自动更新、测试、检查合规性并部署到整个代码库群。

人工编写的拉取请求

其事件链是相同的，只有一个关键区别：Policy-Bot 需要人工审查。

当开发者开启拉取请求时，Label Maker 应用相同的标签，CI 运行相同的检查。Nit 会修复机械性故障，这类故障在人工智能体和人工操作中比在 Excavator 等确定性流程中更为常见。然而，Policy-Bot 对人工编写的拉取请求的判定条件要求至少有一名团队成员给出批准审查，才会将其状态设置为成功。这就是在人工审查变更前阻止自动合并的门控。

一旦获批，流程的其余部分保持不变。Bulldozer 执行合并，Autorelease 生成发布，Apollo 裁定部署。如果该变更修复了严重漏洞，开发者可以调用 Backport 将拉取请求重放到较旧的发布分支，并根据 Apollo 报告的当前部署到代码库群的内容来选择目标版本。

机器人处理繁杂的工作，让开发者能专注于有趣且具挑战性的任务。

其他应用可以替换上述部分工作流，但关键原则是每个应用都能独立运行并做好一件事。例如，dependabot 和 renovate 都有可能作为 Excavator 的开源替代品，用于开启拉取请求。

自动化导致的大规模故障

这些工作流并非总是完美的，审慎思考潜在的故障模式是明智之举。例如，虽然反向移植对于受支持的发布分支来说似乎是一个简单的操作，但曾经有开发者在单体代码库中对过去一年的每个版本调用了反向移植。由此产生的 CI 饱和负载暂时阻止了其他开发者在不同代码库上测试其变更。大量未缓存的单体代码库 CI 构建随后使我们的制品托管服务达到饱和，进一步减慢了构建和本地开发速度。

我们为 Backport 在单次调用中可创建的拉取请求数量设定了上限。拉取请求速率限制也是我们在其他系统中考虑的问题。Excavator 默认应用了类似约束，对拉取请求创建进行速率限制，以限制其对 GitHub Webhook 基础设施造成的负载。

经验教训

我们最初并非为了构建开发者平台。每个工具都填补了特定的空白，系统是在它们周围逐步发展起来的。

在可能的情况下，我们构建了无状态应用。Policy-Bot 和 Bulldozer 是无状态的，用于响应 GitHub Webhook。Excavator 更智能，需要状态，但鉴于其提供的价值，我们接受了这种复杂性。

我们认识到了衡量阻力的重要性：开发者生产力很难可靠地量化，即使有 SPACE 和 DORA 等框架也是如此。阻力更具可观测性：开发者在何处编写变通脚本？手动步骤在何处累积？团队在何处偏离了标准配置？

我们并非一蹴而就地构建了这些机器人，也没有哪个机器人单独带来了巨大的跨越式变化。我们构建 Policy-Bot 来改善 GitHub 审查，构建 Excavator 来大规模管理代码库，构建 Octo-correct 来处理代码库设置。其影响源于持续投入和工具的组合。随着时间的推移，它们改变了基线，使开发者花在运营开销上的时间越来越少。

入门

PolicyBot 和 Bulldozer 是开源的。你今天就可以部署它们：

  • GitHub 上的 PolicyBot
  • GitHub 上的 Bulldozer

两者都作为无状态容器运行，响应 Webhook，且没有外部依赖。先从其中一个开始，看看它如何改变你的工作流。然后再考虑你还可以填补哪些空白。
