Palantir Apollo Orchestration: Constraint-Based Continuous Deployment For Modern Architectures

编者注：这篇博文是关于 Palantir Apollo 系列文章的第二篇，前一篇是《为什么传统的持续部署方法在当今行不通》。

引言

过去 20 年里，软件行业一直致力于通过各种手段提升开发者效率：

  • 尝试不同的组织架构设计（更小、更独立的开发团队、DevOps、平台工程）
  • 采用新的项目管理方法（敏捷）
  • 采用新的软件架构（面向服务、微服务和无服务）

然而，我们在构建软件和组建工程团队方面的快速变化，已经远远将旨在促进这些活动的工具甩在了身后。虽然使用 Kubernetes 等平台可能会简化独立工作负载的部署，但它无法确保这些工作负载之间相互兼容。随着我们向更小、可组合的单元迈进，当每个组件独立演进时，一些流行的持续部署平台在管理分布式系统中相互依赖的组件更新所固有的复杂性时显得力不从心。这些持续部署平台无法管理服务间依赖关系，意味着组织实际上无法独立发布服务。相反，它们只是在“持续部署”一个分布式单体应用，根本无法体会现代软件开发技术所带来的速度提升。

我们设计了 Apollo 来应对现代架构的独特复杂性，并实现高效的独立软件开发。Apollo 管理这种复杂性的核心要素是 Apollo 编排引擎。该组件允许将部署约束以及软件运行环境发生变化时的规则编码到软件中。在将变更应用到受管软件安装时，它会确保满足这些约束。这使得分布式系统组件能够实现真正的持续部署，同时不牺牲稳定性。

在这篇文章中，我们将讨论 Apollo 编排引擎的工作原理，以及我们如何解决面向服务架构持续部署中固有的问题。

Apollo 编排引擎

Apollo 编排引擎负责将环境的当前状态推向 Apollo 为该环境设定的目标状态。这类似于近年来流行的 Kubernetes Operator 设计模式，但有一个关键补充——能够识别分布式系统的相互依赖关系和约束，并在执行操作时验证这些约束。Apollo 应用这种设计模式来管理跨越不同环境的众多软件安装的持续部署，为其编排引擎的控制循环提供三个输入：

  • 来自 Apollo Catalog 的可部署软件版本
  • 保存在环境配置存储中的环境声明状态
  • 由运行在 Apollo 管理环境中的 Apollo 智能体提供的当前（或“报告”）状态

保证将当前状态与目标状态对齐的过程不会影响软件可用性或违反软件及运维人员的预期至关重要。为此，我们设计 Apollo 的编排引擎以接受由软件和环境定义的约束，并且只推荐满足所有约束的向目标状态迈进的步骤。

这种设计自动化了在提供约束内求解的过程，确保过程可重复。此外，开发者和运维人员可以利用其领域专业知识为编排引擎定义适当的约束以供应用。虽然 Apollo 也提供了声明环境何时发生变化规则的能力，但本文的其余部分将仅关注开发者在其软件中编码约束，以及 Apollo 编排引擎对这些约束的应用。

在软件中编码部署约束

在构建较大架构中的单个组件时，开发者会对分布式系统中的周围组件做出隐式假设。将这些预期保留为隐式需要人工在部署时手动保证。Apollo 使得这些预期能够在软件中被显式编码；这样，开发者只需一次编码这些预期，将部署过程交给 Apollo，从而腾出时间专注于构建产品。

在编码软件预期时，开发者会在一组称为清单的元数据中连同代码一起声明约束。此清单在产品发布注册期间提供给 Apollo 的可部署软件注册表——Apollo Catalog。一旦进入数据目录，此信息即可供 Apollo Hub 的其余部分（包括其编排引擎）使用。然后，编排引擎在确定执行给定软件更新的安全性时会应用这些约束。

关键的是，Apollo减少了人工手动保证软件预期的需求。下面，我们将介绍两种类型的约束——它们是如何声明的，Apollo 如何在整个持续部署过程中遵守它们，以及它们为软件开发者提供的安全性。

示例：服务依赖约束

现代软件架构的功能源于其组件之间的通信。反过来，这种通信依赖于组件之间的 API 契约。处理这些相互依赖关系的复杂性促使我们在 Apollo 开发的早期就构建了对跨服务依赖约束的支持。这项投资极大地减少了 Palantir 在功能推出和弃用 API 移除方面涉及的人力。

假设我们正在服务 foo 中开发一个新功能，该功能依赖于另一个独立开发的服务 bar 中的某个新 API。只有与满足此新依赖的 bar 版本一起部署时，我们的功能才能正常工作。我们认识到在合并功能时需要声明这种跨服务依赖，并将此约束包含在服务的清单中：

service-dependencies:
- product-name: bar
  minimum-version: 1.1.0
  maximum-version: 1.x.x

在标记 foo 的新版本（假设为 1.1.0）时，我们会将此服务依赖包含在发布到 Apollo Catalog 的清单中。观察到这个新版本后，Apollo 编排引擎考虑在具有以下报告状态的环境中升级现有的 foo 安装：

- service-name: foo
  version: 1.0.0
  configuration-overrides:
    log-level: debug
- service-name: bar
  version: 1.0.0
  configuration-overrides: {}

Apollo 考虑将报告的 foo 安装从 1.0.0 升级到 1.1.0 的可行性。然而，在加载 foo 1.0.0 版本声明的约束后，它意识到当前 bar 安装的版本违反了新 foo 版本的跨服务依赖约束。在这种情况下，Apollo 将推迟推荐升级到我们的新版本，直到 bar 升级到满足此约束的版本。

Apollo“双向”应用服务依赖约束。假设 bar 的数据主人发布了一个 2.0.0 版本，该版本移除了对 1.x 版本中弃用端点的支持——Apollo 将拒绝部署此版本，除非先升级到满足以下条件之一的 foo 版本：(1) 更新其服务依赖的接受版本范围，或 (2) 完全移除该依赖。

如果没有声明跨服务依赖约束的支持，其他持续部署解决方案通常需要投入人力来确保新功能与其依赖项落在同一个单体发布序列中。如果组织放弃这种手动努力，可能会导致生产中断和耗时的人工修复。Apollo 免除了开发者的这种手动协调，同时也消除了此类工作中固有人为错误。

示例：模式迁移约束

负责向持久存储读取和写入数据的服务在执行这些读写操作时遵守数据库模式。随着时间的推移，功能开发或性能改进工作可能会促使代码主人更改服务的模式。模式迁移错误会带来巨大风险，可能导致不可逆的数据损坏和长时间的生产中断。我们认识到，随着面向服务架构中的组件不断演进，以自动化方式确保安全的模式迁移至关重要。因此，我们决定在 Apollo 中构建对模式迁移约束的支持，并通过为开发者引入两个交互点来实现这一点：

  • 在发布时在版本的清单中声明支持的模式版本
  • 提供生产环境中运行的服务的当前和目标模式版本，以包含在服务的报告状态中

假设我们希望服务 foo 通过多个版本完成从 V1 到 V2 的模式迁移，每个版本支持不同的模式版本范围：

这种方法正确与否取决于持续部署平台是否会等待从 V1 到 V2 的模式迁移完成后再部署 1.4.0 版本。我们将通过在每个版本的清单中包含支持的模式版本范围来声明这些模式迁移约束：

# 1.2.0 manifest
schema-migrations: [V1, V2]# 1.3.0
schema-migrations: [V1, V2]# 1.4.0
schema-migrations: [V2]

作为 foo 的开发者，我们的额外责任是在收到请求时向 Apollo 智能体报告当前模式版本，智能体在向 Apollo Hub 报告状态时将包含此信息。Apollo 预期服务的当前模式版本仅在迁移成功完成后才会递增——对我们来说，这发生在我们的 1.3.0 版本完成向 V2 的模式迁移时。

Apollo 模式迁移的编排逻辑

现在让我们考虑 Apollo 在决定何时推荐 foo 服务升级时对这些信息的使用。应用模式迁移约束等同于询问：

目标版本支持的模式版本范围是否包含为 foo 安装报告的当前模式版本？

关键的是，Apollo 的模式约束带来两个保证：

  • foo 保证会先升级到能够执行迁移的版本（在本例中为 1.3.0），然后才能进一步升级。
  • 升级到 1.3.0 后，foo 必须报告迁移已成功完成，Apollo 才会考虑将 foo 升级到更高版本。

模式约束应用也确保了安全回滚——假设我们在 1.3.0 和 1.4.0 版本中发现了一个棘手的错误，并且发现有必要回滚到我们最后已知的良好版本 1.2.0。Apollo 对模式版本约束的应用将在回滚之前验证 1.2.0 是否支持读取当前模式版本（现在是 V2）。出于同样的原因，Apollo 排除了回滚到 1.2.0 之前的版本，因为 1.2.0 是支持读取使用模式 V2 写入的数据的最旧版本。

如果没有 Apollo 的约束应用，模式迁移迫使工程师看护生产安装并手动根据迁移完成情况拦截升级。更糟糕的是，应用不当的模式迁移可能导致数据损坏。此类生产中断会侵蚀客户信任，从而可能造成不可逆的声誉损害。通过允许开发者告知其编排引擎在涉及模式迁移时的安全升级和回滚路径，Apollo 大大降低了此类灾难的风险。

总结

现代软件开发专注于构建离散、可组合的单元，将大量复杂性推迟到部署时，而传统的持续部署方法无法处理这种复杂性。在 Palantir 经过实战检验的 Apollo，能够理解和应用开发者和利益相关者在执行环境更新时声明的约束，可以帮助工程组织在不投入密集人力的情况下实现稳定性和速度。这释放了工程师的精力，让他们专注于最擅长的事情——构建。
