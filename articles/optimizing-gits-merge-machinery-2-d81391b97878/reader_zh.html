在本系列的第一篇文章中，我介绍了一些背景知识，包括 git 合并机制的工作原理、我为什么要优化它，并展示了一个通过更好地利用精确重命名来加速该过程的简单优化方案。在本文中，我们将深入探讨下一项优化，其核心思想可以概括为“寻找有效的启发式规则来引导重命名检测，从而避免（或至少减少）暴力的穷举式逐对比较”。

为三方内容合并配对文件

首先，我们来快速回顾一下；以下两段引文出自上一篇博文：

合并不是双方操作，而是一个三方操作，git 会参考合并基、你当前的提交以及另一分支末端提交。[……] 因此，合并算法的基本思想是，使用来自这三个提交的每个文件版本，对仓库中的每个文件进行三方内容合并。

以及

重命名检测是一个二次复杂度算法。它首先获取合并基中独有的所有 N 个文件名，然后获取指定侧独有的所有 M 个文件名，接着对所有 N x M 个文件的内容进行相似度比较，并将相似度足够高的最佳匹配标记为重命名。即使只有少量的重命名，也足以让重命名检测成为合并过程中最慢的环节。

上述第二段引文中有一个容易被忽略的有趣细节：git 会将合并基和指定侧中同名的文件视为相关文件，因此会自动将它们配对以进行三方内容合并，而不考虑文件内容的实际相似度。对于 git diff 或 git log 等命令，可以通过命令行参数（--break-rewrites 或 -B）关闭这种自动配对；这被称为“中断检测”。开启“中断检测”后，git 会检查同名文件的相似度，如果相似度不够高，就会将它们视为独立文件，并将其放入重命名检测池中。

合并机制并不会开启中断检测；如果开启的话，速度会慢得多。这本质上意味着 git 认为文件名是三方内容合并中匹配文件的良好指引，但前提是文件名完全匹配。如果文件名不完全匹配，它就会放弃这一策略，转而认为唯一重要的就是两个文件的内容相似度。简而言之，文件名相似度和内容相似度都可以作为三方内容合并中文件配对的依据。我们稍后再回到这个话题，现在先集中讨论内容相似度比较。

在不查看实际文件内容的情况下，是否更容易发现潜在的重命名对？我问了我五岁的孩子，她找出了所有四个可能的配对。当然，我们无法用颜色编码，但这只是一种简单的方式来强调一个事实：重命名通常涉及将文件移动到不同的目录，同时保持文件的基本名（最后一个目录分隔符之后的所有内容，包括扩展名）不变。这算作弊吗？它在实际中的很多情况下有用吗？它是只适用于某些特定仓库的特殊情况吗？所以最大的问题是……

这种情况有多普遍？

我研究了一些真实的仓库，并查看了每个仓库中的历史重命名记录。我统计了在重命名时未更改文件基本名的比例。以下是我的发现：

  • linux: 76%
  • gcc: 64%
  • gecko: 79%
  • webkit: 89%
  • git.git: 16%

我还检查了其他几个仓库，包括公司内部仓库。除了 git.git 之外，我检查的所有仓库在重命名文件时，似乎通常都会保持文件基本名（最后一个目录分隔符之后的文件名部分，包括扩展名）不变。显然，在保持基本名不变的情况下将文件移动到不同目录的的做法非常普遍。即使是 git.git，我也认为这个比例已经足够高，足以支撑一项有用的优化，因为能够匹配高达 16% 的文件并将它们从矩阵比较中移除，可以使需要检查的配对数量减少多达 30%（因为 (1–0.16)² 约等于 0.70）。

优化：基本名引导的重命名检测

基于上述信息，我们可以增加一组中间检查（在上一篇博文讨论的精确重命名检测之后，但在常规矩阵比较之前），包括以下步骤：

  • 为重命名源（即合并基中独有的文件名）创建一个从基本名到全名的简单映射。如果某个基本名不唯一，则将其忽略。
  • 对重命名目标（即指定侧中独有的文件）执行相同操作。
  • 找出两个集合中都存在哪些基本名。对于每一个基本名：
  • 比较两个完整文件的内容相似度。
  • 如果它们足够相似（阈值高于常规矩阵比较所用的阈值），则将它们标记为重命名，并将它们从后续的矩阵比较中移除。
  • 如果它们不够相似，则将它们纳入后续的完整矩阵两两比较中。

请注意，该算法最终可能会改变行为；根据上述规则，docs/ext.txt 文件可能与 docs/config/ext.txt 足够相似并匹配，从而被标记为重命名，即使其内容可能与 docs/ext.md 更相似。因此，这种优化最终是一个判断问题：性能的提升是否能证明行为差异的合理性？用户对新算法结果的理解程度是否能像对旧算法那样清晰？我们确实要求通过基本名相似度匹配的文件达到更高的最低阈值（>= 75% 的匹配行[1]，而矩阵比较只需 50% 的匹配行）作为一种安全系数。但归根结底，这项优化在理论上可能会产生与旧算法不同的结果。

这项改动带来的收益相当可观。我们使用了与第一篇博文相同的测试用例，该用例涉及一次高层目录重命名——基本上是在一个分支上将 linux 内核中的“drivers”重命名为“pilots”，然后将一堆补丁变基到该分支上。对于这个测试用例，添加上述简单算法带来了以下加速效果：

                        优化前                  优化后
超大重命名:     1799.937 s ±  0.493 s   188.754 s ±  0.284 s

这将耗时从半小时降低到了仅仅 3 分钟出头。显然，具体结果会因仓库中重命名的数量，以及这些重命名中有多少只是将文件移动到新目录而不更改基本名而有所不同，但上面收集的统计数据（在“这种情况有多普遍？”部分）表明，这项优化将具有广泛的适用性。

很好！但我们还能做得更好吗？

那些没有唯一基本名的文件怎么办？这种情况多吗？事实证明有很多。仓库中可能有数百个具有相同基本名的文件，尤其是构建系统文件——.gitignore、Makefile、build.gradle 等。源代码文件具有常见名称的情况也很普遍。Linux 内核有 100 个 setup.c、87 个 irq.c 和 112 个 core.c 文件。我知道的另一个仓库有许多 ObjectFactory.java 和 Plugin.java 文件。

处理这些文件的一种解决方案是，为具有相同基本名的文件建立一个小型矩阵，寻找相似度百分比，并将相似度足够高的最佳匹配配对，将其余文件留到最后的大矩阵中重新比较。我实现了这个方案，但有几个缺点。最大的问题是，它与我想做的另一项后续优化结合使用时效果很差，导致整体方案反而更糟。

然而，观察这些非唯一的基本名，它们代表的情况是：人们将文件移动到了新目录而没有重命名文件，但在不同目录中有多个可能的文件可供选择。如果我们知道如何预测给定文件被移动到了哪个目录，并且该目录中存在一个具有相同基本名的文件，那么我们就找到了一个可能的重命名配对。预测文件移动到了哪个目录，听起来像是目录重命名检测的活，这是我们在之前实现并在《Git 中的重命名与深层目录层级》中探讨过的内容。在这里复用那段代码只有一个问题——目录重命名检测是基于对目录内单个文件重命名检测的汇总。如果它是从所有重命名汇总计算得出的最终结果，我们怎么能用它来预测单个重命名呢？

不过，重命名检测是分阶段进行的，这一事实为我们提供了答案。如本系列第一篇博文所述，我们首先检测精确重命名。这并不能找出所有的重命名，但我们可以仅利用这个重命名子集，通过目录重命名检测逻辑，来预测同一目录中的其他文件将被重命名到哪里。对于任何剩余的文件，如果我们通过目录重命名检测逻辑得到了它移动位置的预测，并且目标目录中存在一个同名的文件，我们就认为它们是可能的重命名对，并检查它们的相似度。如果它们满足匹配相似基本名所需的提高后的阈值，我们就将它们标记为重命名，并将它们从后续的矩阵两两比较中移除。

在优化中加入这一调整后，我们的计时结果改善如下：

                        优化前                  优化后
超大重命名:      188.754 s ±  0.284 s   130.465 s ±  0.259 s

这意味着总体而言，这两项优化将耗时从半小时降至两分钟出头。还不错。任何拥有大量重命名且大部分重命名只是将文件移动到不同目录的仓库，都应该能看到类似的效果。如果你的重命名不多，就不会受到速度缓慢的影响。如果你有大量重命名，但你的仓库中大部分重命名也涉及基本名的更改，那么请告诉我；我非常好奇能了解这样的仓库（git.git 除外）。不过，即使在那种情况下，我手中还有更多的优化方案，我将在本系列的后续文章中介绍。

有关这两项优化的更多细节，请参阅我的补丁经过审查的两封上游邮件列表线程，即关于《基本基于名引导的重命名检测》的线程，以及关于《（通过早期目录重命名检测）进一步利用文件基本名》的线程。

[1] 匹配行并不是实际的衡量标准，但对于基础理解来说已经足够接近了。真实的算法要复杂一些。相似度算法将每个文件划分为多个块。块的划分方式是在每个换行符处或每 64 字节后进行分割，以先到者为准。它对每个块进行哈希处理，将其映射为一个整数（因此频繁重复的行，如简单的左括号或右括号，都将映射为同一个整数）。将所有块映射为整数后，每个文件都会得到一个整数列表。对于任何给定的文件，它可以将该文件的整数列表与另一个文件的整数列表进行比较。在此过程中，它会记录我们的列表中出现在另一个列表中的整数百分比。此外还有一些其他细节，例如大小差异过大的文件（例如相差超过两倍）会自动被假定为不相似。
