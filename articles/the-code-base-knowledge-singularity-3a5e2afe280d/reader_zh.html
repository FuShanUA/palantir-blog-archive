The Code Base Knowledge Singularity

2021-02-11 · Original →

奇点是指系统在某种程度上变得不可预测或行为失常的时间点或配置空间状态。例如，在数学中，函数 f(x) = 1/x 在 x=0 处是奇异的，因为它会“爆炸”；在广义相对论中，Stephen Hawking 证明了宇宙大爆炸具有无限大的密度（即空间呈点状），诺贝尔奖得主 Roger Penrose 则指出黑洞必然存在奇点；也许最著名的要数技术奇点，它是“技术增长变得不可控且不可逆，从而给人类文明带来不可预见的变化的假设时间点”（维基百科）。

在这篇博文中，我将论述软件代码库会受到我称之为代码库知识奇点的影响。简而言之，就是组织已经丢失了关于代码的知识，再也没人记得当初设计的权衡取舍或实现细节的那个时间点。基于我在 Palantir 与数以千计代码仓库中的数百个团队合作的经验，我将描述导致奇点产生的情况，并介绍我们为避免奇点所采用的技巧与工具。

这到底是什么鬼？

我想每位程序员在某个时刻都曾面对过代码库知识奇点。当你看着一段代码，心里犯嘀咕这函数到底在搞什么鬼？的那个瞬间就是了。我观察到在这种情况下有两种常见的反应模式：
- 悲观主义者：我绝不去碰那段代码，改起来太吓人了
- 乐观主义者：那段代码看起来太绕了，我确信能把它重构和简化一下

如果我是你所管理的工程组织或团队中的一员，你会希望我通常表现出悲观还是乐观的行为？没错，两者都不是。悲观的做法很糟糕，因为它意味着相关的代码永远不会被更改，它也就是定义上的遗留代码，是一个死气沉沉的代码库。乐观的做法在更微妙的层面上也很糟糕：代码之所以复杂很可能有其充分的原因，而更有可能的是，我并不理解这些原因，非但无法将其简化，反而会把事情搞砸并引入缺陷。这种情况被称为切斯特顿栅栏；引用法纳姆街文章的话来说：

“当我们试图干预他人创建的系统时，不能仅仅将他们的决定和选择视为一阶思维的后果，因为这样可能会在无意中制造出严重的问题。在改变任何事物之前，我们应该思考他们是否运用了二阶思维。他们做出某些选择的原因可能比乍看之下要复杂得多。最好假设他们掌握了我们不知道的信息，或者拥有我们无法揣测的经验，这样我们就不会去贪图速效之策，最终却让事情变得更糟。”

代码库知识奇点

在更改此类代码之前，我应该请教一位了解内情的同事或贡献者来解释其中的细微之处。遗憾的是，往往找不到这样的人，要么是因为他们已经离开了项目或公司，要么是因为他们自己在几年后也忘记了细节。这就是组织触碰代码库知识奇点的时刻，它已经丢失了关于某段软件为何如此编写的制度记忆。为了弄清原委，唯一的希望就是进行源代码考古：我会在 Git 提交日志、GitHub 拉取请求评论、电子邮件、Quip 文档、Slack 频道等各处翻找，绝望地试图探寻那段神秘“这函数到底在搞什么鬼”的代码背后的原因和实现方式。

当然，其后果并不像技术奇点（“技术增长变得不可控且不可逆，从而给人类文明带来不可预见的变化的假设时间点”）那样可怕。然而，当逼近代码库知识奇点时，开发人员变得更有可能引入性能或行为上的倒退，将时间浪费在探索以前走过的死胡同里，或者违反隐含或隐藏的跨组件假设。

那么我们一开始是如何陷入这种境地的，为什么又怎么会最终面临两个糟糕的选择：是将代码视为遗留产物的悲观主义者，还是对其进行重构并可能引入缺陷的乐观主义者？当然，有许多不同的因素决定了代码库逼近奇点的速度，但我想指出两个特别重要的因素。首先是沟通方式。在以同步沟通为主的组织中（常见口头禅：“我们约个视频会议把这个弄清楚吧”、“反正下周我在伦敦，能当面讨论下这个吗？”），决策主要是通过口头、面对面的沟通做出的。除非这类组织培养了强大的记笔记和分发文化，否则这些决策——更重要的是决策背后的理由——都是短暂的，很快就会遗失。

相比之下，采用异步沟通方式的组织（例如查阅 Basecamp 或 Gitlab）通常使用协作式的、基于文本的沟通工具（如 Quip 或维基），这些工具会自动生成讨论和决策的书面产物。但要注意，别被骗去依赖像 Slack 这样的同步沟通工具来捕获决策以供异步消费！即便使用了像 Quip 或 Google 文档这样的协作文档工具，知识组织（例如结构化、打标签、搜索、归档、访问控制等）依然是一项重大挑战。

其次，软件架构（作为组织结构的镜像，符合康威定律）直接影响架构决策的半衰期。例如，微服务代码与组织架构通常在服务内部展现出强大的社交纽带，但同时在每个服务上也存在严重的信息孤岛现象。即使某个特定服务内的技术决策有文档记录，组织可能也缺乏一个显见的论坛来讨论和记录跨服务、系统级的架构决策。力求推动全组织范围内知识共享的组织领导者会发现自己始终在与熵和惰性打一场艰难的逆水之战。

编程与软件工程

上文法纳姆街的引文在讨论切斯特顿栅栏时提到了“一阶思维”和“二阶思维”。在数学或物理学中，一阶近似通常是线性近似，例如函数的 1 阶泰勒展开。线性近似的预测能力很差。例如，它们无法表达我的账户余额在增长（一阶）与我的账户余额的增长率在增长（二阶）之间的区别。类似地，一阶思维关注的是某个行动的直接后果，而二阶思维考虑的则是一项行动或决策后果的后果。二阶思维者会不断追问，然后呢？二阶思维者具有更好的预测能力，他们能产出更具前瞻性的解决方案。

编程（或者说“编码”）是一阶思维，而软件工程是二阶思维。要理解这种区别，可以参考谷歌的洞见：“软件工程是跨越时间的编程”：编程是为了解决眼前的问题，而软件工程是以一种让解决方案经受住时间考验的方式去解决问题，使其能在未来被其他人和系统适应和重用。写下那段这函数到底在搞什么鬼？代码的人可能是一位出色的程序员，但很可能不是一位卓越的软件工程师。

与编程不同，软件工程所需的远不止写下正确的算法：还要选择经得起时间考验的编程语言，编写测试以记录边缘情况的行为并防止倒退，编写 API 文档，为变量和方法赋予直观且一致的名称，记录设计或实现上的细微差别，甚至通过文档、技术讲座和培训计划与他人分享知识和洞察。这些行为有助于避免出现这函数到底在搞什么鬼？的时刻，有助于推迟甚至规避知识奇点。

面向二阶思维者的编码工具

代码库知识奇点是一个过于真实的威胁，仅靠呼吁（“写代码再仔细点！”、“再加把劲！”）是无法抵御的。更重要的是有助于锚定二阶思维的具体工具和常规流程。以下是我在 Palantir 工作时总结的几个简单例子：

  • 在 Palantir，每一项代码或配置的更改都需要代码审查。一阶后果是捕获缺陷，而（可以说更重要的？）二阶后果是分享关于代码库的知识。
  • 我们使用 Quip 来撰写意见征求（RFC）文档，工程师在其中提出技术问题和解决方案。他们的同行可以参与讨论、评论、批评、提问等。在法律和合规性约束允许的范围内，团队的 Quip 文件夹尽可能采取默认开放的访问策略。
  • 许多团队会编写架构决策记录来记录重要的长期架构选择：这项服务使用了哪种数据库技术或存储布局，为什么该服务相较于 REST 风格的 RPC 更青睐 WebSockets，面向用户的资源是如何设计访问控制的，等等。将 ADR 与代码一起存储在源代码管理系统中非常实用，这既有助于发现也有助于考古。旁注：对许多团队来说，一个巨大的挑战是从讨论阶段（RFC）过渡到文档化阶段（ADR）。很多时候，在 RFC 文档上进行了一场富有成效的讨论后，团队却难以整合讨论内容、更新书面产物，并将 RFC 中的洞察固化为 ADR。
  • 我们定期举办内部技术讲座，工程师们会在讲座上以白板形式展示其软件组件的技术底层。技术讲座会被录制成视频，为新员工或新团队成员提供极佳的信息来源。
  • 我们的团队会举行类似于五个为什么的事后复盘，以了解高影响的缺陷或系统中断，并将经验教训记录下来供未来参考。

结论

尽管有这些好的想法和工具，我当然也时常经历这函数到底在搞什么鬼？的时刻，甚至是对我自己的代码……代码库知识奇点是真实存在的，它对软件组织安全、迅速地交付新功能，或根据新兴的基础设施需求以高效且经济的方式演进现有软件的能力构成了重大威胁。然而，与数学中的奇点不同，这是一个我们可以挺身反击的奇点。工程领导者必须认识到这一威胁，并通过推行工程质量倡议和在软件工程中激励二阶思维来努力抗击它。
