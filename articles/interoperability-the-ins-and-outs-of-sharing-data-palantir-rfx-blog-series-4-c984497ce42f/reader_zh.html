Interoperability: The ins and outs of sharing data (Palantir RFx Blog Series, #4)

2022-11-14 · Original →

随着数据生态系统不断演进，组织必须为其软件建立坚实的互操作性预期，以确保在不确定的未来中仍能保持实用性与影响力。

编者按：本文是《Palantir RFx 博客系列》的第四篇，该系列探讨组织如何更好地撰写 RFI 和 RFP，以评估数字化转型软件。每篇文章聚焦数据生态系统中的一个关键能力领域，旨在帮助企业提出正确的问题，从而更好地评估技术。此前发布的篇章包括《本体论》、《数据连接》和《版本控制》。

引言

近二十年来，Palantir 一直将软件部署到一些极其复杂的运营环境中。我们的软件已安装在研究实验室、工厂、石油钻井平台、服务器场以及多租户云环境中。我们应对过从基于 COBOL 的绿屏终端到最新一代边缘计算环境在内的各种数据架构。在这些不同场景下部署软件，让我们深刻认识到互操作性的重要性——具体而言，就是数据架构与其他系统建立稳定且安全连接所需的技术。

“你的系统具备互操作性吗？”已成为大多数 IT 评估中的标准问题。遗憾的是，在评估软件时，这个问题基本起不到什么作用。就像“你是个好司机吗？”这个问题会引来异口同义的自信回答一样，许多软件提供商宣称自家的系统具备互操作性，其笃定程度掩盖了将一个系统与另一个系统连接起来所需的实际工作量。事实上，互操作性有深浅之分，真正能以灵活高效的方式与其他系统连接的数据系统少之又少。

在本文中，我们将探讨实践中互操作性意味着什么，以及组织如何区分仅在技术上可互操作（需要大量定制和高昂成本）的软件解决方案与真正具备互操作性（采用标准化、预置连接器且只需极小工作量）的解决方案。

什么是互操作性？

互操作性是指系统以标准化、高效的方式与其他系统连接的能力。这些连接支撑着许多下游功能，包括数据迁移、跨平台数据共享、联合搜索以及淘汰过时软件。

在探讨“什么是互操作性？”这个问题时，首先需要明确系统的作用，以及它会产生哪些可能被其他系统访问的数据产物。例如，一个主要用于数据采集的系统，不仅需要提供对数据本身的访问权限，还要提供相关的元数据。一个管理数据清洗、转换编写或业务逻辑工具的系统，则需要提供提取底层代码的方法，以便其他系统复用。无论系统执行什么功能——其创建的产物和生成的输出——都必须易于被外部系统访问和使用，从而实现互操作性。

因此，互操作性不仅仅是一个系统与另一个系统连接的能力。数据生态系统会产生多种有用的信息，要实现真正的互操作性，组织必须能够将系统连接起来，并且明确指定共享信息的内容、时间和方式。这第二点至关重要——许多软件系统拥有一些端点连接服务来与其他系统相连，却缺乏足够的灵活性，无法以让这些产物具备运营价值的格式、结构和交付节奏导出所有有用的产物。下图展示了一些常见的数据产品类型，如果一个系统宣称具备互操作性，就可能需要共享或提取这些产品。

互操作性常见问题有哪些？

系统无法与外部系统有效交互的情况屡见不鲜。一些常见问题包括：

  • 系统使用的自定义格式无法被外部工具轻松读取。这种情况常见于内部团队自研的系统或 IT 顾问定制构建的解决方案。在使用高度专业化的专有软件时也会出现这种情况，这可能导致供应商锁定（无论是有意还是无意）。无论初衷如何，结果都是产生了一堆在没有自定义适配器或其他昂贵且脆弱的方法下，其他程序无法访问或读取的数据产物。
  • 系统不允许数据导出。许多系统缺乏简单、可重复的提取生成机制。这通常是由于后端访问缺乏适当的权限方案所致。配置这些系统的导出功能需要耗费大量时间和精力，通常会导致不必要的成本和/或对第三方的依赖（例如，花钱请顾问进行一次性提取）。
  • 系统对导出规模有限制。许多系统并未将所有驻留数据提供给外部系统。技术限制可能会对导出施加大小限制，导致数据和输出无法在不同系统间实时同步。
  • 系统在可提取的数据类型方面可能存在局限。随着系统不断演进，软件可能无法总是适应不断增长的数据量、新的数据类型，或无法提取如元数据等有价值的内容。随着原始系统的范围不断扩大以吸收新的用例和功能，固定或不灵活的 API 端点可能会变得过时。
  • 系统缺乏 robust 的安全功能。创建共享数据的网关会增加数据泄露或数据丢失的风险。任何未经安全加密、身份验证和审计的提取点都会给组织带来数据风险。

互操作性为何重要？

确保数据系统配备互操作性功能，是最大化其未来实用性和效用的关键。这些功能可帮助组织对抗“数据熵”，即数据生态系统随时间推移变得愈发混乱和无序的趋势（详情请参阅我们的《Palantir 解析——数据信任》博文）。强大的互操作性功能还能带来立竿见影的好处，包括：

  • 促进企业内部人员与团队间的协作；
  • 将一个系统的输出共享给另一个系统，以在不同的上下文、工作流或用例中使用；
  • 出于战略或监管目的进行数据迁移；
  • 通过将数据拉入最适合执行特定任务（如建模、报告、分析等）的系统来提升性能和决策能力；
  • 能够关闭（或“淘汰”）过时或不再有效的系统。

维持一个井然有序的数据生态系统需要极大的远见和努力。数据生态系统是自然发展起来的，多年来一点一点地加入各种点解决方案，每个方案都服务于特定的、孤立的功能。随着系统变得日益复杂，IT 组织往往发现自己将重心放在了维护系统上，而不是支持组织更广泛的目标。这种“尾巴摇狗”的倒挂现象在大型组织中尤为普遍，因为多个业务线会利用广泛且重叠的数据源。IT 部门和数据团队投资新软件以满足不同用户群体的需求，从而创造出也需要被管理的新输出和数据产品。与此同时，现有软件积累的数据越来越多，使得这些系统即便在过时或无法满足依赖它们的团队需求时，也难以被替换。结果，许多达到生命周期的软件系统只能依靠定制支持和服务高昂地勉强维持运行。

互操作性通过标准协议或方法与其他系统进行安全、可靠且稳健的信息交换（无论是数据、模型、代码等），为这一难题提供了解决方案。这些功能最大限度地减少了冗余和重复，同时在需要移动数据或到了淘汰特定系统之时，能够实现平滑迁移。

通过确保每个系统都能全面且及时地共享信息，组织可以释放这些系统设计之初所具备的全部能力。如果缺乏这些能力，系统就会在孤立中生长，为组织造就碎片化的数据格局，也为用户营造了孤岛般的环境。

要求

解决方案必须提供开箱即用的功能，支持按需访问关键信息。到目前为止，大多数组织已经明白，数据系统需要保持开放、可互操作的姿态才能长期发挥效用，而基于专有数据格式或封闭/缺失端点的“供应商锁定”再也不可接受了（如果说曾经可以接受的话）。互操作性技术是任何有效数据系统的基本组成部分，这些功能必须从第一天起就融入架构之中。对这些导出操作必须有最低的延迟和限制，以确保组织发展时的最大灵活性，并实现全面的覆盖，以提取和共享所有相关内容和数据产品。

解决方案不仅要提供对数据的全面访问，还要涵盖元数据、代码、业务规则和数据产品。系统间的有效连接和用户间的协作所需的不仅仅是共享数据。企业数据平台存储、提供并产生许多不同类型的信息——从数据本身到数据的属性（元数据），再到模型输出——所有这些都需要能被外部系统和不同用户群体访问。

解决方案必须通过行业标准 API 和接口提供安全的连接点。连接点必须遵循行业标准，使用标准 API 和驱动程序将数据、元数据、逻辑和其他信息提取到第三方工具中。使用行业标准连接（如 REST API、ODBC 和 JDBC 驱动程序）可确保在与其他平台和工具连接时实现最大的可访问性、一致性和安全性。

解决方案必须将所有数据和数据产品导出为开放且可读的格式。许多系统以专有格式生成产物。将这些“封闭”的数据产品转换为有用的信息需要自定义解析器或代码，从而导致时间浪费或输出无用。这极大地削弱了原系统的实用性，因为输出和数据无法轻易在其他地方利用。预先要求开放格式可确保其他系统能够利用输出，同时也让软件达到生命周期终点时更容易提取数据。

用于查询和提取数据的端点必须是安全的、经过身份验证的且可审计的。如果端点连接没有内建强大的安全机制，组织数据可能会处于高度暴露状态。因此，确保每个端点都安全、经过身份验证且可审计至关重要。具体而言，系统必须内置问责机制，以跟踪提取了什么数据、由谁提取以及何时提取。这可以减轻被提取数据被滥用的风险（例如过于宽泛的搜索或提取），并提供一定程度的透明度，从而最大化组织对数据的控制。如果没有这些内建的安全功能，组织将极易遭受数据泄露和数据滥用的侵害。

解决方案必须包含将信息从系统中提取出来的自助服务功能。任何不及自助服务或固定 SLA 的做法都会带来高风险，即依赖第三方提供及时和充分的支持。自助服务意味着用户和平台管理员拥有相关的文档、API、连接点和权限，可以直接在平台中自行操作。为了减少导出数据时常见的时间消耗、压力和组织管理开销，必须具备自助式互操作性功能。

结论

组织要想以连贯且富有韧性的方式实现扩展，就需要对其软件建立坚实的互操作性预期。要求将互操作性作为开箱即用的自助服务能力，对于确保特定系统能持续发挥相关作用和影响力至关重要。通过前置要求这些互操作性功能——包括标准化、预置的连接器，以及指定系统间共享数据的内容、时间和方式的能力——组织可以确保自身拥有必要的工具，从而在数据生态系统日益复杂的情况下，依然能维持对数据的掌控。
