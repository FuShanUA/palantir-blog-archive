Manufacturing with the Connected Edge

2026-03-24 · Original →

工业与国防环境会产生海量数据，这些数据根本等不及传到云端处理。延迟往往以毫秒计，系统的弹性至关重要。制造工厂绝不能因为不稳定 Wi-Fi 或公有云中断而停机。“传统”做法——发运服务器、雇用本地 IT 人员、定制开发、管理一次性部署——根本无法规模化。关键业务操作需要专门为边缘端的复杂性量身打造的基础设施。

在 AI 驱动的自动化时代，挑战不在于搭建单台边缘设备，而在于如何统一、安全地运维成百上千台设备，且无需在每个站点都配备专门的 IT 团队。这需要一套贯穿三个核心维度的整体架构：

  • 数据：将来自视频、PLC、SCADA、MES、PLM、ERP 等传感器的结构化与非结构化数据流聚合到一个可实时访问的统一命名空间中。
  • 逻辑：在本地处理并分析这些数据，将其转化为可执行智能，用于生产控制、物料调度、实时工艺优化、质量检测、预测性维护等。
  • 行动：通过自动化响应和回写实现闭环，这些响应需遵循细粒度的防护栏，并能与跨环境的工作流进行对接。

从功能上看，用于关键业务的边缘基础设施可划分为五个相互紧扣的类别，它们跨越了信息技术（IT）与运营技术（OT）之间的传统界限：

  1. 边缘计算 — 适配规模的硬件，驱动预集成的数据栈
  2. 部署与管理 — 全局设备编排、安全与生命周期管理
  3. 本地/全局上下文 — 嵌入式本体，将边缘决策与企业智能相连
  4. 快速应用开发 — 灵活的工具链，支持基于企业模型进行构建
  5. 规模化自主运维 — 自愈基础设施与智能体工作流，可扩展至数百个站点

这些类别必须整合并治理于一套跨越人类团队、AI 智能体、设备和环境的企业架构之中。

边缘计算

工业环境会产生海量数据——从 PLC 和传感器，到视觉系统和诊断设备。这些数据大多不会离开工厂车间。它们数据量太大，难以回传；对延迟太敏感，无法往返云端；且对业务运营过于关键，不能依赖可能并不存在的网络连接。

在 Palantir，我们正是为这一现实构建了边缘基础设施。

Palantir 与基础设施合作伙伴携手，提供打包的计算、存储和 GPU 加速功能，设备形态从工业 PC 一直到 3U 机架式服务器均可扩展，连接方式涵盖有线、无线和 5G 蜂窝网络。蜂窝通道与其他通道隔离，专用于带外管理，与生产数据完全隔离。

交付给客户的并非等待配置的裸机，而是一个以嵌入式本体为核心的基础设施——这是 Palantir 本体的轻量级、边缘原生实例，专为在各站点独立运行而设计。该平台包含预集成的时间序列、关系型和对象存储栈，以及用于 AI 工作负载的本地大模型和向量搜索功能。工业协议适配器已预先配置，原生支持消息代理、标签自动发现以及第三方兼容性。数据从接入流向本体，最终流向产生投资回报率（ROI）的应用和服务。

边缘环境可独立运行，但在带宽和连接允许的情况下，会与云端保持连接。模型、配置和应用程序更新从 Foundry 和 AIP 下发。运营反馈（包括精炼后的数据集、性能遥测数据和边缘端生成的洞察）则回传至云端。这意味着中央运营中心的团队与工厂车间的团队基于同一本体工作，边缘端充当本地执行层，而非孤立的竖井。

除了流数据，该平台还与本地部署和云端系统集成——整合来自 ERP、CRM、MES 和 SaaS 应用的数据。它支持图像、视频、工程图纸、培训材料和工艺规范等非结构化数据。这些数据被整合到稳健的处理流水线中，以创建全面的运营数字孪生——这是与需要拼凑各种解决方案才能连接这些数据源的架构的显著差异点。

部署与管理

传统边缘基础设施的故障模式往往在意料之中。配置漂移不断累积。更新引发故障。网络发生分区。有人通过 SSH 访问系统并推送未记录在案的更改。这些并非极端的边缘情况，而是传统部署在规模化后的默认状态。

Palantir 采用了截然不同的方法。我们的边缘平台在不可变操作系统上运行单节点 OpenShift（SNO），其中操作系统状态本身就是构建产物，而非一台耗费数月手动配置的服务器。更新属于原子事务，要么完全成功，要么干净回滚，绝不留下需要调试的中间状态。

设备群管理通过 Apollo 进行，经由加密通信将容器镜像、Helm Charts 和操作系统更新从中央仓库下发至边缘设备。部署默认采用“金丝雀范式”：先在一台设备上部署，验证健康检查，然后再扩大范围。如果出现故障，系统会自动回滚——无需过时的运维手册，无需凌晨三点的告警叫醒。Apollo 支持诸如预发、发布和生产等发布通道，以管理气隙环境中的风险容忍度。每个边缘节点都运行着一个 Apollo Operator，负责在本地管理设备群协调、健康检查、更新和回滚。

应用程序在 AIP 和 Foundry 中利用开发者控制台、模型开发工具和标准应用开发流程进行构建和打包。包括容器、Helm Charts、本体定义和模型配置在内的构建产物会发布到中央仓库，由 Apollo 进行分发。

安全在设计上就是零信任，而非事后补充。系统没有 SSH 访问权限，没有 Shell，也没有暴露的 Kubernetes API。边缘设备发起的所有连接均为出站连接，绝不暴露任何入站端口。实现完全的可观测性和控制力，同时不妥协安全性。

客户 IT 只需将一个 IP 加入白名单。发往互联网的流量通过受管理的安全隧道路由；工厂本地的流量（PLC、传感器、MQTT）则留在本地。当需要支持访问时，系统提供经过审计、有时间限制的会话，无需建立 VPN 隧道或更改防火墙配置。

平台专为实现自我管理而设计：软件和模型部署实现零停机更新，无需中断数据采集；具备预测性自我维护监控磁盘、内存和数据库性能；且无需持续的本地 IT 投入即可执行核心运营操作。数据保留机制确保，一旦网络连接中断，数据能在连接恢复后安全推送至 Foundry。我们的许多客户正加速通过与 Edgescale AI 等合作伙伴进行部署。

本地/全局上下文

边缘数据只有在上下文中被理解才有价值——既要在本地用于实时决策，也要在全局作为整体运营视图的一部分。

Palantir 的边缘平台运行着嵌入式本体，对物理资产、传感器流和运营事件之间的关系进行建模。这并非静态模式，而是环境的动态表示，应用程序可对其进行查询、更新并据此采取行动。本体将数据、逻辑和行动连接成以决策为中心的表示形式，使人类和 AI 都能参与运营，而无需深入了解底层数据技术或编程语言。应用程序通过熟悉的 OSDK API 访问嵌入式本体，无论是查询资产还是触发向 PLC 和 SCADA 系统的回写，都能获得一致的接口。例如，一个质量检测应用程序定义了要同步的本体对象，然后在本地查询它们：

import { Line, Sensor, Defect } from "@vision-inspection/sdk";
import { createClient } from "@osdk/client";

const client: Client = createClient(
  foundryUrl, 
  ontologyRid, 
  auth,
  { syncObjects: [Line, Sensor, Defect] }
);
// Query defects on a specific line
const defects = await client.objects.Defect
  .where({ line: "line-01", severity: "critical" })
  .fetchPage();
// Act on results - trigger PLC rejection
for (const defect of defects.data) {
  await client.actions.rejectPart({ partId: defect.partId });
}

一旦完成同步，无论与 Foundry 的连接状态如何，应用程序都能在本地查询并针对这些对象采取行动，并提供完整的离线支持。

CRUD 操作和行动在本地以毫秒级延迟执行，独立于云连接。当连接存在时，本地上下文与 Foundry/AIP 中的全局本体同步，在此处可无缝驱动跨企业的基于智能体的自动化和优化。当无连接时，边缘端自主运行——缓冲更改、解决冲突，并在链路恢复后协调状态。弹性不是一项功能，而是最基本的假设。边缘节点之间还可互相对等连接，实现无需通过云端路由的跨站点协调。

通常，为了获取流程的全部价值，系统必须在边缘和云端之间无缝运行。在单个站点为车间技术人员提供服务的应用程序，其输出在整个企业范围内都具有意义。能够使用同一套工具将工作流的正确部分部署到正确的环境中至关重要。例如，在边缘端生成的警报可由 AI 增强的机器或工厂技术员自动分诊。由此产生的信号、操作员行动以及任何附加上下文，都会与云端本体无缝同步，用于宏观根因分析、跨工厂物料对比等。这种无缝操作通过将本体扩展至边缘端来实现；一个模型，两个环境，统一的上下文。自定义模型和服务与本体并排运行，消费其数据并回写结果，而底层基础设施（包括 Apollo Operator、SNO 和不可变操作系统）则对应用开发者保持抽象。

快速应用开发与交付

生成式 AI 革命改变了可能性的边界——但在工业环境中，可能性取决于运营现实。能够推理、生成和决策的模型，只有在与物理世界相连时才具有价值：读取传感器、理解工艺上下文，并在真实设备上触发行动。

边缘开发的难点不在于编写应用逻辑，而在于处理其他所有事情。连接工业协议；管理凭证；处理间歇性连接；在不停机的情况下部署更新。并且要在成百上千个地点完成所有这些工作。

Palantir 抽象了为边缘构建应用的复杂性。OPC UA、MQTT 等协议适配器已预配妥当。无论底层设备如何，本体都提供一致的 API，因此开发者基于抽象进行构建，而非针对个别站点的特异性。这种简化人类工程师开发工作的抽象，同样使工业环境对 AI 变得清晰易读——模型通过受治理的接口进行查询、推理和行动。部署通过与平台管理相同的 Apollo 流水线进行——版本控制、测试、增量部署，并在健康检查失败时自动回滚。模型在具备 GPU 加速的环境中运行，使用如 ONNX 等标准运行时。最终带来的开发体验更接近云原生，而非传统工业软件，且运行在无需云连接即可运作的基础设施上。

纯代码和低代码选项让应用开发如同为浏览器或移动端构建 OSDK 应用一样简单。让我们一步步了解如何构建一个计算机视觉质量控制应用，该应用与 PLC 集成，用于将缺陷组件从生产线上剔除。

为了可读性和简洁性，代码将使用 TypeScript/JavaScript——但也可以是符合用例非功能性需求的任何语言或框架。

  1. 捕获图像：当每个部件经过流水线时，高分辨率相机会捕获其图像。
import { client } from "./client";
import { Line, Part, Inspection, Defect } from "@vision-inspection/sdk";
import { subscribe } from "@palantir/edge-adapters/opcua";
import { grabFrame } from "@palantir/edge-adapters/gige-vision";
import { runModel } from "@palantir/edge-runtime/inference";

const line = await client.objects.Line.get("line-01");

// PLC triggers frame grab when part reaches station
subscribe(line.plcEndpoint, "ns=2;s=PartReady", async ({ partId }) => {

  const frame = await grabFrame(line.cameraEndpoint, {
    exposure: "auto",
    resolution: [2048, 2048],
  });
  1. 接入与归一化：边缘节点接入并归一化图像，同时处理来自传感器和 PLC 的时间序列数据。
  // Enrich frame with sensor context
  const part = await client.objects.Part.get(partId);
  const sensors = await client.objects.Sensor
    .where({ line: line.id })
    .fetchPage();

  const input = {
    image: frame.buffer,
    partType: part.partType,
    temperature: sensors.data[0]?.temperature,
    lineSpeed: sensors.data[0]?.lineSpeed,
  };
  1. 模型推理：CV 模型在本地运行推理，对每个部件进行分类并检测缺陷。
  // Run CV model locally (GPU-accelerated) 
  const result = await runModel("defect-classifier-v3“, input);
  1. 决策与行动：应用程序决定下一步操作：放行部件并让流水线继续，或触发执行器将其剔除。每次检测都会记录为本体事件，可通过 OSDK API 访问。
// Reject part and save results for QA & model refinement
const inspection = await client.actions.createInspection({
    partId,
    line: line.id,
    classification: result.label,
    confidence: result.confidence,
    imageRef: frame.storageRef,
});

if (result.label === "defect") {
    await client.actions.createDefect({
      inspectionId: inspection.id,
      partId,
      defectType: result.defectType,
      confidence: result.confidence,
    });
    await client.actions.rejectPart({
      partId,
      actuator: line.rejectActuatorId,
    });
  }
});

同一段代码可以处理所有流水线，也可以复制到第二线、第三线直至第 N 线的工业 PC 中，所有这些都通过单一的 Apollo 部署进行管理。同时：结果同步至 Foundry/AIP，用于跨线分析、模型重训练、缺陷看板，以及与 PLM、ERP 和 MES 等 IT/OT 系统集成。纯代码和低代码选项让这一切如同使用 PWA 为浏览器或移动端构建 OSDK 应用一样简单。

这只是众多可能应用中的一个简单例子——从简单的统计过程控制和阈值告警，到结合视觉、传感器数据和历史背景以驱动自主决策的复杂多模态 AI。

规模化自主运维

在单一站点运营边缘基础设施是可控的。但在数百家工厂间运营——每家工厂都有不同的设备、网络条件和运营约束——则需要根本不同的方法。平台必须实现自我管理。

借助 Palantir 的边缘架构，软件和模型部署持续进行，无需中断数据采集。近乎零停机是常态，而非特例。平台监控自身健康状况——磁盘使用率、内存压力、数据库性能——并在问题演变为事件之前进行自我优化。当确实出现问题时，自愈操作器会执行自动修复，无需等待人工干预。

全局设备群的可观测性将每台设备、每个站点的遥测数据汇聚在单一视图中。操作员不仅能看到单个工厂正在发生的情况，还能看到整个设备群正在涌现的模式——哪些站点在发生漂移，哪些部署表现不佳，哪些模型需要重新训练。这不是为了看板而监控。这是实现大规模持续改进的基础。

系统对 IT 支持的需求降至最低。无需现场工程师看管服务器。无需为每个工厂配备运营团队。那些棘手难题——集成、安全加固、更新编排、修复——在构建流水线中一次性解决，并在每次部署中原样复制。在第一家工厂行得通的方案，在第二百家同样适用；在第二百家工厂学到的东西，反过来会改进第一家工厂。

边缘必须与更广泛的企业形成良性循环。边缘是运营发生的地方——传感器触发、机器运转、决策必须在毫秒内做出的地方。但边缘基础设施的价值只有在与整体相连时才会产生复利效应。

本地行动在每个站点自主执行：模型检测到异常、触发警报、调整工艺参数——这一切都无需等待往返云端。嵌入式本体确保这些决策扎根于实时上下文。当连接存在时，这些上下文会向上游流转。

在工厂层面，跨生产线的数据汇聚成统一的运营视图。维护团队可以纵观整个设施，查看哪些设备正趋向故障。质量工程师将缺陷与特定批次、班次或环境条件相关联。工厂经理基于由实时数据构建的数字孪生运行假设场景。

在企业层面，没有任何单一站点能揭示的模式会涌现出来。跨越数百家工厂的绩效基准能凸显最佳实践和异常值。供应链信号以近乎实时的方式从车间传播至计划系统。基于全局数据训练的模型回传部署至边缘，比任何单一站点模型都更聪明。运行在云端的 AI 智能体跨站点协调：优化供应链、重新平衡工作负载、重训模型——而边缘端的智能体则实时处理本地决策。

全局本体充当连接组织——将本地行动与企业智能相连，并确保企业的经验教训回流至本地决策中。
