A PySpark Style Guide for Real-world Data Scientists

2020-10-20 · Original →

TL;DR: We have open sourced our internal PySpark Style guide; you can find it here: https://github.com/palantir/pyspark-style-guide

At Palantir, we work with many different industries. The diversity of data engineering practices and conventions across these industries poses many problems. Different organizations have incredibly different PySpark styles, steepening learning curves and inhibiting both expansion and maintenance. Two years ago, a few Palantirians decided to enable analytical users to standardize their workflows and properly harden their data pipelines. We wrote a PySpark style guide that has been in use since then, evolving and maturing along the way. We’re excited to share it with the world.

The need for PySpark coding conventions

Our Palantir Foundry platform is used across a variety of industries by users from diverse technical backgrounds. From statisticians at a bank building risk models to aerospace engineers working on predictive maintenance for airplanes, we found that PySpark has become the de facto language for data science, engineering, and analytics at scale. A common problem with code written by a variety of users is the lack of best practices in large code bases, which makes maintaining and debugging data transformation production pipelines difficult.

df = df[["a", "b", "c", "key"]].filter(
        df["a"] == "truthiness"
    ).withColumn("boverc", df["b"] / df.c).join(
        df2, "key", "inner").join(df3, df3.key, "left")\
    .drop('c', 'f')

It’s hard to check for correctness when the code looks like this.

Data transformations and modeling running in production extend the realm of data science into the world of software engineering: How can we organize code for maintainability? How do you make sure the code base has a similar look and feel? How do you minimize the learning curve for new developers and guarantee consistent design patterns? At scale, data engineering (e.g., constructing data transformation pipelines, machine learning models, etc.) benefits from the same principles of craftsmanship as more traditional software development. We found that the absence of these principles was especially problematic for data scientists using PySpark:

  • Users from diverse backgrounds and language experience (e.g., SQL, mathematical languages such as MATLAB, R, SAS, Julia) can write well-intentioned code which is nevertheless non-Pythonic and non-performant in both Python and Spark.
  • Data science and advanced analytics departments are often pushed to continually deliver results without a focus on maintainability, stability, and robustness.
  • PySpark concepts and API syntax can be in conflict with common Python guidelines, as Spark’s libraries tend to be based on JVM conventions which were initially written to be SQL-like.

Data science code should follow the rules of good software engineering.
- Joel Grus’ “Why I don’t like notebooks” talk (author of Data Science from Scratch)

Modern data-driven applications rely on traditional code as well as data to function properly. For example, an application that uses a machine learning model can break if the software itself has a bug, or if the quality of the data underlying the model deteriorates. This means that the data engineering code is production code and thus needs the same rigor around testing, documentation, and legibility to ensure easy maintenance and troubleshooting.

Ignoring software engineering principles can accelerate the process of writing and running code in the short term, but this incurs long-term tech debt that impedes releasing to production and increases the cost of system maintenance. In order to raise the quality of our customers’ data engineering code bases, our initial approach was to help them with code reviews in Foundry, thus “throwing people at the problem” and burning engineering time. However, multiple “best practice” principles evolved organically, and largely existed only in people’s heads. We would often find conflicting styles on different files in the same repository.

How we compiled a style guide

We couldn’t find any style guide focused on PySpark that we could use as a baseline, so we set a goal to document an opinionated style guide and best practices for PySpark, along with matching linter rules. Since the first quarter of 2019, we have rolled out the style guide to many Foundry users, and have been evolving the guide based on their feedback. As such, the style guide we developed strives to match real world usage of PySpark:

A style guide that reflects real-world usage gets used, while a style guide that holds to an ideal that has been rejected by the people it is supposed to help risks not getting used at all — no matter how good it is.
- RuboCop

Individual rules within the style guide didn’t come without opposition. Some of the individual guidelines created friction among user groups both internally at Palantir and amongst our customers. Opinions around certain code practice often polarize groups of developers.

Simple things that are de facto practices in software engineering such as keeping a separation of concerns, or isolating logic, can dramatically help the readability and maintenance of PySpark Code. Isolating a particularly complex expression into a variable or a column is something that was easy to converge opinions around:

# bad
df.select(
    ((F.coalesce(F.unix_timestamp('closed_at'), F.unix_timestamp())
    - F.unix_timestamp('created_at')) / 86400).alias('days_open')
)
# good
df.withColumn(
    'days_open',
    (F.coalesce(F.unix_timestamp('closed_at'), F.unix_timestamp()) - F.unix_timestamp('created_at')) / 86400
)

Other more complex cases, such as avoiding performing long chains of expressions, took longer to converge. There were advocates for both extremes: “chain everything” versus “never chain”. A collective opinion ended up forming around recommending to avoid chaining of expressions into multi-line expressions with different types, particularly if they have different behaviors or contexts: for example, mixing column creation or joining with selecting and filtering.

# bad
df = (
    df
    .select('a', 'b', 'c', 'key')
    .filter(F.col('a') == 'truthiness')
    .withColumn('boverc', F.col('b') / F.col('c'))
    .join(df2, 'key', how='inner')
    .join(df3, 'key', how='left')
    .drop('c')
)

# better (seperating into steps)
# first: we select and trim down the data that we need
# second: we create the columns that we need to have
# third: joining with other dataframes

df = (
    df
    .select('a', 'b', 'c', 'key')
    .filter(F.col('a') == 'truthiness')
)

df = df.withColumn('boverc', F.col('b') / F.col('c'))

df = (
    df
    .join(df2, 'key', how='inner')
    .join(df3, 'key', how='left')
    .drop('c')
)

Evolution of the style guide

As the style guide evolved internally we had, at one point, people vocally ignoring it and applying their own practices, and at another we had multiple versions of the same guide, each focused on different aspects of the problem. Initially it wasn’t clear how to reconcile all of that, but the discussion ended up being beneficial in the long term, since settling conflicts and negotiating sensible trade-offs allowed us to evolve the rules towards more balanced and informed style choices.

The point of having style guidelines is to have a common vocabulary of coding so people can concentrate on what you are saying, rather than on how you are saying it. We present global style rules here so people know the vocabulary. But local style is also important. If code you add to a file looks drastically different from the existing code around it, the discontinuity throws readers out of their rhythm when they go to read it. Try to avoid this.
- https://google.github.io/styleguide/cppguide.html

Having this reference focused our code reviews on the content of our data engineering code bases instead of the distractions of parenthesis placement and line breaks. Each Palantir customer might be managing dozens of datasource integrations, handled as different projects that require constant monitoring ranging from data quality to code complexity. Our platforms handle each data pipeline as a first-class software product, so in a single project’s code repository we often have dozens of change requests being worked on against the master branch in parallel. More rigor around code style and choice of best practices eased rotating members between features and projects, and substantially sped up subsequent code changes and refactoring exercises.

Some of the rules we outline might still be polarizing, so as you read (and hopefully use!) the style guide, please provide feedback through issues or change requests on the repository where maintainers will centralize the discussions. Our intention is that this style guide helps data scientists and engineers out there, while continuing to evolve alongside the growth of the Spark and data science communities.

You can find the PySpark Style guide in this GitHub repository: https://github.com/palantir/pyspark-style-guide
