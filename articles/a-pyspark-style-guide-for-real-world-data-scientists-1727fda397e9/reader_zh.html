A PySpark Style Guide for Real-world Data Scientists

太长不看：我们已将内部的 PySpark 风格指南开源，你可以在这里找到： https://github.com/palantir/pyspark-style-guide

Palantir 的业务横跨众多不同行业。各行业在数据工程实践与规范上的差异带来了诸多问题。不同组织的 PySpark 风格大相径庭，这不仅拉高了学习曲线，也阻碍了代码的扩展与维护。两年前，几位 Palantir 员工决定帮助分析用户标准化工作流，并切实强化他们的数据管道。我们编写了一份 PySpark 风格指南并沿用至今，期间不断演进与打磨。现在，我们很高兴能将其分享给全世界。

PySpark 编码规范的必要性

Palantir Foundry 平台广泛应用于各类行业，用户的技术背景也各不相同。从银行构建风险模型的统计学家，到从事飞机预测性维护的航空航天工程师，我们发现 PySpark 已成为数据科学、工程和大规模分析的事实标准语言。由形形色色的用户编写的代码普遍面临一个问题：大型代码库缺乏最佳实践，导致数据转换生产管道的维护和调试困难重重。

df = df[["a", "b", "c", "key"]].filter(
        df["a"] == "truthiness"
    ).withColumn("boverc", df["b"] / df.c).join(
        df2, "key", "inner").join(df3, df3.key, "left")\
    .drop('c', 'f')

当代码长成这样时，很难检验其正确性。

在生产环境中运行的数据转换与建模，将数据科学的范畴延伸到了软件工程领域：如何组织代码以保障可维护性？如何确保代码库具有统一的观感与体验？如何缩短新开发人员的学习曲线并保证设计模式的一致性？在大规模场景下，数据工程（如构建数据转换管道、机器学习模型等）与更为传统的软件开发一样，都能从同样的工程匠心原则中获益。我们发现，对于使用 PySpark 的数据科学家而言，缺失这些原则尤其容易引发问题：

  • 背景和语言经验各异（如 SQL，以及 MATLAB、R、SAS、Julia 等数学语言）的用户可能会写出初衷虽好，但既不符合 Python 规范，在 Python 和 Spark 中执行效率也不高的代码。
  • 数据科学和高级分析部门往往被要求持续交付结果，却忽视了代码的可维护性、稳定性和健壮性。
  • PySpark 的概念和 API 语法可能与通用的 Python 指南冲突，因为 Spark 的库往往基于 JVM 约定，最初编写时偏向于类似 SQL 的风格。

数据科学代码应当遵循优秀的软件工程规则。
—— Joel Grus 在《为什么我不喜欢 Notebook》演讲中的发言（《数据科学从零开始》作者）

现代数据驱动型应用依赖传统代码和数据才能正常运行。例如，一个使用机器学习模型的应用，可能会因为软件本身存在缺陷，或者模型底层数据的质量下降而崩溃。这意味着数据工程代码即生产代码，因此在测试、文档和可读性方面需要同等严格的要求，以确保易于维护和排障。

无视软件工程原则或许能在短期内加快编写和运行代码的速度，但长期来看会欠下技术债务，阻碍产品发布并增加系统维护成本。为了提升客户数据工程代码库的质量，我们最初的做法是在 Foundry 中通过代码评审提供帮助，也就是“堆人力解决问题”，耗费大量工程时间。然而，多项“最佳实践”原则是零散演化出来的，且大多只存在于个人的脑海中。我们经常在同一个代码仓库的不同文件中发现风格冲突的情况。

我们如何编写风格指南

我们找不到一份专注于 PySpark 并可作为基准的风格指南，因此我们设定了一个目标：为 PySpark 编写一份带有明确倾向性的风格指南与最佳实践，并配套相应的 Linter 规则。自 2019 年第一季度起，我们向众多 Foundry 用户推行了这份风格指南，并基于他们的反馈不断演进。因此，我们开发的风格指南力求契合 PySpark 在真实场景中的用法：

反映真实用法的风格指南才会被采用；而如果一份风格指南固守着被目标受众排斥的理想教条，无论它多么出色，都面临无人问津的风险。
—— RuboCop

风格指南中的各项规则并非没有遭遇阻力。一些具体的指导原则在 Palantir 内部以及客户群体的用户群中引发了摩擦。围绕某些代码实践的分歧，往往会让开发者群体产生两极分化。

软件工程中一些已形成事实标准的简单做法，例如保持关注点分离或逻辑隔离，能够极大提升 PySpark 代码的可读性与可维护性。将特别复杂的表达式剥离成单独的变量或列，就是一个很容易达成共识的做法：

# 差
df.select(
    ((F.coalesce(F.unix_timestamp('closed_at'), F.unix_timestamp())
    - F.unix_timestamp('created_at')) / 86400).alias('days_open')
)

# 好
df.withColumn(
    'days_open',
    (F.coalesce(F.unix_timestamp('closed_at'), F.unix_timestamp()) - F.unix_timestamp('created_at')) / 86400
)

其他更复杂的情况，例如避免冗长的表达式链，则花了更长时间才达成共识。当时存在两种极端主张：“全链式调用”与“绝不链式调用”。最终形成的共识是：建议避免将不同类型的表达式串联成多行表达式，尤其是当它们具有不同行为或上下文时：例如，将列创建或连接操作与选择和过滤操作混在一起。

# 差
df = (
    df
    .select('a', 'b', 'c', 'key')
    .filter(F.col('a') == 'truthiness')
    .withColumn('boverc', F.col('b') / F.col('c'))
    .join(df2, 'key', how='inner')
    .join(df3, 'key', how='left')
    .drop('c')
)

# 更好（分步骤执行）
# 第一步：选择并精简我们需要的数据
# 第二步：创建我们需要的列
# 第三步：与其他数据框进行连接

df = (
    df
    .select('a', 'b', 'c', 'key')
    .filter(F.col('a') == 'truthiness')
)

df = df.withColumn('boverc', F.col('b') / F.col('c'))

df = (
    df
    .join(df2, 'key', how='inner')
    .join(df3, 'key', how='left')
    .drop('c')
)

风格指南的演进

随着风格指南在内部不断演进，我们曾经历过一段时期：有人明确无视它并采用自己的实践，而在另一阶段，同一份指南又衍生出多个版本，各自侧重问题的不同方面。起初我们并不清楚如何调和这些分歧，但长远来看，这些讨论最终带来了积极效果，因为解决冲突并协商出合理的折中方案，促使我们将规则推向了更均衡、更明智的风格选择。

制定风格指南的意义在于建立一套通用的编码词汇表，这样人们就可以把注意力集中在你表达了什么，而不是你怎么表达上。我们在此提出全局风格规则，是为了让人们了解这套词汇表。但局部风格同样重要。如果你添加到文件中的代码与周围现有代码的风格大相径庭，这种突兀的断裂感会打断读者阅读时的节奏。应尽量避免这种情况。
—— https://google.github.io/styleguide/cppguide.html

有了这份参考标准，我们的代码评审得以将焦点放在数据工程代码库的实际内容上，而不再被括号位置和换行等细枝末节分散精力。每位 Palantir 客户可能都在管理数十个数据源集成，这些集成作为不同的项目处理，需要从数据质量到代码复杂度进行持续监控。我们的平台将每条数据管道视为一等软件产品，因此在单个项目的代码仓库中，经常会有数十个变更请求并行地在 master 分支上推进。更严格的代码风格规范和最佳实践选择，使得在不同功能和项目间轮换成员变得更加轻松，也大幅加快了后续的代码修改和重构工作。

我们列出的一些规则可能仍会引发争议，因此当你阅读（并希望使用！）这份风格指南时，请通过仓库的 Issue 或变更请求提供反馈，维护者将在那里集中讨论。我们的初衷是希望这份风格指南能帮助广大数据科学家和工程师，同时随着 Spark 和数据科学社区的发展而继续演进。
