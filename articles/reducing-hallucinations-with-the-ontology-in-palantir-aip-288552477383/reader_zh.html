在过去的二十年里，Palantir 一直致力于通过我们的原则、公开立场以及产品来推动技术的负责任使用。这一点在我们当前的工作中体现得尤为重要：我们正与各类组织合作，通过 Palantir AIP 发挥 AI 的强大威力。从确保安全到提升可解释性，再到解决公平性与偏见问题，Palantir AIP 为用户提供了一套先进的功能，助力在实践中落地“负责任的 AI”。在这一系列博文中，我们将分享如何将我们对 AI 伦理和“负责任的 AI”的承诺直接贯彻到 Palantir AIP 的产品开发中。

在这篇博文中，我们将探讨 AI 安全领域对话中最普遍的挑战之一：幻觉。

幻觉

当生成式 AI 模型给出虚假或具有误导性的信息时，就会发生“幻觉”[1]。例如，你可能向大语言模型（LLM）询问今天的天气，尽管你正置身于倾盆大雨中，它却回答天气晴朗且温暖。又或者，你可能要求它提供某个研究领域的学术文献引用，结果却发现模型返回的是一个拼凑而成的合成引用——它由一些真实的零碎信息构成（如正当的期刊、真实的作者以及看似合理的文章名称），但拼凑在一起，却成了一篇实际上根本不存在的假文章。“幻觉”一词（尽管存在一些争议）指的正是 LLM 生成此类虚假信息的倾向。

为了更好地理解幻觉及其成因，让我们回顾一下 LLM 的工作原理。LLM 的设计和训练目的只有一个：基于给定的文本序列，预测下一个最可能的“词元”。词元可以是一个词、词的一部分，甚至是一串字符。LLM 之所以具备这种能力，是因为它们在长文本序列上进行了“预训练”，并经过进一步微调，以提高预测下一个词元的相关性和质量，从而让这些模型能够回答问题或响应任务。LLM 预测最可能的下一个词元的能力，使得这些模型能够生成逼真的文本，完成诸如回答问题、进行对话以及起草看似真实的文本等任务。然而，在回答问题方面，预测下一个词元并不要求 LLM 返回“真实的”或“准确的”答案。LLM 的任务仅仅是基于其训练数据，生成在统计学上看似合理的文本。

幻觉的产生有多种原因。例如，模型的训练基于有限的数据，而回答您问题所需的相关数据可能并未包含在模型的训练数据中。此外，LLM 架构内部的随机性（或随机因素）可能导致模型为某些下一个词的预测分配比我们预期更高的概率。

尽管 LLM 的某些改进可能有助于在未来降低幻觉发生的概率，但幻觉的某些特性本身就是这些模型生成能力的一部分。正是这些让 LLM 能够生成流畅且富有创意文本的特性，同时也带来了引入幻觉的风险。那么，在无法修改训练数据或 LLM 架构的情况下，我们应如何利用 LLM 执行可靠且有用的任务，同时降低幻觉的发生频率呢？

一个关键洞见在于，重点关注 LLM 真正擅长的领域：发挥其预测下一个词元的能力，并以此扩展开去生成合成文本。重要的是，这不应将我们局限于聊天机器人或邮件助手等应用。借助合适的工具来约束模型并将其锚定于现实，我们可以利用 LLM 的生成能力来支持更广泛、更强大且更实用的工作流，涵盖搜索、编排和自动化。在这篇博文中，我们将重点介绍几种在实践中降低幻觉风险的技术，并演示如何使用 Palantir AIP 将这些方法集成到您的工作流中。

在 Palantir AIP 中降低幻觉

Palantir 构建软件平台的核心洞见之一是：当数据准确映射组织内部的概念和流程时，其对于企业决策的价值才达到最大化。我们将这种融合了数据与逻辑的软件模型称为本体，并且我们已经进行了详尽的论述，说明这种软件架构模型为何对高效使用 AI 至关重要。在生成式 AI 的语境下，本体使用户能够将 LLM 与支撑组织决策的受信任数据、逻辑和操作连接起来。这种 LLM 与本体的集成有助于降低幻觉发生的概率和影响。让我们通过三个实际案例来说明这一点。

查询本体获取受信任数据

本体作为受信任的数据源，将 LLM 锚定在您组织的特定上下文中。首先来看一个 LLM 产生幻觉并返回误导性信息的示例。在下面的 AIP Logic 函数中，我们要求 LLM 回答一个关于 Titan Industries 的问题，这是我们《使用 Palantir AIP 构建》系列博文中假设的制造企业。我们希望模型返回一些美国分销中心所在城市的列表。

当我们运行这个 AIP Logic 函数时，模型生成了一个它认为最可能的城市列表。然而，这个输出结果是幻觉。尽管关于 Titan Industries 的信息是虚构的，但我们的企业数据表明，Titan Industries 主要在较小的城市运营，集中在美国东北部、中西部和加利福尼亚州。LLM 返回的城市中没有一个是分销中心的实际所在地。

让我们剖析一下这种情况为何会发生。关于我们这家假设的公司 Titan Industries 的详细信息并不存在于互联网或任何其他地方。这些信息仅存在于 Titan Industries 的企业数据中，并已集成到其本体中。这意味着 LLM 在训练期间从未接触过准确回答此问题所需的任何信息。因此，当被问及美国分销中心所在的城市时，模型基于其训练数据预测了统计学上最可能的词元。这些城市被视为“幻觉”，因为它们是对该问题的事实性错误回答，尽管它们看起来可能很有说服力。

为了解决这种幻觉，用户可以用受信任的、经过验证的数据来“锚定”LLM。在实践中，这意味着允许模型直接从本体请求数据，以补充原始提示词中的信息。在 AIP Logic 中，我们可以通过为 Logic 函数提供一个查询本体对象的工具，来实现将 LLM 锚定于本体。这一过程利用本体元数据（关于对象类型及其属性的信息）来扩充用户提供的提示词，从而引导 LLM 生成搜索查询，随后针对本体执行该查询。

让我们看看实际效果。通过在添加该工具后重新运行此 Logic 函数，我们看到模型现在返回了正确答案。

需要注意的是，即使相关信息已包含在 LLM 的训练数据中，仍可能发生上述描述的幻觉。在这种情况下，LLM 可能只是未能预测出正确的下一个词以产生“符合事实的”答案，这凸显了将 LLM 锚定于本体的重要性。如需深入了解此方法，请阅读我们关于《本体增强生成 (OAG)》的博文。

交接给受信任的逻辑工具

赋予 LLM 查询本体以获取受信任数据的能力，是降低模型响应中幻觉发生概率的重要一步。但这并非万灵药。幻觉的产生有多种原因，让我们来看另一个成因：过度依赖 LLM 执行其并不擅长的计算任务而导致的幻觉。如前所述，LLM 旨在预测下一个最佳词元。虽然这种能力有助于生成看似真实的文本，但对于求解方程、预测或运行模拟等更复杂的任务而言，它并不总是合适的工具，这些任务更适合由专门构建的模型或函数来处理。

与为 LLM 提供查询本体的工具类似，我们也可以让 LLM 编排并向其他类型的模型或规则进行交接。对于 LLM 并不擅长的任务，通过利用更具可解释性、确定性或可靠性的逻辑形式，我们可以进一步降低整个 Logic 函数中发生幻觉的可能性。

让我们继续以 Titan Industries 为例。这一次，我们的一辆卡车在某处抛锚了，我们希望通过与 AIP 交互来找出距离它最近的是哪个分销中心。正如我们从第一个示例中所学到的，我们现在已为 Logic 函数提供了查询本体的工具，使模型能够获取我们分销中心的位置。运行 Logic 函数后，模型返回纽约州奥尔巴尼作为距离抛锚卡车坐标最近的城市。然而，这属于幻觉。卡车的实际位置在马萨诸塞州斯普林菲尔德，它距离罗德岛普罗维登斯的分销中心更近。

让我们思考一下为什么会发生这种情况。首先，LLM 没有计算坐标之间距离的内置能力。预测最可能的词元只能在近似简单算术方面发挥作用，更不用说计算旅行距离所需的更复杂的计算了。预测下一个词元对于此类任务而言是错误的计算方式，如果我们要求 LLM 执行它不擅长的任务，只会增加输出误导性结果的概率。

为了降低此类基于逻辑的幻觉发生的概率，我们可以为 LLM 提供一个“工具”，将计算委托给某个模型、函数或其他逻辑部件。在此案例中，我们可以为 LLM 提供一个位于我们本体中的自定义函数，该函数利用 Haversine 公式计算任意坐标点与已知分销中心之间的旅行距离。就像查询本体工具一样，LLM 随后可以创建对该工具的请求，AIP Logic 将根据给定的请求调用该工具，并将结果传回给 LLM。这种 LLM 与工具之间的交接机制扩充了 LLM 预测下一个词元的核心能力，为模型配备了适宜的工具，以应对其可能需要执行的任何类型的逻辑运算。

在这种情况下，使用自定义函数重新运行相同的 Logic 函数，我们看到 Logic 函数正确输出了罗德岛普罗维登斯作为最近的分销中心。这是因为它能够依赖自定义函数来执行距离计算，而不是凭空捏造一个错误的答案。

审查 AI 生成的操作

本体是以决策为中心的企业模型，将数据、逻辑和操作融为一体。到目前为止，我们已经讨论了如何使用数据工具和逻辑工具来降低幻觉发生的概率和影响。但是，无论我们设计得多么周全，幻觉始终是使用生成式 AI 的一种潜在副作用，因此我们必须考虑到模型在采取这些措施后仍返回幻觉响应的可能性。现在让我们来看看如何利用本体来更好地监督和监控 LLM 建议操作中的潜在幻觉。

继续以 Titan Industries 为例，我们可以编写一个 AIP Logic 函数，利用 LLM 帮助我们寻找解决方案，以应对关于库存短缺的内部支持工单，如下图所示。我们将从支持工单中提取详细信息，将其映射到本体中的组件，引入文档以帮助查找相关信息，并最终提出一个在分销中心之间重新分配或替换库存的解决方案（如果适用）。

随后，我们可以将这个 AIP Logic 函数接入到我们的库存管理应用中。每当本体中创建新的支持工单时，我们的 Logic 函数就会被自动调用以寻找解决方案。尽管我们为 LLM 提供了查询数据和向受信任逻辑交接的工具，模型仍有可能产生幻觉响应。例如，支持工单中的说明可能含糊不清，且与我们本体中的文档信息不匹配，从而导致无用的建议。或者 LLM 可能未使用正确的参数调用重新分配模型。在任何这些情况下，将人工审查和监督纳入我们的工作流至关重要，以此作为防范幻觉的又一道防线。

AIP Logic 不会直接采取建议的重新分配或替换操作并将其写回外部系统，而是可以排队提交一份建议的重新分配方案以供审批。此时，领域专家可以审查建议的重新分配方案，确认其在支持工单的语境下是否合理，并明确批准或拒绝该操作——甚至可以调查该特定 AI 生成的解决方案是如何得出的。纳入这种形式的人工审查可以提供额外的防御层，以降低幻觉对运营决策的影响。

这种“人机协同”的方法允许用户无缝地交互、监督和修改 AI 生成的操作，对于在实践中应用“负责任的 AI”原则至关重要。而且，这不仅仅局限于降低幻觉的影响：这种设计模式还能提升问责制，阐明 AI 生成的结果，并确保 AI 的使用始终以人为本。我们将在本系列的未来博文中探讨这些主题。

结语

降低幻觉风险对于有效且负责任地使用生成式 AI 至关重要。Palantir AIP 使您能够将 LLM 锚定于来自本体的数据中，让您能够无缝地向模型提供受信任的数据，并帮助其产生更具相关性的结果。AIP Logic 还允许您使用工具扩展 LLM，使 LLM 能够委托某些更适合通过其他计算形式来处理的逻辑任务。同样重要的是，Palantir 本体让您能够利用生成式 AI，同时仍确保对 AI 建议的决策拥有适当的监督和控制权。这三种方法——建立在您本体的数据、逻辑和操作之上——能够降低幻觉发生的概率和影响，并使组织能够在 Palantir AIP 中将 LLM 应用于最具挑战性的问题。

[1] 当我们使用诸如“要求模型响应查询”之类的表述时，我们借用了日常用语，这些用语可能为人际交流的语境所熟悉。在关于语言模型的探讨中，“响应”和“回复”等词已成为描述与模型交互的便捷方式。尽管我们在本博文中使用了这些术语，但必须警示将模型拟人化的风险。这些模型无法在真正意义上进行响应、回复或推理；相反，我们向模型提供一些文本作为输入，而模型则返回一些在统计学上具有可能性的文本作为其输出。
