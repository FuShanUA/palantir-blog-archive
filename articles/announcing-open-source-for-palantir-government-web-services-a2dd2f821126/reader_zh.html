Announcing Open Source for Palantir Government Web Services

2023-10-25 · Original →

编者按：本文是《技术附录：发布 Palantir 政府网络服务 (GWS)》的后续文章，该文预告了我们将为 GWS 推出的一些即将开源的产品。在接下来的几个月里，我们将分享最引人注目的开源工具，以加速国防科技生态系统中的构建者。首先是用于合规变更管理的《Policy Bot》。

为什么选择开源？

用于支持作战人员的软件门槛很高。期望软件能在 10 万多个异构环境中始终保持可用，从大型涉密数据中心到 Replicator 机器人，从云端到边缘。构建、部署和维护基础设施与应用程序，将这些期望转化为现实，是整个国防科技生态系统共同的任务。为了助力这一使命，我们将开源两类能力：

  1. 通用实用工具，帮助国防科技公司更快实现规模化扩展，从构建安全的软件应用程序到通过零配置应用服务器简化复杂软件的运行。
  2. 持续降低锁定风险并降低客户切换成本的技术。政府应始终能够采用最佳解决方案，同时保持运营基础设施和访问其数据的能力，而无需在决定放弃时支付许可费用。

过去二十年来，Palantir 为开源做出了巨大贡献。Spark 最初并不支持 Kubernetes。我们贡献了代码以促进兼容性，使 Spark 能够自动扩展并充分发挥云的价值。我们在 GitHub 上拥有超过 220 个开源 仓库。这些仓库不仅仅代表了一份功能列表——总体而言，它们是帮助公司更好地为作战人员服务的变革性构建块。我们想重点介绍的首个构建块是用于合规变更管理的 Policy Bot。

Policy Bot

对于服务于国家最关键机构的公司而言，严格的合规性是不容妥协的。同时，快速提供软件解决方案至关重要。Policy Bot 是一款开源的 GitHub 应用，它能在拉取请求上强制执行审批策略，消除对特定审查人员的依赖瓶颈，使软件开发在不牺牲速度的前提下符合合规要求。

Palantir 使用 Policy Bot 按照政府的 DevSecOps 最佳实践开发软件，并将其作为满足 FedRAMP、IL5 和 IL6 合规性变更管理流程的一部分。具体而言，Policy Bot 能够满足这些认证所要求的以下控制措施，同时保持快速的发布节奏：

  • CM-3 — 配置变更控制
  • CM-3(1) — 变更的自动化文档记录、通知与禁止
  • SA-10 — 开发者配置管理

除 Palantir 外，Policy Bot 还有多个企业贡献者。我们希望 First Breakfast 的成员公司能从中受益，持续为客户大规模交付影响力。

技术细节

任何合规体系的基本要求之一是，任何变更（无论是软件还是基础设施）在进入生产环境前都必须经过审查。鉴于 Palantir 遵循基础设施即代码实践，这一过程几乎总是以 代码审查 的形式进行。

要求代码审查听起来是一个简单的问题，但随着我们对具有不同工作流的开发团队进行调查，需求变得更加细致。例如：

  • 针对特定分支（如预发布分支）的变更不需要审查。
  • 某些类型的变更可能需要信息安全或法务团队的额外批准。
  • 某些类型的变更（如文档变更）已预先获得批准，无需审查。
  • 当多个环境的配置集中在一个仓库中时，开发人员应能在同级批准的情况下更改各自的环境（例如通过仓库内的子文件夹识别），而不必等待 DevOps 团队成员的批准。

未能自动化上述任何条件都会拖慢开发人员的进度，迫使他们四处寻找并打断他人以获取批准，从而造成不必要的变更延误。当团队地理位置分散或忙于其他工作时，合并一项变更可能需要数天时间。

我们最初寻求 GitHub 现有工具来寻找解决方案，但没有足够精细的工具能满足我们的约束条件。因此，我们构建了自己的与 GitHub 集成的审批工具。
