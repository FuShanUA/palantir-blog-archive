将工作站安全策略实现为代码

编者按：在这篇博文中，来自我们信息 安全团队的 Chad 和 Tyler 探讨了 Palantir 在企业范围内管理浏览器扩展的方法演进，并详细介绍了如今这套低阻力流程的运作方式。

浏览器扩展：不断演进的威胁

Web 浏览器是许多敏感业务任务的主要交互界面；它们如此普遍，以至于人们很容易忽视与之相关的 安全 风险。几乎每位 Web 浏览器用户都会使用浏览器扩展。常见的扩展包括密码管理器（可代为填充凭证）和广告拦截器（可根据规则列表阻止特定内容，即广告加载）。尽管看似有益，浏览器扩展却对个人和企业构成了潜在的灾难性风险。

关于扩展，目前存在两个重大的未决问题：

  • 若不加管控与监控，浏览器扩展会带来极大的风险。从设计上讲，扩展是在浏览器会话中运行的软件，并根据其被授予的权限与该会话进行交互。虽然大多数浏览器扩展的预期功能都是为了造福用户，但执行这些功能所需的权限却非常强大，为潜在的滥用打开了大门。与任何软件一样，功能的增加意味着代码量的增加，从而导致更大的攻击面和更高的漏洞发生率。

    这意味着，恶意扩展可以轻而易举地收集用户的敏感信息、监控击键和文件上传、为 nefarious 目的重写页面内容，或以其他方式滥用查看和操纵网络流量的能力。即便是起初良性的扩展也可能变坏：拥有一定用户基础的扩展被转让或出售给其他方是常有的事，在现有的目标受众群中，只需一次软件更新，就能实现瞬时且广泛的分发。
  • 在整个企业范围内管理浏览器扩展非常困难。默认情况下，用户可以随意安装任何想要的浏览器扩展，无论是来自网络应用商店还是其他来源，而对其请求的权限和带来的风险几乎一无所知。显然，这对任何组织都是一个重大隐患，但管理哪些扩展应该或不应被允许却是一个复杂且耗费资源的过程，尤其是当终端存在多种操作系统和/或浏览器时。

    举例来说，只要组织为其用户配备了 Google Workspace 账户并且只使用 Chrome，Google 就能在一个管理后台中让扩展的管理变得相对简单——但这并不适用于绝大多数组织。Microsoft Edge 和 Mozilla Firefox 完全依赖于直接部署到工作站的 安全 策略，这意味着任何关于允许或拒绝扩展的更改，只能由域管理员基于单纯的扩展 ID 进行手动更新管理——这种做法极易发生人为错误。

    在缺乏更低阻力方案的情况下，那些担心浏览器扩展带来风险的组织最终往往倾向于直接全面封禁，而不是去试图应对看似必不可少的管理开销，这也就不足为奇了。

Palantir 如何实现低阻力的浏览器扩展管理

从黑名单转向白名单

Palantir 历史上一直维护着一份已知恶意扩展的黑名单，其来源主要是 安全 行业的新闻报道和其他安全通报。曾几何时，这似乎是唯一可行的选择，因为转向“默认拒绝”模型会破坏大量既有的用户工作流，除非我们愿意投入大量的管理开销，为 Windows 终端的组策略和 macOS 终端的 Jamf plists 分别手动维护 Edge、Chrome、Firefox 的独立白名单。

尽管如此，我们还是开始调查全网络中已安装的扩展。我们求助于我们的 osquery 数据集，该数据集包含一个表格，记录了这两种操作系统上所有三种浏览器已安装的浏览器扩展。运行快速查询并按安装量排序后，哪些扩展常见、哪些是个例便一目了然。数据中透露出一些意外发现：数百个独特的扩展仅安装在一两台工作站上；有些扩展几年前就已从网络应用商店下架，但仍然处于启用状态；还有一位员工安装了 40 多个扩展（被问及时，他们对其中的大多数全然不知）。

除了这些意外发现，这次调查还揭示了几个关键要点：

  • 由于许多用户使用个人 Google 账号登录 Chrome，他们一直在个人电脑和工作电脑之间同步扩展（及其他设置）。这意味着，如果员工家用电脑上的恶意软件强行在该浏览器中安装了扩展，它就会自动出现在其工作电脑上，从而有效地将恶意软件延续到企业网络中。虽然我们早得多之前就禁用了 Chrome 配置文件同步功能，但让我们感到沮丧的是，在切断同步之前引入的任何内容依然存在。
  • 在 Chromium 内核浏览器（Chrome 和 Edge）中，从相关网络应用商店下架的扩展并不一定会从已安装它的浏览器中移除或禁用。有时，如果扩展因恶意行为被下架，浏览器会将其禁用并提示相应信息，但大多数情况下，扩展被下架时 Google 或 Microsoft 不会发表任何评论。而 Firefox 则会从浏览器中移除任何无法从原始安装源获取的扩展。
  • 大多数员工实际依赖的扩展只有三四个——几乎所有安装了更多扩展的员工，要么是不了解这些扩展，要么是工作中根本用不到它们。

掌握了 osquery 指标后，我们根据独特安装数量、宣传的功能以及在企业环境中的实用性等标准，精心挑选了一份初始的可接受扩展白名单。随后，我们将注意力转向如何随着时间推移高效维护这份名单的问题。

将白名单定义为代码：打造用户驱动的流程

在流程初期我们就意识到，可能永远无法将扩展白名单一成不变地固定下来：用户经常会发现能支持关键业务流程的新扩展，而如果怀疑存在滥用行为，先前已批准的扩展也可能需要迅速被封禁。

我们认定，在声明单一事实来源的同时赋予员工请求变更的能力，最佳方式是将其定义为内部 GitHub 代码仓库中的代码。在这个仓库中，我们为三种 Web 浏览器分别编制了一个相对简单的 json 文件，格式如下：

[
    {
      "ExtensionDescription": "Approved Password manager",
      "ExtensionId": "dppgmdbiimibapkepcbdbmkaabgiofem",
      "ExtensionName": "1Password – Password Manager",
      "ExtensionType": "Security"
    },
    {
      "ExtensionDescription": "Another Extension - used by the frontend dev team",
      "ExtensionId": "aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaam",
      "ExtensionName": "Another Extension",
      "ExtensionType": "Productivity"
    }
 ]

归根结底，ExtensionId 是白名单中唯一重要的字段（因为白名单不过是一份简单的扩展 ID 列表），但我们加入其他字段是为了在解析事实来源文件时更符合人类阅读习惯。

有了这份名单，员工可以直接提交 Pull Request (PR) 来更新名单；如果他们不习惯使用 GitHub，也可以提交工单让我们的团队代为提交 PR。这不仅赋予了员工一定程度的自主权和流程参与感（稍后详述），也让我们有时间在做决定前对提议的扩展进行调查。

我们典型的审查流程包含以下一般步骤：

  • 提交 PR 时，CircleCI 自动化会执行基础检查，以确保 json 格式正确、条目未重复，且该扩展确实存在于相关的网络应用商店中。除了验证扩展 ID 外，CircleCI 还会对提供的 ExtensionName 与网络应用商店中列出的名称进行基础比对，以帮助发现任何分配错误或误导性条目。
  • 无论是否有未处理的 PR，这项自动化检查都会每天运行，并在出现任何失败时向团队发送警报，这表明某个已批准的扩展已从相应的网络应用商店下架，因此需要展开调查。
  • 以下代码片段展示了对 Chrome 网上应用商店的检查——Edge 也会应用类似的检查，Firefox 则应用更简单的检查，因为该网络应用商店拥有公共 API。它包含一个 exceptions 列表，用于存放不应通过网络应用商店检查的扩展，例如自行发布或私有/未公开的扩展。
steps:
      - checkout
      - run:
          name: Parse JSON file for syntax errors
          command: cat *.json | jq

      - run:
          name: Check extensions are not duplicated
          command: |
            set +e
            extensions=$(jq -r ".[] | .ExtensionId" chrome_extensions.json)
            duplicates=$(echo "$extensions" | sort | uniq -c | sort -nr | grep -v '^ *1 ' | cat)
            set -e
            if [[ -n "$duplicates" ]]; then
              echo "There are duplicate extension ids"
              echo -n "$duplicates"
              exit 1
            fi
            
      - run:
          name: Check all extensions exist
          command: |
            extensions=$(jq -r ".[] | .ExtensionId" chrome_extensions.json)
            rc=0
            # These extensions are not required to be in the Chrome Web Store
            exceptions=(space_separated_extensions_list)
            
            for extension in $extensions; do
              if [[ " ${exceptions[@]} " =~ " ${extension} " ]]; then
                echo "$extension is not required to be in the Chrome Web Store"
                continue
              fi
              extension_name=$(jq ".[] | select(.ExtensionId==\"$extension\").ExtensionName" chrome_extensions.json)
              detail_url="https://chrome.google.com/webstore/detail/$extension"
              # Validate that the chrome webstore knows about this extension id
              http_code=$(curl "$detail_url" -s -o /dev/null -w "%{http_code}")
              if [[ $http_code -gt 399 ]]; then
                rc=$((rc+1))
                echo "Could not validate $extension_name ($extension)"
                exit $rc
              fi
              # Validate that the extension names matches the value in the chrome webstore
              effective_url=$(curl "$detail_url" -Ls -o /dev/null -w %{url_effective})
              effective_name=$(echo $effective_url | sed 's#https://chrome.google.com/webstore/detail/##g' | sed "s#/$extension##g")
              # some postprocessing to make the human readable name match the extension webstore
              converted_extension_name=$(echo "${extension_name}" | tr '[:upper:]' '[:lower:]' | sed 's/(//g' | sed 's/)//g' | sed 's/ & /-/g' | sed 's/://g' | sed 's/ – /-%E2%80%93-/g' | sed 's/ - /-/g' | sed 's/ /-/g')
              if ! [[ "$converted_extension_name" =~ "$effective_name" ]]; then
                echo "${converted_extension_name} does not belong with ${effective_name}"
              fi
            done
            exit $rc
  • 一旦 PR 通过这些基础检查，我们就会调查该扩展所需的权限。虽然这可以在实验室环境中或通过下载并提取清单文件来完成，但我们倾向于利用在线免费提供的检查工具，例如 Duo Labs 出色的 CRXcavator。此类工具能提供关于请求权限、扩展预期通信的外部网络流量目的地、用户群规模统计等有价值的信息摘要，甚至还能渲染扩展源代码，供我们这样的应用 安全 团队在有疑问时进行检查。
  • 然后，我们会调查官方扩展网络应用商店页面上的隐私政策。在 Chrome 网上应用商店中，这被称为“隐私惯例”。在 Edge 加载项商店中，该政策位于“条款”部分。虽然这显然不能保证扩展的实际行为，但我们会留意任何提及从浏览器出售数据的内容。
  • 接下来，我们会快速进行互联网搜索 (OSINT) 以寻找“不良”行为的迹象，作为直觉上的校验。随着时间的推移，我们发现，在大多数搜索引擎中，搜索“extension ID + malware”似乎最具价值；但 Twitter 是个例外，在其中搜索“extension name + malware”往往更可靠。有时这会揭示出关于行为不端的软件和不可信开发者的整篇文章，而有时迹象可能很微妙——这不过是一种快速直觉检查罢了。
  • 最后，我们会权衡生产力收益/潜在的业务正面影响与风险。例如，使用一家声誉极高的供应商发布的密码管理器扩展，其好处要大于不得不从桌面应用或 Web 仪表板手动复制/粘贴密码所带来的不便——或者更糟的是，根本不使用密码管理器。

有了这套流程，请求浏览器扩展往往会成为一项令人惊讶的个性化行为，因为请求者需要有足够的信念，并愿意论证为什么应该允许某项扩展。我们会尽可能偏向于批准，因为我们看重成为值得信赖的伙伴，而不是守门人。当我们确实拒绝请求时，我们不会武断行事——我们总是会与对方交流，探讨为什么我们认为某项扩展的风险大于收益。我们也会尝试建议替代方案：我们发现，实际独特的用例数量少得惊人，很可能我们已经允许了一款具有同等或更小风险的类似扩展。只要能给请求者带来他们寻求的结果，他们通常并不在意具体是如何实现的。

工作站策略管理员怎么办？

在流程的这个阶段，我们确信能够充分支持并与终端用户互动，但还有一个问题：如何确保我们的工作站策略管理员也满意？我们认为，要求他们交叉比对并手动复制/粘贴单个扩展 ID 到受控的组策略和 Jamf 配置文件仪表板中太过繁琐，但与此同时，我们又找不到自动化这类工作流的先例。我们并未气馁，而是从其他企业基础设施即代码的实现中寻找灵感。

当我们找到 PowerShell cmdlet Set-GPRegistryValue 时，我们取得了突破。我们早已知道，在 Windows 中，这三种浏览器的白名单策略不过是一个带编号的注册表项列表。经过大量的反复试错，我们围绕这条 PowerShell 命令开发了一条流水线，能够在 PR 获批并合并后约一小时内，自动将新的白名单部署到所有 Windows 终端，无需任何人工干预。

其工作原理如下：

第一步：发布至 Artifactory
一旦 PR 合并且 json 列表更新，CircleCI 会自动将新版本发布到我们的内部 Artifactory。以下是 Edge 的示例；Chrome 和 Firefox 几乎完全相同。

publish:
    docker:
      - image: container.palantir.internal/circle-build/ubuntu
    steps:
      - checkout
      - run:
          name: Publish extensions
          command: |
            curl --fail -u "${USERNAME}:${PASSWORD}" \
            -X PUT "https://files.palantir.internal/files/extensions/latest/edge_extensions.json" \
            -T edge_extensions.json

第二步：计划内的 PowerShell 脚本对比并更新组策略
我们编写了 PowerShell 脚本来检查 Artifactory 中发布的 json 列表，并与当前组策略进行比对；如果存在差异，这些脚本会从头重建相关列表（这同时支持添加和移除操作）。

这些 PowerShell 脚本被打包到 Docker 镜像中，通过 Hashistack 部署到我们的 Tier-0（相当于域控）网络，并通过计划任务每五分钟运行一次脚本。这台单功能机器在组托管服务账户的上下文中执行脚本，该账户对控制浏览器策略的组策略拥有 UPDATE 权限（且绝对仅有此权限）。由于这台工作机器能够修改可能影响众多机器的组策略，我们采用与实际域控制器和其他 Tier-0 服务相同的安全考量来处理它。

# Applying Registry w/ PowerShell:
#   https://docs.microsoft.com/en-us/powershell/module/grouppolicy/set-gpregistryvalue?view=win10-ps

# Limited Scope GPO:
#    "{YOUR GPO GUID}"

# Allowlist location:
#    Computer\HKEY_CURRENT_USER\Software\Policies\Google\Chrome\ExtensionInstallAllowlist

# The GPO operates as a simple ordered list of REG_SZ incrementing from one (not zero).


$GPO_Id = "{YOUR GPO GUID HERE}"
$chrome_rules_file = "chrome_extensions.json"
$extensions_uri = "https://files.palantir.internal/files/extensions/latest/" + $chrome_rules_file
$RegistryPath = "HKEY_CURRENT_USER\Software\Policies\Google\Chrome\ExtensionInstallAllowlist"

Import-Module GroupPolicy

Write-Host $(Get-Date -Format u) "[info] Chrome Policy Comparison Starting"

function Get-CurrentJSON {
    Try {
        (Get-Content -Raw -Path ($chrome_rules_file) | ConvertFrom-Json) | Select -ExpandProperty ExtensionId
        Write-Host $(Get-Date -Format u) "[info] Retrieved Settings from the JSON input file"
    } Catch {
        Write-Host $(Get-Date -Format u) "[warning] Failed to read JSON input file"
    }
}

function Get-CurrentGPOExtensions {
    Try {
        Get-GPRegistryValue -Id $GPO_Id -Key $RegistryPath | Select -ExpandProperty Value
        Write-Host $(Get-Date -Format u) "[info] Retrieved group policy settings"
    } Catch {
        Write-Host $(Get-Date -Format u) "[warning] Failed to read group policy settings"        
    }
}

#DOWNLOAD THE EXTENSIONS FILE FROM ARTIFACTORY
Invoke-WebRequest -Uri $extensions_uri -OutFile $chrome_rules_file
$f = get-item $chrome_rules_file
if ($f.Length -lt 500){
    Write-Host $(Get-Date -Format u) "[warning] Something is wrong with the JSON download. Exiting."
    Exit 1
}

# CHECK IF ANY CHANGES HAVE BEEN MADE,
# EXIT HERE IF NO CHANGES DETECTED:
$current_gpo_state = @(Get-CurrentGPOExtensions)
$current_JSON = @(Get-CurrentJSON)
$differences = Compare-Object $current_gpo_state $current_JSON
if ($null -eq $differences){
    Write-Host $(Get-Date -Format u) "[info] No changes in the JSON. Exiting."
    Exit 0
}

#CONTINUE, JSON INPUT IS DIFFERENT THAN GPO

#LAST SAFETY CHECK: Is the current JSON input valid?
if ($current_JSON.Length -lt 20){
    Write-Host $(Get-Date -Format u) "[warning] Failed safety check. JSON input invalid. Aborted before making changes."
    Exit 1
}

#DROP EXISTING SETTINGS:
Remove-GPRegistryValue -Id $GPO_Id -Key $RegistryPath

#APPLY THE NEW SETTINGS AS A SINGLE GPO UPDATE:
Try {
    Set-GPRegistryValue -Id $GPO_Id -Key $RegistryPath -ValuePrefix 0 -Type String -Value $current_JSON
    Write-Host $(Get-Date -Format u) "[info] Applied registry changes to GPO $GPO_Id"
} Catch {
    Write-Host $(Get-Date -Format u) "[critical] Failed to apply registry changes to GPO $GPO_Id"
    Write-Host $(Get-Date -Format u) "[critical] Old settings were deleted and new settings have not been applied"
}

第三步：不要忘记 macOS
虽然我们很高兴为 Windows 工作站打造了一条高功能流水线，但我们也意识到大多数员工使用的是 macOS，因此无法从这项自动化中受益。最终，我们利用了 Windows 流水线的部分环节来简化 macOS 配置文件的管理，但并未完全实现自动化。

Palantir 此前开发了一项名为 Excavator 的服务，该服务可针对内部 GitHub 仓库执行任意脚本，并能根据结果提交 PR。利用 Excavator，我们编写了一个脚本，用于解析来自 Artifactory 的 json 文件，并自动将任何更改转换为预格式化的 plist 文件，以便我们的 Jamf Pro 管理员轻松应用。该流程最终仍是手动的，但已得到精简，因为他们可以一次性上传整个更新后的 plist 文件，而不必在数组中逐个排查更改。

Excavator 运行的用于生成 Edge plist 的脚本如下所示：

/bin/env python3
''' Generate a plist file in the endpoint configuration profile repository. '''

import datetime
import logging
import optparse
import sys
import os
import requests
import jinja2
import plistlib

BASEPATH = 'https://files.palantir.internal/files/extensions/latest'
EDGEFILE = 'edge_extensions.json'
directory = os.path.dirname(os.path.realpath('__file__'))
file = '/Microsoft Edge/com.microsoft.Edge.plist'
filedir = directory + file

def render_template(path, extensionids):
    """ Render the template from the list of extension ids """
    templateloader = jinja2.FileSystemLoader(searchpath=os.path.dirname(path))
    templateenv = jinja2.Environment(loader=templateloader, trim_blocks=True, keep_trailing_newline=True)
    templatefile = os.path.basename(path)
    template = templateenv.get_template(templatefile)
    outputtext = template.render(extensionids=extensionids)
    logging.info("file directory detected as %s", filedir)
    logging.info("writing file")
    with open(filedir, 'w') as filewriter:
        filewriter.write(outputtext)

def test_plist():
    try:
        with open(filedir, "rb") as plist:
            plistlib.load(plist)
    except:
        logging.critical("Plist is not valid, aborting: %s")
        sys.exit(1)

def get_published_file():
    """ Pull down file from Files"""
    logging.info("pulling down Edge extension file from Files...")
    logging.debug(BASEPATH + EDGEFILE)
    ## try or die
    try:
        results = requests.get(BASEPATH + EDGEFILE)
    except requests.exceptions.RequestException as message:
        logging.critical("Was unable to get policy: %s", message)
        sys.exit(1)
    return results.json()

def parse_args():
    """Parse the arguments"""
    parser = optparse.OptionParser()
    parser.add_option("-v", "--verbose", help="increase logging output",
                      action="store_true")
    # pylint: enable=line-too-long
    parser.description = """
    Generate new Edge plist file from published policy
    """
    (options, args) = parser.parse_args()
    if options.verbose:
        level = logging.DEBUG
    else:
        level = logging.INFO
    # pylint: disable=line-too-long
    logging.basicConfig(level=level,
                        format='%(asctime)s %(levelname)s %(name)s: %(message)s',
                        datefmt='%m/%d/%Y %H:%M:%S')
    # pylint: enable=line-too-long
    return (options, args[0])

def get_ordered_ids(blob):
    """ Take json blob and return ordered list """
    ids = []
    for extension in blob:
        ids.append(extension['ExtensionId'])
    return ids

def main():
    """ Main function """
    start = datetime.datetime.utcnow()
    # options and args
    (options, path) = parse_args()
    logging.info("Starting Jamf Pro policy updater: %s", start)
    edge_blob = get_published_file()
    ids = get_ordered_ids(edge_blob)
    render_template(path, ids)
    test_plist()

if __name__ == '__main__':
    main()

该脚本依赖一个模板文件，以便将扩展 ID 正确插入现有的配置文件中。这种方法的一个局限性在于，Excavator 假定只有扩展 ID 是动态的，而所有其他组件相对静态。从我们的经验来看确实如此，但也有一些边界情况需要考虑。下面是一个模板 plist 文件示例：

Name: edge_plist.tpl
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>ExtensionInstallAllowlist</key>
    <array>
{% for extensionid in extensionids %}
        <string>{{ extensionid }}</string>
{% endfor %}
    </array>
    <key>ExtensionInstallBlocklist</key>
    <array>
        <string>*</string>
    </array>
    <key>SyncDisabled</key>
    <true/>
</dict>
</plist>

Jamf Pro 提供的 API 允许直接交互和操纵现有的配置文件 plist 配置，从理论上讲，这可以实现 macOS 安全 策略的自动化，以匹配 Windows 的自动化程度，但迄今为止，我们还无法以一种足够 安全 的方式执行该功能以将其部署到生产环境中。
