# 数据共享中的技术挑战 177 COVID-19 危机催生了前所未有的信息共享需求与渴望。决策者如今所需的信息背景，早已超出了其自身组织、数据库、标准以及常规沟通渠道所能供给的范围。例如，英国各行政区域之间原本职权互不干涉，如今却开始[相互交换数据](https://healthtech.blog.gov.uk/2020/03/28/the-power-of-data-in-a-pandemic/)，共享各自医院网络内呼吸机等关键医疗设备的储备与使用情况。我们观察到了一个关键的新兴挑战：如何在提供这种跨领域的广泛信息访问权限的同时，切实保护隐私并尊重数据权属。 内容共享是一个微妙的问题：内容所有者往往将访问权限视为非此即彼的“全有或全无”开关。在过去的 15 年里，我们深耕于此道，提供的软件平台能够让组织在“完全无权访问”到“完全有权访问”的渐进区间内共享内容。尽管这种平滑的权限过渡看似复杂，但我们的工程实践始终起步于一套基础原则，力求尽可能逼近理想的解决方案。下文将探讨我们认为优秀解决方案必备的核心原则，简要阐述我们在 Foundry 和 Gotham 平台上构建的解决方案，并介绍目前如何利用这些平台在[我们 COVID-19 相关工作](https://www.palantir.com/covid19/)的背景下实现安全的数据共享工作流。 ![图片](https://miro.medium.com/v2/resize:fit:700/1*r1Ta8tc5mt-WJ_WyqhUcYg.png) ## 原则 我们围绕三个主要维度来考量访问与共享，这些维度既限制又赋予了特定的访问权限：*什么（内容）*、*为什么（目的）*以及*谁（用户）*。 **什么。** 首先，我们需要明确内容的具体指向。这些内容可能是*数据*、*逻辑*，甚至是*决策*等高阶信息。数据的形式与规模千差万别；在抽象层面上，我们可以将其视为*文件*、*表格*乃至*数据库*。在某些系统中，我们可能会考虑*对象*（与特定世界观相匹配的语义模型）以及*对象关系*。但“什么”涵盖的范畴远不止数据：我们还可以考虑共享元数据、逻辑、模型、数据管道，或诸如*决策*等高阶信息产物。 此外，还必须考量我们是否意指上述内容的全部。举例来说，如果政府希望了解商业航空公司的货运能力，通报运力与航线是一回事；而通报机组人员排班则完全是另一回事。这种共享方式需要*细粒度*的视图。 在探讨“什么”时，我们需要思考信息的使用者将如何信任其真实性。在高质量的数据系统中，真实性通常是一定数量的断言不变量（或健康检查）*以及*高度的血缘关系综合决定的结果：即数据、逻辑和决策是如何产生的。在细粒度共享的语境下，我们需要使数据主人能够决定有多少血缘关系可以或应该被共享。 **为什么**。其次，在共享信息时，我们需要明确其目的。从业者常常深切感受到《数据使用协议》带来的复杂且有时极为繁重的合规要求。我们认为，一个优秀的工程解决方案不应仅仅停留在展示和跟踪协议层面。事实上，它应该赋能并执行该协议，使得用户能够专注于分析与业务产出，由系统来确保数据仅被用于数据主人所规定的预期目的，即便跨越系统边界也应如此。 为了对“为什么”施加限制并保持用户对生产性工作的专注，优秀的系统会帮助用户在会话开始时以及工作推进过程中声明其目的。也就是说，工作的目的既受限于数据主人的限制，*也*取决于用户自身的意图。必须尽早直面这种复杂性，因为不同的目的可能相互排斥，因此对某些内容的访问可能会影响对其他内容的访问。例如，我们可能希望限制在同一会话中同时获取机组人员行程*和*飞机运力的权限，即便这两项内容均已共享给同一用户且该用户本身具有阅读权限。 **谁。** 第三，共享是针对特定用户（和/或用户组）的。共享并非一项不透明的访问凭证认证练习。相反，它涉及授权特定用户，正如上文所述，在特定时期内为特定目的共享特定内容。我们还应考虑对用户进行更抽象的定义。在某些情况下，我们可能只知道具有特定主管的员工应获得内容的访问权限；为了保持访问控制列表的更新，一个优秀的系统应当支持将*规则*编码，而非仅仅依赖静态列表。 ## 能力 接下来，我们将探讨一系列软件能力，这些能力有助于依照上述原则实施数据共享工作流。 **资源与用户模型。** 作为起点，我们需要为信息资源以及与这些资源交互的用户建立显式模型。以尽可能宽泛的视角来看待信息资源是有益的，例如包括数据库、数据集、表格、索引、时间序列、分析、可视化、AI/ML 模型、数据转换、数据管道与调度、健康检查、Git 仓库、报告、仪表板、数据驱动应用等。理想情况下，用户的建模也应结合其组织背景，如用户组。 我们还应慎重考虑信息资源模型的粒度。下文将看到，资源是锚定访问控制规则的天然位置，因此资源模型的粒度直接关系到权限系统的粒度，以及具有权限感知的数据共享工作流的粒度。例如，如果仅在数据库或表格级别追踪资源，那么很难设计出针对单行或单列的共享工作流。 **血缘关系。** 信息资源并非孤立存在；它们通过数据导入导出、数据管道、机器学习训练或信息共享等过程相互连接。资源及其关系的图谱被称为资源*血缘关系图*。数据血缘背后的概念十分直观：对于每一个资源，具备血缘感知能力的数据系统都能回答它是如何派生的、来自何处（例如，是从哪些其他信息派生而来的）、由谁生产以及何时生产。 就上述三个基本问题而言，全面的数据血缘关系使我们能够回答*什么*。它迎合了人类的直觉：为了信任信息与数据，我们希望了解其来源、是否可靠、是否为最新、由谁生产以及如何演进。围绕这些问题的透明度建立了信任，也是跨越数据孤岛进行协作的基础。 血缘关系不仅限于回溯过去，还可以延伸至未来。例如，数据管道系统可以记录所有已调度的数据转换作业及其对下游数据集、索引或应用中派生数据资产的传递影响。在数据共享的语境下，未来的血缘关系尤为重要，因为它允许数据主人了解哪个数据版本将在何时共享给哪个接收方。 **访问控制。** 访问控制机制的任务是对一个简单问题给出是/否的答案：*特定用户能否为特定目的访问特定的信息资源*？要回答此问题，我们首先可以将访问控制规则关联至资源，然后在暴露信息资源的每个 API 上设置授权屏障来评估相应规则。两个有趣的技术考量点是授权系统的覆盖范围和规则语言的表达能力。就前者而言，显然访问控制系统在一个集成系统中覆盖的 API 和服务越多，就越有用。 关于访问控制系统的表达能力，访问控制列表或基于规则的授权是一个不错的起点。更具表现力的访问控制模型是[基于属性的访问控制](https://en.wikipedia.org/wiki/Attribute-based_access_control)（ABAC）。ABAC 规则兼顾了其应用数据的上下文（例如，规则可能会根据销售目标的值来区分对*银行*对象的访问，如*消费者、小型企业或大型商业企业*）以及请求访问的用户的上下文（例如，我所属的 LDAP 组或我签署的保密协议集合）。 获取 Palantir 的故事到您的收件箱 免费加入 Medium 以获取该作者的更新。 此外，我们可以区分[自主](https://en.wikipedia.org/wiki/Discretionary_access_control)和[强制](https://en.wikipedia.org/wiki/Mandatory_access_control)控制。简而言之，*自主控制授予访问权限*，例如当用户与同事共享可视化链接时；而*强制控制限制访问权限*，例如通过组织范围内的数据治理规则。许多 ABAC 系统能够表达特定上下文规则的布尔组合。复杂的自主 ABAC 规则可以实现对数据使用*目的*的约束（参见上文的*为什么*原则）。举个具体的例子，我们可以防止具有特定属性的信息资源被混用，在概念上表示为*属性 A 异或 属性 B*。 **具备血缘感知的访问控制。** 在数据系统中，将强制控制附加到输入数据集通常是最便捷的做法，以镜像源数据库中存在的权限结构。强制控制还可以反映组织、地理或法规上的约束。例如，源自特定国家的数据可能只能从该国内的工作站或由该国公民访问。 数据血缘关系对于在整个信息生态系统中执行强制控制尤为有用：给定源系统上的强制控制，数据血缘可用于通过数据转换将这些控制传播到所有派生的数据资产和应用程序。继续上面的例子，假设特定国家的数据被用于训练机器学习模型；数据血缘和强制控制将共同确保该机器学习模型只能在国内和/或由相应公民访问，*因为*它是基于受这些控制约束（并因此包含）的源数据训练的，或者受制于不同的区域监管限制。 请注意，强制控制优先于自主访问：如果上例中该国的公民与非公民共享数据集，从而授予其自主访问权，该非公民仍然无法访问该数据集，因为其不满足组织范围内的强制访问策略。换言之，强制访问控制与系统级血缘关系的结合，使数据主人能够控制其数据资产在下游的使用。 **资源作用域会话。** 我们的访问控制系统工程实践得益于我们在政府领域早期的经验，在那里基于属性的访问控制是常态，并且已存在对*什么*、*为什么*和*谁*的天然建模。同样地，我们继承了这样一种理念：用户应在会话开始时声明*意图*，并在会话过程中随着分析迭代循环的演进或变化而更新意图。此外，正是从这些环境中，我们首次认识到强制分离意图的必要性，以免原本强大的数据系统被滥用。 **审计。** ABAC、自主与强制访问控制以及数据血缘都非常重要，但我们如何证明它们的实现对具体的数据治理决策产生了预期效果呢？所有数据和 API 访问都应形成审计追踪，记录*谁*访问了*什么*数据以及*为什么*授予该访问权限。换言之，审计追踪精确地回答了支配数据访问的公理性原则。 审计追踪的两个示例用例如下：首先，审计追踪可用于自动验证数据治理不变量，从而捕获无意或恶意的权限提升。其次，审计人员可以在事件后调查中分析审计日志。在这两个示例中，都可以从以用户为中心（“该用户做了什么？”）或以数据为中心（“谁访问了这条信息？”）的视角来解读审计追踪。 我们坚信，透明度是建立数据治理信任的关键所在。为此，我们与客户共享访问控制框架的语义以及实际的审计追踪，并明确帮助他们进行系统与数据使用建模。 ## 工作流 为应对 COVID-19，我们在政府和工业界的客户正以非传统的方式重新分配资源（例如，使用客机运送货物，改造工厂以生产口罩或急需药品），而汽车或航空航天等整个行业都在规划其业务的产能缩减（例如，在停工期间我应如何保持设备处于安全运行状态？）与产能恢复（例如，考虑到物流限制，我能交付哪些产品？考虑到可用资源，我能生产哪些产品？）。这种重心的转变意味着对跨领域信息共享有着前所未有的需求，以便理解这个（新）世界。为了将理论与实践相结合，我们将探讨几个具体的数据共享工作流，并说明如何应用前几节的原则与能力。 **脱敏与发布。** 让我们从相邻图表所示的简单脱敏工作流开始。源数据集 A 和 B 通过某种数据转换（比如数据集连接或合并）组合成派生数据集 C；蓝色和绿色小方框表示通过血缘关系图从 A/B 传播到 C 的强制控制：假设采用合取控制，访问 C 需要同时具有对 A 和 B 的访问权。 ![图片](https://miro.medium.com/v2/resize:fit:700/1*tFNwWg2zk3NLC6TwHd-5_g.png) 为了将数据集 C 的子集共享给没有源数据集 A 访问权限的用户，数据主人可以进一步派生出数据集 D，在其中对敏感信息进行脱敏，例如通过过滤掉特定的行、属性或列。然后，数据主人可以下调对 D 的强制控制，并仅基于数据源 B（绿色框）的派生强制控制将其发布。 请注意，脱敏与发布工作流属于高度特权操作，通常只能由满足所有强制控制并具有特殊数据发布权限的数据主人执行。当然，此类脱敏工作流也会被审计系统记录。脱敏始终需要明确的人工监督和签字确认，因为目前没有通用的算法途径来确定某些派生信息是否包含来自输入数据集的某些特权信息。例如，可以通过代码审查和批准流程来建立针对脱敏与发布工作流的人工监督。 **透明的数据流。** 数据共享工作流的起点当然是要共享的数据集。可共享的数据集通常不会凭空出现，它们本身也是从其他源数据集派生而来的，例如通过连接、合并或丰富来自不同数据库或组织的数据。通常，这种数据派生并非静态的，而是会随着源数据的更新或派生逻辑的改进而发生变化。例如，在我们协助应对 COVID-19 的政府项目中，我们正在整合每天多次更新的公开感染人数开源数据。 共享数据集的动态特性带来了一个有趣的挑战：如何让数据消费者获知数据集的变更？简单的情况是增量变更，如表格数据集中增加了若干行；但更复杂的变更则是模式改变，如增加或删除列、对先前共享的数据进行更新，或对属性的语义进行细微更改。当下游消费者开始构建应用程序并因此依赖共享数据的结构时，静默传播此类可能破坏性变更的做法便不可接受了。 例如，[Foundry](https://www.palantir.com/palantir-foundry/) 支持多种安全传播数据变更的机制。首先，血缘关系图捕获了每个共享数据集的现状：即通过何种逻辑从哪些输入数据集派生而来。Foundry 的[管道编排系统](https://medium.com/palantir/rethinking-the-foundry-job-orchestration-back-end-from-crud-to-event-sourcing-658716afdd7c)会按用户定义的调度计划并在满足完整性约束的前提下传播变更。（例如，对输入数据集的[菱形依赖](https://en.wikipedia.org/wiki/Dependency_hell)会得到一致性解析。）由于所有数据集都有版本控制，下游消费者可以选择在新的数据版本可用时接收，也可以坚持使用固定版本并手动升级。数据主人可以指定完整性和一致性检查，由编排系统在将变更传播给消费者之前进行验证。最后，[Foundry 将分支和代码审查等软件工程工作流延伸至数据工程领域](https://www.palantir.com/)，实现了数据主人与消费者之间透明的协作。 ## 结论 通过与政府及私营部门合作伙伴共同开展 [COVID-19
