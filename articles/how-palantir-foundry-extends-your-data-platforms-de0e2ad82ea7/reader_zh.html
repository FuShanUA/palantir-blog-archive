How Palantir Foundry Extends Your Data Platforms

2022-04-10 · Original →

数据平台产出洞察，Foundry 驱动决策

作者：Akshay Krishnaswamy，Palantir 首席架构师

编者按：本文是系列文章的首篇，探讨 Palantir Foundry 如何与企业软件架构中的现有方案形成互补。

人们有时会误将 Palantir Foundry 视为一款单纯的数据平台方案。这种误解在某种程度上情有可原。对于处于数字化转型初期的企业，Foundry 确实能够担此重任，提供数据集成、管理、数据治理以及搜索与分析等核心功能。

然而，Foundry 的重心不止于此，其终极目标是赋能构成企业核心的运营决策。尽管数据平台在集中化存储与单点管理数据方面发挥着重要作用，但 Foundry 的设计初衷在于将数据分析与实际业务运营紧密相连——在复杂多变的环境中实现持续、动态的协同。因此，最好将其视为一套操作系统，用于统筹协调企业内部数据、模型与决策之间的交互运作。

这种定位需要一套截然不同的架构。下文将探讨 Foundry 的各项能力如何通过构建运营级连通性，极大拓宽数据平台传统的价值边界。

数据平台产出洞察

诸如 Snowflake 平台、Google BigQuery、AWS Redshift、IBM Db2、Azure Synapse、Teradata 等数据平台，非常适合汇聚来自多源的海量数据，为其赋予结构并提取洞察。

然而，面对瞬息万变且宏观冲击频发的环境，众多企业意识到，仅停留在输出洞察已远远不够：他们希望为决策者提供付诸行动的技术抓手。落实到具体业务中，这意味着要打通跨职能的决策链路，使其能够被模拟、被底层系统执行，并最终回流至统一的数据底座，从而实现基于反馈的持续优化。

鉴于当今运营环境高度复杂，软件系统需要以动态（即敏捷响应外部条件变化）且可扩展（即在团队、区域乃至业务线层面实现全局优化）的方式协调各项决策。这种能力要求系统具备传统数据平台之外的诸多特性，例如模型集成、动态语义层、敏捷应用构建，以及决策模拟与回写。

Palantir Foundry 驱动决策

Foundry 专为深度的运营级连通性而设计，旨在帮助企业打通数据、分析与运营决策之间的闭环。

Palantir Foundry 能够通过以下四个核心维度，将企业的数据平台延伸至运营领域：

模型与人工智能的集成及创建
Foundry 中的模型开发与集成是一套核心级能力，贯穿整个 MLOps 生命周期。它将机器学习、人工智能、统计学及数学模型与 Foundry 生态中的其他组件融为一体，支持将多个模型串联并部署于多样化的业务流中。该层的核心能力涵盖模型的开发、发现、管理与部署。

  • 自动化、软件定义的集成机制能够快速对接 Foundry 外部的数据与模型。既可以通过原生平台集成（如 AWS Sagemaker、Azure ML、DataRobot、Databricks 平台）进行连接，也可以将模型制品以代码、依赖库或已训练模型的形式直接导入 Foundry。对 PyTorch、TensorFlow、SKLearn 等主流开源框架以及 ONNX 等互操作标准的支持，大幅简化了集成过程。
  • Foundry 代码工作簿为模型构建者提供了原生且安全的数据访问机制，使其在端到端的模型开发过程中，能够运用数据集与本体范式进行操作。
  • Foundry 模型管理提供对已部署模型和候选模型的持续评估与监控，并支持多角色参与的审批与发布流程，确保改进成果顺利推向生产环境。Foundry 为集成的数据与模型提供完整的版本控制、分支管理、可复现性、安全性以及血缘关系保障。
  • 模型目标相当于 Foundry 工作流中模型的“任务控制中心”，确保业务逻辑与特定的业务关键绩效指标（KPI）紧密绑定，并在不同用例中保持部署目标的一致性。

动态本体与语义层
Foundry 将所有相关数据、逻辑和模型整合为企业的数字化映射，即本体。该本体超越了单纯的业务对象及其关联关系，进一步封装了高级数据语义（空间、关系、时序）、“动力学”（反映复杂的写入操作链与系统集成），以及细粒度权限控制——所有这一切均可通过用户界面与安全的 API 加以调用。本体能够快速初始化，并通常随着新工作流的接入而不断演进，持续吸纳新的数据源、对象、关系、交互及流程。

  • 业务的数字化映射不仅反映了真实世界的关键语义（对象与关系），也体现了其关键的动力学特征（函数、动作与模型）。
  • 决策与数据捕获机制确保工作流在扩展时不会破坏核心数据底座，使其始终作为企业可信赖的事实来源。
  • 原生的“场景感知”功能支持假设分析与复合模拟运行。

模块化工作流与应用
Foundry 内置一套丰富的构建模块，支持快速组装工作流及具备读写能力的运营级应用（而非简单的只读仪表盘）。这些模块涵盖了前述的语义与动力学基础组件，使团队能在数小时内为前线用户配置出高质量的交互式工作流。

  • 无代码/低代码应用构建器依托本体，为用户提供了一个高度灵活的框架来与企业数据资产进行交互，同时在底层自动管理存储、计算、本体数据、模型绑定及安全范式。
  • 所见即所得（WYSIWYG）的组件驱动构建器支持本体数据以及表格化/SQL 格式数据的交互。
  • “低门槛、高上限”的灵活性使用户既能构建服务于小团队的轻量应用，也能打造位于运营中心核心的关键任务系统。
  • 开放 API 确保平台能够适应不断演进的技术生态。

决策编排
该层在企业的数据分析与运营工作流之间搭建了技术桥梁。随着运营人员、业务流程及系统做出决策并采取行动，该层会将结果回写至本体中，为高速运转的组织学习提供反馈闭环。

  • 可扩展的双向连接器允许 Foundry 记录所有决策并将其回写至企业的本体以及运营/事务系统（如 ERP、CRM、MES、资产配置、边缘端）。Foundry 完整记录决策与状态日志，确保审计可行性与全面透明度。
  • 模拟功能为用户提供了强大的假设分析与决策探索能力，使其能够模拟潜在行动并评估其对业务的影响。模拟既可是战术性短期的，也可是长期持续的，并随数据和模型同步刷新。
  • 全链路血缘关系贯穿从数据到决策的每个环节，使用户能够追溯某项数据或元数据在外部被写入时“世界的状态如何”。决策始终以非破坏性方式进行，因为 Foundry 通过系统集成引入底层数据，并对所有变更进行版本控制。

这些能力赋予了系统一种独特的运营级连通性，驱动各行各业的业务走向成功——从世界粮食计划署提供拯救生命的援助，到大型饮料企业的定价优化，再到全球制造商的库存优化，以及更多其他案例。
