Palantir 下一代审计日志系统

编者按：本文是 Palantir Foundations 系列文章的第二篇，探讨我们如何定制基础设施软件以实现大规模的可靠运行。

每天，各类组织都将最敏感的数据和关键运营工作托付给 Palantir 平台。从协调国家安全任务的政府机构，到保护患者信息的医疗机构，再到检测欺诈的金融机构，客户依赖我们帮助他们做出关乎重大的决策。这份信任得之非易，守之更难。它建立在透明、问责的基础之上，以及我们自创立之初便坚守的一条原则：系统中的每一个用户操作都必须可审计。

在 Palantir，隐私与安全绝非事后补救的措施。我们的隐私与公民自由（PCL）工程团队是全球首创，长期以来一直倡导一个理念：隐私保护技术必须是核心能力，而不仅仅是锦上添花。落实这一理念最关键的抓手之一就是全面的审计日志。审计日志在我们的平台上创建了关于“谁在何时何地做了什么”的不可篡改记录。安全团队借此调查安全事件，合规人员据此证明对法规的遵从，组织也能维持复杂协作环境所必需的问责制。

监督监督者：将可审计性作为核心价值

在探讨权力与监督时，人们常引用一句拉丁语：Quis custodiet ipsos custodes? ——即“谁来监督监督者？”在 Palantir，这不是一句反问，而是一项设计要求，确保我们的客户能够独立监督自身的组织运作。

Palantir Foundry、AIP 以及我们的领域专属产品，赋予用户处理敏感数据、运行复杂分析并做出重大决策的强大能力。能力越大，责任越大；而责任所在，问责并行。全面的审计日志正是我们将这种问责制落到实处的手段。

但可审计性不仅仅是为了监督或勾选合规清单，它从根本上关乎信任。当政府机构与情报分析师共享机密情报数据时，当医院授权临床医生访问病历时，当金融机构允许调查员查询交易数据库时——他们都在授予权限与信任。只有留存清晰、可靠的记录来追踪这些信任的使用情况，这种信任才具有可持续性。

这一理念深刻影响了我们构建软件的思维方式。隐私保护与运营效能并非相互掣肘的目标，而是相辅相成。安全团队借审计日志检测内部威胁，隐私官同样借此确保数据最小化原则得到贯彻。系统追踪特定用户访问的资源，同时也在跨组织的协作工作中维持清晰的权责归属与监督。

我们的 PCL 团队以及横跨产品开发与前线部署工程师（FDE）的团队，将这一原则贯穿于整个平台。平台上的每一个操作——无论是“读”操作、“写”操作、分析探索还是权限变更——都会生成审计日志。在开发新功能时，可审计性从一开始就是一项核心要求。

今天，我们宣布这一承诺迈出新的一步：下一代审计日志系统（称为“audit.3”）正式全面可用。历经多年研发，audit.3 能够更快速地交付日志，并带来更完善的结构与更强的可访问性。它标志着我们在大规模环境下捕获、交付和构建审计数据方式的根本性重构。

大规模审计日志的挑战

要理解我们为何踏上这场长达数年的演进之旅，有必要先了解我们要解决的问题。Palantir 平台每天在我们可观测的集群中产生数百 TB 的审计日志。这些日志绝不仅仅是技术痕迹。在各组织可选择在共享基础设施上开展协作的部署环境中，这些按照环境严格保障安全的日志，包含了用户操作的详尽记录——涵盖数据访问请求、导出、本体操作、分析查询、权限变更等。

旧版审计日志系统多年来运行良好，但随着平台演进和客户需求日益复杂，系统在发展过程中暴露出了根本性的局限。Palantir 联邦式的产品开发模式虽然带来了快速创新与高效迭代，却在规模化扩展时引发了两大挑战。

首先，我们旧版的 audit.2 日志交付管道依赖批处理。各产品会根据自身的活动规律，以不同间隔写入审计日志。日志由各产品写入后，必须流经复杂的多阶段管道：首先收集并批量写入存储桶，接着处理成多个中间数据集，逐步对审计数据进行格式化与富化，最终交付给分析师的数据集。每个阶段都需要大量的存储和计算资源，增加了架构复杂度与延迟。正如我们的许多客户从周期性批处理集成转向具有实时业务数据流的流式数据架构一样，我们意识到这种方式已无法满足现代安全运营的响应速度。

其次，audit.2 日志模式本身是随着时间推移自然演进的，各产品团队定义了自己的事件结构。这种联邦式做法提升了开发速度，却让分析师难以跨平台编写一致的查询语句。举例来说，要追踪环境中所有的数据导出操作，你必须频繁更新查询参数，以支持不断演变的字段结构，从而应对我们开发新导出功能时引入的新模式。系统虽能运行，但要达到最佳使用效果，往往需要精通平台深层知识的资深领域专家来导航。

这两个挑战均源于早期曾行之有效的架构决策，但随着平台规模扩大和客户需求演变，这些决策已触及极限。要向前迈进，我们必须从第一性原理出发，重新审视交付机制和数据结构。我们面临一个根本性的问题：如何重新设计一个已经在处理海量规模的关键系统？

新基石：Audit.3 与遥测收集流式管道

要解决上述问题，我们必须从第一性原理出发，重新审视整个审计日志技术栈。我们扪心自问：如果今天我们带着对规模、性能以及安全与合规团队实际工作流的既有认知来设计审计日志，它应该是什么样子？

答案就是我们现在推出的 audit.3，它包含全新的审计日志模式以及经过彻底重构的流式管道架构。变革始于审计日志创建的那一刻。通过对审计日志库的增强，我们现在能够在写入日志时直接捕获日志元数据，而无需事后通过繁杂的检索来确定。这个看似微小的改动，直接消除了整个处理阶段，而该阶段此前正是延迟和架构复杂度的主要来源。

随后，日志流经遥测收集流式管道。该管道作为分布式智能体系统部署，与我们基础设施中的每个容器化服务同步运行。这些智能体将日志高效路由至其他遥测收集智能体，由后者进行聚合与标准化处理（即处理并结构化数据）。紧接着，日志被直接写入对象存储，并采用了改进的存储桶前缀。这些改进后的前缀带来的最终结果是，我们不再需要存储审计历史数据集的中间副本——而这正是旧版 audit.2 管道中的旧有弊病。新流程更精简，也更快速。

通过将日志可用性“左移”——在收集后即刻开放审计数据访问，而非等待漫长的处理周期——我们将审计日志从用于回溯分析的资源，转变为近实时的运营工具。在许多环境中，日志最快在几分钟内即可调用（具体视使用模式而定）。这不仅是性能的跃升，更是分析能力的质变。Palantir 自身的信息安全团队依赖这些日志开展主动调查。当客户的安全分析师需要了解潜在安全事件期间究竟发生了什么时，他们可以直接分析最新鲜的数据，而无需苦苦等待。

面向未来的结构：Audit.3 模式

单凭速度并不足够，我们还必须解决模式一致性的问题。audit.3 模式标志着从基于事件的日志记录向基于类别的日志记录的哲学转变。不再由各个 Palantir 产品自行定义事件结构，audit.3 的核心标准是：每一个可审计操作都映射至一个明确定义的类别。这些类别代表了安全与合规的核心基础操作：数据导出、身份验证事件、权限变更、分析查询等。这就是审计领域的本体时刻！

这种标准化带来了深远影响。例如，当 Palantir 产品开发者要在 Foundry 中引入一种全新的数据导出方式时，他们会选择“dataExport”审计日志类别。一旦选定该类别，编译器就会通过报错引导开发者，确保将所有相关数据片段包含在正确的预定义字段中。合规人员此前若已编写了 categories contains "dataExport" 的规则来监控日志，则完全无需更新逻辑。他们的分析会自然延伸并覆盖新功能——即便在我们的平台每天向客户环境推送数百项升级的情况下也是如此。这一原则适用于所有 audit.3 日志类别。这种方式从根本上实现了审计分析的前瞻兼容，旧系统对此无能为力。

对于在众多相互关联的 Palantir 产品线开展研发的工程师而言，这种标准化也大大降低了实现一流可审计性的难度。我们构建了开发者工具库来辅助研发人员落地 audit.3 模式，该工具在构建阶段即强制落实类别要求，确保新功能在设计之初即可审计，而非事后补加。这是一种强制约束机制，与我们创立之初的理念一脉相承：隐私与安全必须自始融入。

  • origins：网络请求来源，有助于区分用户发起与服务发起的请求。

  • users：审计日志中出现的所有用户 ID 及部分附加上下文字段（即“操作主体”）。

场景对比：改造前后

为了理解这些改进的实际影响，不妨设想一个典型的安全事件调查场景。

过去：audit.2 时代

安全监控系统标记出可疑的数据访问模式。某用户账户似乎在短时间内访问了异常大量的敏感数据集——这可能是凭证泄露或内部威胁。

安全分析师展开调查，但立刻遭遇了 audit.2 时代典型的复杂困境。要全面了解该用户访问了哪些数据，他们必须弄清用户与哪些 Palantir 产品发生了交互。在合规但繁杂的 audit.2 模式中，各产品定义了各自的审计日志事件命名约定和字段结构。

分析师开始构建 audit.2 日志查询，通过对 name 列进行过滤：对某服务使用 name = "READ_DATASET"，但这会漏掉另一服务的 LOAD_OBJECT 事件、第三方的 FETCH_DATA 等等。他们需要查阅文档，寻找平台上所有可能代表数据访问的事件名称。有些服务将资源标识符放在 request_params.datasetRid 中，有些使用 result_params.objectId，还有一些则将其嵌入需要专门解析的自定义字段中。

更令情况雪上加霜的是，合规团队正试图查明是否真有数据从平台导出。他们需要跨不同产品核查多种不同的事件类型：某服务的 EXPORT_DATASET、另一服务的 DOWNLOAD_FILE、第三个服务的 EXTRACT_REPORT、第四个服务的 GENERATE_ANALYSIS_OUTPUT，以此类推。导出事件的审计日志字段结构也各不相同——从单一产品逻辑出发最合理的日志记录方式，却让整体分析工作变得举步维艰。因此，要编写一个能捕获所有导出方式的综合查询语句，需要对系统的审计日志框架有深入的了解，并且在新功能及其特有事件名称不断推出时保持高度警觉。

现在：audit.3 时代

同样的场景：安全监控系统标记出可疑的数据访问模式。但如今，分析师的调查方式已截然不同。

他们只需编写一条简单直观的 categories contains "dataLoad" 查询语句。这就能返回所有 Palantir 产品中的所有数据访问事件，无论使用的是哪个具体服务或功能。得益于标准化的 audit.3 模式，分析师无需了解内部架构；相反，类别本身直接告诉他们发生了什么。

在调查过程中，你的团队能够更快速地：

  • 确认用户身份，验证其凭证是否已被泄露。
  • 使用 dataLoad 类别的简单过滤器，审查其访问过的每一个数据集。
  • 运行额外的 categories contains "dataExport" 查询，借助不同的审计日志类别核实：尽管该用户加载了大量数据集，但并未发生实际导出。一次查询，全面覆盖。
  • 检查每条日志顶层的 users 和 entities 字段，精准掌握谁对哪些资源做了什么。
  • 联系用户了解业务背景。
  • 记录所采取的步骤，并在补救完成后有效结案。

借助 audit.3，调查过程变得更简单、更快速、更可靠。分析师不再需要掌握 Palantir 内部架构的专门知识。结构化的类别承担了原先繁琐的整理各类相关审计事件名称的工作。这让分析师能够更快速、更准确地确定在调查过程中究竟发生了什么。

你的合规团队同样受益于此次迁移。他们持续运行的导出监控查询现在变得十分简单：只需过滤 categories contains "dataExport"，无论涉及哪个产品或功能。更重要的是，当 Palantir 研发团队发布新的数据导出功能时，他们会将新功能的审计日志归入同一个 dataExport 类别下。查询语句无需更新；在设计上已具备前瞻兼容性。

从产品专属的事件名称向通用类别的转变，不仅简化了查询，更从根本上改变了安全与合规团队处理审计数据的方式。无需再逐一理解每个 Palantir 服务的实现细节，他们可以按类别对操作进行推理：数据已加载、数据已导出、身份验证已核查、权限已变更。底层平台的复杂性被抽象隐藏，团队得以专注真正重要之事：理解用户行为并确保合规性。

顺应客户的现有体系

我们深知，许多组织已在自身的安全基础设施和工具链上投入巨资。他们拥有安全信息与事件管理（SIEM）平台、受过训练的分析师以及成熟的工作流。我们的目标并非取代这些系统，而是与其无缝集成。

为此，我们引入了新的公共 API 端点，无需让审计日志先流经 Foundry 即可直接访问。审计日志 API 提供两个直观的端点：一个用于按日期范围过滤并列出可用的日志文件（list-log-files），另一个用于检索特定日志文件的内容（get-log-file-content）。

安全团队可以自动化持续轮询新日志，并按需将其直接摄入首选的 SIEM 中进行处理和告警。底层遥测收集流式管道的低延迟特性，确保这些日志新鲜且具备可操作性。

这种“多模态”方式充分尊重了不同客户的差异化需求。对于在安全运营中心（SOC）开展日常工作、希望将审计日志接入现有安全工具栈的客户，新 API 提供了一条低延迟的直达通道。对于已有成熟监控架构和专用 SIEM 工具的重度日志消费者而言，通过 API 访问审计日志是首选方案。而对于未配置专用 SIEM 工具或分析需求较轻的组织，Palantir Foundry 依然是进行审计日志分析的可行平台。组织可以将新的 audit.3 模式日志导出至新的 Foundry 数据集中。

迁移之旅

如此规模的变更必须严谨对待。audit.3 模式的研发与落地历经数年的精心规划。重构后的遥测收集流式管道标志着更广泛的下一代解决方案的最新演进。我们与早期采用者进行了长期测试，借此不断迭代优化可用性。随着此次正式全面发布，我们正开始有计划地推动更广泛的采纳。

从 audit.2 迁移至 audit.3，代表着从模式结构不一致的时代，迈向强制一致且具备前瞻扩展性的新时代。audit.3 模式在设计上能够容纳 Palantir 产品、平台与工作流在未来的长远演变。在过渡期间，audit.2 与 audit.3 日志的摄入可并行运行，允许组织在完全切换前验证其新的分析逻辑。

我们通过全面更新的文档来为此次过渡提供支持，包括详尽的模式参考、迁移文档以及面向 SIEM 使用的 API 文档。我们期待更多组织体验新系统带来的益处。

一旦完成迁移，组织不仅能享受到延迟与模式一致性方面的即时改善，还能在未来基于此基础享受到持续的增强。可扩展的类别系统意味着未来的产品迭代不会破坏既有分析逻辑。公共 API 将能持续兼容未来引入的新产品功能。高效的遥测收集流式管道确保 Palantir 平台在扩展的同时，审计日志也能同步扩容。

夯实根基

审计日志常被视为基础设施中的底层管道——但它实际上深刻反映了组织的价值观。Palantir 投入多年工程力量，使审计日志更及时、更结构化、更易于访问。我们的客户所处的环境对信任的要求极高，任何失误都可能造成严重后果。他们需要能够快速、准确、全面地回答“发生了什么？”这一问题。

我们的隐私与公民自由团队秉持一个原则：隐私保护与运营效能并非互斥。我们不将两者视为相互竞争的优先事项，而是坚信通过用心设计能够调和这两大重要关切。有效的审计日志是负责任的数据运营的基石。它支撑着“监督监督者”的原则，在人类能够访问敏感信息和强大分析工具时，这一原则至关重要。审计日志帮助我们落实问责保障——强调出于某一目的合法的数据访问，并不等同于获得将其挪作他用的全面许可。

audit.3 模式与遥测收集流式管道的正式全面可用，标志着我们对 Palantir 可审计性创立原则的承诺达到了一个顶点。我们打下的这一基石——标准化的类别、高效的交付以及灵活的访问模式——为我们在未来的长远发展提供了广阔空间。

对于我们客户与合作伙伴的安全团队、合规人员以及信息安全专家而言，这一演进意味着他们能以更快的速度工作，并对审计数据的完整性与一致性抱有更高的信心。它彰显了 Palantir 在产品每一个层面践行透明与问责的承诺。

信任源自长年累月始终如一的行动。借助下一代审计日志系统，Palantir 确保平台上的每一个操作都可见、及时且可审计——因为这正是大规模运营的责任所在。
