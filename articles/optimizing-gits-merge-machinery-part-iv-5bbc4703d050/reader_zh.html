Optimizing Git’s Merge Machinery, #4

2021-07-01 · Original →

编者按：这是 Palantir 软件工程师关于扩展 Git 合并与重命名检测机制系列的第四篇文章。点击阅读第一篇、第二篇和第三篇博文。

本系列的第一篇文章包含了一些背景信息，介绍了合并机制的工作原理以及我为何要对其进行优化和重写。这些背景对于理解本文至关重要。因此，如果你不知道 Git 除了为提交记录存储哈希值外，还为文件存储了哈希值；或者不知道合并涉及三方内容合并；又或者不了解变基和拣选实际上包含一系列特殊的合并操作，那么你可能需要先回过头去阅读那篇文章。如果你已经了解了这些知识，让我们继续。

在这篇文章中，我将探讨另一种优化方案，我在第一篇文章中将其简单概括为：“在对一系列提交进行变基或拣选时，不要重复检测上游分支已经完成的重命名。”然而，实际情况要稍微复杂一些。

再次探讨抽象层

在上一篇文章中，我们讨论了抽象层。具体而言，由于每次合并都涉及执行两组独立的重命名检测（从合并基础到参与合并的两个提交），而且重命名检测的设计初衷是仅针对两个端点进行操作（对于合并而言，即合并基础和我们要为其检测重命名的那一侧历史），因此重命名检测天生就不知道另一侧历史中文件发生了什么。然而，为了实现“跳过无关重命名”的优化，我们需要两组重命名检测操作都能了解所有三个端点（合并基础以及参与合并的每个分支末端的提交）的信息。为了实现该优化，我们必须打破重命名检测抽象层，并传入来自历史另一侧的某种信息。这种打破抽象层的思路与本次优化同样息息相关。

正如本系列第一篇文章中所详细讨论的，对一系列提交进行变基和拣选涉及一系列三方合并。每次合并都包含两组重命名检测——从“合并基础”到历史的一侧（即上游侧），以及从“合并基础”到历史的另一侧（即我们要移植的提交）。因此，在变基或拣选操作中移植 N 个提交时，将会有 2*N 次对重命名检测机制的调用，其中 N 次调用是为了检测上游侧历史中发生的重命名。这是否意味着我们将相同的重命名检测了 N 次？

重复检测相同的重命名？

让我们深入探讨一下，看看上游侧历史的相同重命名重复检测是如何发生的。让我们重新审视变基或拣选一系列提交时的情形。考虑以下设置：

             A---B---C topic
            /
       D---E---F---G main

将 topic 变基或拣选到 main 之后，呈现如下状态：

                     A'---B'---C' topic
                    /
       D---E---F---G main

提交 A'、B' 和 C' 的创建过程经历了一系列合并，其中变基或拣选按顺序将 A-B-C 这三个提交中的每一个用于一次特殊的合并操作。让我们将合并操作中的这三个提交标记为 MERGE_BASE、MERGE_SIDE1 和 MERGE_SIDE2。针对此图，三次合并中每一次的三个提交如下：

创建 A’：

  • MERGE_BASE: E
  • MERGE_SIDE1: G
  • MERGE_SIDE2: A

创建 B’：

  • MERGE_BASE: A
  • MERGE_SIDE1: A’
  • MERGE_SIDE2: B

创建 C’：

  • MERGE_BASE: B
  • MERGE_SIDE1: B’
  • MERGE_SIDE2: C

现在，如果原始提交中的上游侧在 E 和 G 之间发生了多次重命名，那么创建 A’ 的三方合并将反映这些重命名。这意味着在第二次合并中，当比较 A 和 A’ 时，所有这些相同的重命名都会被重新检测……并并入 B’。然后，在第三次合并中，当比较 B 和 B’ 时，所有这些相同的重命名又会被再次重新检测，然后并入 C’。因此，在变基或拣选过程中，我们每移植一个提交，就要重复重新检测上游侧历史的重命名。

重复重新检测重命名已经够糟糕了，但在这种情况下情况甚至更糟：通常，上游侧历史的提交和变更要比 topic 分支多得多，因此上游侧历史比较的是历史中间隔相对较远的两个提交。提交间隔较远意味着变更更多，包括需要配对进行重命名检测的文件也更多。需要配对进行重命名检测的文件更多，意味着上游侧历史是检测重命名时计算开销大得多的一侧……然而，这也正是我们反复重新检测重命名的那一侧历史。不过，事情还有一线希望：这暗示着一个简单的解决方案。

缓存重命名！

意识到在拣选或变基 N 个提交的序列时，我们将上游侧历史中发生的重命名检测了 N 次，这启发了一个非常简单的解决方案：在第一次重命名检测后记住这些重命名（在内存缓存中），并在后续提交中直接使用这些缓存的知识。

听起来很简单。我们以前为什么没这么做？

这个想法相当简单。我认为它之所以被忽视，是出于以下几个原因的综合作用：首先，为拣选或变基执行的合并序列略显特殊，这很容易让人不小心假定错误的“合并基础”，并进而认为该优化完全是不安全的（正如我发布本系列文章后收到的第一条回复所证明的那样）。其次，它违反了现有的抽象层（由于抽象层映射了我们执行这些操作的思维定势，我们不得不从另一个角度来审视该操作）。或许最重要的一点是，尽管想法很简单，但要验证它不会引发任何奇怪的 Bug 或行为差异（除了运行速度变快之外），实际上相当复杂。事实证明，确实存在一种轻微的行为差异（如下所述）；只是它介于无害和有益之间。

关于试图验证缓存重命名不会导致奇怪 Bug 的这一点非常重要。尽管这个想法是本系列博客中最容易描述的（看看“缓存重命名！”那一节有多短就知道了），但要证明它是一项有效且安全的优化，所需的工作比我在本系列中迄今描述的任何其他优化都要复杂。例如，为了确保这种优化是安全的，必须回答以下一些问题（请注意，我将使用“移植提交”作为变基或拣选提交的简写）：

  • 移植给定提交时的所有重命名，是否也是移植后续提交时的重命名？仓库中是否存在特定文件集会导致情况并非如此？（如果是，这表明某些缓存的重命名可能无效。）
  • 给定提交中的重命名是否足以代表后续提交中的所有重命名？是否存在一种特殊的仓库设置导致情况并非如此？（如果是，这表明我们需要的不仅仅是缓存的重命名。）
  • 即使后续移植的提交中的重命名是先前移植提交中重命名的子集，如果我们没有检测到上一个（或几个）提交中的所有重命名怎么办？上一篇博文有一项关于跳过无关重命名的优化——如果那些被跳过的重命名对于移植上一个提交是无关紧要的，但对于移植下一个提交却是相关的呢？
  • 上述两个问题表明，通常输入到重命名检测中的源文件有三种类型：将作为重命名源文件名的文件、将对应删除操作的文件，以及可能属于重命名或删除但我们未花时间检测到底是两者中哪一种的文件。既然类型不止两种（重命名、删除或未知），我们是否也需要像缓存重命名一样缓存删除操作？
  • 是否还有其他我们需要缓存的元数据，例如目录重命名检测元数据？
  • 说到目录重命名，它们是如何处理的？当历史的一侧将文件添加到旧目录，而另一侧重命名该目录时，我们是只缓存目录重命名，还是还需要缓存相关特定文件的重命名？在历史的哪一侧？
  • 更复杂的目录重命名情况如何：历史的一侧将文件重命名到旧目录中（即 original/file → old-dir/file），而另一侧重命名旧目录（即 old-dir/ → new-dir/）。对于这种情况我们要缓存什么？original/file → old-dir/file？original/file → new-dir/file？old-dir/file → new-dir/file？这是否取决于上游侧历史是重命名旧目录的一方，还是将文件重命名到旧目录的一方？
  • 如果移植操作（变基或拣选）存在冲突，我们不得不停下来让用户有机会编辑工作树中的所有文件并将他们的更改包含在提交中，这会导致我们需要为下一个要移植的提交获取更多重命名吗？用户编辑完成后，之前移植产生的重命名是否仍然有效？
  • 如果变基是交互式的，并且用户停下来进行编辑怎么办？我们是否拥有所有有效的重命名，还是还有更多需要检测？我们现有的重命名是否还有效？
  • 如果用户移植的是非线性的历史怎么办？或者如果用户正在使用‘merge’或‘reset’指令进行更复杂的交互式变基（即变基合并或变基多个分支）怎么办？总的来说，我们如何知道对合并机制的顺序调用具有使该优化生效的性质？
  • 与上一个问题相关，除了移植线性序列的提交外，是否还有其他情况使缓存重命名成为一种有效的优化？
  • 这种优化是否会导致我们得到不同类型的冲突、不同的合并结果，或者得到没有这种优化就不会出现的冲突，亦或是导致原本会出现冲突的地方不再出现冲突？

回答这些问题可能会变得相当棘手。如果你对细节感到好奇，可以阅读我撰写的用于解释这些细节的详尽文档。该文档实质上是对这种优化可能产生和不可能产生的结果的证明，连同为避免意外而需应用的重要限制，以及对这种优化无法避免的一对罕见行为差异的解释。你可以阅读该文档了解细节，但在这里我只涵盖潜在的行为差异和施加的限制。

行为差异

如上所述，鉴于对该优化触发条件的限制，只有一种极其罕见的情况会导致行为差异。新的行为仍然非常合理；事实上，我认为它提供了改进的行为。不幸的是，试图解释这种罕见情况有点复杂……

前置条件

考虑一个满足以下所有条件的情况：

  • 上游侧历史既修改了文件又对其进行了重命名（例如 A → B）
  • 上游侧的修改使得 B 中相对于 A 的行数净增加。
  • 非上游侧历史大幅缩减了 A 的大小，使得剩余的行数严格少于上游侧添加到 B 的净行数。
  • 尽管双方都进行了重大的内容更改，但它们都能干净地合并；没有冲突。结果将是一个文件 B，其中包含来自双方的干净合并的更改。让我们将合并后生成的该文件称为 B-merged。

示例测试用例

如果很难想象一个满足上述条件的示例，你可以查看我添加到 Git 测试套件中的“拣选一个提交及其直接撤销”测试用例。（请注意，该补丁中测试的行为是优化前的；在启用记忆重命名优化的提交中对该测试进行了微调。）该测试用例涉及内容的简要总结如下：

  • A 是一个最初有 20 行的文件
  • 上游在 A 的开头添加了 10 个新行，然后将其重命名为 B（这样 B 总共有 30 行）
  • 非上游侧删除了 A 中除前三行之外的所有内容（因此它从 20 行减少到 3 行；注意：3 < 10）

上面的示例将干净地合并。B-merged 将是一个名为 B 的文件，长度为 13 行，由上游侧添加的 10 个新行及其后非上游侧完整保留的原始文件 A 的前 3 行组成。

附加条件与可能观察到的差异

现在，对于这种情况，如果非上游侧历史有另一个也修改了 A 的提交，那么有两种可能：

首先，如果这些对 A 的进一步更改也能与 B-merged 干净地合并：

  • 使用该优化，移植操作可以继续不间断地进行。
  • 不使用该优化，移植操作将因 A 的修改/删除冲突而停止，工作副本中将同时存在最新版本的 A 和合并版本的 B，并且不会向用户发出 A 和 B 可能相关的信号。用户将不得不手动解决变基或拣选冲突并继续操作。

其次，如果这些对 A 的进一步更改无法与 B-merged 干净地合并：

  • 使用该优化，将报告内容冲突，并在 B 中找到更改的三方合并（包括冲突标记）供用户整理。B 在序列早期是 A 的重命名这一事实表明这很可能对用户来说具有直观意义，尽管如果他们愿意，他们可以将文件分开，因为不同的文件将存在于索引中针对 B 的不同阶段。
  • 不使用该优化，将报告 A 的修改/删除冲突。工作副本中将同时存在最新版本的 A 和合并版本的 B，但不会向用户提供 A 和 B 可能相关的信号。用户将全权负责尝试调和冲突，并希望能自己发现 A 和 B 的关系（如果相关的话）。

扩展示例测试用例

这一切什么时候会发生？好吧，让我们继续看我们的示例测试用例。如果被移植的提交序列只涉及两个提交，第一个是将 A 从 20 行缩减到 3 行的提交，紧接着是将其撤销并将 A 从 3 行恢复到原始 20 行的提交。你会认为，如果你能干净地拣选一个提交，那么你应该能够在紧接着之后干净地拣选它的直接撤销，有了这个优化你就可以做到，但如果没有这个优化，这就是一个独特的情况，合并算法将无法做到这一点，因为它不会认为缩减后的 A 和 B-merged 是重命名。

限制

如上所述，对于何时应用该优化存在一些限制（要么是为了确保优化保持安全，要么在一种情况下是为了实现简单）。这些限制是：

  • 缓存重命名优化仅适用于被移植（即拣选或变基）的线性提交序列
  • 缓存位于内存中，不会存储在磁盘上。
  • 缓存会在移植操作结束时丢弃，以工作树和索引被更新为标志。（注意：变基和拣选需要重构，以使其仅在移植操作结束时才更新工作树和索引，而不是在每移植一个提交后就无用地更新。）
  • 如果历史两侧以相同方式重命名文件（即 A 在历史两侧都被重命名为 B），则缓存将被丢弃。

请注意，第二项，即缓存仅存在于内存中，会产生几个后果：

  • 如果仅移植一个提交，该优化无效
  • 一旦出现任何冲突，该优化就会被关闭，因为停下来等待用户输入会导致缓存被丢弃
  • 当存在目录重命名时，该优化会被关闭，除非用户将 merge.directoryRenames 配置设置为“true”（默认值为“conflict”）
  • 一旦用户通过“edit”或“break”指令暂停变基，交互式变基的该优化就会被关闭，因为暂停会导致缓存被丢弃。

优势

尽管确实有不少因素限制了该优化的适用时机，但它的优势在于能够广泛应用于本系列博客中先前优化无法覆盖的多种情况。请注意其他那些优化的以下限制：

  • 《优化 Git 的合并机制，第一部分》：仅适用于文件在重命名时没有任何内容更改，且同一侧历史中的其他提交在重命名前后均未修改该重命名文件的情况
  • 《优化 Git 的合并机制，第二部分》：仅适用于文件被移动到不同目录而未重命名文件名的情况
  • 《优化 Git 的合并机制，第三部分》：仅适用于未重命名文件的那一侧历史未对文件进行任何修改的情况

相比之下，即使这些条件都不满足，该优化也适用。只要只有一侧重命名了文件并且合并是干净且自动的，缓存重命名优化就适用于拣选或变基线性历史中所有剩余的提交。

结果

对于我在本系列中一直关注的特定测试用例，即涉及将 35 个补丁变基到一个重命名了约 2.6 万个文件的分支的测试用例，这将整体运行时间加快了约 13%：

                         Before                  After
mega-renames:     11.435 s ±  0.158 s    10.127 s ±  0.073 s

然而，这个特定的测试用例对于先前的三个优化来说几乎是最优的；它在上游侧添加了一个仅重命名顶级目录的提交，这导致在 Git 看来就像 26,000 次重命名。当我最初实现这个优化时，我在同一个测试用例上实际上看到了 2 倍的加速，但那是因为我还没有找到对之前博文中的优化进行的多个额外调整，而这些调整使它们更加有效。由于该优化倾向于在其他优化无法发挥作用时弥补空缺，让其他优化更有效反而使得这个优化看起来不那么有效。但是约 13% 的加速仍然是一个相当不错的收益。

看看如果抛弃本系列第二部分和第三部分的优化，仅观察我们从该优化中获得了多少性能提升，也会很有意思。在这种情况下，我们看到了 8 倍的速度提升。

            Before Basename Series   After Just This Series

mega-renames: 1799.937 s ± 0.493 s 205.709 s ± 0.457 s

因此，在其他优化无效的情况下，该优化能带来显著帮助。

该优化将于 2021 年 8 月出现在 Git 2.33 中。有关此优化的更多详细信息，请参阅上游线程。

凭借目前涵盖的四种优化，对于上述海量重命名测试用例，重命名检测花费的时间已降至总运行时间的约 9%。本系列剩余的两篇文章将重点讨论优化合并机制的其他部分。
