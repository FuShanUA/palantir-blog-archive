Democratizing Security Detection

2022-06-02 · Original →

引言

安全检测项目面临巨大的扩展挑战。随着组织规模和业务范围的增长，安全告警的数量和多样性也随之增加；这种涌入会对组织的安全态势产生负面影响，并可能导致负责告警分类的安全人员出现倦怠。团队通常试图通过增加人手来解决告警量问题，但单纯增加人数并非长久之计。

在 Palantir，我们在全球拥有超过 3000 名员工和外包人员。我们运行着一个复杂的企业生态系统，在数以万计的主机上运行着数百个应用程序。我们还在包括 Amazon Web Services、Microsoft Azure 和 Google Compute 在内的超大规模云服务提供商上提供 Palantir 软件即服务。

尽管 Palantir 的技术基础设施相对复杂，但所有这些基础设施上的信息安全监控、告警分类和事件响应都集中由不到 10 人的团队负责管理。这篇博文的目的有两个方面：

  • 首先，分享我们通过安全告警大众化来扩展检测项目的经验。
  • 其次，提供适用于大多数环境的可操作的检测策略。

检测架构

在深入探讨安全检测大众化之前，我们先讨论一下检测架构。不同组织的架构会有很大差异，但基本要素是相似的。

Palantir 计算机事件响应团队（CIRT）负责企业内部安全检测和响应的方方面面。这包括主机、网络、应用程序和云基础设施上的检测。

在日志源方面，CIRT 定义了所有遥测要求和来源。他们会根据与安全相关的日志和数据的实用性，对遥测源进行验证和审批，并拥有最终决定权。

与安全相关的日志和数据通过数百条日志管道接入集中式安全信息和事件管理（SIEM）平台。然后，CIRT 开发告警和检测策略（ADS），以显现这些数据源中的异常或恶意活动。

生成的告警随后流入我们的安全编排、自动化和响应（SOAR）平台。根据告警的不同，我们应用代码剧本根据告警的来源、上下文和有效载荷执行特定操作。这些代码剧本的一个子集用于通过用户安全意识和分类来识别和响应安全事件。

InfoSecBot

InfoSecBot 是 SOAR 平台与 Slack 租户之间的应用程序集成，是我们将异常活动直接推送给用户的最成功实践。自 2016 年以来，我们一直在全公司范围内运行 InfoSecBot，其功能、特性和范围随着时间的推移不断增强。

InfoSecBot 处理几个关键操作。这些操作在我们的 SOAR 平台中作为模块实现，ADS 作者在开发和实施告警时可以选择一个或多个模块。这些模块包括：

  • 个人通知： 通过直接消息（DM）向用户发送包含任意告警有效载荷的通知。
  • 个人挑战： 通过直接消息（DM）向用户发起包含任意告警有效载荷的挑战。挑战由一条消息和一个确认/拒绝按钮组成，以收集用户的响应。
  • 群组通知： 通过消息向 Slack 频道发送包含任意告警有效载荷的通知。
  • 群组挑战： 通过消息向 Slack 频道发起包含任意告警有效载荷的挑战。挑战由一条消息和一个确认/拒绝按钮组成，以收集用户的响应。
  • MFA 挑战： 发出 Duo 多因素认证提示，以收集用户的响应。

这些模块可以串联起来构建复杂的检测工作流。例如，通过直接消息向个人用户呈现告警。系统向用户发起挑战，他们有 15 分钟的时间通过 Slack 按钮进行确认或拒绝。用户点击确认，但由于告警的严重性，系统会发出 Duo 多因素认证提示以确认其身份。一旦 Duo 提示被批准，告警就可以被关闭、降低优先级，或提交给 CIRT 进行人工验证。

如果用户点击拒绝按钮、超时或 Duo 提示失败，告警将立即升级，其他自动化程序也将随之启动。

检测机会与策略

虽然向用户推送这些告警会耗费时间和精力，但它提供了一些可能的最强信号，并显著缩短了异常或未经授权活动的响应时间。此外，根据告警的有效性和严重程度，可以在响应用户信号时纳入额外的自动化操作，包括密码重置、账户锁定、令牌撤销或设备隔离。

本文的以下部分包含了告警和检测策略的示例，这些策略承担了我们总体告警量的很大一部分。通过直接与操作方解决大量告警冲突，这些策略使 CIRT 能够扩大其攻击面覆盖范围，而无需按比例增加人员。

我们鼓励其他组织开发自己独特的 ADS，并找到一种机制与更广泛的信息安全社区分享。虽然公开承认和记录内部告警存在操作安全方面的考量，但我们希望这些例子能促进更多的分享和合作，激发其他防御者的检测增强，并最终增加攻击者的操作成本。

检测账户接管

针对影响用户账户实体的异常活动，可以快速检测到账户接管行为。用户直接了解他们使用账户执行的主要操作，特别是在行为发生后短时间内如果能联系到他们，效果会更好。以下检测策略旨在帮助显现使用账户接管技术的攻击。

检测账户接管：MFA 失败 —— 用户拒绝或不认可 MFA 提示

攻击概述： 攻击者获取了用户的密码，并试图以该用户身份进行身份验证。然而，由于启用了 MFA，攻击者在没有用户第二重身份验证因素的情况下无法通过身份验证。

检测策略： MFA 通常被吹捧为一种强化账户防御态势以抵御账户接管的控制手段。然而，除了作为一种强化控制外，我们认为 MFA 还应作为告警和检测的基础。用户拒绝的任何 MFA 提示都应被视为潜在的账户接管事件，并应生成相应的告警。

响应： 当用户（或试图以用户身份进行验证的行为者）尝试登录时，他们会收到 MFA 通知；用户可以选择接受或拒绝该通知。如果用户拒绝了 MFA 提示，告警将传播到 CIRT 的告警和检测管道，分析师将审查相关活动。

检测账户接管：用户认证到敏感资源

攻击概述： 攻击者接管了受害者账户，目的是利用受害者对敏感目标资源的访问权限。

检测策略： 监控安全/审计日志，查看针对敏感平台、主机或资源的离散身份验证事件。

响应： 每当用户认证到范围内的资源时，InfoSecBot 就会向用户发出直接消息通知；消息包括：

  • 活动的时间戳。
  • 其账户认证到的资源。
  • 发起身份验证尝试的 IP 地址。

相关用户必须对 InfoSecBot 消息提示做出肯定回应，并且还必须确认来自 InfoSecBot 的 MFA 提示。如果用户未能完成提示/MFA，告警将升级至 CIRT。

检测凭证重置

密码是个糟糕的东西；它们通常很弱或被重复使用，而且很容易把密码交给攻击者。然而，在无密码身份验证普及之前，密码是必不可少的无奈之举。以下检测策略旨在帮助显现使用凭证重置技术的攻击。

检测凭证重置：用户更改密码

攻击概述： 被攻陷的用户可能会尝试轮换其凭证，以获取对该账户的独占访问权。

检测策略： 监控安全日志（如 AzureAD、Okta、Windows 事件日志），查看由用户发起的离散密码更改工作流。

响应： 每当用户更改其企业账户的密码时，InfoSecBot 就会向用户发出直接消息通知。呈现的告警有效载荷包括：

  • 活动的时间戳。
  • 密码轮换的来源，包括设备主机名和 IP 地址。

如果活动是意外或不寻常的，系统会向用户提供如何快速将其上报给 CIRT 的说明。

检测凭证重置：用户密码被管理员更改

攻击概述： 具有提升权限的被攻陷用户可能会尝试对另一个账户执行凭证重置，以进行横向移动或权限提升。

检测策略： 监控安全日志（如 AzureAD、Okta、Windows 事件日志），查看由管理员发起的离散密码更改工作流。

响应： 每当管理员强制更改用户企业账户的密码时，InfoSecBot 就会向用户发出直接消息通知。呈现的告警有效载荷包括：

  • 活动的时间戳。
  • 密码轮换的来源，包括设备主机名和 IP 地址。
  • 管理员的用户名。

如果活动是意外或不寻常的，系统会向用户提供如何快速将其上报给 CIRT 的说明。

响应策略可以扩展为包括通知执行此操作的管理员。这提供了额外的检测属性，并作为具有特权访问权限用户的监督机制。

检测凭证重置：为用户创建了无密码绕过码

无密码身份验证理想情况下会使用硬件安全令牌。然而，用户可能会丢失或忘记其安全令牌，从而无法访问信息系统。此外，用户在开始使用安全令牌之前需要先进行注册。为了实现这些工作流，可能需要向用户提供绕过码或临时访问密码。这些实际上充当了用户的第一和第二身份验证因素，并且通常有时间限制或使用次数限制。

攻击概述： 具有提升权限的被攻陷用户可能会尝试为用户创建绕过码或临时访问密码。

检测策略： 监控安全日志（如 AzureAD、Okta、Windows 事件日志），查看由管理员发起的离散绕过码创建或关联工作流。

响应： 每当管理员强制更改用户企业账户的密码时，InfoSecBot 就会向用户发出直接消息通知。呈现的告警有效载荷包括：

  • 活动的时间戳。
  • 关于绕过码或临时访问密码的详细信息。
  • 管理员的用户名。

如果活动是意外或不寻常的，系统会向用户提供如何快速将其上报给 CIRT 的说明。

响应策略可以扩展为包括通知执行此操作的管理员。这提供了额外的检测属性，并作为具有特权访问权限用户的监督机制。

检测凭证重置：用户尝试密码恢复

出于安全原因，我们不允许自助密码恢复。然而，这是攻击者滥用以获取合法账户初始访问权限的非常常见的技术。

攻击概述： 攻击者滥用自助密码恢复门户以获取账户的初始访问权限。

检测策略： 监控安全日志（如 AzureAD、Okta、Windows 事件日志），查看由用户发起的离散密码恢复尝试。

响应： 每当发生密码重置请求时，InfoSecBot 就会向用户发出直接消息通知。呈现的告警有效载荷包括：

  • 活动的时间戳。
  • 密码恢复过程的来源，包括设备主机名和 IP 地址。
  • 密码恢复过程是否成功。

如果活动是意外或不寻常的，系统会向用户提供如何快速将其上报给 CIRT 的说明。

如果启用了自助密码恢复，多次失败的尝试可以是一个高信号检测。然而，攻击者可能会在第一次尝试时走运，因此向受影响的用户发送强制性通知是适当的。

检测凭证钓鱼

认识到密码（以及由密码衍生的人类行为）可以说是身份安全态势中最薄弱的环节，Palantir 的信息安全人员试图构建一个专用工具，以增强我们打击凭证钓鱼和重复使用的方法。结果就是我们的浏览器扩展程序“PhishCatch”，它后来已经开源；在 GitHub 上或我们的博客上阅读更多关于 PhishCatch 的功能及其在 Palantir 的作用。Palantir 的 PhishCatch 告警和检测工作原理如下：

检测凭证钓鱼：PhishCatch 告警

如果/当用户在 Palantir 专用 IDP 以外的页面输入其密码时，PhishCatch 将创建一个告警。然后，PhishCatch 告警将传播到 CIRT 的 SOAR，此时将针对该告警运行一系列自动化操作：

  • InfoSecBot 将通过 Slack 请求用户更改密码；此消息将每 1 小时重复一次，持续 72 小时。
  • 同样的每小时消息也将发送到用户的电子邮件作为冗余。
  • 用户的领导/主管也将收到同样的每小时消息。
  • 如果/当用户更改了密码，他们将停止收到通知。
  • 如果用户在告警后 72 小时内未能更改密码，其账户将被锁定。

经过 PhishCatch 告警完整的 72 小时生命周期后，特定用户需要由 Palantir 的支持人员手动重新激活其账户。

检测异常行为

即使告警实体不能完全归入某一类攻击方法，对于任何由恶意方执行可能对 Palantir 构成风险的活动，我们仍会为其编写临时的告警和检测策略。为了应对此类活动，CIRT 编写了如下告警和检测策略：

攻击概述： CIRT 的告警和检测策略识别出了一项临时操作，该操作不一定被归类为明确的攻击尝试，但其可疑或敏感程度足以需要向用户咨询有关该活动的情况。

检测策略： 虽然 CIRT 的告警和检测覆盖范围不断扩大，但以下列表代表了我们认为可疑或敏感程度足以需要向用户提出质疑的活动的总体概况：

  • 用户通过移动设备管理注册新设备。
  • 用户注册新的多因素身份验证设备。
  • 用户登录高风险基础设施（如 VDI）。
  • 用户账户被锁定或被标记为高风险。

响应： 每当检测到范围内的活动时，InfoSecBot 就会向用户发出直接消息通知。呈现的告警有效载荷包括：

  • 活动本身的摘要。
  • 活动的时间戳。
  • 关于该活动为何可能令人担忧的简要描述。

如果该活动是意外或不寻常的，系统会向用户提供如何快速将其上报给 CIRT 的说明。

检测后渗透

虽然信息安全团队应力求不让攻击者达到后渗透阶段，但忽视这种可能性将是失职。健康的告警和检测态势应覆盖整个攻击生命周期，而就系统上的活动咨询用户，可以成为将生命周期后期告警与其根本原因相核对的有力工具。以下告警和检测策略代表了 Palantir 围绕后渗透活动的以用户为中心的告警示例。

检测后渗透：进出敏感目的地的数据移动或暂存

攻击概述： 已经使用被攻陷账户在环境中运行的攻击者，利用该账户暂存（即归档、压缩或加密）或移动/传输数据。

检测策略： 如果通过用户账户发生异常的数据移动、归档或加密操作，端点日志遥测将记录相关活动。CIRT 的告警逻辑将查询上述日志遥测以显现相关活动。

响应： 当发现用户账户以告警范围内的方式与数据进行交互时，用户将收到包含以下内容的 InfoSecBot 提示：

  • 活动的时间戳。
  • 发生活动的系统。
  • 相关数据交互的摘要。

相关用户必须对 InfoSecBot 消息提示做出肯定回应，并且还必须确认来自 InfoSecBot 的 MFA 提示。如果用户未能完成提示/MFA，告警将升级至 CIRT。

检测后渗透：基于主机的侦察或扫描

攻击概述： 在特定用户端点上运行的攻击者，或使用特定用户账户在任何端点上运行的攻击者，可能会尝试运行发现/侦察命令，以努力查明有关我们环境攻击面的更多细节。

检测策略： CIRT 查询 EDR 日志遥测，查找可能被解释为侦察的活动或命令，或与发现或端口扫描（如 nmap）一致的任何命令。

响应： 当从用户账户或用户工作站检测到此类活动时，该用户将收到详细说明以下内容的 InfoSecBot 提示：

  • 从其账户或工作站运行的特定命令。
  • 活动的时间戳。
  • 发生活动的系统。

相关用户必须对 InfoSecBot 消息提示做出肯定回应，并且还必须确认来自 InfoSecBot 的 MFA 提示。如果用户未能完成提示/MFA，告警将升级至 CIRT。

众包合理使用

虽然大多数 InfoSecBot 提示是发送给特定个人的，但在某些情况下，向更广泛的群组发送消息更有意义。考虑以下因素：

  • 如果攻击者接管了账户，他们可能会采取行动静默、删除或以其他方式隐藏发送给被接管账户的 Slack 消息，这可能会阻碍合法账户所有者看到该消息。
  • 在内部威胁场景中，给定用户对自己的行为非常清楚；向该用户发送 Slack 消息不会提供任何安全收益。
  • 同事团队，包括这些团队的领导，通常对团队成员采取的行动有大致的了解。

认识到上述因素，Palantir 的 CIRT 发现将一些 InfoSecBot 提示定向发送到 Slack 聊天室而不是个人是明智的。一般来说，目标聊天室是用于与给定团队进行通信的专用/集中频道；因此，团队的所有成员（包括领导）都在该频道中。此外，频道成员资格由 Palantir IDP 中的群组成员身份支持，如果成员因任何原因被移除，他们将被自动重新添加到该频道。

对于众包告警，Palantir 的方法通常遵循以下做法：

  • 告警通常反映针对敏感集合资源的特权活动。
  • 例如，使用紧急凭证 SSH 登录到后端服务器。
  • InfoSecBot 通知通常包含用户活动的完整上下文。
  • 例如，谁采取了行动、采取了什么行动、时间戳等。
  • 告警将向包含行为者整个团队的频道发送消息。
  • 如果活动与行为者的工作预期不一致，行为者的团队会收到将问题升级至 CIRT 的指示。

软件合规性

Palantir 维护着一份针对端点工作站的严格且经批准和未批准的软件列表。如果用户安装了与我们的安全标准不符的产品，InfoSecBot 会通知他们违规行为，并允许他们在信息安全团队成员介入前有一段短暂的宽限期来删除违规内容。

此外，Palantir 还维护着用户工作站上的软件版本清单。如果 Palantir 用户的工作站包含具有已知漏洞的软件版本，InfoSecBot 将提示该用户更新其软件。用户将有一段短暂的宽限期来更新其软件，然后将该问题升级至 Palantir 的信息安全团队。

紧急补丁

在高严重性或关键严重性 CVE 需要立即进行软件补丁或操作系统更新的情况下，Palantir 的安全工程师利用 InfoSecBot 通知用户其系统可能因更新而即将停机。用户也会通过其他方式收到通知，但 InfoSecBot 提供了额外的通信冗余层，以确保用户不会因紧急补丁而措手不及。
