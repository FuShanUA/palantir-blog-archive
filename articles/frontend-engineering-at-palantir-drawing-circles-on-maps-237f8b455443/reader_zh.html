地图可视化

在许多开发场景中，我们利用开源地图库 MapLibre 来可视化地理空间数据。为了绘制形状（线、多边形、点），我们向 MapLibre 输入 GeoJSON 格式的几何数据。

GeoJSON 非常适合 Web 应用：它本质上就是 JSON，因此在 JavaScript 中原生支持，无需转换。它具备易读性、轻量级的特点，并且是 Web 地图领域事实上的标准。MapLibre 的 API 就是围绕它构建的。但问题在于，GeoJSON 规范并不包含圆。该规范支持 LineString（线）、Polygon（多边形）、Point（点）以及它们的各种集合，但没有圆。

这并非规范的设计疏忽。与纯靠坐标顶点定义的多边形不同，圆需要一个半径，这意味着必须选择一种度量单位。半径应该用米？英里？还是纬度度数？每种选择都会导致不同的结果，而 GeoJSON 有意避免将这类测量决策编码到其几何类型中。该规范严格坚持普遍适用的基于坐标的形状，将距离计算和单位留给了应用层处理。

但是，Web 地图上的圆还隐藏着一个更深层次的问题，只有当你真正动手实现时才会显现：地图投影从根本上扭曲了“圆”的真实含义。

尝试实现

最直观的朴素解法是：圆只是一个拥有许多顶点的多边形。在圆周上采样点，连接它们，你就得到了一个圆。

完美！嗯，几乎是——眼尖的人能看出单独的采样点和连接它们的直线。我们可以通过提高采样率来缓解这个问题。对于大多数应用场景来说，这已经足够好了。现在，让我们用几千英里的半径尝试同样的方法。

等等，发生了什么？在平面地图上，生成的形状看起来像个椭圆。但当我们把相同的点渲染在三维地球仪上时，它们构成了一个完美的圆。事情开始变得有趣了。是时候谈谈地图投影了。

地图投影

在制图学中，地图投影试图将地球仪的曲面呈现为二维平面的地图。根本问题在于：你无法将一个球体平铺到平面上而不产生变形。将球体投影到平面上的所有方法，都必然会在某种程度上扭曲表面。你见过的每一张地图都是一种妥协。所有投影都会扭曲某些东西——面积、形状、距离或方向。

不同的投影会做出不同的取舍：兰伯特正形圆锥投影为航空图保留了角度，但扭曲了面积；阿尔伯斯等面积投影为专题地图保留了面积，却使形状变形；罗宾森投影在美学上折中，但完全牺牲了精确度。

最常见的投影是墨卡托投影。它是谷歌地图、大多数 GPS 系统以及许多课堂地图所使用的投影。墨卡托投影在 1569 年问世后便成为了主导性的世界地图，因为它对导航极其有用——地图上的任何直线都代表恒定的罗盘方位，使得水手能够轻松规划并遵循航线。这一实用优势使其在几个世纪内一直是标准。与其他投影一样，墨卡托投影也扭曲了地球表面的真实布局。它放大了远离赤道的地区；越接近地球极点，变形越大。一个常见的例子是，格陵兰岛和非洲看起来一样大，而实际上非洲的面积是格陵兰岛的 14 倍。

还有一个在线工具可以让你查看不同国家的真实大小：https://thetruesize.com/

如今，大多数在线地图服务（包括 MapLibre）都使用 Web 墨卡托投影，这是一种为数字地图优化的现代变体。它计算效率高，并且能很好地保留局部形状，这就是为什么你在谷歌地图上看到的街区样貌是准确的。但它仍然极大地扭曲了极地附近的面积。这正是我们的圆看起来像椭圆的原因。在大比例尺下，投影的变形扭曲了可视化结果。这个圆在地球球体上是正确的，但在我们渲染的平面地图上却显得变形了。

实现细节探析

实际的实现比我在上面描述的高层解法要复杂得多。让我们将其拆解为几个关键部分：在圆周上采样点、创建圆轮廓以及生成彩色填充。

在圆周上采样点

首先，我们需要一种方法，在给定圆心、半径和方位角的情况下找到一个点。这涉及一些球面三角学（使用半正矢公式），具体数学细节我就不赘述了，你可以直接使用像 Turf.js 的 destination 函数这样的开源解决方案：

现在，获取圆周上的点就很简单了：将 360 度分成 numPoints 等份的角度，并计算每个方位角对应的目的地点：

function getSampledCirclePoints(center, radius, numPoints) {
  const coordinates = [];
  
  for (let i = 0; i < numPoints; i++) {
    // It is -360 because we go in counterclockwise order
    // as per geojson polygon spec!
    const bearing = (i * -360) / numPoints;
    coordinates.push(
      destination(center, radius, bearing)
    );
  }
  
  return coordinates;
}

处理圆轮廓

还记得跨越极点的圆不是闭合多边形，而是正弦曲线吗？这意味着轮廓需要使用 GeoJSON 的 LineString 类型，而不是 Polygon。棘手的部分在于找到截断圆的两个坐标。也就是当线应该跨越反子午线（即经度 180 度的国际日期变更线，它将地球分为东半球和西半球）时的点。以下是核心逻辑：

function createPolarAwareCircleOutline(sampledPoints) {
  const lineCoordinates = [sampledPoints[0]];
  
  for (let i = 1; i < sampledPoints.length; i++) {
    const point = sampledPoints[i];
    const prevPoint = sampledPoints[i - 1];
    
    if (shouldCrossAntimeridian(point.lon, prevPoint.lon)) {
      // Adjust longitude to render on the next world copy
      const adjustedLon = point.lon + (point.lon < 0 ? 360 : -360);
      lineCoordinates.push([adjustedLon, point.lat]);
    } else {
      lineCoordinates.push([point.lon, point.lat]);
    }
  }
  
  return {
    type: "LineString",
    coordinates: lineCoordinates
  };
}
// If 2 points are more than half the world apart, we should cross the antimeridian!
function shouldCrossAntimeridian(lon1, lon2) {
  return Math.abs(lon1 - lon2) > 180;
}

这里的巧妙之处在于，MapLibre 允许经度超出标准的 -180 到 180 度范围。例如，经度 200 度会告诉 MapLibre 将点渲染在东边的下一个世界副本上。这就是我们让圆正确环绕的方法。

处理彩色填充

彩色填充更棘手，因为它必须是一个闭合多边形。如果圆包含极点，我们需要显式地将多边形延伸到该极点，以确保填充覆盖整个区域。根据你采样点的方式，跨越反子午线的点的索引总是相同的。这里我用伪代码来抽象化寻找这些点的过程，因为我们前面已经讲过了。以下是我们处理跨越极点的圆的方法：
