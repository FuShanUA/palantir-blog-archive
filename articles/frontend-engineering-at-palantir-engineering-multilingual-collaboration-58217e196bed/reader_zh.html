在这篇文章中，我将分享我们是如何构建 Dossier 翻译模式的、面临的技术挑战，以及为什么这是一个很好的端到端、高影响力前端项目的典范，正是这类项目让我在工作中始终保持兴奋和动力。

为什么要做翻译？为什么是现在？

在当今几乎所有重要的新闻头条中，你都会发现大多数行动都需要深度的多国协作。这些不同的盟国通常使用不同的语言，但我们不能让语言成为高信噪比协作和团队合作的障碍。用户、分析师、规划人员和操作人员比以往任何时候都更多地与全球队友协同工作。但他们中的许多人处于封闭或涉密网络中，标准的公开消费级翻译工具在这些环境中根本无法使用或被明令禁止。

为了让我们的生产力套件具备真正的协作能力，我们需要将翻译功能引入富文本文档编辑器 Dossier，并尽可能做到无缝集成。我们此前已经在 Chat 和 Slides 中推出了翻译功能，但 Dossier 带来了一系列全新的挑战。与聊天消息或幻灯片不同，Dossier 文档可能长达数百页，包含复杂的格式、表格以及实时的协同编辑。我们的目标是让用户能够切换到翻译模式，以选择的语言查看整个文档，并且依然能与内容进行交互（复制文本和添加评论)，同时不丢失上下文或牺牲性能。

翻译功能设计

我们的愿景很简单：只需点击一次，用户就能以翻译后的只读模式查看任何 Dossier 文档。在底层，这需要：

  • 一致的 UX：翻译体验需要与用户在 Chat 和 Slides 中已经熟悉的体验保持一致。
  • 安全：所有翻译都必须通过我们内部的语言模型服务 (LMS) 运行，绝不能离开安全网络。
  • 性能：即便是庞大的文档也要能快速翻译，不能导致浏览器卡顿或使后端过载。
  • 协作：翻译模式必须与实时编辑、评论和实时用户状态良好兼容。

技术实现

1. 共享组件

我们此前已经为 Slides 和 Chat 构建了翻译 UI 组件，因此我们将这些组件提取到一个共享库中，以确保各应用间的体验一致。但 Dossier 的文档模型要复杂得多——我们必须扩展这些组件，以处理深层嵌套的富文本结构、表格和自定义组件。

2. 用于翻译的 Redux 状态

翻译并非一个可以简单附加的功能——它涉及整个应用。用户的语言偏好、每个翻译块的加载状态、缓存、在不丢失文档阅读位置的情况下开启和关闭翻译——所有这些都需要进行全局协调。因此，我们对状态建模方式进行了深思熟虑，在 Dossier 的 Redux store 中添加了一个专门的 slice 来对其进行清晰管理。

export interface TranslationsCacheValue {
  originalText: string;
  translatedText: IAsyncLoaded<string>;
}

export interface TranslationsCache {
  [elementId: string]: TranslationsCacheValue;
}

export interface TranslationState {
  isInTranslationMode: boolean;
  supportedLanguages: IAsyncLoaded<LanguageMetadata[]>;
  translations: TranslationsCache;
  detectedLanguage?: string;
  sourceLanguage?: string;
  targetLanguage?: string;
}

该状态还负责跟踪哪些块正在加载、已成功或已失败，以便我们向用户展示详细的反馈。

3. 分块：平衡翻译质量与速度

翻译必须让人感觉快速且无缝，即便是对于长达多页的 Dossier 文档也是如此。一次性发送整个文档会让用户盯着加载屏幕干等；逐词或逐句翻译虽然更快，但会降低质量，因为大多数翻译模型需要依赖更广泛的上下文才能生成自然的结果。

我们最终决定在块级别进行分块——如段落、列表项、表格单元格——这取得了恰当的平衡。一旦块翻译完成，用户就能立即看到它们以翻译后的形式出现，从而感受到渐进式的进度，而无需等待整个文档翻译完毕。

在下一版的分块策略中，我们希望解决一些有趣的边缘情况：

  • 超长段落：如果用户写了一个长达数页的单独段落
  • 多语言块：如果一个段落包含多种语言，模型可能无法正确检测目标语言
  • 非语法片段：用户创建的块可能只是项目符号、首字母缩写或碎片化内容，在没有周围上下文的情况下，模型可能难以解析

目前，我们已经推出了块级分块方案，但随着获取更多使用数据和反馈，未来仍有很大的改进空间。

4. 延迟增量翻译

除了分块策略，我们还必须考虑对于长达 100 多页的文档应优先处理哪些块。我们使用了 Intersection Observer API 来实现翻译的“延迟加载”。只有用户当前视口中可见的块才会被发送到 LMS 进行翻译。随着用户滚动页面，新的块会按需进行翻译。

// IntersectionObserver API hook for "has entered screen" detection
function useHasEnteredScreen(ref) {
  const [hasEntered, setHasEntered] = React.useState(false);

  useEffect(() => {
    if (hasEntered || ref.current == null) return;

    const el = ref.current;
    const observer = new IntersectionObserver(([entry]) => {
      if (entry.isIntersecting) {
        setHasEntered(true);
        observer.disconnect();
      }
    });

    observer.observe(el);
    return () => observer.disconnect();
  }, [hasEntered]);

  return hasEntered;
}

这使得 UI 保持响应，并避免大量请求冲击后端。我们经常遇到多个用户同时访问同一文档的情况，因此我们实现了批处理、防抖和缓存，以避免对同一个块进行多次重复翻译。

每个段落或块都会显示一个状态指示器：加载中、失败或已翻译。如果由于翻译资源限制或模型限制导致翻译失败，用户只需点击一下即可重试失败的翻译。

5. 处理协作与实时编辑

Dossier 是一个协同编辑器，多个用户可以同时编辑同一文档。如果在查看翻译时有人编辑了某个段落，我们会检测到更改、使缓存的翻译失效，并重新加载更新后的文本进行翻译。

我们使用按块缓存（以元素 ID 为键)来跟踪翻译状态。当某个块被编辑时，只要该块仍在用户的屏幕上，其缓存条目就会失效并重新触发翻译。这确保了即使在文档不断演变的情况下，翻译视图也能始终保持最新。

6. 剥离格式以保障翻译质量

这里有一个艰难的权衡：为了获得最佳的翻译结果，我们在将文本发送给 LMS 之前会剥离所有内联格式（加粗、斜体、超链接、提及)。这是因为翻译模型在获得完整、未被打断的句子上下文时效果最好。

如果我们围绕格式拆分文本，比如翻译“I love frontend engineering”时，将“I”和“frontend engineering”单独发送，而将“love”作为加粗的块发送，翻译模型就会丢失句子结构。这可能导致错误或荒谬的结果，尤其是在词序或词形变化取决于整个句子的语言中。

示例：

考虑以下英文句子：

I love frontend engineering

假设我们尝试通过拆分来保留格式：

  • “I”
  • “love”
  • “frontend engineering”

如果我们将这些片段分别翻译成法语：

  • “I” → “Je”
  • “love” → “amour”（名词，而非动词)
  • “frontend engineering” → “ingénierie frontend”

如果我们将其重新拼接起来：

Je amour ingénierie frontend

这不仅不正确，而且毫无意义。但如果我们把完整的句子发给翻译模型：

“I love frontend engineering.” → “J’adore l’ingénierie frontend.”

在这里，“J’adore”是正确的动词形式，句子读起来也很自然。这就是为什么保留句子上下文对翻译质量至关重要，也是我们目前在翻译前剥离格式的原因，否则用户可能会看到令人困惑或具有误导性的结果。

正因如此，目前的翻译视图是纯文本，但我们正在探索更多方法，以便随着模型和 API 支持的改进，保留更多的格式。例如，未来支持对 HTML 或 Markdown 感知的翻译。

这项工作的有趣之处在哪？

该项目涵盖了前端开发的多个方面：

  • 组件设计：通过让生态系统中的下一个基于文本的应用无缝接入为 Dossier 完成的组件工作，为其成功做好准备。
  • 状态管理：在数百个块之间协调翻译状态，并提供实时更新。
  • 性能工程：利用浏览器 API（如 Intersection Observer 和 LRU 缓存)保持对用户的极速响应，同时善待我们的后端服务器。
  • 安全与部署：确保一切都能在物理隔离、涉密或其他受限网络中正常运行。

成果：为每个 Dossier 提供翻译

如今，用户可以在任何 Dossier 中切换翻译模式，选择源语言和目标语言，并即时查看文档的只读翻译版本。所有文本内容都会随着你的滚动进行“延迟”翻译，任何传入的编辑都会实时重新翻译。
