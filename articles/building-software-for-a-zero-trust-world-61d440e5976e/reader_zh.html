Building Software for a Zero Trust World

2023-04-19 · Original →

自 2003 年成立以来，Palantir 一直与政府机构以及处于高度监管行业内的商业客户合作，致力于从他们最敏感的数据资产中挖掘价值。在此过程中，我们开创了许多如今被称为零信任的安全理念与实践。

时至今日，安全依然是我们的产品开发、企业文化和内部运营的基石。这也是我们软件的核心差异化优势所在——它提供一流的安全与数据治理能力，使客户能够充满自信地做出数据驱动的运营决策。

什么是零信任？

在网络安全威胁快速演变的今天，我们认为绝不应默认授予信任。相反，应在应用程序和基础设施的各个环节对信任进行持续评估和验证。这一被称为“零信任”的理念，是对“纵深防御”和“最小权限”等长期安全概念的演进与综合。

零信任架构是一种企业安全框架，它利用关于身份和资源的上下文信息，制定策略并在整个企业或机构内动态执行。

Palantir 的零信任策略，与美国政府国防部（DOD）《零信任参考架构》中指导定义与示例的理念一脉相承。该国防部参考架构确定了零信任理念的五个核心原则，转述如下：

  1. 假设环境处于敌对状态：将用户、设备和网络视为不可信。
  2. 假定已被入侵：在运营和防御资源时，假设对手已经存在于您的环境中。
  3. 永不信任，始终验证：默认拒绝访问。确保每台设备、每个用户、每个应用程序和每条数据流都经过身份验证和授权。
  4. 显式审查：应以安全方式访问所有资源，并使用多种因素来判定置信度。对资源的访问是有条件的，可根据置信度和上下文动态调整。
  5. 应用统一分析：应使用统一分析（包括行为分析）来监控数据、应用程序、资产和服务。每笔交易均应记录以备分析与审计。

以上述五个核心原则为基础，我们可以通过考量《NIST 特别出版物 800-207》中定义的两个关键组件，从架构层面理解零信任：

  • 策略决策点（PDP）：将上下文信息输入信任算法，以作出针对某资源访问权限的决策；以及
  • 策略执行点（PEP）：根据 PDP 的决策，启用或禁用对一组资源的访问。

策略决策点的一个实施示例是 Azure Active Directory 利用“条件访问”功能。应用程序和基础设施可以在实施策略执行点时，利用策略决策点提供的上下文。我们的软件平台在应用程序组件之间以及托管基础设施中均设有多个策略执行点；本文将深入探讨这些细节。

各组织可用于描述其零信任策略并指导其实现成熟的第三大架构概念，是隐式信任区。

隐式信任区

隐式信任区描述了 PDP 与 PEP 之间存在的受保护资源集合，以及在存在多个执行点时，各 PEP 之间的区域。

谨慎定义隐式信任区，有助于组织在零信任策略上实现迭代改进。它们可以通过力求缩小信任区来实现这一目标，即减少无需重新评估即可访问的资源。换言之，企业可以设法减少存在于 PEP 之间的资源范围，如此一来，便能不断趋近于确保系统内的每一次交互都接受上下文信息审查这一理想目标。

以下摘自 NIST SP 800-27 的示例很好地说明了这一点：

“以机场的旅客安检模型为例。所有旅客通过机场安检站（PDP/PEP）才能进入登机口。旅客、机场员工、机组人员等在航站楼区域走动，所有人都被视为可信。在此模型中，登机区即为隐式信任区。”

Palantir 的零信任

Palantir 企业安全

在过去 24 个月里，为了保护 Palantir 自身（包括我们的软件开发流程），我们大幅提升了自身企业环境中的零信任架构成熟度。鉴于供应链安全在任何组织的整体安全态势中都发挥着至关重要的作用，我们认为政府及商业企业应评估其供应商和服务提供商在企业零信任方面的投资与成熟度。

国防部《零信任策略》强调了“自动化动态策略”，这正是我们在企业环境中日常使用的功能，被视为一种高级别的零信任能力。我们是 Microsoft Azure Active Directory 条件访问作为策略决策点（PDP）的重度用户。我们特意利用条件访问功能来确保发起身份验证请求的设备符合要求，并禁止来自其他设备的验证。我们能够获取每台设备上软件包和安全更新的信号，并确保合规性设置落实到位。

作为零信任策略决策点，Azure Active Directory 使我们能够确保用户请求访问的应用程序所处的源网络和地理位置是可接受的，包括满足其所需要的权限级别。在下游，许多应用程序位于代理服务之后，这些服务实现了策略执行点，并允许对向服务发出的请求进行持续评估。

我们还在企业环境中实行了全面的多因素身份验证——使用 YubiKey Series 5 FIPS 140-2 加密令牌实现无密码验证，以进一步提升我们对身份验证功能的信任度。您可以在此处阅读更多关于我们迈向无密码验证历程的内容。

保护我们的客户

在 Palantir，我们深切致力于保障客户的安全。尽管我们始终在响应技术能力的发展并了解新的对手技术，不断改进企业安全，但目前我们对零信任的大部分投资集中在客户的网络及其数据上（在下文中简称为“生产”环境）。

Palantir 的软件以及我们的生产托管基础设施在设计上既能独立保证安全，又能与客户的企业网络和身份基础设施无缝集成。没有任何单一的工具、解决方案或服务提供商（包括 Palantir 在内）能够将组织的安全态势彻底转变为零信任。但 Palantir 的软件包含诸多功能，通过支持国防部《零信任策略》中确定的七大支柱，为构建零信任架构做出贡献：

  1. 用户
  2. 设备
  3. 应用与工作负载
  4. 数据
  5. 网络与环境
  6. 自动化与编排
  7. 可见性与分析

在本节的剩余部分，我们将更详细地描述 Palantir 的零信任策略如何为客户支持上述每一个支柱。

用户与设备

国防部零信任策略中的“用户”与“设备”支柱，核心在于用户与设备的身份验证和授权。Palantir 已将零信任设计作为这些层面的优先事项。

身份验证

无论请求来自哪个网络或设备，Palantir 软件都会将其视为不可信，直至其通过显式身份验证。我们的软件开发团队利用持续集成（CI）来确保任何试图添加未经验证端点的行为都会被标记并拒绝，直至问题解决。托管基础设施的变更也采用类似方法。Palantir 基础设施以代码形式定义，变更须通过 CI 检查，以确保不会因每次变更而引入特定条件。我们使用商业云和基础设施扫描产品作为兜底手段，一旦在任何生产环境中检测到未经验证的资源或配置错误，便会向事件响应团队发出警报。

我们在零信任架构中评估并设计了多种身份验证场景，并为每种场景制定了成熟路线图。主要场景包括：

  1. 许多客户选择提供自己的零信任策略决策点和执行点。这些通常属于企业身份提供商（IDP）或身份、凭证与访问管理（ICAM）解决方案的一部分，但也可扩展至包含 ZScaler、Microsoft 的 Azure 应用程序代理以及 Cloudflare 的零信任网络访问（ZTNA）等解决方案中的零信任功能。
  2. 部分客户选择依赖我们团队管理的基础设施来获取类似功能。我们承担 IDP 角色，并将其功能作为零信任策略决策点加以利用。
  3. 我们对托管基础设施也负有管理和支持责任，并为此场景设计了零信任身份验证方法。

为了提供更多关于其实际运作方式的细节，我们可以对最后一种场景进行更详细的阐述。

Palantir 为管理员向生产系统进行身份验证而选定的身份提供商（IDP），是一个在无密码模式下运行的 Azure Active Directory GCC 高级别租户。我们将 YubiKey Series 5 FIPS 140-2 加密设备作为 Palantir 员工身份验证的唯一支持方式，旨在执行“生产”环境中的管理任务。IDP 充当 PDP，并在提供可由应用程序和托管基础设施中的 PEP 使用的令牌之前，验证一系列条件。我们评估的部分条件包括：

  • 发起身份验证请求的设备已知并已注册到我们的租户；
  • 设备已安装最新的安全更新，且未托管任何对于所请求的生产环境访问级别而言不适当的软件包；
  • 用户和设备所处的网络或地理位置，对于其尝试访问的应用程序或基础设施组件而言是可接受的；
  • 设备从安全角度被视为“健康”。宏观而言，这意味着该设备在我们的突发事件响应平台中没有任何未处理的警报或检测。

授权

一旦确信地掌握了谁在请求访问，我们就需要持续评估他们应该拥有哪些访问权限。传统授权是指将用户身份和所请求的资源与该用户被允许访问的资源集进行比较的过程。在零信任模型中，我们将此决策扩展至包含上述策略决策点提供的一些属性。

以指示用户是否拥有资源访问权限的传统访问控制列表（ACL）为例。为了与我们的零信任策略保持一致，我们还必须考虑是否应允许用户从其当前所处的网络进行访问。我们必须确保她的培训和合规确认是最新的。这些额外信号由应用程序中的策略执行点角色消费，对传统 ACL 构成了重大增强。

除了 Microsoft 开箱即用提供的工具外，我们实施的一项关键增强措施是使用自定义辅助作业和函数，为我们的策略决策点提供额外的上下文信息，这些信息可包含在应用程序和基础设施接收的令牌中。我们从合规服务中获取此上下文，该服务从多个系统中提取员工数据，以丰富我们零信任基础设施可用的信息。

此额外上下文包含以下因素：

  • 用户的公民身份是否与其尝试操作的环境要求相匹配；
  • 他们在基础设施团队的任期是否达到特定门槛；
  • 团队负责人是否已担保某位成员确实有正当的访问需求。

此额外上下文包含在从 Azure GCC High 租户中的策略决策点向下传递至 Palantir 应用程序策略执行点的令牌中，最终由策略执行点允许或拒绝访问。

身份验证完成后，用户上下文的任何变化都会导致在策略决策点进行直接且基本透明的重新验证。一个常见的例子是检测到客户端源网络发生变化。这可能只是用户在旅行热点和 WiFi 网络之间漫游这种无害行为，但也可能表明令牌被盗并企图被重复使用。无论哪种情况，请求都需要在策略执行点重新处理。未能重新验证请求将导致令牌失效。

为了使我们的零信任架构更具适应性，我们设计了一种模块化策略执行点，可以“插入”隐式信任区之间。随着我们整体产品架构的成熟，新功能不断添加，信任边界被重新定义，实例也可以被重新部署以适应这些不断发展的需求。在目前的实现中，我们的模块化 PEP 旨在识别状态变化，例如检测到先前批准的客户端在通过策略决策点时出现了新的 IP 地址，或者尝试执行的管理操作需要较高级别的权限。在这两种情况下，请求都会通过策略执行点透明地重新路由以进行重新评估。

应用与工作负载

Palantir 在全公司范围内采用了软件开发生命周期（SDLC）来管理产品的软件开发过程，旨在生产高质量软件，同时将处理错误和安全漏洞降至最低。SDLC 是一个基于敏捷方法的迭代过程，专注于适应性开发，并通过在开发过程中的多个节点整合反馈来确保客户满意度。Palantir 的 SDLC 涵盖：

  • 协作式的路线图规划和严格的设计审查流程；
  • 自动化测试与验证，以在开发过程中实现持续集成（CI）；
  • 分层的治理检查，以确保软件在部署前安全且正确；
  • 通过 Palantir 的自主部署平台 Palantir Apollo，实现软件的自动化部署、更新和召回；
  • 对性能、稳定性和安全进行持续监控。

“应用与工作负载”支柱的一个基本要素，是监控在软件开发过程中不可避免地引入或发现的零信任架构漏洞。Palantir 的全职应用安全团队维护着一项全面的漏洞管理计划，用于识别和修复漏洞。Palantir 还开发了容器漏洞扫描器（CVS），提供对 Palantir 软件中潜在漏洞的深度检测和可见性。结合 Palantir Apollo，这能对许多类别的漏洞实现自动修复操作。

数据

Palantir 通过确保客户数据在静态和传输过程中保持加密，并通过数据标记、标签和访问控制来赋能客户的数据治理，从而支持“数据”支柱。

我们的软件对传输中的数据实施强制加密原则。所有网络连接均使用可配置的强密码算法列表来执行 TLS 加密（1.2 及以上版本）。即使可信用户建立的网络连接穿过敌对环境（例如公共互联网），这种应用层加密也能确保数据的私密性。Palantir 还对静态数据实施强制加密原则。这是践行“假定已被入侵”原则的一种方式——即使对手获得了存储数据的服务器和硬盘的物理访问权限，他们也无法窃取信息。

Palantir 软件还实现了以数据为中心的安全模型，支持细粒度的数据标记、标签和访问控制。这种灵活的安全模型包含强制访问控制（MAC）和自主访问控制（DAC），可用于配置基于属性的访问控制（ABAC）和基于角色的访问控制（RBAC），以根据客户要求保护数据。Palantir 平台在存储、处理和传输过程中，始终保持数据标签、标签和访问控制策略与信息对象的关联。访问控制策略是细粒度的，可在项目（文件夹）、数据集（表）、对象（行）或属性（单元格）级别应用。这种访问控制模型将策略执行点引入应用程序内部，从而将受保护数据周围的隐式信任区缩小至最小尺寸。

网络与环境

我们通过在主机、容器和网络级别实施带有防火墙的微分段，来支持“网络与环境”支柱。微分段缩小了每个受保护资源周围的隐式信任区。进出我们软件的所有网络流量，都必须遵守一系列假设环境处于敌对状态并审查所有流量的网络安全控制。所有发往面向互联网服务的传入请求，均由 Web 应用防火墙（WAF）检查，该防火墙可防御 OWASP Top 10 等类似攻击，实施平台特定保护，缓解拒绝服务攻击企图，并拒绝未经授权的连接。

可见性与分析

零信任范式强调，静态的安全方法不足以应对现实中不断适应的威胁。因此，零信任架构的一个关键组件是实施动态检查，持续评估当前策略和配置的有效性，并随着威胁的演变对其进行更新。

Palantir 的零信任架构包含一套动态的自动化工具，用于检测和缓解恶意活动，包括 Web 应用防火墙（WAF）、网络入侵检测系统（NIDS）、基于主机的入侵检测系统（HIDS）、云安全态势管理（CSPM）、执行策略的服务网格，以及主机、容器和网络级别的防火墙。所有这些工具发出的警报和遥测数据均被输入至安全信息与事件管理（SIEM）系统中，该系统为 Palantir 的信息安全团队提供了对架构的全局可见性，使其能够实时预防、检测、缓解和响应威胁。

自动化与编排

Palantir 在整个零信任架构中实施了自动化与编排，包括在请求于隐式信任区之间移动时，利用策略决策点在授权和检查任务中提供的信号。Palantir 的信息安全团队还包含一个专门的安全运营中心（SOC）和事件响应能力。我们还将 SIEM 与安全编排、自动化与响应（SOAR）系统相配合，使得信息安全团队能够在发生安全事件时，使用自动化的行动手册和 API 驱动的操作快速响应。

超越零信任

Palantir 不断寻找创新方式来扩展零信任范式，即便这意味着需要彻底重新思考我们的基础设施。举例而言，我们必须“假定已被入侵”的原则，是我们开发 Rubix——Palantir 基于 Kubernetes 的云原生计算架构——的重要驱动因素。在临时性基础设施上运行 Rubix 这项高投入、高回报的决策意味着，即使发生入侵，攻击者想要维持立足点也会面临巨大困难——尤其是在结合严格的出站限制的情况下。临时性基础设施意味着每台主机会定期被销毁，并从可信镜像中重建，因此即使是最高明的恶意软件，也无法在主机被销毁后继续驻留。

我们还计划在未来 12 个月内对零信任策略决策点进行进一步的重大投资，并致力于实现性能优化，以便我们进一步缩小每个网络和应用程序内的隐式信任区。

支持并集成客户引入的第三方零信任组件始终是我们的首要任务，我们将继续向带来新技术的合作伙伴学习。

结论

简而言之，零信任承认网络环境处于敌对状态，威胁行为者不会被静态策略或单一加固的边界所震慑或击败。应对这一挑战需要动态且无处不在的防护措施，而自成立以来，Palantir 正是围绕这一原则开展业务的。

零信任原则（包括身份验证、上下文感知授权和检查）始终是我们开发团队关注的重点。跨越数十个行业，并与客户自身的网络安全和企业架构团队通力合作，我们正在帮助构建零信任架构，以保护世界上最受保护的数据。
