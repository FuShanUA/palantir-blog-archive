PhishCatch: Detecting password reuse from the inside out

2021-03-23 · Original →

用于强制执行密码规范的浏览器扩展

为了加强 Palantir 的密码规范，我们的安全团队在 SpecterOps 团队的深入协作下，致力于打造一种解决方案，旨在即使终端不在网络遥测检测范围内，也能检测密码复用情况。随着 COVID-19 的爆发以及绝大多数员工转向远程办公，这一需求变得尤为迫切。例如，许多 Palantir 员工的网络条件或带宽根本不足以使用全隧道 VPN。结果就是，我们全套的网络安全控制和检测手段（包括密码复用检测）可能只覆盖了他们极少部分的网络流量。

在考虑了多种解决该问题的方法后，我们发现浏览器扩展最能有效减少整个终端设备群体的密码复用和网络钓鱼风险。PhishCatch 本质上是一款浏览器扩展，它将用于企业资源的密码与在其他地方使用的密码进行比较，但比以往类似工具更加稳健和通用。这篇博文解释了我们开发 PhishCatch 的动机，并介绍了其底层的工作原理。

PhishCatch 现已在 Chrome 网上应用店上架。此外，作为我们对开源安全承诺的一部分，我们正在公开的 GitHub 页面上发布 PhishCatch 的源代码、策略模板及相关文档。我们将负责披露计划的范围扩展至包含 PhishCatch，并承诺定期进行应用安全评估以验证该扩展的安全性，同时公开评估结果。我们深切希望其他网络防御者能够使用 PhishCatch，帮助其环境抵御网络钓鱼和低劣的凭证管理习惯。

企业中的密码复用

良好的密码规范（或其缺失）对任何组织的安全态势都有着极其重大的影响。身份验证流程的改进（例如强制多因素身份验证、密码审计与禁用列表、账户锁定阈值和无密码身份验证）使得账户盗用既更加困难，破坏性也更低。然而，大多数系统从根本上仍然依赖密码进行身份验证。

浏览器日益成为处理工作和个人事务的主要场所，且常常同时进行。桌面应用程序和命令行界面服务于更专业的功能，但大多数用户一天的时间都在完全通过浏览器与企业及生产力应用交互。单点登录 (SSO) 帮助企业应对了 Web 生态系统固有的身份验证复杂性，但这同时也将风险集中到了一个易受网络钓鱼的单点故障上。

Google 在 2019 年 12 月的一项研究表明，只有 35% 的受访者在他们使用的每个账户上都使用了唯一密码——更令人担忧的是，有 13% 的人在所有账户上都复用了同一个密码。

在传统的（即 COVID-19 之前的）办公环境中，安全团队可以在网络边界部署安全工具，以控制和监控员工行为。在更复杂的设置中，这些工具会对网络流量进行解密，以识别密码复用或窃取行为。既然远程办公已成为新常态，传统的密码复用检测策略就已失效或完全无关紧要。

为了更好地理解问题的范围，让我们先来看看当前的风险态势：

密码泄露的三种途径

密码泄露很少孤立发生——人类行为是主要因素，无论是出于故意还是单纯的人为失误。

蓄意的账户盗用： 不法分子想要你的密码，并设计方法诱骗你交出密码。最常见的形式是发送网络钓鱼邮件，诱导用户访问虚假的登录页面，受害者在其中输入真实的凭证。

密码复用： 计算机系统要求人们按照对人类来说极难记忆的规则来设置密码。现实世界中常见的做法是创建少数几个满足大多数复杂性要求的密码，并根据需要在它们之间轮换。不幸的是，这也意味着一旦某个站点被攻破，就可能导致其他所有站点也被攻破——如果其中一个复用密码的站点是企业环境，这就是一个尤为棘手的业务风险。

肌肉记忆： 我们当中有谁没有心不在焉地在错误的地方输入过真实密码呢？登录失败，你对错误一笑置之，然后继续过你的日子——没造成什么伤害，对吧？尽管身份验证失败了，但无法完全保证没有发生泄露。该网站可能会以明文形式记录所有身份验证尝试，可能在运行键盘记录器，或者存在任何其他潜在的问题。没人喜欢轮换密码，但即使是由这种愚蠢的错误，也必须假定已被攻破。

阻止泄露的三种方法

幸运的是，人类行为在一定程度上是可以预见的，并且可以被善意地引导至期望的结果——或者至少在出现问题时可以进行监控。

防范网络钓鱼： 适当的安全工具可以保护用户免受威胁行为者的侵害。垃圾邮件过滤器可以在钓鱼邮件到达收件箱之前将其拦截，Web 过滤器可以在用户做出糟糕决定之前限制其对可疑域名的访问。然而，这些工具在很大程度上是被动的，可能无法预料到不法分子采取的新奇手段。

培训用户： 大多数组织都要求进行一定程度的安全意识培训，通常还会辅以经过授权的钓鱼演练来测试和培训员工。对抗密码复用最稳健的解决方案是使用像 1Password 或 LastPass 这样的数字密码管理器，它们可以根据需要为尽可能多的平台生成并存储强大的唯一密码。不幸的是，密码管理器并未被广泛使用，尤其是那些粗心或不太懂技术的用户（他们正是攻击者的理想目标）。

检测泄露： 像 Have I Been Pwned? 这样的服务会监控数据泄露中的用户信息，但这本质上是一种检测泄密的最后手段。许多泄露事件从未为公众所知，而且很难追溯特定泄露的源头。一种更先进且实时的选择是在企业网络边界监控网络流量。可以对网络流量进行检查、解密，并将提交的密码与哈希处理的域密码进行比较。这种策略很复杂，需要大量硬件，要求组织管理公钥基础设施，并会引发严重的隐私问题。

控制措施失败的三种方式

不幸的是，即使是最好的安全措施也可能不准确、被轻易绕过，或者仅仅是因为不够稳健而无法正确处理意料之外的边缘情况。

电子邮件盗用： 合法的电子邮件地址被不法分子接管并用于发送钓鱼邮件。这些邮件可能会躲过安全工具的检测，而之前建立的任何关系都会为该请求增加合法性。

域名复用： 一个已知且 previously-trustworthy 的外部域名被不法分子接管，可能是因为所有者疏忽了域名续费，或者是因为不法分子能够在不被发现的情况下暗中上传了自己的内容。另外，恶意内容也可能托管在通用的共享 Web 托管域名上，例如 azurewebsites.net。这些域名不太可能被 Web 内容过滤器足够快地检测和拦截以防止滥用。

监控被规避： 网络边界的安全工具和监控非常脆弱。出于隐私考虑，组织可能会选择不对敏感网站类别（如银行或医疗保健）的网络流量进行监控——有动机的不法分子可以利用这一点，提前将一个良性网站归入此类，随后再替换为恶意内容。密码提交也很容易不被注意，要么是因为提交方式不符合预期，要么是因为网站可能不以明文形式提交密码。最后且最明显的一点是，如果用户不在网络上，再多的基于网络的安全工具也无法保护他们——在当前这个全天候远程办公的世界里，任何人只要断开 VPN 就失去了保护。

新安全工具的目标

考虑到当前的风险态势和终端用户最可能的行为，我们开发了一个框架，我们相信 PhishCatch 浏览器扩展和告警服务器非常适合解决这些问题。

我们的技术目标是：

  • 无论网络状态如何都能检测密码复用事件：
    即使在未连接到企业网络时（例如关闭 VPN 或离开办公室），保护措施仍然有效，且不强制要求任何企业面向互联网的基础设施，以免造成额外的攻击面。

  • 所有复用检测均在本地执行：
    凭证绝不会以任何形式通过网络传输——毫无例外。即使恶意行为者能够窥探来自终端的任何网络流量，也不会增加实质性风险。

  • 支持多对凭证：
    大多数现有的密码复用工具都假设员工只有一个企业密码，但这不符合针对特权或敏感工作流采用账户隔离的最佳实践。

  • 最小化误报：
    这对于可能响应告警的事件响应团队来说最重要，但对终端用户不产生不必要的摩擦也至关重要。

  • 延伸目标：首先防止泄露发生：
    减少因密码泄露或复用导致的安全事件。这是事件响应者和用户双方理想的结局。

在可信度和可用性方面，我们也有一些理念上的目标：

  • 保护个人账户的隐私：
    员工对隐私的担忧是合理的，因此我们希望从根本上保证用户的个人凭证保持私密——同时不妥协企业安全。

  • 低门槛的部署和配置管理：
    让部署安全团队的测试和更新工具尽可能简单——不需要新的仪表盘、专用的查询语言或编写代码。

  • 无缝的用户体验：
    该工具无需任何用户交互即可运行——毕竟，一个无形的安全工具才不会让感到挫败的用户想要去绕过它。

PhishCatch 初次亮相

PhishCatch 是一款开源的 Chrome 浏览器扩展和后端告警 API 服务器，用于识别和防止企业密码泄露。

在浏览器中检测密码复用

PhishCatch 的主要配置是一个企业域名列表，用户预期会在这些域名输入其企业凭证。该列表可能包括组织的主要身份提供者 (IdP)，例如 Azure AD、Active Directory 联合服务、Salesforce、Okta 等，但也可能包括未启用单点登录的独立域名。

当在企业域名列表中的网站上输入凭证时，PhishCatch 会捕获密码输入，对其进行哈希处理，并使用每个密钥对应的盐值在本地（即在浏览器存储中）存储该哈希值。PhishCatch 还会保存与该域名关联的输入用户名。此数据默认在浏览器存储中保存 30 天后过期，尽管可以选择将此滚动窗口配置为任何持续时间。

当在不在企业域名列表中的网站上输入凭证时，PhishCatch 会再次捕获密码输入，但这次会使用存储的企业哈希中的盐值进行哈希处理，然后将结果哈希与存储的企业哈希进行比较。如果匹配，则表明企业密码已在非企业域名上输入，并触发复用告警（详见下文）。如果存储了多个企业哈希，PhishCatch 会依次使用每个哈希的盐值，以便与每个唯一加盐的哈希进行比较。

PhishCatch 会在每次击键时对输入到非企业域名的密码进行哈希和比较，而不是等待输入完成，这样即使外部密码仅仅包含企业密码，也能被识别出来。此功能是多线程的，并且与浏览器中呈现的页面完全分离，因此不会影响用户体验，也最小化了时序攻击的攻击面。

PhishCatch 还会对在非企业域名上输入的用户名进行哈希处理，并将其与缓存的哈希进行比较。这可以检测并防止用户意外将密码输入到用户名字段中。由于用户名字段在事件日志中通常不进行哈希处理或受到保护，这有助于防止凭证的进一步明文泄露。

来自非企业域名的哈希绝不会在本地存储或远程传输。只有企业域名的密码哈希在本地缓存，并且所有比较均在本地执行，因此在所有域名中维护了终端用户的隐私，同时也最大程度减少了对额外计算资源的需求。

关于密码复用的告警

PhishCatch 具有两种可配置的告警机制。

终端告警 会通过弹出通知立即通知受影响的用户，以便他们采取相应行动。

远程告警 对于安全团队和大规模监控更有用。PhishCatch 扩展旨在通过 API 调用将告警元数据（如用户名、外部域名和时间戳）发送到后端服务器。该服务器可以进行本机配置，随后将传入的告警中继到 Slack 或 Microsoft Teams 的 Webhooks 中。安全团队可以直接监控服务器日志，或将其摄取到 SIEM（如 Splunk 或 Elastic）中。

浏览器扩展和 API 服务器需要配置相同的预共享密钥 (PSK)，以提供基本的防滥用基准。我们目前正在研究更强的身份验证机制，以帮助防止机会性滥用。

只有关于复用事件的元数据（用户名、外部域名、引荐来源、时间戳等）被发送到服务器进行日志记录和告警。哈希仅存储在本地，绝不会发送到 PhishCatch 服务器或任何其他外部位置。

如果无法连接到 PhishCatch 服务器，扩展将缓存告警并尝试定期重传。这在 PhishCatch 服务器只能通过企业 VPN 访问，或者路由或网络连接暂时中断时非常有用。

密码的保护方式

PhishCatch 在保存到浏览器存储之前，使用了 PBKDF2（一种众所周知的抗暴力破解密钥派生函数）。默认情况下，它执行 100,000 次迭代的加盐 PBKDF2-HMAC-SHA512 哈希处理，尽管可以使用自定义值进行配置。这种哈希配置类似于 1Password 中对主密码执行的哈希处理。PhishCatch 还可以选择截断存储的哈希值，以进一步提高逆转存储哈希的难度。

PhishCatch 使用 Chrome 存储将企业哈希保存到磁盘。网站或其他扩展无法访问此存储。PhishCatch 开发团队认为 PhishCatch 的代码不包含任何源于浏览器的漏洞（例如跨站脚本攻击），不会让攻击者访问到存储的哈希。此外，如果攻击者能够直接访问你计算机的磁盘，我们认为目前从技术上讲逆转这些哈希是不可行的。

个人密码，或者更一般地说，在未被列为企业域名的站点上输入的密码，只在内存中进行哈希处理，绝不会持久化到磁盘上。

主动防止密码泄露

我们开发 PhishCatch 扩展的部分目标是帮助用户在输入密码之前识别恶意网站。

PhishCatch 通过获取所访问企业域名 HTML 代码的代表性哈希（称为模糊 DOM 哈希）并将其存储在本地来实现这一点。然后，该扩展对外部域名执行相同的哈希处理并进行比较。这可以识别克隆的或视觉上相似的域名，从而在用户察觉之前潜在地发现网络钓鱼活动。

与密码复用告警一样，只有企业域名的哈希在本地存储，并且如果匹配的哈希触发了告警，则只有关于该事件的元数据会被发送到 API 服务器。

大规模部署 PhishCatch

该扩展使用 Chrome 管理架构来获取部署配置设置。这些设置应使用适合终端的适当企业管理机制进行部署：对于加入域的 Windows 系统使用组策略，对于受管理的 macOS 系统使用 Jamf Pro 配置文件，或其他配置部署系统。

这具有几个明显的优势：

  • 扩展中没有硬编码的配置数据：
    PhishCatch 可以直接从 Chrome 网上应用店安装，无需从源代码编辑和构建你自己的扩展副本。如果你更愿意构建和部署自己的扩展副本，源代码可在我们的公开 GitHub 仓库中获取。

  • 无需自定义管理仪表盘：
    所有配置都直接在你已经在使用的企业管理平台中完成。

  • 无需账户：
    该扩展不需要加入到你的单点登录提供程序中，也不存储任何访问令牌——如果使用（可选的）API 服务器，则仅存储通过配置管理部署的明文 PSK。

  • 终端用户无需任何操作：
    PhishCatch 扩展是对用户网页浏览体验的一个被动、无形的补充（直到触发告警为止！）。

很少有其他 Chrome 扩展能将企业管理利用到这种程度，但我们认为这为组织和终端用户都提供了最佳体验。

在 Palantir，我们使用 Webhook 集成来持续关注发生的钓鱼报告。我们还将 PhishCatch 日志摄取到我们的 SIEM 中，并使用安全自动化直接通知用户，如果确认了钓鱼或凭证复用事件，则帮助他们走完凭证轮换流程。
