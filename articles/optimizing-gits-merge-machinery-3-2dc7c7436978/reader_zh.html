努力终有回报，懒惰立竿见影

面对在海量输入上运行 O(N²) 算法的代码，且希望使其运行更快时，大多数人大概会在脑海中浮现以下问题之一：

  • 是否存在可以在此采用的 O(N log N) 甚至 O(N) 的算法？
  • 即使无法改善算法复杂度，我们能否通过优化代码使其速度提升数倍？
  • 是否存在可以采用的近似方案，虽然解决的是稍有不同的问题，但能提供足够好的答案，并且速度更快？
  • 我们能否通过某种方式随时间推移缓存相关信息，从而在需要时降低计算成本？

你可能会注意到，本系列的前一篇博客文章本质上属于近似方案这一类。然而，我并不是按照发现这些优化的先后顺序来介绍的；第一篇博客文章实际上是我的第二个优化，而第二篇博客文章是我的第三个优化。我最初的优化并非来自回答上述任何一个问题，而是来自完全不同的另一类问题：

  • 如果我们干脆不费心做这个计算，情况会有多糟？在某些情况下我们能侥幸逃脱吗？

实际上，你可能会说我正在借用计算机科学中惰性求值的概念——即直到结果被请求时才进行求值——并试图将其提升一个档次，变成超级惰性求值，直接说：“不，你不需要那些；我干脆就不算它们了”——然后试图弄清楚在什么时候我可以这样做而不产生不良后果。

（目光敏锐的人可能会将我之前的两个优化以及当前的优化都视为“让 N 变小”的不同形式。前两个优化是通过寻找替代方法来计算或近似计算某些重命名来实现的，而当前的优化则是直接决定不去计算某些重命名。）

到底为什么需要重命名？

合并机制将重命名用于两个目的：

  • 三方内容合并：为了对文件进行三方内容合并，我们需要三个不同版本的文件。如果历史记录的其中一方重命名了文件，那么该文件的部分内容将出现在与合并基线不同的路径下。
  • 目录重命名检测：当历史记录的一方重命名了目录，而另一方在该目录中添加了新文件时，我们希望能够提醒用户需要选择是将这些新文件保留在旧目录中，还是移动到新目录中。

如果不进行重命名检测，被重命名的文件很可能会在旧文件名下显示为修改/删除冲突，并在新文件名下显示为新文件。两者都会出现在工作副本中，都包含编辑内容，合并基线中的原始文件版本将变得不明确，用户只能尝试手动调和两者。此外，如果我们不进行重命名检测，当旧目录中的所有其他文件都已移动到其他位置时，放入旧目录的新文件可能会被“遗留”在原地。

好吧，我们需要重命名；但我们需要所有的重命名吗？

如果为了进行三方内容合并或为了进行目录重命名检测而需要某个重命名，那么我们就需要它。让我们深入探讨这两种情况，看看它们是否在所有情况下都需要重命名，或者是否存在可以跳过重命名检测的情况：

三方内容合并

三方内容合并旨在调和双方历史记录中所做的更改。如果只有一方修改了文件内容，那么我们不需要进行三方内容合并。让我们看看两种可能性，以了解为什么会出现这种情况。简而言之：

  • 如果文件内容在发生重命名的一方未被修改 ⇒ 我们可以进行精确重命名检测，其成本很低
  • 如果文件内容在未发生重命名的一方未被修改 ⇒ 对该源文件进行重命名检测是无关紧要的

让我们更详细地逐一分析。我们使用两个文件名，old.c 和 new.c，以及如下简写的对象 ID（其中值 '000000' 用于表示该文件在对应提交中缺失）。第一种情况类似于这样：

                 old.c     new.c
   MERGE_BASE:   01d01d    000000
   MERGE_SIDE1:  000000    01d01d
   MERGE_SIDE2:  5e1ec7    000000

在这里，精确重命名检测将判定 MERGE_BASE 中的 old.c 和 MERGE_SIDE1 中的 new.c 是相关的（实际上是相同的）。这表明我们应该在 MERGE_BASE:old.c、MERGE_SIDE1:new.c 和 MERGE_SIDE2:old.c 之间进行三方内容合并。然而，这三个文件中有两个具有相同的内容（通过相同的哈希值注明），因此根本不需要进行三方内容合并；我们已经知道结果将是一个哈希值为 5e1ec7 的文件——我们已经在仓库中存储了这个文件。而关于重命名的知识让我们知道，在合并结果中，5e1ec7 是名为 new.c 的文件的哈希值，而 old.c 应该从结果中移除。

第二种情况稍微复杂一些，认为重命名无关紧要的说法可能令人惊讶。让我们来看看这种情况是什么样的：

                 old.c     new.c
   MERGE_BASE:   01d01d    000000
   MERGE_SIDE1:  01d01d    000000
   MERGE_SIDE2:  000000    5e1ec7

既然我声称对于这种情况重命名检测是无关紧要的，那么让我们比较一下检测到重命名和未检测到重命名分别会发生什么：

如果未检测到重命名 ⇒

  • 那么 old.c 看起来像是在一方未被修改而在另一方被删除，因此应该被移除。
  • new.c 看起来像是一个我们应该原样保留的新文件。

如果检测到了重命名 ⇒

  • 代码会在 MERGE_BASE:old.c、MERGE_SIDE1:old.c 和 MERGE_SIDE2:new.c 之间设置三方内容合并。其运行过程如下……
  • 由于 MERGE_BASE 和 MERGE_SIDE1 中的文件版本是相同的（都是哈希值为 01d01d 的文件），三方内容合并将准确产生简写对象 ID 为 5e1ec7 的文件版本。事实上，代码会查看哈希值并直接返回答案，甚至都不会去查看文件内容，因为它知道答案，并且知道仓库中已经存储了具有所需哈希值的文件。
  • 由于文件从 old.c 重命名为 new.c，它会在 new.c 处记录哈希值为 5e1ec7 的文件内容（这意味着将其原样保留），并移除 old.c。

如果你比较上述两个结果，你会发现它们得出了完全相同的结论，即使它们使用的逻辑不同。因此，无论我们是否检测 old.c 的重命名，结果都是一样的。我们无法先验地知道 new.c 会与哪个文件配对，以及该文件是否已被修改，因此在搜索重命名时，new.c 将不得不留在我们庞大的比较矩阵中。然而，old.c 在历史记录的一方未被修改，这一事实足以让我们知道它不需要被包含在内——至少不是为了三方内容合并的目的。

目录重命名检测

作为目录重命名检测的一部分，我们仅在以下情况下需要源文件的重命名：

  • 包含该文件的目录不再出现在我们正在检测重命名的那一方历史记录中（如果该目录仍然出现，说明它没有被重命名，至少在那方历史记录中没有）。
  • 在我们不检测重命名的那一方历史记录中，包含该文件的目录中添加了新文件（如果该目录中没有添加新文件，那么目录重命名检测就没有什么可以警告或移动到新位置的内容）。

此外，即使在这些情况下，我们也不需要重命名目录中所有源文件的重命名，我们只需要足够的重命名来满足“多数占优”规则即可。因此，如果我们通过精确重命名检测或基名引导的重命名检测（参见本系列之前的两篇博客文章）获得了足够的重命名，那么我们就不需要再检测更多的重命名——至少不是为了目录重命名检测的目的。

与三方内容合并非常相似，目录重命名检测的需求也集中在重命名源上，这为我们提供了一种将某些源文件标记为与重命名检测无关的方法。我们没有办法减少潜在的重命名目标数量。

抽象层

人们常说计算机科学中的每个问题都可以通过再加一层抽象来解决，除了抽象层过多这个问题本身。作为软件工程师，我们喜欢抽象事物；它使我们能够创建可重用的组件。然而，有时抽象会在程序的不同组件之间筑起一道无形的墙，后来我们发现自己想要创建一座横跨墙两端的建筑。还有些时候，我们的可重用组件是为了解决某个问题而设计的，但当我们将它们用于另一个问题时，会发现它们虽然为我们解决问题提供了足够的信息，但却提供了超出我们实际需要的内容。

在 git 中，有一个专门的组件负责重命名（和复制）检测。它是 diff 机制的一个子组件，该机制接收两棵树（一个起始状态和一个结束状态）并计算差异。这对于 git diff OLD_COMMIT NEW_COMMIT 或 git log -p 之类操作非常方便，因为它们想知道两个提交之间的所有差异；diff 机制能很好地解决这类问题。

合并需要两组重命名——从 MERGE_BASE 到 MERGE_SIDE1 以及从 MERGE_BASE 到 MERGE_SIDE2。因此，简单地重用我们优秀的组件两次来为每一方进行重命名检测似乎是顺理成章的。这给了我们需要的答案……但我认为这种抽象在很长一段时间内掩盖了优化的机会。注意，我们上面所做的所有关于何时可以跳过重命名检测的工作都涉及关于另一方历史记录的信息。Diff 组件不了解三个提交的情况；它只知道两个，因此它没有另一方历史记录的概念。

实现这些优化基本上意味着我们需要从合并组件向重命名检测组件传递大量额外信息，以便它能够根据这些额外信息做出决策。这同时也意味着将所有目录重命名检测逻辑从合并组件移至重命名检测组件中（尽管名称相似，但目录重命名检测一直以来都是合并特有的，而且由于它是一个后处理步骤，它一直处于重命名检测的范畴之外）。

这一切听起来真的很复杂且属于特殊情况；值得吗？

至此，你可能会注意到一些让你质疑此项优化可取性的因素：

  • 此优化仅对合并机制有帮助（因此适用于 merge、revert、cherry-pick 或 rebase 等 git 命令）；它对普通的 diff 系列命令没有帮助（因此对 diff、log 或 status 等 git 命令没有帮助）。
  • 此优化将涉及大量代码和对特殊情况的检查，以及跨越我们旧有的抽象边界传递大量信息。
  • 这一切都是为了文件在未重命名的一方未被修改且该文件不可能是目录重命名的一部分这些特殊情况。第二个条件可能很常见，但第一个听起来像是一种特殊情况。

那么，这值得吗？

在我看来，这是我所做的所有优化中最好的一项。为了解它为何如此强大，假设我们需要拣选一个旧提交，但与此同时上游已经重命名了 5K 个文件。让我们暂且忽略之前的优化（精确重命名检测和基名引导的重命名检测），或者假设在这些优化之后我们还剩下 5K 个文件。git 中的非精确重命名检测将需要进行 5K * 5K 次文件比较来匹配重命名对。然而，如果我们需要拣选的提交很典型，那么它只修改了约 10 个文件，并且很可能甚至没有添加任何新文件。对于这样的情况，我们的 5K * 5K 比较锐减至约 10 * 5K——缩减了约 200 倍。

对于我们感兴趣的测试用例，基名引导的重命名检测恰好非常有效，这意味着已经没有太多剩余的优化潜力了。但尽管如此，使用这种额外的优化仍然产生了一些相当惊人的结果：

                        Before                  After
mega-renames:    130.465 s ±  0.259 s    11.435 s ±  0.158 s

有趣的是，如果我们关闭上一篇博客文章中的基名引导优化，那么单靠这种优化就能为我们感兴趣的测试用例带来以下速度提升：

                        Before                  After
mega-renames:   1799.937 s ±  0.493 s    16.148 s ±  0.169 s

因此，至少对于典型的变基和拣选操作而言，它比基名引导的优化要强大得多。但最棒的是，某些路径只能从其中一种优化中受益，而我们可以针对任何给定的路径选择最适合的那一种。

有关此优化的更多详细信息，请参阅我的补丁接受审核的两个上游线程，即关于避免检测无关重命名的线程以及关于避免检测更多无关重命名的线程。

此优化将出现在 git-2.32 中，该版本计划于 2021 年 6 月发布。
