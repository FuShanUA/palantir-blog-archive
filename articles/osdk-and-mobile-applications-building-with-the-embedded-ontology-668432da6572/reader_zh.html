OSDK and Mobile Applications: Building with the Embedded Ontology

2026-03-31 · Original →

嵌入式本体（Embedded Ontology）让你能够为在边缘作业的团队构建强大的企业级应用。在设备本地运行完整的、具备上下文信息的本体。在行动触点上，发挥 Palantir 的强大威力。

传统企业级平台功能强大。它们汇聚数据、执行数据治理、编排工作流，并为企业提供“统一全景视图”来运营业务。但这幅全景图挂在温控办公室里，连着稳定可靠的 Wi-Fi，呈现在宽大的显示器上。

现在，换个视角，跟随外勤服务技术员体验一番。他们驱车穿梭于各个站点之间，在没有手机信号的机房里检查设备，站在轰鸣的暖通空调设备旁，用平板电脑记录检查结果，繁杂任务不一而足。边缘侧的真实情况截然不同：网络连接最多算时断时续，延迟是最大的敌人，而手持设备的人需要立刻做出决策，绝不是等转圈加载结束之后。

传统的移动端架构——那些完全基于 REST 调用和 API 优先模式构建的架构——在这些环境中会彻底崩盘。用户的每一次操作都变成一次网络请求。每一次网络请求都可能失败。每一次失败都会导致体验降级、被迫寻找替代方案，甚至更糟——错失决策良机。你可以硬塞入缓存层和重试队列，但底层假设依然是“服务器可达”。一旦不可达，应用就成了一具空壳。

嵌入式本体彻底颠覆了这一模式。它不再将移动设备视为窥探远程平台的瘦客户端，而是将其当作分布式数据系统中的一个节点。一个经过范围限定、具备丰富上下文信息的本体投影驻留在设备本地，囊括了用户实际所需的对象类型、关系和操作。读取在本地进行。写入在本地完成。网络允许时进行同步，重连时自动处理冲突解决。

在这篇文章中，我们将快速梳理嵌入式本体背后的架构、支撑其运转的思维模型，以及构建渐进式 Web 应用（PWA）的实操步骤。该应用由 OSDK 驱动，无论设备是否联网都能流畅运行。我们将以一个简单可用的示例——外勤服务技术员应用——来落地讲解。但这种模式适用于任何运营型组织的边缘场景：仓库、车辆、偏远站点等等。

通过本体实现本地与全局上下文

脱离了上下文，数据毫无用处。扫描出来的设备标签只是一串字符。但是，如果将扫描的标签与本地资产登记册匹配、与今日工单交叉验证，并作为与特定站点和服务合同关联的巡检事件记录下来……这就是可操作的情报。

两者的区别在于上下文，而上下文需要来自多个范围、多个源头的数据。这是嵌入式本体背后的核心设计原则：本地数据驱动实时决策；全局数据提供全局运营视角。从逻辑角度看，设备上的数据可视为存在于两个层级：

本地对象是设备的工作集。它们的创建、更新和查询延迟在毫秒级，完全不受联网状态影响。技术员记录一条巡检结果时，写入操作直接作用于本地存储。没有网络往返，没有转圈等待，没有体验降级。

全局上下文对象是组织背景板。它们变动不频繁，设备也不会向其写入数据，但它们对于赋予本地操作意义至关重要。AssetRegistry 告诉技术员该站点有哪些设备。WorkOrder 告诉他们此行的任务。ServiceContract 告诉他们合同覆盖范围。

联网时同步；断网时自主运行。

同步模型很直接：

  1. 首次连接时，设备引导启动本地存储，拉取所需的限定范围对象集。
  2. 联网期间，变更在后台进行双向同步。
  3. 断网期间，设备继续正常运行。读操作命中本地存储。写操作（操作动作、数据变更）作为意图在本地捕获。
  4. 重连时，排队的写操作向上游核对。冲突解决和状态核对自动完成。

关键在于，第 3 步并非降级模式。应用不会弹出横幅提示“您已离线，部分功能可能不可用”。应用照常运转。技术员甚至可能根本不知道自己是联网还是断网，这正是其要义所在。韧性是基线假设，而非附加功能。

概念：嵌入式本体

在动手构建之前，我们先确立思维模型。如果你刚接触 Palantir 的生态系统，本体（Ontology）是一个语义系统，将数据、逻辑、操作和安全原语融合在一起。它定义了：

  • 对象类型：领域中的名词（Asset、Technician、WorkOrder、InspectionEvent）
  • 属性：这些对象的特征（assetId、status、scheduledDate、severity）
  • 链接类型：对象间的关系（InspectionEvent → Asset、WorkOrder → Site）
  • 操作：改变状态的动词（logInspection、flagDefect、completeWorkOrder）

在典型的 Foundry 部署中，本体在 AIP + Foundry 服务网格内运行，供特定客户端查询。你的应用调用本体 API，获取返回的对象并进行渲染。这在网络连接稳定且能容忍网络延迟的情况下运作良好。嵌入式本体采取了不同路线：你无需远程查询本体，而是定义一个限定范围投影，即全量本体的一个子集，直接物化在设备本地。

该投影包含：

  • 对象集：仅包含该设备、用户或角色实际运行所需的对象类型和实例。外勤技术员的平板电脑不需要采购团队的供应商对象。
  • 本地操作和变更：在本地捕获并执行的写入意图，待网络连通时向上游核对。

与缓存的区别

你可能在想：“这听起来像多了几步操作的缓存。”并非如此，且区别至关重要。

缓存是一种性能优化手段。嵌入式本体是一种分布式数据架构。设备拥有自己的本地存储、自己的查询引擎以及自己的操作执行层。它通过同步——以及必要时通过实时 API 调用——参与到更广阔的本体中。

基础设置与开发

开始构建吧。我们将走完整个设置流程：在开发者控制台进行应用设置、搭建 PWA 脚手架、配置嵌入式本体以及实现应用逻辑。我们设定的场景是：技术员驱车前往客户站点检查、维护和修理设备。他们的工作环境网络连接极不稳定。应用需要展示今日工单、允许查阅资产详情、记录巡检结果、标记缺陷，并在行车上路或返回办公室时将所有数据同步回去。

前提条件

  • 拥有具备开发者控制台访问权限的 Foundry 环境
  • 拥有包含需在本地同步的对象类型（如 WorkOrder）的本体（或愿意自行创建）
  • 本地已安装 Node.js 和 npm（或使用 Foundry 中的 Code Workspaces）

步骤 1：创建应用

在开发者控制台中，创建一个新的面向客户端的应用：

  1. 选择你的本体以及应用所需的对象类型
  2. 生成你的 TypeScript SDK，它为你的对象类型和操作提供类型化绑定
  3. 配置带有子域名的网站托管（部署时需要用到）

详细步骤请查阅 Foundry 文档中的《创建嵌入式本体应用》。

步骤 2：使用嵌入式本体模板进行引导

通往可用脚手架的最快路径是预配置模板：

  1. 导航至 Start developing → Bootstrap in Foundry
  2. 从模板下拉菜单中选择 Embedded Ontology React
  3. 生成代码仓库

这将为你提供：

  • 预装好的 @palantir/lohi-ts（嵌入式本体）
  • 配置好支持 WebAssembly 的 Vite 插件
  • 可直接定制的同步模式示例

步骤 3：配置本地同步的对象

在此定义你的对象集，即驻留在设备上的本体限定范围投影（即特定的嵌入式本体）。打开 src/client.ts 并指定同步目标：

import { createClient } from "@palantir/lohi-ts";                                                                                                                                                           
import {                                                                                                                                                                                                   
  InspectionEvent,
  ServiceNote,                                                                                                                                                                                             
  DefectReport,                                                                                                                                                                                            
  WorkOrder,
  AssetRegistry,
  ServiceContract,
  SiteLayout,
} from "@your-app/sdk";

const client = createClient(
  import.meta.env.VITE_FOUNDRY_URL,
  import.meta.env.VITE_ONTOLOGY_RID,
  tokenProvider,
  "field-service-ontology",
  [
    // Reference data — snapshot sync all
    AssetRegistry,
    ServiceContract,
    SiteLayout,

    // Filtered snapshot — only open work orders
    {
    objectType: WorkOrder,
    objectSet: (c) => c(WorkOrder).where({ status: "Open" }),
    },

    // Peering — bidirectional sync for field-created objects
    {
    objectType: InspectionEvent,
    usePeering: true,
    },
    {
    objectType: ServiceNote,
    usePeering: true,
    },
    {
    objectType: DefectReport,
    usePeering: true,
    },
]
);

export default client;

步骤 4：实现同步模式

import { SyncState } from "@palantir/lohi-ts";
import { useState, useEffect } from "react";
import client from "./client";                                                                                                                                                                              

function FieldServiceApp() {                                                                                                                                                                                
const [ready, setReady] = useState(false);
const [syncError, setSyncError] = useState<string | null>(null);

useEffect(() => {
    (async () => {
    try {
        const state = await client.syncState();
        if (state !== "Ready") {
        await client.sync();
        }
        setReady(true);
    } catch (err) {
        setSyncError("Initial sync failed. Check connectivity and refresh.");
    }
    })();
}, []);

if (syncError) return <div className="sync-error">{syncError}</div>;
if (!ready) return <div className="sync-loading">Syncing field service data...</div>;

return <FieldServiceWorkflow />;
}

代码逻辑如下：

  1. client.syncState() 检查本地存储是否已引导启动。首次启动时尚未完成。
  2. client.sync() 拉取已配置的对象类型并填充本地存储。这是嵌入式本体在设备上的初始“注水”。
  3. 一旦 ready 变为 true，所有后续的读写操作都将命中本地存储。查询以毫秒级延迟执行。操作在本地捕获。设备完全自主。
  4. 后台同步负责持续核对，只要网络连通，便将本地写入推送至上游并拉取更新。

步骤 5：构建离线工作流

随着嵌入式本体引导完成，无论设备在线还是离线，你的应用代码看起来完全一样。以下是一个极度简化的巡检工作流：

import {        
    InspectionEvent,
    LogInspection,
    AssetRegistry,
    WorkOrder
  } from "@your-app/sdk";
  import type { Client } from "@palantir/lohi-ts";

  async function handleInspection(
    assetTag: string,
    workOrderId: string,
    client: Client
  ) {
    // 1. Look up the asset from the local registry (millisecond latency)
    const asset = await client(AssetRegistry)
      .where({ assetTag })
      .fetchOne();

    if (!asset) {
      return { status: "UNKNOWN_ASSET" as const, assetTag };
    }

    // 2. Validate against today's work order
    const workOrder = await client(WorkOrder)
      .where({ workOrderId })
      .fetchOne();

    // 3. Log the inspection event locally
    await client(LogInspection).applyAction({
      assetId: asset.assetId,
      workOrderId: workOrder.workOrderId,
      siteId: workOrder.siteId,
      technicianId: getCurrentTechnician(),
      timestamp: new Date().toISOString(),
      status: "INSPECTED",
      notes: "",
    });

    // 4. Return result to the UI immediately
    return { status: "OK" as const, asset, workOrder };
  }

注意这里没有什么：没有网络检查，没有针对 API 调用的 try/catch，没有针对离线模式的降级逻辑。代码既不知道也不关心联网状态。它查询本地本体，执行本地操作，然后返回。剩下的交给同步层处理。

步骤 6：作为 PWA 部署和安装

在你的代码仓库中打上发布标签，通过 UI 或运行几条 git 命令即可触发部署。例如：

git tag 1.0.0
git push origin 1.0.0 - tags

Foundry 的 CI 会构建你的应用并将其部署到你配置的子域名。用户随后可以直接从浏览器安装该 PWA：

  • 桌面端：在地址栏寻找安装图标
  • 移动端：从浏览器菜单中选择“添加到主屏幕”

安装完成后，该应用会像原生应用一样启动：全屏显示、自带图标，并借助嵌入式本体实现完全的离线功能。

结语

嵌入式本体代表了我们对边缘企业级应用思维方式的根本性转变。设备不再是窥视平台的视窗。在行动触点上，它本身就是平台。

当技术员在没有信号的地下室检查设备时，他们不应该苦等服务器响应。当他们在屋顶上记录缺陷时，不应该草草记在纸上留待后续录入。当他们在站点之间查阅资产的服务历史时，检索不应受制于手机信号。

韧性、速度和上下文智能理应处于边缘侧，嵌入式本体让这一切变得切实可行，而非纸上谈兵。在 Foundry 中驱动仪表盘和工作流的同一套强类型本体，如今在设备本地运行，包含相同的对象类型、相同的操作和相同的关系。开发体验几乎如出一辙。用户体验则比传统方式有了质的飞跃。

这仅仅是个开始。发展轨迹指向更智能的同步（基于连接模式和数据优先级的自适应策略）、AI 增强的本地操作（利用本地本体进行上下文感知决策的端侧模型），以及网格化本体（设备不仅与平台同步，还彼此同步）。除了 PWA 之外，嵌入式本体还可通过引入时序、关系型和对象存储来增强，并针对 AI 工作负载集成本地 LLM 和向量搜索能力。数据从接入到本体，最终流向能带来 ROI 的应用与服务，这一切皆可实现。

边缘正变得越来越智能，而本体是驱动其进化的引擎。
