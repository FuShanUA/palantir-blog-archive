Defensive Databases: Optimizing Index-Refresh Semantics

2025-09-23 · Original →

编者按：这是探讨 Palantir 如何定制基础设施软件以实现大规模可靠运行的系列文章的第一篇。本文由 Foundations 组织撰写——该组织负责支撑我们所有软件的基础技术，包括我们的存储基础设施——文章详细介绍了我们在不分叉源代码的情况下调整和定制 ES 的经验。我们有两个主要目标：一是展示我们的存储基础设施团队为确保 Palantir 的成功所做的工作；二是为 Elastic 社区提供讨论切入点，探讨这些修改如何纳入主线版本，或者 ES 如何为高级用户提供更强的可定制性。

引言

任何企业的大规模软件服务背后，都需要一套基础设施作为基石，支撑其他生态系统的运转。在 Palantir，Foundations 团队致力于为 Foundry、Apollo、Gotham 和 AIP 平台提供大规模软件交付服务。Foundations 广泛提供两类支持：一是用于构建和交付其他软件的基础设施（如 CI、RPC 标准化等）；二是可用于全平台数据持久化的存储基础设施软件（如 Cassandra、Oracle、Postgres 等）。此类基础设施的稳定运行至关重要，一旦该层发生故障，极易引发连锁反应。

Elasticsearch（简称 ES）是一款开源的分布式搜索与索引引擎，也是 Foundations 负责在存储基础设施层面运维和交付的技术之一。我们在各类环境中运维着至少 300 个 ES 集群，涵盖容器化与非容器化基础设施、物理隔离环境，并支撑跨政府与商业领域的业务流。ES API 功能强大，但其极高的灵活性使得工程师能够构建 ES 客户端，甚至是多级嵌套的客户端，这些客户端与 ES API 的交互方式可能会危及平台稳定性。我们在现场处理此类性能退化问题时积累的经验，以及无法预测服务网格中何时出现新的调用链并重蹈覆辙的困境，促使我们定制 Elasticsearch，以增强其防御能力。

在软件工程领域提到“防御性编程”时，你可能会想到增加更多的空指针检查、编写防范注入攻击的 SQL 代码，或者实施遥测，以便运维人员能在异常行为发生时及时检测并采取行动，防止即将来临的服务中断。这些例子都无可厚非——但如果需要增强防御性或可观测性的应用程序并非由你的组织编写呢？

Palantir 对全栈软件的弹性有着极高的标准。平台上的大多数服务器都支持归因和限流等机制，以防范不良访问模式，同时配备详细的遥测数据，接入我们的告警和可视化基础设施。在自身能力允许的范围内，对于非我们编写的软件，我们同样秉持同样严格的弹性标准。

在这篇博文中，我们将通过重写不当索引刷新策略的案例，展示如何通过插件增强 ES 的防御能力，而无需分叉 Elasticsearch 源代码。我们分享此案例，一方面是希望开源 Elasticsearch 项目未来能融入类似功能，另一方面也是为了分享 Palantir Foundations 存储基础设施开发团队的工作性质。

Palantir 中的 Elasticsearch

要理解我们为何需要增强 Elasticsearch 的防御能力，首先要了解 ES 在 Palantir 架构中的角色及其带来的挑战。

部分依赖 ES 的应用场景包括但不限于：

  • 用于在组织内查找资源的快速搜索
  • 国防工作流中的地理空间分析与操作
  • 批处理作业编排

Palantir 平台由微服务构成。其中一小部分微服务充当平台的数据库客户端服务，对外暴露类似数据库的 API，底层由存储软件支撑。这些服务可将数据放入关系型数据库等持久化存储中，并利用 Elasticsearch 作为二级索引提供搜索能力。数据库客户端可在其 API 中提供与 ES 相关的参数，这些参数会被直接转发给 Elasticsearch。

这些数据库客户端服务可能拥有任意数量的上游服务作为依赖。数据库客户端通常设计有灵活的 API，以支持多样化的业务流。当用户设定的条件与服务网格中各服务间的交互相互叠加时，往往很难预测何时会导致 ES API 的低效使用。“不良访问模式”指任何导致 API 提供方（此处为 ES）或其客户端发生性能退化的 API 使用方式。

此外，虽然对这些数据库承担运维责任的工程师可能具备一定的 ES 专业知识，但管理上游服务的工程师由于精力有限且专攻其他技术领域，往往对 ES 了解较少。

在最底层，我们的存储基础设施在维持防御性和稳定性方面往往面临最大挑战。此类软件多为开源且并非由我们编写，ES 也不例外。ES 有一套自己的遥测理念（基于 Elastic APM），与我们的遥测生态不兼容，且 ES 未对隐蔽的不良访问模式设置防护边界。当发现此类开源软件存在上述缺口时，我们的存储基础设施开发团队会提供新颖的解决方案，以满足我们严格的稳定性标准。

这些团队也致力于与开源社区协作。我们首先在内部构建这些定制化方案，以高速达成交付目标。理想情况下，长远来看，我们在该层做出的诸多改进可以贡献回主线代码。我们也可以提议增强 ES 插件框架，以便更轻松地支持我们的定制功能。在可行的情况下，我们尽量避免分叉软件，因为这不仅需要付出高昂的成本以保持分叉版本与主线代码的同步，还会影响我们与开源社区建立信任。

案例研究：重写不当的索引刷新策略

Elasticsearch 的核心在于将文档存储（索引）到索引中，从而高效检索这些文档的内容。我们将首先介绍如何让 Elasticsearch 在索引写入时抵御不良访问模式。

Elasticsearch 中的文档存储在分片中，分片对索引进行分区，使并行线程能够高效执行搜索。ES 中的分片以 Lucene 索引的形式持久化，每个 Lucene 索引进一步划分为段。需要注意的是，虽然段越多并行度越高，但过小的段会导致索引和搜索效率低下。

ES 会周期性地处理批量写入，并将其转换为大小最优的段。为此，ES 首先将写入缓冲到两个位置：

  1. 一个名为 translog 的基于磁盘的高效写入存储，数据在此被持久化，确保即使节点宕机，写入也不会丢失
  2. 一个内存索引缓冲区，使得转换为段的过程无需读取磁盘上的 translog

然而，此时数据尚不可搜索。数据会定期从内存索引缓冲区转移到段中，通过 ES 称为刷新的操作使其对搜索可见。后台执行索引刷新的周期称为刷新间隔，可通过索引设置进行配置。

由于将数据写入段存在延迟，索引化的数据可能无法立即用于搜索。ES 从根本上建立在最终一致性模型之上。尽管如此，索引 API 可通过以下刷新策略之一进行参数化，这些策略会导致新索引数据在可搜索性方面呈现以下后置条件：

1. None（默认策略）- 仅将数据持久化到内存索引缓冲区（和 translog）并立即返回。在分片下一次刷新之前，不保证写入的数据可被搜索到。

2. true（在 ES 源码中也称为 immediate）- 将写入持久化到内存索引缓冲区（和 translog），然后同步强制刷新被索引操作修改的分片。根据 ES 文档，通常不建议在测试环境之外使用此策略，但在实践中我们仍能看到它的应用，主要面向 UI 交互场景（此类场景要求索引化后即刻具备可搜索性）。

3. wait_for- 将数据持久化到内存索引缓冲区（和 translog），然后注册一个回调，该回调仅在 ES 完成后台索引刷新后才完成客户端 API 调用。这保证了一个后置条件：当 ES 返回结果时，写入的数据在搜索中可见，代价是客户端需要等待最多配置的刷新间隔时间才能完成 API 调用。

一个关键细节是，同一时刻仅允许一个线程对某个分片执行刷新，刷新代码路径受排他锁保护。

基于上述背景，我们现在可以探讨在使用刷新策略进行索引写入时可能出现、而 ES 原生未加防范的不良访问模式。

不良访问模式 1：并发同步刷新

ES 文档明确指出，使用同步刷新代码路径是一种反模式。我们将证明，滥用此选项不仅是表面上的反模式或轻微的性能退化那么简单。

Elasticsearch 的写入受限于固定大小的线程池，意味着同一时间仅有特定数量的线程可用于写入。使用同步刷新选项参数化的写入，会使用该线程池中的同一个写入线程来执行数据持久化和刷新操作。回顾前文，此刷新是在排他锁下完成的。所有这些因素叠加，使得以下场景成为可能：

  1. 假设分片（S0、S1……Sn）为某节点上的分片
  2. 某个分片 Sk 接收到大量配置了同步刷新的索引写入。（当写入热点集中在单个分片、分片配置不当，或者索引太小不值得分片以避免微小分片反模式时，都可能出现这种情况）
  3. 因此，任何此类写入线程都必须在 Sk 的刷新锁上阻塞
  4. 考虑同一节点上对任何其他分片 S' 的传入写入。由于所有线程都被 Sk 的写入占用，对 S' 的写入无法分配到线程
  5. 结果：该节点的写入线程池耗尽，所有写入都在等待刷新锁，没有其他写入能够继续推进

不良访问模式 2：长刷新间隔下的 wait_for

wait_for 是一项实用功能，可避免导致段结构糟糕的同步刷新，更不必说还能避免写入阻塞在排他锁上。其缺点是客户端需要等待后台刷新完成才能返回 API 调用。

客户端应当清楚索引上设置的刷新间隔，从而在使用 wait_for 选项时获得可预期的结果。然而，在复杂的工程组织中，不能指望所有人都对刷新间隔与写入刷新策略之间的交互了如指掌。在索引上设置刷新间隔的服务，可能并非决定写入这些索引刷新策略的服务。或者，有人编写代码调整索引写入的刷新策略时，并不了解生产环境中索引刷新间隔的当前设置，也不清楚可能改变刷新间隔的潜在代码路径。

当这些情况发生时，如果某个代码路径意外延长了刷新间隔，另一代码路径的写入 API 调用就会陷入停滞。如果刷新间隔被设置为 -1，写入 API 调用可能会无限期阻塞。

解决方案：在 TransportInterceptor 中更改不当刷新策略

期望行为

短期内，我们可以通过修补 ES 客户端的代码路径来阻止此类 ES API 交互的发生。然而，我们认为这不足以提供深度防御。服务网格演进迅速，随时可能出现对这些访问模式的新调用。

关键在于：我们希望在存储层（ES）本身构建更严格的约束，即使这意味着会在客户端底层更改 ES API 的行为。我们认为，以牺牲部分功能性为代价来换取关键任务工作流的极致稳定，是值得的。

尽管如此，短期内我们也不希望完全禁用 immediate 和 wait_for 刷新策略的合理用法。如果我们通过直接抛出错误来强制禁止这些选项，对现有客户端造成的影响将不可预测且波及广泛。那么问题来了：我们如何保证这些选项能被安全使用？

我们首先设定了以下不变式，并让我们的 Elasticsearch 安装实例强制执行，以减轻这些不良索引刷新访问模式在现实中的影响：

  1. 绝不允许两个并发尝试同步刷新同一分片的情况。允许“immediate”刷新，但绝不希望在多个写入线程间争用刷新锁
  2. 绝不允许针对刷新间隔设置长于默认值或设为 -1 的索引，在写入时参数化 wait_for 刷新策略。允许 wait_for 刷新，但其延迟时间必须限制在最多 1 秒

因此，我们最终期望的行为是：

  1. 如果检测到针对某分片的传入写入参数化为 immediate 刷新，且该分片已存在一个进行中的写入，则将后者的刷新策略更改为 wait_for
  2. 如果检测到针对某分片的传入写入参数化为 wait_for 刷新（包含因前述步骤而重写刷新策略的情况），且刷新间隔索引设置长于默认值，则将该分片写入的刷新策略重写为 none

值得注意的是，在上述两种情况中，我们都考虑过直接抛出错误而不是尝试更改刷新策略的替代方案。我们最终决定不这么做，因为我们认为抛出错误会导致工作流彻底中断，从而在生产环境中引发更大的不稳定性。

在任何更改刷新策略的场景下，我们还会发出遥测信号，提醒我们此类事件正在发生。随后，我们会指导 ES API 的使用者修改相关代码路径上的刷新策略配置。

选择插件切入点

我们希望在 Elasticsearch 中实现这些不变式，但不想分叉代码库。因此，我们研究了允许定制此行为的插件 API。ES 服务器的核心代码相对精简，大部分 ES 逻辑以插件或模块的形式加载。插件实际上是“外部编写”的模块，ES 会扫描其安装文件夹树来发现要加载并绑定在内存中的插件和模块。我们创建了自己的插件，并将其与我们部署在全集群的每个 Elasticsearch 实例一同发布。

我们必须选择正确的位置添加重写刷新策略的逻辑。确定合适的 ES 插件 API 需要深入理解向 ES 写入索引时调用的代码路径。对 /_bulk 端点的调用遵循以下（简化版）控制流：

  1. 协调节点接收客户端写入批量数据的请求
  2. 协调节点将其转换为由 ES 操作支撑的 ES 任务。（ES 任务是逻辑工作单元，操作是实现任务的代码。在此阶段，协调节点上调用的操作是 TransportBulkAction）
  3. [插件切入点] 在实际进入操作代码之前，会调用零个或多个 ActionFilters，这些过滤器可以装饰生成操作的执行过程
  4. 写入被分区到主分片，然后扇出到持有这些分片的数据节点
  5. 数据节点接收到写入分片的请求
  6. [插件切入点] TransportInterceptors 收到通知，得知节点已接收到执行写入的请求
  7. 数据节点创建任务以完成分片上的写入，并从写入线程池中取一个线程执行写入
  8. [插件切入点] 每次写入都会以最简参数通过已注册的 IndexingOperationListeners
  9. 随后写入实际持久化，并执行任何需要进行的写后刷新操作（wait_for 或 immediate 刷新）
  10. 如果分片有副本，数据节点将写入传播到副本。持有副本分片的数据节点在本地重复步骤 6 至 9，然后将成功或失败的结果传回主分片节点
  11. 回调将写入完成事件通过协调节点传回，最终到达客户端

步骤 (3)、(6) 和 (8) 是我们可以通过插件挂钩的位置。经过排除法筛选，我们最终选定了 TransportInterceptor：

  1. 我们不能使用 ActionFilter，因为我们需要在数据节点上检测并发的同步刷新写入，但在此代码路径上，ActionFilters 不会在持久化数据的数据节点上被调用
  2. 我们不能使用 IndexingOperationListener，因为该 API 未提供更改刷新策略所需的写请求句柄

算法

因此，我们的 TransportInterceptor 实现可拆解如下：

  1. 拦截器维护一个数据结构，用于追踪“当前正在同步刷新的分片 ID”
  2. 接收到分片级别的批量写入时，检查刷新策略
  3. 如果刷新策略为 immediate，通过检查该分片是否在步骤 (1) 的数据结构中，来判断该分片是否已存在带有 immediate 刷新策略的未完成写入。如果存在，则将刷新策略更改为 wait_for。如果不存在，则在步骤 (1) 的数据结构中开始追踪此未完成的写入，并在写入完成后通过回调将其从追踪集合中移除
  4. 如果刷新策略为 wait_for（可能源自上述步骤 (3)），则查找刷新间隔索引设置。如果刷新间隔为 -1 或长于默认值，则将刷新策略更改为 none

如前所述，对刷新策略的任何更改都会记录在我们的遥测系统中，并标记给 ES 客户端工程团队以供重新评估。

评估

上述对 ES 扩展的优势在于，如今在任何情况下，ES 都能防范对索引刷新的不良使用。这并不意味着我们不应努力改善 ES 的使用方式，避免在一开始就错误地选择这些索引刷新策略。我们的遥测系统会向我们发出异常告警，以便我们与 ES 客户端工程团队跟进，改善他们的 ES 用法。理想情况下，我们希望逐步过渡到仅使用 none 作为刷新策略，并让 ES 客户端将最终一致性视为常态。

这种权衡也有一些弊端：

  1. TransportInterceptors 是通过扩展已废弃的 ES 内部 NetworkPlugin 提供的。该插件随时可能发生变化，一旦 ES 版本升级导致兼容性问题，我们需要及时进行适配。
  2. 如前所述，我们在客户端底层隐蔽地更改了 ES 的 API 语义。例如，依赖写后一致性来保证正确性的客户端，可能会在后端发生重写时处理不当。此外，我们不会直接向客户端提供发生此类情况的信号。我们建立的用于检测这些重写的遥测机制，正是为了减轻此类风险。
  3. 我们必须确保在 ES 版本升级后行为依然正确，因为底层处理写入和刷新的 ES 代码会随时间演进。我们使用 ES 测试框架编写了大量测试，该框架通常用于 Elasticsearch 自身的测试，但对我们验证 ES 底层逻辑也非常实用。随着我们推进 ES 升级，这些测试应能捕捉到回归问题。

我们已论证了 Palantir 为何要为 Elasticsearch 注入额外的弹性，以防范不良访问模式对 ES 使用的未来影响。我们展示了一种新颖的解决方案：通过 TransportInterceptor 监控并抵御索引刷新策略的不当选择。数据库是我们工作流的核心，防御能力是确保系统随时为关键任务用户可用的关键所在。
