Optimizing Git’s Merge Machinery, #6

2021-09-20 · Original →

Editor’s note: This is the sixth and final post in a series by a Palantir Software Engineer on scaling Git’s merge and rename detection machinery. Click to read the first, second, third, fourth, and fifth blog posts.

The first post in this series included some background information on how the Git merge machinery works and why I have worked on optimizing and rewriting it. That background may be helpful in understanding some parts of this post. However, unlike the previous posts, this one doesn’t delve into the intricacies of merges. While the earlier posts used clever ideas that exploited various properties of merges to reduce the work required, this post instead looks at more traditional, straightforward optimizations. Also, unlike the previous posts centered around specific themes that each took a fair amount of effort to explain, this post will include optimizations that are pretty easy to explain and group them into a single post.

Large optimizations

First, let’s take a look at the optimizations, grouped into two categories:

Fundamental changes in merge-ort design

Fundamental to Git is an in-memory data structure mirroring the index (i.e., the .git/index file). It’s just an array of entries for each path tracked in the repository, sorted by pathname. This data structure was at the heart of merge-recursive and most of the rest of the Git codebase. In the fifth blog post, I discussed how the in-memory index data structure would have prevented trivial directory resolutions; here, I discuss how it also caused some performance problems for merges.

The fact that the in-memory index is an array makes for fast iteration and relatively fast searching (O(log N)) but is somewhat slow for insertion/removal since O(N) items will have to shift. Usually, that isn’t much of a problem because it’s an array of pointers, and shifting lots of pointers can be done relatively cheaply (using the memmove() function, which is likely optimized for the system). Also, the initial iteration traverses all three trees simultaneously, adding multiple entries for each path when there is a conflict. The initial traversal takes care to only insert entries at the end of the array. The combination of these characteristics mean that most merges will only have a small number of insertions and removals to handle.

However, if there are lots of three-way content merges needed (and they resolve cleanly, so the three conflicting entries will be removed and a new entry put in their place), or if there are a lot of renames (causing paths to need to be removed and added elsewhere in the array), then we may have O(N) insertions and/or removals that we need to make. Since each of those is O(N), we spend O(N^2) time shifting elements in the array.

The second problem with the old algorithm came from a low-level function named unpack_trees(). This function would walk one or more trees simultaneously to create or update an in-memory index and invoke various callbacks. This design was very useful for commands such as checkout or reset. In simple cases, it also provided what looked like a solid foundation for merging (because it would only need a “few simple fix-ups for special cases”). In reality, the information provided in this scheme was insufficient for the needs of a merging algorithm. Merges need more data and the ability to change traversal orders with deferring and skipping of subtrees entirely, mainly to handle renames well. My early attempts to tweak all the tricky logic in unpack_trees() to address the extra requirements eventually made me realize that we just needed a different abstraction entirely for merge-ort. Since merge-recursive was built on top of unpack_trees(), it did have functions that would re-iterate trees either entirely (to search for specific information, e.g., directory/file conflicts) or re-traverse from the top-level tree down to individual paths (e.g., as done for every rename), to collect more information. That was expensive and we wanted to avoid that expense in merge-ort.

merge-ort avoided both the above problems by side-stepping the in-memory index and any use of unpack_trees(). merge-ort was written to create a new merge result as a tree (which possibly included files with conflict markers if the merge had conflicts) without touching the working tree or index. Only after the merge result will merge-ort use the checkout logic to move from the current tree to the merge result. This design avoided the quadratic behavior from the insertions and removals of index entries and the numerous expensive mini-tree traversals done by merge-recursive. I believe these two improvements were responsible for the lion’s share of the speedup seen between the starting point for merge-recursive and the starting point for merge-ort:

                     merge-recursive           merge-ort
  mega-renames:   5964.031 s ± 10.459 s  5504.231 s ±  5.150 s

As a reminder, the “mega-renames” test case involved rebasing 35 commits across a branch with about 26,000 renames. This speedup technically also includes numerous “micro-optimizations” listed in the section below. However, they are subsecond optimizations, so nearly all of the speedup above came from the two big optimizations of skipping quadratic array entry shifts and avoiding additional tree re-traversals. From the above results, these two optimizations provide a savings of nearly 459.8 seconds or 7.66 minutes. See commit 557ac0350d (merge-ort: begin performance work; instrument with trace2_region_* calls, 2021–01–23) if you want to read about other test cases and how far ahead of merge-recursive merge-ort started on those other test cases too.

Prefetching missing objects in batches for partial clones

The next optimization will only benefit folks using partial clones, but it’s a pretty substantial optimization for them. Partial clones operate by only downloading a partial or incomplete set of objects. Typically, it skips all the files and downloads the trees, commits, and tags with the associated branch names and the commits they point to. Then, when any operation needs to access certain files, it will download them on the fly from the original repository.

The fallback code to ensure partial clones works will download any given object the first time it is accessed by any Git command, which can result in many objects being downloaded in separate connections to the server, one at a time. Using separate sequential connections for each needed object can be slow. It is helpful if operations that will need many objects can first do some computation to determine which objects it will need and then call a function to prefetch all those objects.

A few years ago, Jonathan Tan from Google added such prefetch functionality to the rename detection logic. However, after the optimizations in my first, second, third, and fourth blog posts, we often need far fewer objects to complete rename detection. That provided an opportunity to modify the prefetch logic in rename detection first to do a more limited set of prefetching and then fall back to the more thorough prefetching only in cases that needed it. This modification could potentially increase download operations but often significantly reduce the number of objects downloaded.

While at it, I also added some prefetching to merge-ort for the three-way content merges to further reduce the number of download operations.

The combination yielded some pretty impressive results in a partial clone:

Number of `git fetch` ops during rebase                     Before Series   After Series
merge-recursive:          62              63
merge-ort:                30              20
     Number of objects fetched during rebase                     Before Series   After Series
merge-recursive:         11423          11423
merge-ort:               11391             63

See the upstream thread discussing this optimization if you want to see more details. In particular, these results got Microsoft talking about potentially enabling rename detection for their big monorepos. Their monorepos have more than 25 times as many files as the Linux kernel, forcing them to need partial clones, sparse checkouts, and a pile of special configurations to make things function well. One of those things was turning off rename detection because it was too costly for them. Being able to show somebody that far fewer files would be needed with these new algorithms has them talking about enabling it again.

Micro-optimizations

There was a long list of micro-optimizations, which include all the sub-second optimizations I made. These ranged in magnitude from hundreds to tens of milliseconds to even just a few milliseconds. There were also some for which I didn’t even have measurements (e.g., because it was part of the basic rewrite and I didn’t have a way to test just that item in isolation). I won’t discuss all these micro-optimizations in detail but will provide links in many cases if you want to dive deeper.

  • Avoid re-comparing hashes (cached in match_masks)
  • Reduce the size of the (heavily used) merged_info struct by eight bytes, identified by pahole

The combined effect of micro-optimizations

As noted above, several of these optimizations were in the initial timings. For several of them, clicking on the links will provide some details about performance improvements. However, for those closely following all six blog posts in this series, there are two sets of micro-optimizations that I’d like to call out. First, the replacement of string_list_df_name_compare was submitted in the middle of the other optimizations from the other blog posts and made the following difference:

Preview-only optimizations (not merge or rename detection related)

Some optimizations will not yet be available in released versions of Git but foreshadow future work.

The machinery for applying a sequence of patches (for either rebasing, cherry-picking, or reverting) is in sequencer.c. Since no merge backend could operate without updating the working tree as it went, sequencer was built with that as a hard assumption — in a way that permeates the code. Further, since all the sequencer logic originally existed as shell scripts that were migrated over time to C, it also had assumptions about re-reading and verifying the state with each patch and still has a design that forks several subprocesses.

The rebasing machinery also has a multitude of checks which could be viewed as part feature, part attempt to workaround slowness in the former merge machinery. Even though these checks originally had performance as their goal, they can be particularly slow and painful in very large repositories and/or when using partial clones.

Trying to change all this was a significant undertaking that was out of scope for my initial work. As a result, I created a simple fast-rebase tool with very limited applicability (and in particular would error out if it hit conflicts) but was designed to operate very quickly and show the possibilities for Git’s rebase and cherry-pick. In particular, fast-rebase would:

  • avoid updating working-tree/index with every intermediate patch
  • avoid reading/writing rebase metadata until conflict or other stopping points
  • avoid checking index matches HEAD with every patch being rebased; do it at the start only
  • avoid unnecessary process forking (a side-effect of the original shell-script design)
  • avoid per-commit hooks that were only triggered due to a design accident of history
  • avoid unnecessary history traversals (from various can-fast-forward and is-linear-history checks) and patch inspections of commits not being rebased (due to --no-reapply-cherry-picks being the historical default)

These changes add up to about six seconds of execution time, which may not sound like a lot, but is when you compare it to the final optimization results we achieved on the “mega-renames” test case, as shown below.

Overall Results

Here I’d like to look at the overall improvements achieved across all optimizations, from where the old merge algorithm started (using Git version 2.30.0). I’ll include a few more test cases as well (note that “no-renames” was a misnomer; it should have been “few-renames”), which are explained in more detail here.

For each of the three test cases, I’ll provide the original timings using the old merge algorithm (merge-recursive), how much merge-recursive improved (as a side effect of optimizing the new merge algorithm, merge-ort), what kind of timings we would have gotten with a theoretical implementation that replaced the rename detection code with an infinitely parallelized version (i.e., an implementation where rename detection was instantaneous), and what timings I achieved with the new merge algorithm.

So there’s still work to do, but outside of the rename detection logic and merge machinery.

Many of the optimizations in this post already appear in released versions of Git, accessible via passing the --strategy ort option to merge/rebase/cherry-pick or by setting git config --global pull.twohead ort (lame configuration option name due to backward compatibility). Other than the “Preview only” ones, the remaining optimizations will appear in Git 2.34 in late 2021.
