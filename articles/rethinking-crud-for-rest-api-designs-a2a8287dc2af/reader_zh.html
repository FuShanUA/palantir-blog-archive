Rethinking CRUD For REST API Designs

2021-08-11 · Original →

当我听到有人说“我们即将创建新服务 X”时，后面通常跟着一句话：“V0 版本会使用简单的 CRUD 接口。”虽然这种设计方法曾一度吸引过我，但我越来越觉得它并非最佳的设计选择。在这篇文章中，我将探讨我在使用 CRUD 时遇到的一些实际问题，以及这些问题如何改变我们对优秀 API 设计的思路。

回顾一下，CRUD 这个词代表的是 创建、读取、更新、删除，它试图描述“持久化存储的四种基本操作”。该术语源于早期关系型数据库的工作。它看起来非常像标准 SQL 中的 INSERT、SELECT、UPDATE 和 DELETE 操作，而且由于它与 HTTP 动词 POST、GET、PUT 和 DELETE 紧密映射，其用法在 Web 和库 API 设计中也很普遍。对于任何以资源为中心的工作流来说，CRUD 感觉非常直观——你会先“创建”它，之后你可以“读取”它的当前值，“更新”为新值，或者“删除”它。

问题所在

虽然 CRUD 乍一看似乎很容易理解、实现和使用，但在实践中我发现它远没有那么吸引人。为了解释我的意思，让我们来看看 CRUD 的各个功能：

  • Create 意味着不存在则创建：如果资源已存在，则操作失败。如果同时到达两个有效但不同的创建操作，至少有一个必须失败。
  • Read 就是读取：获取最新值，如果不存在则为 null。
  • Update 意味着存在则更新：将当前值整体覆盖为提供的新值。如果资源不存在，则操作失败。如果同时到达两个有效但不同的更新操作，两者都会被接受，但会选择一个“最新”的值来处理所有后续的读取。
  • Delete 意味着存在则删除：如果资源不存在，则操作失败。如果同时到达两个有效的删除操作，则删除资源，并允许两者都通过。

那么问题出在哪里呢？我看到了几个问题。

首先，许多客户端不需要区分 Create、Update 和 Delete——它们想要一个包罗万象的“set”操作。当 API 使用随机生成的 ID 来标识资源且碰撞概率极低时（例如 UUID 或哈希），情况尤其如此。我遇到过一些 API，几乎每个客户端调用都包含类似下面的代码块，这在某种程度上显得很愚蠢：

if (newValue != null) {
    try {
        backend.insertRecord(id, newValue);
    } catch (ResourceAlreadyExistsException e) {
        backend.updateRecord(id, newValue);
    }
} else {
    backend.deleteRecord(id);
}

其次，虽然有些资源是原子性的且无法拆分，但许多其他资源可以分解为多个子组件或字段。大多数客户端一次只想更新资源的一个字段或一部分，但 CRUD 要求他们读取并写入整个资源才能正确更新单个字段。这使客户端逻辑变得复杂，引入了不必要的性能开销，并极大地加剧了我在下一步中描述的并发问题。想象一下，如果在文档编辑器中写一个字就需要将整个文档重新发送回服务器！

第三，许多客户端的编辑工作流是三步事务。它们会对资源执行读取以获取其当前值，基于该值和用户输入运行一些计算，然后在更新中写回结果。服务本质上是多访问的，但更新命令只是一个覆盖操作，没有内置的冲突或同步语义。在任何时刻，多个用户可能同时读取、计算、更新资源，并互相覆盖彼此的编辑内容。这通常通过单独的锁定或事务开始/结束命令来处理，但这些往往很难做到既正确又高性能。此外，由于它们不属于 CRUD 定义的一部分，往往会被遗忘，直到出现头几个 Bug 之后才被想起来。

最后，在实践中，特别是对于我们 Palantir 来说，CRUD 遗漏了其他两个对任何初始 REST API 设计都至关重要的操作：搜索和安全变更/硬删除。如果在项目初期没有考虑到这两个操作，它们往往会导致服务后端被迫进行大幅度的调整。例如：

  • 搜索操作接收一些搜索条件（例如，今日修改过的，或距离某点 50 公里以内的），并返回符合条件的一页资源。
  • 安全变更和硬删除操作接收一组资源，主动更改它们的安全属性或将其从持久化存储中彻底清除，并且以这种方式导致现有客户端失去对这些资源的访问权限。

解决方案

综合来看，这说明 CRUD 并非通常号称的那种一刀切式 API。在初次设计 API 时，我们应该仔细审视服务的工作流，并认真思考它应该采取何种形式。我提出了一小套替代模式供 API 设计者考虑：

Get-and-Set

这是最简单的解决方案，我认为 API 设计者应该始终将其纳入考虑。它包含两个操作：

  • Get：与 CRUD 的 Read 相同。返回资源的最新值，如果不存在则返回 null。
  • Set：包罗万象的创建/更新/删除操作，也称为最后写入者胜出。将资源设置为提供的值。如果提供的值为 null，则删除它。如果两个不同的 set 操作并发到达，则选择一个“最新”的胜出。可选地，也返回其先前的值。

大多数文件服务 API 天生遵循这种模式。例如，索引工作流（将数据库对象索引到 ElasticSearch 中）就遵循此模型。

Get-and-Patch

对于可以分解为多个可单独更新的字段的资源，Get-and-Set 可以轻松扩展为一种略有不同的模式。Get-and-Patch 包含两个操作：

  • Get：与 Get-and-Set 中的 Get 相同。
  • Patch：一种包罗万象的创建/更新/删除操作，但允许你只设置资源上的某些字段。缺失的字段意味着“保留现有值”。通过一个合成的“deleted”字段来删除资源。

我们的配置服务 API 广泛使用了这种模式。JSON Patches 也是一个这样的例子。

Timestamp-Checked

对于遵循三步读取-计算-更新模式，或者需要事务或并发编辑保护的应用程序，可以使用逻辑时钟来实现乐观并发。

除了每个资源的当前值之外，还要存储该资源最近一次写入的时间戳。然后暴露两个操作：

  • Read-Return-Timestamp：类似于 Get，但同时返回最近一次写入的时间戳。
  • Write-Check-Timestamp：类似于 Set，但除了提供新值之外，还要提供客户端首次读取资源时的时间戳。如果资源在服务端的时间戳与提供的时间戳匹配，则执行写入并返回新的时间戳。如果时间戳不同，则拒绝写入并返回服务端的值，以便客户端重新运行其计算并重新提交更新后的值。

你也可以将其视为 Check-and-Set (CAS) 的一种变体。Firebase 的 firestore API 也包含这种模式作为前提条件检查。

与大多数乐观并发模式一样，这种方法也有局限性；如果预期的并发量过高，或者客户端的计算成本昂贵，这可能会导致在重试中损失过多的工作。在这些情况下，最好选择一种不同的模式，并围绕它添加事务性 API。

Append-Only-Log

对于具有明确定义编辑的应用程序，或者客户端确实不应该直接修改值的应用程序，完全摒弃传统的“用新值更新”操作可能是值得的。

与 Timestamp-Checked 相同：同时存储资源的当前值和最近一次写入的时间戳。然后暴露两个（或三个）操作：

  • Read-Return-Timestamp：与 Timestamp-Checked 相同，返回资源的当前值和时间戳。
  • Append-Return-Updated：客户端不提供要写入服务的值，而是由服务定义一组可应用于其资源的编辑操作。然后客户端提供其中一个操作让服务在资源上运行。随后将更新后的值返回给客户端。
  • Load-Since-Timestamp：可选操作。假设客户端能够自行计算编辑操作的结果。在这种情况下，客户端可以提供其最近一次读取资源时的时间戳，然后接收自该时间戳以来应用于该资源的所有编辑操作。

操作转换和基于 CRDT 的应用程序可能是这种模式最高级的应用，这种模式在我们的协作应用中得到了广泛使用。也存在更简单的应用，例如银行 API 允许你取款、存款和转账，但不能手动设置你的账户余额。

注意事项

关于上述模式有两点需要注意：1）我没有为其中任何一种模式包含搜索或重新设置 ACL/硬删除操作，因为它们对所有方法都是通用的，并且在很大程度上不受你选择哪种模式的影响；2）这些模式并非互斥——完全有可能在一个 API 中同时拥有“不存在则创建”和“更新检查时间戳”操作。

这绝不是说 CRUD 在服务 API 设计中毫无立足之地——只是它不应成为首选模式。特别是，当客户端认为资源的创建或销毁意义重大且不同于编辑该资源时，或者当服务需要进行在编辑时不需要的大量验证或计算时，CRUD 非常有用。

几个例子包括资源的Primary Key由现实世界信息组成的情况，例如社会安全号码，在这种情况下，无论是有意还是无意，资源键的冲突都是可能发生且危险的。另一个例子可能是创建操作接受的信息比更新操作多得多，比如创建在线外卖订单与更改司机小费之间的区别。

结论

在为以资源为中心的服务创建初始 API 时，请考虑：

  • 你是否需要为创建、更新和删除提供单独的操作，或者客户端工作流是否一视同仁地对待它们。
  • 多个客户端同时查看和编辑资源时将如何处理并发编辑。
  • 客户端是应该直接设置资源的全部属性、部分资源，还是应该为它们定义允许的编辑。
  • 客户端在一开始将如何搜索和处理与资源相关的安全变更，而不是事后才想起。

从长远来看，这样做会为你和你的服务带来更好的结果，也大大降低了最终变成一堆糟糕代码的可能性。
