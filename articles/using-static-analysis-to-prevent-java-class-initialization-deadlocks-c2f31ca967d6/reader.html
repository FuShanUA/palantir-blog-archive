Using Static Analysis to Prevent Java Class Initialization Deadlocks

2021-02-02 · Original →

This blog post tells the short story of the discovery of a race condition in a popular Java library and how we decided to prevent similar bugs through static code analysis. Along the way, we’ll talk about the JVM internals that led to this bug.

Discovery and fix of a JavaPoet race condition

At Palantir, we use Square’s fantastic JavaPoet library for many code generation tasks, first and foremost to generate the Java bindings of our RPC framework, Conjure. Our colleague Justin had reported back in 2018 that JavaPoet suffers from a race condition in multi-threaded environments: non-deterministically, two threads would get deadlocked in JavaPoet library code.

As Justin had conjectured, the cause of the deadlock is a race condition between two class initialization code paths. Assume that we have a class Base whose initialization code (for example via static fields) depends on its own subclass Sub:

class Base {
  private static final Sub SUB = new Sub();
  static class Sub extends Base {}
}

In order to initialize Base, the JVM must first initialize Sub. This works as expected in a single-threaded environment, but breaks if one thread initializes Sub directly (i.e., by calling new Sub()) while simultaneously a second thread tries to initialize Sub via the static dependency from Base (i.e., by calling new Base()). It turns out that JavaPoet uses exactly this pattern: the TypeName class has a static field of type ClassName and ClassName is a subclass of TypeName.

Background

The Java Language Specification defines class initialization in jls-12.4.2. In particular, the statement “Synchronize on the initialization lock, LC, for C. This involves waiting until the current thread can acquire LC.” directly opens the door for the described deadlock. It was also pointed out in JDK-8037567 that this deadlock is expected behavior according to the specification and has to be prevented in application and library code rather than in the JVM. If you’re interested in trying for yourself, JDK-8037567 contains a minimal code example that reproduces the race condition and deadlock reliably.

While the JVM could in theory provide a more complex locking protocol, this would result in performance changes to class initialization, penalizing startup time as well as potentially breaking existing code. Many components or programming patterns rely directly on class initialization details, for example the initialization-on-demand holder design pattern.

Fixing and preventing the race condition

Of course, such deadlocks can be fixed by reorganizing static fields or methods. However, as demonstrated in the case of JavaPoet, a fix may introduce API breaks when public fields or methods get shuffled around. To prevent both the race conditions and subsequent API breaks when fixing them, we decided to put together an error-prone rule that bans a number of known bad static class dependencies at compile time.

In a nutshell, the rule works as follows:

  • Given a class definition C, inspect each static block and static field.
  • If the code contains a reference to a subclass of C which results in initialization based on jls-12.4.1, we have a potential deadlock scenario. Initialization occurs when: a new instance is created, a static (non-constant) field is read, a static field is written, a static method is invoked.
  • If the subclass is enclosed by C and private, the subclass cannot be instantiated outside of C and thus the deadlock is impossible.
  • If the subclass is visible, but has a private constructor and no accessible static fields or methods, the deadlock is also impossible, however this assumption is fragile because adding non-private constructors, methods, or fields can allow deadlocks to occur!

Note that this algorithm does not catch every bad pattern; there are scenarios with layers of indirection that are more difficult to detect, but in our experience have not resulted in deadlocks. Unfortunately, as is often the case with notoriously intractable static analysis problems, it seems unlikely (although we have not written down a proof) that one can identify all such patterns.

Examples

The error-prone rule bans the following patterns, for example:

class Base {
  private static final Sub SUB = new Sub();
  static class Sub extends Base {}
}class Base {
  static {
    new Sub();
  }
  static class Sub extends Base {}
}class Base {
  static Sub SUB = new Sub();
  static class Sub extends Base {
    Sub() {} // can be initialized externally through instantiation
  }
}

Conversely, the rule allows patterns of this form:

class Base {
  private /* not static */ final Sub SUB = new Sub();
  static class Sub extends Base {}
}class Base {
  static Sub SUB = new Sub();
  static class Sub extends Base {
    private Sub() {} // cannot be instantiated externally
  }
}

We have applied the error-prone rule to many of our internal code repositories and have — sadly, but unsurprisingly — found a number of violations. Of course we have fixed those potential race conditions in order to prevent them from causing damage in the wild.
